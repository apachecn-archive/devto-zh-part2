# 观点鲜明的框架

> 原文：<https://dev.to/polyov_dev/frameworks-with-strong-opinions-5g02>

[![some people are strongly opinionated.](../Images/b9e9192f2215623bb5a82666c80f3765.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--ARvIutea--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/jheb01e4an0wurhqrhxx.jpg)

我学习的第一个框架是 Ruby on Rails。我是在高中学到的，后来，我把学到的所有东西都与 Rails 进行了比较。我经常心烦意乱，郁郁寡欢地离开。其他框架不像 Rails。我不能轻易地让一个项目进行下去。人们是如何处理这个问题的？我搞不清这些模式。

为什么使用 Rails 对我来说更容易？因为 Rails 是一个固执己见的框架。

当在 Rails 中开发时，rails 有一套特定的准则要遵循:使用 MVC。你以某种方式组织你的文件和文件夹。您使用`routes.rb`文件进行路由。等等。

后来，当我离开 Rails-world 并开始使用以 Javascript 为中心的框架时，我意识到其他框架并不那么固执己见。有了`package.json`，我可以在我的项目中包含任何东西。这并不意味着它会工作，但对我来说，这意味着我必须自己找出结构。而那个[竟然是](https://github.com/skwh/podcastr) [硬的](https://github.com/skwh/TypeScriptEGI)。

我后来也因为这个原因迷上了 Angular:它的项目有一个定义良好的结构。该框架清楚地知道应该如何编写它的应用程序，需要哪些文件，把它们放在哪里，以及输出是什么样子。

我还纠结于可定制性和可靠性之间的区别。当谈到可定制性时，大多数框架都可以进行调整；他们意见的力度很重要，因为这改变了做出这些调整的难度。

但是一个固执己见的框架也可能更可靠，因为创建一个工作应用程序或项目只需要几个步骤。您从最初的项目中采取的步骤越多，应用程序就越不稳定。除非你真的知道自己在做什么，否则从最低限度创建一个非常可靠的应用程序可能不是你想要的方向。

最终，一个框架观点的力量是一个人在开始学习这个框架，或者用它做一个项目时应该考虑的事情。

你对制作自己的项目结构有多满意？当 bug 和主要变更突然出现时，你能够考虑到你已经放入项目中的所有不同的结构和观点因素吗？

如果这些问题的答案是否定的，您可能会考虑一个更固执己见的框架。这意味着开发一个项目更容易，因为程序员不需要一直担心这些事情。这意味着错误修复和堆栈溢出问题变得更加容易，因为每个人都有每个 Rails 应用程序或每个 Angular 应用程序的共享知识。

如果这些问题的答案是肯定的，那么你可以按照自己的方式工作。这种风格非常适合小型项目，因为您不必管理额外文件和结构的开销。你可以把你的代码放在任何地方，任何你喜欢的结构。

不管怎样，你可能会找到一个 Webpack 模块将它重新组合在一起。
