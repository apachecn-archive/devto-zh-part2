# 标准很无聊

> 原文：<https://dev.to/dwd/standards-are-boring-1pld>

## 创新容器

让我们从容器开始。

集装箱无处不在，它们已经成为“可运输”的简写，实际上，你能想到的几乎所有东西都变成了一个长方形的集装箱，每个角落都有紧固件。

[![Containers](img/fa2cf675fe5d378fb98b8cdadb4468d2.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--MQLY1QNz--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/iuuzfl95wddlz36bph3n.jpg)

是的，我真的是在谈论物理运输容器，而不是 Docker、Kubernates 或任何这些。这里没有软件，这些是大的金属盒子，最常用来放其他东西的大盒子。但它们不仅仅被用作一个哑盒子——事实上，这种格式已经成为你能想到的几乎所有东西的普遍形状。

使用范围从军用无人机到运输坦克(如大型流体容器、*或*配有大炮的履带式车辆)。有冷藏集装箱。有包含机械车间的容器。有几组包含大型工程车间的容器。有可展开的披萨小屋(真的！)，用集装箱装运。有可运输的厕所和狗窝。甚至还有我最喜欢的，名字很有趣的战术侦察部署图像系统，或 TARDIS -这是一种皇家空军战术图像的移动办公室。这就是“集装箱化”一词的由来。

[![Pizza Hut](img/b55faee11feeb4896587aa2761d64c31.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--47bwXUql--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/vgs1jo41zijaephnijmt.jpg)

所有这些都非常有效，因为一旦某样东西变成了一个集装箱，运输和部署就变得很容易了——因为每个人都在为这个不起眼的集装箱而努力，因为在 ISO 668 和 ISO 1496-1，一群人花了无数个小时才弄清楚集装箱到底应该是什么。

## 标准集装箱

是的，多式联运集装箱，正如它的正式名称，是一个开放的标准。众所周知，标准很无聊，一点都不创新。作为一个在开放标准领域工作了很长时间的人，几十年来我的工作一直受到这样的指责。你可能会认为我会拒绝它，但这里有大量的真理，这实际上是一件值得骄傲的事情。

容器本身并不是很刺激。如果说有什么不同的话，那就是故意让它变得无聊。通过鼓励每个人坚持相同的尺寸，在每个角落使用相同的扭锁紧固件，没有人可以创新和创造，比如说，稍微大一点的容器来挤出几英寸的空间。

另一方面，通过提供一个稳定、标准化的可运输平台，每个人都可以充满信心地进行创新——你知道，如果你建造一个由 ISO 海运集装箱构成的医院，它就可以通过船只、卡车(美国人用卡车)、火车，甚至一些飞机运输到世界各地。每个角上有趣的积木可以在运输过程中保护你的医院单元，并作为起重机的吊点。你不必真的检查每一艘船，每一辆卡车，每一列火车。

如果你想知道的话，是的，有人把医院塞进了集装箱。然后把它放进一架飞机里。

[![Hospital unloading from Anatov](img/0505272eff83ae1ea337cbdb7ad212c1.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--762XH4B---/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/ilsxcbteffa5aojobhf2.jpg)

所发生的是，国际标准组织，ISO，已经把集装箱变成了一种商品。容器本身一点也不有趣。创新已经全部完成，然后被标准工厂磨掉。通过设计，你不能得到好的或坏的容器。他们就是这样。

但是既然你可以在几天内将一个集装箱或者任何伪装成集装箱的东西运送到整个地球而不用担心，人们已经疯狂地创新了这个概念。虽然容器和它们的标准化本身并不十分创新，但建立在它们之上的创新是巨大的。

## 互联网标准

很容易指向网络并产生同样的想法——但是网络在很多方面是一个糟糕的标准。首先，它是巨大的——如今的浏览器必须处理实时视频、事务数据库、用于布局、样式、文档的多种重量级语言，当然还有有史以来发明的最复杂的编程语言。球门柱一直在移动，因为这是一个要么全有要么全无的事情。

因此，让我们来看看电子邮件。

当电子邮件出现时，几乎每个人都有自己的电子邮件系统。古老的 Sendmail 只是为它们之间的网关而编写的。IETF 很快为电子邮件设计了一个标准格式，因此至少当你从一个系统收到一条信息到另一个系统时，你可以理解它。

然而，最终，传输和路由都被标准化了，基本的 ASCII 消息变得乏味了——这是所有标准化工作的最终目标。

然而，革新者并没有停止，他们设计了将二进制文件编码到这些 ASCII 消息中的方法。他们开发了不仅仅是发送 ASCII 码的神秘方法。他们完全滥用消息来发送命令，例如在电子邮件列表和文件服务器上构建整个系统(前者仍然存在，后者早已消亡)。

渐渐地，其中许多被采用并变得无聊，所以我们最终使用了 MIME，例如，它给我们带来的不仅仅是 ASCII 消息、附件，以及——最终——恼人的 HTML 电子邮件，超过了整整一代市场商人认为的真正好主意。但是所有这些事情都变得，嗯，很无聊。

你也可以看到反例。

推特、谷歌和脸书都是我们未能创造出基本概念的商品化版本的例子，结果是尽管推特、谷歌和脸书本身具有创新性，但它们根本没有推动多少创新。

你可以用分形的术语来思考它——随着商品化、标准化概念泡沫的增长，我们创新的表面积以更快的速度增长——但是创新本身并不能提供同样的基础。

## 无权限创新

然而，所有这些都对我们如何设计标准产生了影响。为了最大化创新的表面区域——如果你愿意，可以称之为无聊的视界——我们必须确保标准有足够的扩展点。

例如，TLS 刚刚以 TLS 1.3 的形式经历了一次重大修订，而且大部分工作都是为了添加更多的扩展点。

在 XMPP 领域，我们已经坚持使用 XML 将近 20 年了——不是因为与 JSON 相比，它设计和使用静态结构更快，而是因为它对名称空间的内置支持意味着任何人都可以轻松地扩展它，而不会与任何人发生冲突，也不必协调他们的工作。这最大化了创新的潜力。

因此，虽然标准可能很无聊，但这种可预测、令人欣慰的无聊为一些惊人的创新飞跃提供了基础。

## 接下来呢？

如果你有兴趣做一些无聊且毫无创新的事情，IETF 互联网工程任务组和 T2 XSF XMPP 标准基金会完全可以免费参与，在这两种情况下，你对无聊这个重要世界的贡献将会得到极大的赞赏。
