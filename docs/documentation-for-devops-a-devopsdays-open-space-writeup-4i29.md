我从我在 Chef 的朋友那里借用了这个形象，因为假装有可能只是撒上“devops”，这是关于每个层面的文化变化的东西，这真的很有趣。

在我去的每一个开发日(相当多)，我都会提议/争论/经营两个开放空间。一个是关于特性标志/金丝雀发布/假设驱动的开发，另一个是这个，我通常称之为“DevOps 的文档”。

DevOps 文档很难编写，因为它不是面向外部受众的，所以很难找到雇用作者的理由，但它也是至关重要的，这意味着它不能被忽视或忽略。

我也有一个关于这个话题的演讲。你可以在这里找到明尼阿波利斯和明尼苏达州[NDC](https://youtu.be/U_-wCQWXgbM)的唱片。但这是特别关于开放空间的。我很感激去年来到开放空间的许多人，他们分享了他们的经历，提出了问题，并挑战了我的想法和假设。

我将最常听到的问题分成了以下几类:

*   我怎样才能找到任何东西？
*   我如何让人们阅读这些文件？
*   我如何让人们写文档？
*   我的团队很小，你有什么特别的建议吗？
*   我的团队很大，我们该怎么办？
*   分布式团队最佳实践？
*   为什么是 sharepoint？
*   什么工具最有用？

# 怎么才能找到东西？

通常，人们遇到的第一个问题不是什么都没有写下来，而是很多事情都写了下来，问题是可发现性和准确性。有说明是一回事，当你需要时能找到说明又是另一回事。许多人的操作文档分散在一个或多个 wikis、票务系统和文档存储平台上。

这个问题的简短答案是“搜索”。这也是大部分的长答案。我认为我们忽略的一点是，我们不必使用维基百科自带的搜索引擎。我们可以在它前面放点别的东西。随着谷歌搜索工具的消亡，你仍然有很多服务选择。他们中的大多数会给你选择索引和搜索多个网站。

作为一名帮助公司组织文档使其可用的顾问，我有很多关于如何改变架构和文化的具体建议，但老实说，如果你能让你的公司雇佣一些人来帮助你，你可能会让他们雇佣一名作家并以另一种方式解决问题。

这个答案的最后一部分是，你可以获得一些咨询费，并聘请一名档案管理员/图书管理员来帮助你找到有助于识别数据类型并使其更容易搜索的标签。当我们试图回忆数据时，我们经常通过关键字或标签这样做，档案管理员可以帮助你列出人们应该使用的规范列表。

# 如何让人们阅读我们所拥有的东西？

首先，看上面，让它容易找到。

第二，尽可能让它容易阅读。在一个小组中坐下来，讨论应该包含什么类型的信息才能使一个主题/文章/页面对其他人有用。一旦你知道你需要包括什么，你就可以制作一个模板，这样人们就会记得把它都包括进去。

下一个问题是找出如何在不轻蔑或让他们觉得你太优秀而不能回答他们的问题的情况下将人们引向文档。这可能是聊天机器人或充满资源的单个网页的好地方。

许多人宁愿自己找到答案，也不愿麻烦同事去得到答案，所以如果他们没有阅读文档，很可能是系统问题阻碍了他们。

你如何让人们写这些文件？

你知道有哪个系统管理员或 SRE 曾经因为缺乏文档而被解雇吗？

你知道有谁仅仅因为写内部文档的能力而得到提升吗？

我们大多数人都不知道。无论组织如何声称他们关心文档，他们并没有在这些文字背后隐藏任何有意义的东西。这不是恶意的，但是它确实告诉你文档实际上不是他们关心的东西。

解决这个问题的方法之一是使用奖励系统。买一包星巴克卡，或者牛逼贴纸，或者冰激凌券，每当有人完成一个文档任务，就马上给他们发。给他们的最终评估加上一个分数比工作的直接奖励更没有激励性。要做到这一点，你不必是一名经理——任何人都可以激励他们的同事。奖励不一定要有现金价值，但我认为这是表达你感激的最简单的方式。

另一件需要注意的事情是编写文档的难易程度。如果是复杂系统，进入门槛高。如果是在团队已经使用的工具中，用他们已经知道的语言，说服他们加入几行就容易多了。如果有一个表格让他们填写，这样他们就不必去想他们应该包括什么，那就更好了。降低认知准入门槛和工具准入门槛同样重要。毕竟，如果我们不得不手写错误报告，并且没有任何字段可以填写，我们会有更多的麻烦。文档也是如此。

# 我的团队很小

考虑只用一页真相。不，说真的。如果你和一个 8 人以下的团队一起工作，并且*特别是*如果他们是分散的，考虑把所有的东西都放在一个网页上，当你需要找东西的时候搜索它。当你的团队很小的时候，管理一个文档系统或者 wiki 的开销并不是你所需要的。当你写新的东西时，它们会放在顶端，旧的东西会被推到底端，变得明显不那么重要或相关。如果你需要找什么东西，你可以很容易地搜索页面。没有人有团队其他成员不知道的秘密文件。这不是一个优雅的解决方案，但它是非常实用的。

# 我的团队很大

雇佣一个作家。和/或图书管理员。我曾经和一个 IT 组织超过 7000 人的人交谈过。他们显然需要一些结构、一些工具、一些信息架构。与其在 it 上花费内部 IT 周期，我建议雇佣专家。这个世界上有成千上万的人接受过管理、校对、撰写和争论大量文档的培训，试图自己去做是一种浪费。

# 分布式团队的最佳实践有哪些？

拥有一个分布式团队的唯一有效方法是改变团队文化，让你写下问题和答案，而不是突然出现分享它们。重要的是要确保关键数据总是被写下来，而不是通过口头传播，因为一旦你有了口头文化，你就把远程成员排除在外了。

至少有一个具有分布式团队经验的团队成员远程工作是很有用的。你可以向他们询问过去对他们有效的方法，并在重复你的实践中寻求他们的帮助。例如，团队是否很难记住为会议安排电话会议？更改会议模板以自动包含共享会议室。没有人能马上得到一个分布式团队，就像我们没有一个协同定位的团队一样。只是分布式的团队更容易在有人察觉之前就以支离破碎的状态收场。

# 为什么是 SharePoint？

因为范例和核心类比是文件柜。"如果我能看到操作档案柜里面会怎么样？"因为它认为自己是一种组织纸张的方式，所以它不太擅长需要跨组织边界同时共享和修改的文档。这在当时是一个奇迹，非技术用户可以通过这种方式从共享驱动器和文件柜中获取文件，并允许其他人有限制地访问，但它从根本上反对搜索。

# 什么工具最有用？

看情况吧？工具存在于一个滑动的标尺上，一端是绝对的零进入壁垒，另一端是税收的有用性。每个团队都要找到自己的最佳位置，可能全公司都不一样。但是如果一个团队不喜欢某个工具或者觉得它很麻烦，他们就不会使用它。

最好的工具是你能让人们采用的工具。如果你的团队是 100%知道如何使用 LateX 的人，并在其中创建了他们的简历和游戏角色表，那么它将是一个很好的工具。如果你有一个完全由 20 岁以下的人组成的团队，也许 youtube 频道是个不错的选择。既然我们都没有完全统一的团队，我们能做的最好的事情就是找到一个每个人都勉强同意只有一点点糟糕的工具。

# 总之

*   使用更好的搜索引擎
*   尽可能雇佣专家
*   通过使用模板使写东西更容易
*   迭代并持续改进您的 devops 文档流程。没有解决办法，只是暂时的解决办法。
