## 概念

正如我前面提到的，这个项目的目标是开发一个友好的体育赛事管理应用程序，供那些聚集一些朋友和他们的朋友一起玩的人使用。最初的想法是围绕足球(或英式足球)，但我会尽量使它通用。

考虑到这一点，我试图找到适合这个想法的主要概念:

*   用户——嗯……这是意料之外的吧？😆
*   团体——一群朋友聚在一起玩
*   玩家——玩家是一个不同于用户的概念，它允许一群人使用这个应用程序，虽然不是每个人都想使用，但是想使用的用户仍然可以管理一切，即使有人不是注册用户
*   比赛/事件/游戏-代表已经/正在/将要发生的特定事件
*   团队——参加比赛的团队——这可能是固定的，但也可能是比赛范围内的，每次都定义了团队
*   统计——有趣的是保存各种比赛的统计数据，如比赛分数、进球/得分/其他得分、犯规等

## 期望的特性

当写下这些概念时，想要的特性开始浮现在脑海中(以及将在下一节中写下的一些组件):

*   创建/管理用户
    *   定义电子邮件，姓名，昵称，个人资料图片，…
*   沟通
    *   发送电子邮件、通知等
    *   管理首选项
*   创建/管理组
    *   邀请用户
    *   创建玩家
    *   将玩家与用户相关联
    *   为小组设定特定的运动
    *   …
*   比赛
    *   日程安排
    *   管理信息
    *   何时，何地，参与的球员，选择的球队，…
    *   添加事件
    *   进球得分，遭受犯规，…(更多和运动依赖)
    *   活着
    *   能够在比赛中添加比赛事件，并能够看到统计演变实时

## 组件

好了，对于这类项目，有了尽可能好的概念和期望的特性(如果一个业务分析师把这样的东西给我们中的一个，我们可能会被激怒😝)我们可以开始描绘我们过度工程化的系统，在那里我们试图融入每一种可能的潮人技术……因为这是那种项目，那将会发生什么！

下面是一个非常高层次的架构。它肯定会随着时间的推移而改变(希望是最好的)，但我认为可以总结最初的想法。

[![High level architecture](../Images/088186c323769468c76014219942e429.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--FBJYPRgd--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/2xnss8l9ed19dr11agga.jpg)

我们可以看到各种组件的基本组织，即使我们过度设计了解决方案，我们也不希望组件像 CRUD 服务一样，而是尽可能成为能够实现特定功能集的自包含服务。

考虑到这一点，我们有:

*   `Users` -处理注册、认证和其他用户管理要求
*   处理所有与团队相关的事情，从联系用户到管理玩家。
*   `Communication` -处理所有通信需求，包括用户的通信偏好。
*   处理所有比赛细节，比如时间和地点，比分，比赛事件等等，除了直播部分。
*   很可能会从主匹配组件中分离出来，因为它会有实时性的要求，可能会对分离有用。
*   这将是一个处理所有统计数据的中心位置，即使这些数据最初来自于比赛事件。我们将会看到，从长远来看，如何(以及是否应该)将这个服务与 matches 协调起来。

除了负责应用程序逻辑的组件之外，我们还需要一个前端来查看它的工作情况。为了支持前端，我在考虑用“后端支持前端”，以最好的方式为网络前端提供一切。移动部分不在范围内，只是作为一种可能性添加到图像中。
