# 如何构建强大的移动性能监控策略

> 原文：<https://dev.to/apium_hub/how-to-build-a-robust-mobile-performance-monitoring-strategy-4ii>

在一个完美的世界里，所有的应用程序都会像他们的开发者想象的那样运行——快速加载，易于使用，没有错误，没有崩溃。但是这个世界并不完美，移动应用也是如此。事实上，[一些最流行的应用程序](http://download.blancco.com/download/en-rs-q3-2017-state-of-mobile-device-performance-and-health.pdf?mkt_tok=eyJpIjoiWVROaU5XVXdNVFl4WlRreiIsInQiOiJHWEdFQ1RNWUJ3bWtDWWtJQ1U5bE1UdzN4XC81R01DWGxcL3Vzeml1eFI4UFwvK244WDBoRW1pWkp6M2dINmdcL2YwSVRib3hxUU9KMWNQRzJPXC96dWt2RTdsQ2pwMXNTb1h6dmpBR3pSUW5mRTVhRXlBUktJbjhMTTNNNUNHWERhV2dnIn0%3D)有最高的崩溃率。无论我们经历突然的崩溃，缓慢的加载时间，还是令人沮丧的用户体验，应用程序都像其他技术一样容易受到故障的影响。不幸的是，这可能会给移动应用程序的制造商带来可怕的后果:收入损失、客户不满、品牌声誉受损等等。随着[应用市场的成熟](https://www.recode.net/2016/6/8/11883518/app-boom-over-snapchat-uber)，确保你的应用运行变得更加重要。今天，我们将探讨如何构建一个强大的移动性能监控策略。

**移动性能监控**帮助您控制移动应用的性能，以便您能够解决客户最关心的问题:高可用性、高可用性和高性能，并保护您的公司免受性能不佳的移动应用的影响。

虽然听起来很简单，但应用程序性能监控不仅仅是检查应用程序的速度、流量统计、错误率，甚至是 KPI。真正的移动应用程序性能监控是为了找到应用程序性能问题的根本原因，并需要一种系统的(有时是复杂的)方法来解决问题，这需要专业知识、洞察力、正确的信息和正确的应用程序监控工具集。

## 移动性能监控策略将如何帮助您？

持续监控您的应用不仅能让您掌握其性能(以及任何影响可用性的潜在问题)，还能让您:

*   **确保品牌声誉**:据 Gartner 称，许多品牌发现他们的[移动应用并没有像他们希望的那样吸引用户](https://www.gartner.com/binaries/content/assets/events/keywords/cio/ciode5/top_strategic_predictions_fo_315910.pdf)。使用一款表现符合预期的应用程序来维护公司的正面形象。
*   **保护收入**:多达[1/4 的用户](http://info.localytics.com/blog/23-of-users-abandon-an-app-after-one-use)会在你的应用程序不工作时放弃它。虽然技术缺陷只是问题的一部分，但客户对此几乎没有容忍度。不要因为可预防的问题而失去客户和收入。
*   **收集有用的数据**:一个稳定的应用程序既能让用户满意，又能成为从真实用户那里收集数据的可靠工具。
*   **最大限度地减少中断**:在移动性能问题(如加载速度慢)造成问题之前，识别并解决这些问题，从而最大限度地减少甚至完全消除对最终用户的影响。

通过正确的策略和正确的监控平台，您可以建立一个系统来监控您的移动应用的性能，并确保您的用户和您的公司保持满意。

## 创建稳健的移动性能监控策略的 6 个步骤

为了创建可靠、强大的移动性能监控策略，我们建议遵循以下六个关键步骤:

1.  收集数据
2.  设置基准监控
3.  创建警报机制
4.  获取数据报告
5.  分析你的数据
6.  分享您的信息

正确执行时，每个步骤都有助于创建应用监控策略，保护应用的完整性，并让客户满意。

**第一步:收集数据**

没有关键数据，你就是在暗中操作。

收集到的数据量从少到多，多得令人不知所措。知道如何收集相关数据是建立有效策略的关键。您可以收集应用程序生成的所有数据，但是如果没有一个适当的策略将数据与您的绩效 KPI 联系起来，数据就会变得无关紧要，并且对您的业务几乎没有价值。

了解您的数据应该来自哪里，创建事件来监控和跟踪该数据，然后将数据输入到一个可以轻松分析的存储库中，这对于从您的工作中获得最大价值至关重要。

关于数据收集频率的说明:以不同频率收集的相同类型的数据可能会显示两种截然不同的结果。如果您想从数据中获得有意义的见解，确定轮询频率至关重要。

管理数据历史是数据收集的另一个关键组成部分。随着时间的推移，保留这些数据将让您清楚地了解您的应用程序的执行情况，包括峰值、错误率、解决时间等。有了这段历史带来的清晰性，您将能够在扩展应用时制定未来战略，利用过去的趋势和经验来开发下一组里程碑。

**关键要点**:最终，数据的价值来自于您收集关键数据的能力。选择一个平台，让您能够灵活地收集您需要的所有数据，提供可变轮询，并具有开箱即用的功能来存储数据一段时间(至少一年或更长时间)。

**第二步:设置基准监控**

对您的数据进行基准测试，以获得您真正需要的信息。

所以你已经开始收集和存储你的数据。太好了！下一步是什么？

对这些数据进行标准化和基准测试。

要理解您的数据，规范化是第一步。规范化会自动告诉你什么时候你的应用表现“正常”，或者什么时候它的数据超出了常规参数。

在确定关键数据基准之前，与受该数据影响的利益相关方(业务领导、技术领导、营销经理等)交流。)来确定他们认为可接受的基准。根据您的具体情况，偏离行业规范可能是可以接受的。

如果没有基准监控，您将会不断收到警报，即使您的应用程序数据完全符合常规参数。通过适当的基准测试，您可以比较任意两组数据，了解当前数据是否遵循过去的趋势，如果不是，则启用正确的措施来纠正偏差。

但是，请记住，基准测试并不是静态的。它们可能会根据历史数据发生变化。这是好事吗？是啊！你的应用程序今天和明天、一年后或两年后都不会处于相同的状态。随着应用程序的发展，您的基准也需要发展，以提供新的“标准”

**关键要点**:选择一个应用程序监控平台，它不仅可以使用人工生成的数据，还可以访问虚拟数据中心，使数据监控既灵活(按需)又无需人工干预，从而加快您的基准测试和监控过程。

**第三步:创建警报机制**

将静态和动态警报结合起来，以便及时了解情况。

所有这些数据和基准都需要投入使用。但是在哪里？先说预警机制。

因为移动性能监控需要仔细跟踪您系统的健康状况，以确保一切顺利运行，所以如果出现问题，需要有一个即时的警报机制，以确保在客户受到影响之前采取及时的行动。实施良好的警报机制可以显著减少用户遇到的错误数量。

警报本身可能包括两种类型:静态和动态。当您希望得到参数(如 CPU 或内存使用量)随时间推移或多或少保持不变的通知时，静态警报就派上了用场。为这些参数设置静态阈值是一个好主意，应该根据您想要测量的参数设置上限和下限。

动态警报基于您为应用监控设置的基准。这些警报没有固定的阈值。当你的系统开始从历史数据中学习时，它必须学会何时提醒你。这意味着今天可能引发警报的一个值可能在 6 个月后成为标准，而不会引发相同的警报。

静态和动态警报都有助于您持续监控应用的状态。

对于随时间变化的应用程序参数，包括流量、pings 等。—不要定义人为的门槛，而是让它随着时间的推移自然发展。这样可以避免误报，只有当应用程序出现真正的问题时才会收到警报。您的角色将是定义在触发警报之前，您认为多少标准差是可接受的。

**关键要点**:优秀的应用监控平台提供基于静态和动态数据的警报机制。这确保了完整的系统监控覆盖范围，能够在环境发生任何变化时立即向您发出警报。

**第四步:获取数据报告**

自动化、定制的报告让利益相关方了解情况并提供长期价值。

警报机制对于异常情况非常重要，允许您快速识别和修复问题。但是，当您需要在正常的日常条件下保持应用程序的最新性能时，该怎么办呢？这就是报告的用武之地。

出色的移动性能监控工具不仅能提供标准指标监控的自动化打包报告，还能针对您或业务利益相关方需要了解具体细节的情况提供特别报告。这些可以包括外部团队的系统审计请求、管理层的报告请求、关于意外峰值的报告等等。

与常规的自动化报告相比，临时报告提供了一个独特的、定制的视角。良好的即席报告可以无限制地组合不同的数据点，提供数据的历史视图，并且可以根据您的需求进行定制。

快速提示:记得考虑如何存储您的数据。随着时间的推移扩展应用程序，您也需要扩展数据。由于存储了大量数据，将所有数据存储在一个集中的位置会在几年内降低报告生成速度。分散数据可确保在几秒钟内生成报告。选择提供分散数据存储的应用程序监控平台。

**关键要点**:为确保您的组织在需要时拥有所需的信息，应定期生成一系列自动化和可定制的报告，并将其存储在分散的环境中。

**第五步:分析你的数据**

让你的数据发挥作用。

那么，您如何处理所有这些数据，以及如何使用这些数据来推进您的应用程序监控策略呢？你分析一下！我们说的不仅仅是被动的分析，而是主动的、深入的分析。毕竟，你需要可操作的见解，能够帮助你的应用实现下一个[业务里程碑](https://dev.to/apium_hub/software-project-milestone-what-why--how-588j)。

假设您需要对客户的问题进行故障诊断，甚至在客户看到这些问题之前。主动数据分析向您展示了潜在的棘手问题，以便您可以集中精力立即解决这些问题。借助定义完善的自动化分析，您将领先客户一步，最终为他们提供出色的体验。

最后，决定如何显示、格式化和查看数据。移动性能监控工具不仅应该能够显示不同的数据，还应该帮助您以易于理解的格式显示这些数据。确保你的数据突出事件的原因。

**关键要点**:主动数据分析不仅能让您在有问题的问题上领先一步，还能为您提供做出明智决策所需的信息。

**第六步。分享你的信息**

传播知识——为了每个人的利益。

最终，你的应用程序监控策略应该决定如何在不同的人群中分享信息。您的受众将包括各种角色，包括 IT 领导、营销分析师、财务经理，当然还有 cxo。每个组都希望访问不同格式的不同信息集。

请确保您的移动性能监控策略考虑到了所有这些需求，并且您选择了一个能够以多种格式提供信息共享的平台。

**关键要点**:不同的利益相关者需要不同格式的不同信息。一个好的应用程序监控平台允许您以他们喜欢的格式发送信息，以便在整个组织内共享关键数据和见解。

**总之**

最终，一个出色的移动性能监控策略会提出 6 个关键问题:

*   正在收集哪些数据来帮助您有效监控您的应用？
*   这些数据是如何进行基准测试的？
*   您的警报机制设置正确吗？
*   您的数据报告是否涵盖多个场景？
*   您是否有一个可以让您领先一步并主动解决问题的预测策略？
*   你能有效地分享你的信息吗？

真正的移动性能监控允许您监控和测试移动应用的性能，包括其可用性和用户体验。借助正确的应用监控策略和工具，您可以主动识别性能问题，并在出现任何意外问题时快速找到并解决。它有助于确保你的应用程序的声誉，并最终确保你的品牌。
