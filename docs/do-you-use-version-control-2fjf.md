# 你使用版本控制吗？

> 原文：<https://dev.to/thaiwood/do-you-use-version-control-2fjf>

你可能以前听说过，“你应该使用版本控制”。这是一个关键。在许多情况下，其他的建议可能取决于你的团队的规模、你的组织、它的目标或者它的价值观。

不是这个。这是一个基本的建议。如果你想实现任何合理的、**有益的**自动化，你必须使用版本控制。这是为什么呢？因为如果需要，您必须能够重新创建以前的状态。您需要能够回滚。你的团队需要能够回顾过去，看看发生了什么变化。看看他们今天是怎么到达这个州的。

如果你把你的车拿到机修工那里，他会问在你的车到达那里之前发生了什么。你怎么开车？你经历了什么？什么变了？想象一下，每次去都只是说“我不知道，我每次开车都昏昏沉沉的，从来不听发动机的噪音。”没有版本控制就是这样。你不能回答基本的问题。

这不仅仅是为了当事情出错的时候。您的团队可以在开发功能时使用这些信息。例如，他们可以检查该功能或类似功能以前是否存在，但可能已被删除。

这也是新团队成员复习的好基础。它可以帮助他们理解特性是如何发展的。即，快或慢，一次许多，或一次一个。他们也可以通过这种方式感受代码评审过程(或者首先开始代码评审)和团队风格。理解什么被询问或者最终被修改可以帮助他们更快地融入团队。

版本控制还允许您测试应用程序的不同版本。您的团队可以找出各种特性或特定实现如何经得起基准测试或您的测试套件的检验。

即使你觉得你的团队现在还不够大，不能保证这一点，从一开始就开始这个过程将会更容易扩展团队或在以后获得帮助，并且比允许潜在的特性在单个开发人员的机器上枯萎要安全得多。

最后，重要的是实际使用版本控制，而不仅仅是在某个地方可用。如果你的团队不使用它，它允许代码和知识丢失的可能性，并且不会帮助你。
