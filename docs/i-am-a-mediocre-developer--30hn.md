# 我是一个平庸的开发者

> 原文：<https://dev.to/sobolevn/i-am-a-mediocre-developer--30hn>

我个人认识一些非常有天赋的开发人员，他们可以不费吹灰之力就创造出精彩的软件。因为这些有天赋的个人，我们的行业充满了很高的期望。但可悲的事实是:不是每个人都是忍者/大师/摇滚明星开发者。

而这正是我:一个平庸的开发者。如果你不是天才，这篇文章将指导你如何在这个行业生存。

## 我总是用谷歌搜索最简单的东西

我记不起很多事情了。比如标准库中的函数和方法、参数位置、包名、样板代码等等。

所以，我得去谷歌一下。我每天都这么做。我也重用旧项目的代码。有时候我甚至会从 StackOverflow 或者 Github 上复制粘贴答案。对，其实是个东西: [StackOverflow 驱动开发](https://meta.stackoverflow.com/questions/361904/what-is-stack-overflow-driven-development)。

但我并不孤单。很多其他开发人员也这么做。有一个由`Ruby on Rails`的创造者发起的[热门推特讨论](https://twitter.com/dhh/status/834146806594433025)。

但是为什么一开始就不好呢？嗯，它有几个缺点:

1.  它允许你从其他人那里复制糟糕的设计决策或易受攻击的代码
2.  它形成了一种特定的思维模式:如果我们不能用谷歌搜索某些东西，那么“休斯顿，我们有问题了”
3.  当网络瘫痪时，我们将无法工作

但是，我不认为这是一个大问题。它甚至可以作为你的秘密武器。我有一些建议来减少负面影响。

**如何生存**:

1.  使用`IDE`获得自动补全和建议，这样你就不必谷歌语言基础了
2.  记住你已经在哪里(而不是如何)解决了这个问题。所以你可以在那里寻找解决方案
3.  您粘贴到项目中的所有代码都应该在以后进行分析、重构和审查。这样我们就不会用糟糕的代码来损害项目，而是用快速的解决方案来帮助它

## 我保持事情简单明了

机器总是做他们被告知的事情。有时他们只是被告知去做错误的事情。所以，软件开发的主要问题不是机器，而是开发人员的脑力。非常有限。因此，我们——平庸的开发人员——不能浪费它来创建复杂的抽象、晦涩的算法或不可读的长代码块。保持简单就好。

但是我们如何判断这段代码是简单的，那段代码是复杂的呢？我们需要使用`WTFs/Minute`方法来衡量代码质量。

[![wtf/minute code quality](../Images/93dae8e5374e8e01fd569c182b8289f8.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--RTnc6j1i--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://i2.wp.com/commadot.com/wp-content/uploads/2009/02/wtf.png%3Fresize%3D550%252C433)

这个原理非常容易理解。每当你在代码中发现你不理解的东西——它太复杂了。你能做什么？

1.  重写它，让 cleanner 设计
2.  提供文件
3.  在最棘手的部分添加注释。但是请记住，注释本身就是代码味道

**如何从头开始写简单的东西**:

1.  为变量、函数和类使用正确的名称
2.  确保你程序的每一部分都只做一件事
3.  比起常规函数，更喜欢纯函数
4.  更喜欢常规函数而不是类
5.  只有在非常需要时才退而求其次

## 我不信任自己

一些开发人员被证明能够交付高质量的代码。比如这个女人:玛格丽特·哈米尔顿，阿波罗计划的首席软件工程师。在这张照片中，她正站在她为登月任务编写的代码旁边:

[![Margaret Hamilton](../Images/cbceb039d23dff3c663c3340b2326c05.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--G4sZPf_1--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/http://cdn8.openculture.com/2017/08/29205628/margaret-hamilton-mit-apollo-code_0.jpg)

但是，每当我写代码时，我都不相信自己。即使是项目中最简单的部分，我也会把事情搞砸。这可能包括:

1.  语言错误
2.  逻辑错误
3.  设计错误
4.  风格错误
5.  安全错误
6.  WTF 错误(我的最爱！)

而且也没有什么“学习如何编写无 bug 代码”的魔法书。这是完全正常的。[所有软件都有 bug](https://m.signalvnoise.com/software-has-bugs-this-is-normal-f64761a262ca)。不过除了[这个框架](https://github.com/kelseyhightower/nocode)。处理好它。

问题是:任何人都不应该被允许写有明显错误的代码。至少，我们应该试试。但是我怎么才能保护项目不受自己的伤害呢？做这件事有多种方法。

**如何生存**:

1.  编写测试。写很多测试。从集成测试到单元测试。在每个拉取请求之前，在`CI`中运行它。这将保护你免于一些逻辑错误
2.  使用静态类型或可选的静态类型。比如我们用 [`mypy`](http://mypy-lang.org/) 搭配`python`[`flow`](https://flow.org/)搭配`javascript`。积极效果:更干净的设计和“编译时”检查
3.  使用自动样式检查。每种语言都有大量的样式检查器
4.  使用质量检查。一些工具在你的代码基础上运行一些复杂的启发式算法来检测不同的问题，比如这行代码里面有太多的逻辑，这个类是不需要的，这个函数太复杂了
5.  检查您的代码。在合并到`master`之前检查它。合并后的某个时候
6.  付钱给其他人来审核你的代码。这种技术有巨大的积极影响！因为当开发人员第一次看你的代码时，他们更容易发现不一致和糟糕的设计决策

## 它不应该只在我的电脑上工作

[![Works on my machine](../Images/5349b4c64d4ec9ec72f213d93e835784.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--h027pWFE--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://www.ca.com/us/products/excuse-free-testing/worked-fine-on-my-machine/_jcr_content/page/adaptiveimage_855e.img.620.high.jpg/1484844865861.jpg)

大约十年前，当我的团队开发出我们的第一个大型软件项目时，我们将它作为`java`源文件发布。并且无法在目标服务器上编译。那是在向客户演示之前的几个小时。这是一个大失败！不知何故，我们设法让它运行起来，但这是一次改变人生的经历。

这是因为在构建管道中有大量的配置和复杂性。我们无法恰当地管理这个系统的复杂性。从那天起，为了降低这一步的复杂性，我尝试将我的程序打包在隔离的环境中。并在实际部署之前在这个环境中测试它们。

最近几年，随着`docker`(以及一般的集装箱)的兴起，这变得和以前一样容易。`docker`允许您在同一个隔离环境中运行开发、测试和生产。所以，你永远不会错过任何重要的事情。

你受伤了吗？说到我自己，我总是在创建服务器、最初配置它们或者将它们链接在一起时忘记一些事情。有太多的事情需要记住！希望我们仍然可以自动化它。有不同的令人惊叹的工具来自动化您的部署过程。如: [`terraform`](https://www.terraform.io/) 、 [`ansible`](https://www.ansible.com/) 、 [`packer`](https://www.packer.io/) 。阅读它们，找到你真正需要的任务。

我也尽量尽快设置 [`CI` / `CD`](https://about.gitlab.com/features/gitlab-ci-cd/) 。因此，如果我的构建在测试或部署中失败，我会得到报告。

**如何生存**:

1.  自动化您用于部署的任何东西
2.  使用`docker`进行应用程序开发、测试和部署
3.  使用部署工具

## 应用程序部署后，我仍然不信任自己

哦，我的应用程序终于投入生产了。它现在正在工作。我可以小睡一会儿，不会有事的。等等不要。一切都将破碎。是的，我是认真的:**一切**。

实际上，有一些工具可以让查找和修复现有问题变得更加容易。

1.  [`Sentry`](https://sentry.io/welcome/) 。当一个错误发生在你的任何一个用户身上时，你会得到通知。几乎可以绑定到任何编程语言
2.  不同的[服务](https://papertrailapp.com/)和[工具](https://www.elastic.co/products/kibana)将来自多个进程和服务器的日志收集到一个地方
3.  [服务器监控](https://grafana.com/)。您可以在这里为 CPU、磁盘、网络和内存配置监视器。您甚至可以在用户真正中断您的服务之前发现扩展的时间

**简而言之**，我们需要在生产中监控我们的应用。我们有时使用所有这些工具，有时只使用最需要的部分。

## 不断学习

哇，要学的东西真多。但事情就是这样。如果我们想写出好的软件，我们需要不断地学习如何去做。没有捷径或魔术。每天都要学习如何变得更好。

总之，我们需要理解两件基本的事情:

1.  每个人都会遇到问题。唯一重要的是我们对这些问题准备得如何
2.  我们可以将问题的根源减少到一些可接受的程度

这与你的心智能力或心态无关。
