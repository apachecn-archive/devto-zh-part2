# 投资者尽职调查

> 原文：<https://dev.to/perttisoomann/investor-due-diligence--80f>

这里有一个小小的曲线球，来自通常的开发导向的讨论，关于开发-你如何准备投资者的技术尽职调查？

几乎没有背景，我已经在英国的一家 QA 公司担任高级/全栈开发人员 6 年了，从公司一开始就一直在公司工作。我们使用内部开发的问题管理网络应用程序从远程测试人员那里收集缺陷，在内部审查，然后向客户报告有效的分类问题。

我们正处于业务运营良好的阶段，方正正在寻找潜在的合作伙伴，将公司带上一个新的台阶。这个过程的一部分当然是合伙人找到他们信任的人进行技术尽职调查。

我想知道是否有人有最好的准备方法的建议或第一手经验？

我有一个小小的担忧，这是因为我们的前任 CTO，他和我们在一起的时间很短，他说我们的 PHP/CodeIgniter 解决方案无法扩展，投资者无论如何都想把它重写为默认值。作为一个老派的 Java 人，他也不喜欢 NodeJS，所以我们在技术方向上被搁置了大约一年。

除了引发该死的冒名顶替综合症，我现在正试图主动化解这一问题。

当然，对我来说有偏见，即使在低端 AWS 实例(2-3%的 CPU)上，应用程序也能流畅运行，而无需我们专门对代码或架构进行任何优化——所以我们仍然有一些开发技巧，我们仍然可以在上面投入很多硬件。

然而，仅仅说“嗯，会好的”并不是任何审计师想要听到的，不管他们是否偏向于技术。

根据应用程序的大小和现有的内部技能，我个人会避免完全重写，更不用说用不同的语言重写，但如果需要改变，我宁愿现在就做，并在 1-2 年内处于正确的位置，而不是走上错误的道路，然后遇到存在的问题。

从收入目标(知道我们需要多少用户来提交报告，我可以计算出我们需要交付多少项目来达到收入目标，等等)中逆向工程出非常高端的估计用户数，证明我们可以处理这些数字，和/或列出达到这些数字时我们将改进平台的具体方法，这是否足够？

除此之外，我认为他们希望看到的最重要的事情是我们从开发方面发展的能力(坦率地说，就是有效利用投资者的资金)。

因此，从现在开始，我们的时间可能更好地用于确保我们有更大的关注点分离，更多的文档，更多的结构，以便现有的技术栈快速而毫不费力地加入全职开发人员/设计师或自由职业者。和/或已经有了特性/变更路线图，只是等待额外的开发资源来快速实现变更。
