# 为行为编程面试问题讲述更好的故事

> 原文:[https://dev . to/interview cake/telling-better-stories-for-behavioral-programming-interview-questions-35 a4](https://dev.to/interviewcake/telling-better-stories-for-behavioral-programming-interview-questions-35a4)

## *显示*，不要告诉

你可能以前听过这个建议。也许是你十年级的英语老师。也许是大学里的职业服务。“记住:*秀*，不要说。”

这是个好建议。当谈到在编码面试中回答行为问题(比如“告诉我关于你自己”)时，一个好的答案和一个很好的答案之间的区别在于 T2 表现出 T3 而不是告诉别人。

问题是，那些给你“展示，不要说”建议的人自己并没有遵循这个建议。他们是*在告诉*你去表演，但他们应该是*在展示*你*如何表演*。那是最难的部分！

因此，这里有三个具体的提示，让你展示更多，讲述更少。

## [](#1-sprinkle-in-specific-details)1。点缀具体细节

想象一下对股票面试问题“说说你自己吧”的两种回答

首先:

> 大约两年前，我从一些个人项目开始编程。我最终在家乡的一家小科技公司找到了一份工作，并且已经在那里工作了一年半。我喜欢我的工作，但我正在寻找新的挑战，我认为贵公司可以提供这种挑战。

然后:

> 我开始编程是因为我想为猫建立一个社交网络。这并没有成功，但原型帮助我在家乡的一家小型科技公司找到了一份工作。
> 
> 上个月，我在《黑客新闻》上读到了一篇关于贵公司正在建设的社交网络的精彩文章。你所面临的挑战看起来会帮助我比我现在的角色成长得更快更强。

第二个回答是关于候选人的更多信息。

为什么？因为*具体细节*。面试官不会记得第十个说“我在寻找新挑战”的人。他们*会*记得那个试图为猫建立社交网络的人，并在黑客新闻上读到他们公司的消息。

所以不要忽略细节。寻找机会使用细节，尤其是那些古怪的、有趣的、令人惊讶的或者其他*难忘的*。

## [](#2-tell-a-story-from-your-life)2。讲一个你生活中的故事

再拿一个常见的问题来说:“你为什么想在这里工作？”

人们倾向于将自己的价值观与他们面试的公司或团队的价值观相互参照:

> 我对技术博客和开源非常感兴趣。所以我喜欢你的公司有一些开源工作，并对社区有所贡献。

这是一个很好的回答。但是要真正让你的面试官惊叹，试着围绕这些价值观添加一个具体的故事:

> 几年前，当我还是编程新手时，我正在研究这个棘手的 bug。我在一家公司的博客上发现了一个帖子，其中一名工程师解释了她的团队如何解决这个问题。她加入了一段她开源的代码片段。我很感激她花时间来写她的团队的经验，并分享他们的解决方案。它帮助了我！
> 
> 我就是这样开始接触开源的。我真的很想和更多这样的工程师一起工作——他们写下自己的工作，并试图帮助社区中的其他人。因此，我很高兴看到你的团队在你的博客和公司的 Github 个人资料上分享的所有东西。

第二个回答听起来更真实。它*展示了*与开源和技术博客的个人联系，而不仅仅是*告诉*它。

任何人都可以查阅一家公司的核心价值观，并在面试中重复这些价值观。更有意义的是**讲一个你生活中的故事**，展示这些价值观如何让你受益或教会你一些东西。

## [](#3-use-someone-elses-voice)3。用别人的声音

这是一个巧妙的诡计。再考虑一个标准的行为问题:“你最大的优点是什么？”

你可以告诉面试官:

> 我与他人合作愉快。即使在艰难的环境下，我也能确保我的同事感受到支持。

但是一个不太详细的故事更适合展示这种力量:

> 我有一个同事叫安娜，她做工程师已经快十年了。我们一起完成了这个非常棘手的项目。
> 
> 快结束时，她告诉我，“对于这样一个地狱般的项目，你真的让事情变得理智了。”我认为这是我最大的优点——即使在艰难的环境下，我也能与他人合作愉快。

当你用一个故事来回应时，你可以**参考*其他*人对你最好品质的评价**。在这种情况下，一个十年的技术老手说你让一个项目感觉不那么糟糕。这种赞扬如果来自别人，会可信得多。

## [](#practice-practice-practice)练习，练习，练习

记住这些展示而不是讲述的技巧:

1.  **使用具体的、令人难忘的细节**。“猫的社交网络”而不是“个人项目”
2.  **讲一个你生活中的故事**。“我正试图解决一个棘手的问题……”而不是“我重视开源贡献。”
3.  **用别人的声音**。“‘你真的让事情变得理智了’”，而不是“我和别人合作得很好。”

在下面的问题中尝试这些策略。请记住，有时最简单的方法是以“告诉”开始，然后修饰成“展示”

*   说说你作为工程师最大的弱点？
*   描述你遇到的一个棘手的 bug。
*   你运过的最大的项目是什么？
*   你最喜欢的编程语言是什么？为什么？
*   你如何克服与同事的人际冲突？
