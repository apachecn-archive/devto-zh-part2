# “我不想维护他们的代码”

> 原文：<https://dev.to/pbeekums/i-dont-want-to-maintain-their-code-1od5>

我最近听说了一个情况，由于人手不足，一个团队需要雇人。

其中一个开发者的反应很简单:

**“我不想维护他们的代码。”**

[![](img/b426dc0e610f6b6c653cdf871276da12.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--k2GlTG86--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.professorbeekums.com/img/2018/2/screaming.jpg)

我理解这种心态。最差的代码永远是别人的代码。对于大多数开发者来说，这是简单的事实。

事实上，很少有重要的软件是由独狼开发的。

构建软件是一项艰巨的任务。这也是一个耗时的过程。在某种程度上，为了完成更多的工作，必须增加更多的开发人员。这并不是说在团队中增加新的开发人员很容易。它不是。

然而，添加新开发人员的大部分困难都可以减轻。只有当一个拥有错误技能的开发人员被雇佣时，维护另一个开发人员的代码的恐惧才是合理的。这并不是说招聘是一件容易的事，但是努力做这件事会有很大帮助。

新开发人员面临的最大困难实际上是起步阶段。每个软件系统都有不同的模式和命名约定。这些都要向一个新的开发者解释。即使当他们被解释的时候，光是要知道的东西的数量就会导致一个新的开发者在 100%之前花费数周或数月的时间。

这可以通过良好的软件架构来缓解。好的软件架构让事情更加清晰明了。不需要一个新开发人员到处摸索，只要看看代码，事情就有意义了。

不幸的是，“好的”软件架构是极其主观的。我让多个开发人员加入一个团队，其中一些人非常喜欢我做的决定，而一些人非常讨厌这些决定。什么“有意义”取决于一个人怎么想，不同的人怎么想……不一样。

这就是添加新开发人员的最大问题之一:现有开发人员的自负。新的开发者也意味着新的观点。新的观点挑战我们已经做出的决定。在这一点上，害怕维护另一个开发人员的代码并不是因为代码不好。这是基于代码与我们编写代码的方式不同。

在许多方面，新观点是一件好事。新的视角可以帮助我们完善我们的思维，让团队中的每个人，无论是现有的还是新的成员，都比以前更加强大。

这只有在现有成员的自负能够承受的情况下才会发生。
