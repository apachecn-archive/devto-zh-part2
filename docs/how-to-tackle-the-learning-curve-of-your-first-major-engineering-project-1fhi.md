# 如何应对你的第一个主要工程项目的学习曲线

> 原文：<https://dev.to/nylas/how-to-tackle-the-learning-curve-of-your-first-major-engineering-project-1fhi>

在我第一份全职工程工作的第六周，我被分配了一个测试我编程能力的大项目。我不得不放弃我最熟悉的语言和思维方式，Javascript，一头扎进 [Nylas 的](https://www.nylas.com/) 4 年前的 Python 代码库，以便构建一个 API 来同步超过 8 亿个联系人(并且还在上升)。接下来是一个为期 16 周的项目，它加深了我对大规模工程 fullstack 系统的理解，并教会了我处理您的第一个主要项目所面临的挑战。

许多工程师很早就质疑自己的能力。进入一家拥有全新代码库以及一套独特的基础设施、监控系统、最佳实践和工作流程的新公司，可能会感觉力不从心。我写这个博客的目的是给其他处于职业生涯早期的工程师提供建议。但首先，让我提供一些关于 Nylas 提供的产品和我负责的项目的背景:更新我们的联系人 API。

如果你对这个项目的技术细节感兴趣，[看看我之前的帖子](https://www.nylas.com/blog/lessons-learned-syncing-800-million-contacts-to-our-database)。你现在需要知道的是，Nylas APIs 允许开发人员将电子邮件、日历和联系人功能(以及双向同步)构建到他们的应用程序中。从早期的产品反馈来看，我们发现客户要求我们的 contacts API 增加新的特性。我们以前只有非常有限的联系人数据的只读功能，而客户要求更多的功能。简而言之，这是迄今为止我最大最复杂的 fullstack 项目。我很兴奋，尽管也很紧张，来处理这个项目，我在这个过程中学到了很多。

## 外卖

### 编写好的测试是确保你编写好代码的最好方法。

写好测试的重要性是我参与的每一个项目中反复出现的教训之一。作为程序员，你的目标是写出好的代码。评估您的代码并随着代码中的其他因素或环境的变化继续评估它的最佳方式是编写好的测试。假设这是一个简单的概念，但实现起来却出奇的困难，即使对于对这个行业不熟悉的人来说也是如此。

在这个项目中，我学会了欣赏单元测试对于调试的力量。顾名思义，单元测试隔离并测试感兴趣的特定单元。以前，当我遇到一个我不确定的 bug 时，我会立即尝试解决它。这通常是在我很好地理解正在发生的事情之前，在我能够复制它之前。这意味着我花了更多的时间调试，对我的解决方案信心不足，对原因了解甚少。

使用单元测试来调试问题是一个不同的过程。单元测试就绪后，我开始隔离问题，并编写一个测试，在开始调试之前模拟所有其他东西。这样做是必要的，因为您希望确保错误行为不受其他元素的影响。一旦隔离了问题，测试就会因为 bug 而失败。现在，这个失败的测试是一个可靠的方法来衡量错误何时被修复，以及它是否会一直被修复。一旦我准备好了测试，我就可以自信地投入到调试中，因为我知道我的目标是通过测试。

> 尽管编写测试花费的时间与产品没有直接关系，但最终会节省时间。

### 利用你的导师作为资源。

加入一家新公司可能会令人生畏，因为要学的东西太多了。为了缩短学习曲线，Nylas 将新员工与高级工程师配对。对于 Contacts v2.0 项目，我与 Karim 作为我的编码伙伴、导师和总资源一起工作。

在项目开始时，我们专门通过结对编程来工作，我打字，他一路指导和解释。随着我获得自信和理解，我慢慢开始接手并独立完成更大的作品。有一段时间，我在工作时仍然非常依赖他回答问题。随着时间的推移，我的问题量减少了。最终，我成了项目负责人，也是回答来自项目经理、我们的客户和我的工程师同事的问题的主要资源。

在 Nylas 的最初几个月，这是我成长的重要部分。如果你的公司还没有进行结对编程，我建议你这么做，因为这对你非常有益，尤其是对新员工。

> 如果没有 Karim 的全程指导，我的项目不会如此成功，我也不会对自己的能力充满信心。

### 即使不知道要问什么也要提问。

在 Contacts v2.0 刚开始的时候，我经常在迷茫的时候不说话，因为我觉得我连该问什么都不知道。有些概念超出了我的理解范围，以至于我无法拼凑出一个问题。我不好意思问可能被视为基本的问题，更担心这可能暴露我能力上的一些致命缺陷。因此，相反，我会在困惑的阴霾中默默跋涉，希望时间会带来最终的清晰。

我很快意识到这不是最好的策略。虽然时间可能会带来清晰，但通常不会。不管怎样，如果我问了，事情会更快明朗。我的沉默没有让我看起来明白到底发生了什么，而是简单地隐藏了我没有跟上。我理解中的这些差距几乎肯定会在项目后期更关键的时候适得其反。此外，如果在那种情况下，我不仅不理解当前的概念，而且一直被蒙在鼓里，这将对我的整体表现产生不良影响。

然而，仅仅知道你*应该*问一个问题并不总是有助于知道*该问什么*。我的策略是，当你不容易拼凑出问题时，如何提问。如果你深陷在某个复杂的过程中，继续后退一步，直到达到你觉得舒服的水平，这是有帮助的。如果您对某个函数有一些不确定性，请沿着堆栈跟踪向上，直到到达您熟悉的进程。如果所有这些都失败了，那就坦率地说出你的困惑，让你的导师从基础开始。

> 同样重要的是要记住，你可能不是唯一一个有这些问题的人。提问不是软弱的表现，而是学习的欲望。

### 反思需要时间，但时间花得值吗？

当我第一次加入 Nylas 时，我觉得每天都充满了新的术语、流程、技术、技巧和项目。我会问很多问题，并得到很好的答案，但我觉得很多信息都没有留下来。

为了获得更多的信息，并感觉我可以回答关于以前工作的问题，我开始写工作日志。很简单；只是一个谷歌文档，我在那里写下每个工作日出现的事情——我学到的技巧，如何使用某些脚本，我遇到的错误，出现的问题，以及这些问题的后续答案。

在快节奏的创业环境中，你很容易一直向前，完成一个项目，然后进入下一个项目。写日志迫使我放慢速度，反思我的工作。有时候，写东西的过程会暴露出我理解上的差距。

> 在一个我们需要不断掌握新技术的领域，简化学习过程的策略是非常重要的。写尼拉斯学习日记是帮助我的一件事。

我在这个项目中的经历——从质疑我作为程序员的能力，到慢慢成为首席工程师，到直接与客户进行 beta 测试，以及更新 SDK 到最终发布——给了我更大的信心。我对下一个项目中的所有知识感到兴奋。每个工程师的经历都不一样，但我希望分享我在第一个重大工程项目中学到的东西，对别人有所帮助！
