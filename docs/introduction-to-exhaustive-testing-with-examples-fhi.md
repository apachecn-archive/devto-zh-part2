# 通过示例介绍穷举测试

> 原文：<https://dev.to/dohertykornelia/introduction-to-exhaustive-testing-with-examples-fhi>

## 什么是穷举测试？

当您团队中的所有测试人员都筋疲力尽，并且所有计划的测试都已执行完毕时，就发生了彻底测试，也称为完全测试。这是一种质量保证测试技术，其中所有场景或数据都经过测试。用一种更容易理解的方式来说，详尽的测试意味着确保在测试阶段结束时没有未被发现的错误。测试一切(输入和前提条件的所有组合)是不可行的，除了琐碎的情况。作为测试人员，我们经常说“我总是没有足够的时间来测试”。即使你有世界上所有的时间，你仍然没有足够的时间来测试所有可能的输入组合和输出组合。

## 详尽测试的策略

我们知道软件测试的 [7 测试原则之一](https://testautomationresources.com/videos/software-testing-principles-tutorial/)说穷举测试是不可能的。现在的问题是:由于详尽的测试是不可能的，我们应该如何得到足够的覆盖率来告诉被测试的应用程序趋向于没有任何缺陷？

我们不是试图用我们倾向于做的事情和我们想要应用的一些有用的方法来测试一切，而是使用风险和优先级来集中测试工作。我们可能会使用诸如成对测试这样的技术来减少组合的数量，或者像因果图这样的东西，并且更多地使用基于逻辑的方法来优化我们的测试。以下是可以明确包含在测试计划中的基本策略:

**对业务的影响:**这不是唯一的，但却是理解风险的最重要的参数之一。测试人员应该质疑，如果被测试模块的某个功能失败了，会给整个模块带来多大的风险。测试人员应该不断地与模块的业务分析师合作，以理解每个功能的失败所带来的威胁，以便在测试中优先考虑它。

**故障概率:**该参数在了解功能的健康状况方面也起着重要作用。与其他功能相比，有些功能很有可能失败，例如，让我们考虑一个电子商务网站，它有一个作为用户登录的功能，另一个功能是确定用户的浏览历史，并正确地给用户更多的选项(交叉销售/追加销售)。作为测试人员，我们可以确定第二个功能被破坏的可能性更高，因为与失败可能性较小的登录页面相比，它包含许多要实现的业务算法。

**回归:**当 AUT 的某些方面发生变化或增加时，测试人员应该通过识别必须更频繁回归的功能来查看该参数。应进行正确的影响分析，以正确确定区域。影响分析应该总是包括开发人员、QA 和业务分析师团队，以获得对受影响区域的正确理解。

**恢复:**如果某个功能中断，多快可以恢复，或者恢复需要多长时间。QA 人员可以从开发人员那里获得关于被测应用程序每个功能的恢复时间的信息。

## 为什么详尽的测试是不切实际和不可能的？

不可能进行完整的测试或详尽的测试。对于大多数系统来说，这几乎是不可能的，原因如下:

一个程序的可能输入域太大，不能完全用于测试一个系统。存在有效输入和无效输入。程序可能有大量的状态。输入可能存在时序限制，即输入可能在某个时间有效，而在其他时间无效。一个有效但未正确计时的输入值被称为不合时宜的输入。一个系统的输入域可以非常大，完全用于测试一个程序。
设计问题可能过于复杂，无法完全测试。该设计可能包含了隐含的设计决策和假设。例如，程序员可以使用全局变量或静态变量来控制程序执行。
不可能创建系统所有可能的执行环境。当软件系统的行为依赖于真实的外部世界，如天气、温度、海拔、压力等时，这变得更加重要。
