# 软件支持的便捷技巧

> 原文：<https://dev.to/zeke/handy-tips-for-software-support-2iap>

*这篇文章最初发表在我的博客上，[这里](https://zekehernandez.com/post/support/)* 。

当我开始软件工程师的职业生涯时，我没有意识到要花多少时间来调查问题和修复错误。在我现在的公司，我和我的队友轮流负责生产问题和支持票；这不是我最喜欢的转变，但它需要做。

负责几个问题和 bug 可能会让人不堪重负，尤其是如果你一天要在几个支持票之间来回切换多次的话。在过去的几年里，我采纳了一些我认为有用的实践，也许它们也能帮助你！

## 1。有条理

这听起来是显而易见的，但是组织可以极大地提高你作为程序员的效率。当早上或周末或其他工作结束后返回工作岗位时，我们希望花更少的时间问“这又是怎么回事？”或者“我做了多长时间了？”花更多时间解决问题。我喜欢这样保持有条理。

### 文件夹

我喜欢有一个文件夹，其中包含我正在处理的每个班次的一个文件夹。每个文件夹都标有票证编号和简要描述(通常是票证的标题)。在每个文件夹中，我都保存了从我们的文件服务器上复制的所有数据文件。如果我做了任何不需要在票上分享的笔记，我也会把它们留在这里。最后，我在这里保存了一个 SQL 文件(稍后会详细介绍)。

[![Folder Screenshot](../Images/2d89c8328a6bf85c3ff63c1bc37c26b9.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--f6d_EeNB--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/m3tsfwcjp7uy95hkak7g.png)

关于文件的一个额外注意事项:如果您正在调查一个数据问题，并且您有一组文件，其中有好的数据，有些有坏的数据，请清楚地将它们标记为好的和坏的。这听起来显而易见，但我才刚刚开始这么做，它真的很有帮助。你越少接触作为你长期记忆的硬盘，你的工作速度就越快。更重要的是，没有什么比仔细检查一个文件，认为它是一个坏文件，而事实上，它不是。

### 书签

与我们的文件系统文件夹类似，最好有一个清晰标记的书签文件夹，其中包含与票据相关的所有页面的书签。阅读材料的链接、堆栈溢出问题和文档都是值得保存的有价值的项目。如果您正在开发一个 web 应用程序，那么问题页面的书签也非常方便。

### SQL

最后，我希望有一个 SQL 文件，其中包含我在调查票据时使用的任何查询。就个人而言，这是最有帮助的练习。当在票之间来回跳转时，我发现我每次都必须重新创建某些查询来赶上我之前的位置。这绝对是浪费时间。如果你有任何在功能上不明显的疑问，就在上面加个注释；你可能记得他们现在在做什么，但在一个漫长的周末后你可能不记得了。

## 2。谦虚一点

### 求助

你想在寻求帮助和自己解决问题之间找到一个最佳平衡。根据我的经验，这种平衡通常是在寻求帮助中找到的。我所说的“寻求帮助”并不是要求别人解决你的问题，而是给你指明正确的方向。如果你担心你会打扰到你的队友，记住，花 15 分钟的时间来解决问题比你花一个小时来解决问题要好。如果你的队友已经在游戏中呆了一段时间，我相信他们明白这一点。

你的工程师同事不是唯一可以帮助你解决问题的人。如果问题是由技术支持或客户代表报告的；一定要和他们聊票上任何不明确或不清楚的地方。避免浪费时间解决错误的问题。

### 不要自大，就是不要

如果客户将您的应用程序中的行为报告为错误或问题，并且技术支持人员将其发送给您，但结果证明应用程序正在按预期工作，或者问题是由于客户错误造成的；你的回答要亲切。

1.  **不要让技术支持人员或客户因为没有意识到这一点而感到愚蠢。**
2.  **请**以此为契机，重新评估你的申请，看看是否有办法防止此类问题再次发生。
3.  说真的，善良一点。它走了很长的路。

## 3。乐于助人

### 与你的支持团队合作

如果您没有直接与客户联系，那么其他人会，他们可能就是报告故障单的人。意识到这个职位可能带来的压力。对罚单的第一反应要快；如果你能立即提供任何信息，请尽快提供。此外，保持他们对任何更新的关注；你不想让你的客户觉得你没有解决他们的问题。此外，如果有技术支持/客户支持经理，与他或她交谈，看看他们是否有一个优先列表，列出应该首先解决的问题。从事对你有吸引力的问题可能很有诱惑力，但这对你的团队没有帮助。相信您的支持团队能够准确判断客户的感受。

### 向你的工程师同事伸出援手

在更大的工程师团队中，很难知道每个软件组件的来龙去脉。当您完成调查并解决问题后，将其记录下来。如果您发现自己在调试一个文档记录很差的系统，请修复它！不要让下一个开发者不得不经历和你一样的事情。你可以把它记录在一个永久的地方，你的队友很容易找到它，但我也鼓励你给你的队友发一条消息，说一些类似这样的话:“嘿，我刚刚在调查 *x* 并且发现了 *y* 和 *z* ，如果你想知道更多，请告诉我。”
