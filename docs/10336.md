# 博文:团队章鱼——一种反模式？

> 原文:[https://dev.to/jonasbn/blog-post-team-octopus-安-反模式-3946](https://dev.to/jonasbn/blog-post-team-octopus--an-anti-pattern-3946)

我们的许多日常站立会议和许多日常桌上交流听起来像是:

–"*有人需要更新 QA 数据库*

–"*回归测试失败*"

–"*您测试的是什么版本的应用程序，您确定是最新的吗*

经典的是:

–"*我已经在本地安装了组件，并且它可以在我的机器上工作*

我们还有更严重的问题，在 SCRUM 中这些被称为障碍，这些是 SCRUM 大师要处理的，可能需要 SCRUM 团队以外的人的帮助，甚至是升级。

我上面给出的所有例子都不在这个层次，它们可以由团队来处理，也应该由团队来处理，但有时它们会落在两把椅子之间。为了不将这些与阻碍混淆，我称之为障碍，它们可以被克服，但需要一些努力。

现在让我们来搭建舞台。

我们是一个小团队。我们在某些领域都是专家，而在另一些领域则较弱。同时，我们的团队档案由一组功能角色组成，确保我们的软件开发策略的所有需要的方面都被覆盖。

随着时间的推移，团队不断壮大，我们不时有顾问作为团队成员开展工作。我们有些人在公司工作了很长时间，有些人工作时间较短。这意味着，除了个人基于教育和以前的经验为团队带来的东西之外，我们公司积累的知识也对我们个人贡献什么和如何贡献起着重要作用。

总的来说，这给出了一个很好的平衡，在许多失败、实验、星座、计划、回顾、改变、调整之后，我们有了一个好的软件开发过程。

但它并不完美。

我们最大的障碍，也是实际上意味着我们某些部分的总线数量较低的障碍，是落在椅子之间的东西，没有人的工作描述中的任务，将所有东西粘在一起的东西，现代软件团队的无形工作。

我们的团队由以下人员组成:

1.  –1 名团队经理
2.  –1 名前端 UI / UX 开发人员

–3 名开发人员

–2 名测试人员，其中 1 名还担任测试经理

–1 名产品经理—*即我* [![🙂](../Images/2ea3454d48c385c4b31b2c45b77b4229.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--zCyXRrdx--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://s0.wp.com/wp-content/mu-plugins/wpcom-smileys/twemoji/2/72x72/1f642.png)

产品经理和前端开发人员设计特性，开发人员开发，测试人员测试。我们的团队经理负责确保运营问题与新功能的平衡，一切都很顺利——除了似乎永无止境的积压，但这似乎是游戏的一部分。

这是一个很好的软件发布、bug 修复、新特性的设置，但是它确实需要开发软件的基础设施到位。这里我的意思是:

–票务系统

–版本控制

–构建系统

–持续集成系统

–测试系统(应用服务器和数据库服务器)

我们的票务和版本控制系统由我们的运营部门控制，并且几乎一直在工作。如果出现问题，我们只需提出问题，问题就会得到解决。

我们的构建系统高度集成到我们构建的应用程序和框架中，并且基于我们部署到的平台。因此，这是在版本控制之下，工作得很好。在某个时候，当苹果切断了与 OpenSSL 的联系时，我们遇到了很多问题，我们通过为开发人员引入 Docker 解决了这些问题，并且工作得非常好——但稍后会有更多内容。

对于 CI，我们也有一个相当稳定的设置。我们的建筑偶尔会垮掉。通常我们可以根据触发构建的提交来指责罪魁祸首。有时我们的测试环境失败或定时 CI 作业失败，这有许多原因，通常是基于对示例数据的更改或多个团队成员正在处理同一个数据集。有多种方法可以克服这一问题，我们可能有一个很好的技术解决方案——我们有一个问题，下载外部组件会导致我们的 Jenkins 设置无响应，这让我很困惑，为什么这是可能的，但我们只观察了几次。

总而言之，我们有一个运作良好团队、开发流程和基础设施。

但我确实观察到以下问题:

–当事情突然发生时，它往往会落在椅子之间

–非产品代码维护似乎没有人感兴趣

–共享数据库的维护没有人负责

–开发平台的长期战略所有权没有人感兴趣

我喜欢构建软件，我喜欢整个 SDLC，我经常最终解决上面列出的问题——这很糟糕，因为它不在我的工作描述中，但我是团队中最资深的，它是我的 *cryptonite，*从这个意义上说，当它不工作时，我不能不管它，也许我应该寻求专业帮助。

这让我想起了 Docker——这是一个很好的例子。

Docker 拯救了我们，在这种情况下，由于堆栈的问题，开发环境的维护耗费了大量的时间。苹果切断了与 OpenSSL 的联系，所以我们所有的本地开发环境都遇到了问题。幸运的是，我们部署到 Linux，所以我们没有遇到问题。

所以我引入了 Docker，我们可以开始行动了，这需要大量的实验，我们学到了很多，但我们已经稳定下来，一切都在控制之中。

这让我认为 Docker 可以扮演更重要的角色——是的，我对开发平台和部署平台有长远的考虑，因为我在乎。

–如果我们可以跳过目前的整个部署打包策略，而只使用 Docker 映像进行部署，会怎么样？

当它不在你的工作描述中时，很难进行这样的推进，它需要时间和努力，远远超过在开发团队中引入 Docker，因为它涉及经理和其他部门。

如果没有呢？

如果你看看团队公鸡和一些问题，也许团队中的操作配置文件可以做到这一点，如果团队扩大了 devops 角色。

从历史上看，由于 IT 政策和标准化强调这一点，我们在运营和开发之间有着紧密的分离。

目前，我们发布过程相当繁琐，涉及到一定数量的繁文缛节，是的，它有所改进，但它远不是你所听说的一键部署，甚至是连续部署设置。

–如果我们能够将 Docker 映像直接部署到生产环境中会怎么样？

这肯定会促进反馈循环，从而促进团队和生产力。

–但这是个好主意吗？–与在某些领域有专长的角色合作是一个好主意，我们可以从我们当前的团队中看到哪些专家可以做出贡献，但我们的目标不应该是与我们现有的运营团队更好地整合吗？

如果你看看这个团队，我们没有支持职能，我们的组织只有第一级支持，所以我们充当第二级。或者第三。水平取决于你如何描绘它。我们在这里做的非常有趣，我们让支持角色在开发人员之间的团队中巡回演出。这增加了知识共享和责任。这不是我的功劳，我们现在的团队经理介绍了这一点，但我真希望我几年前就介绍了这一点…

因此，解决我们面临的问题的正确解决方案可能是对运营责任应用相同的策略，我认为这叫做 devops。因此团队可以自己处理*所有*的事情——问题解决了！

我仍然在思考我作为产品经理的角色，我应该走多远。我看到许多角色变得抽象和便利——而我根本不是那种类型。它对我不起作用，也许这是我的技术背景，但我需要有一个更低层次的理解。我就像是我们系统的一部分的架构师，我学到的一件事是，不编码的架构师会失去与他们正在构建的东西的耦合。你不应该是一个贡献代码的团队成员，而是原型，例子等。可以在不干扰日常软件开发周期的情况下进行编码。

我仍在努力寻找我作为产品经理的领域，我可能会在未来的帖子中回到这个话题。现在，我正试图让车队找出如何克服我们的障碍…
