# 您的公司正在使用这 8 项预防措施来防止移动应用程序安全漏洞吗？

随着市场转向移动，大大小小的企业都看到移动安全漏洞的增加。从[网络钓鱼攻击到](https://securityintelligence.com/six-major-data-breach-trends-from-2017/)无法解决已知漏洞，各行各业都在报告违规事件。

> Ponemon Institute 最近发布了调查结果,表明至少 25%的企业将遭遇重大安全漏洞。为了解决这个问题，企业需要开发更强大的安全测试和评估程序，以解决整个企业环境中的漏洞。

不管你的公司是开发[聊天机器人工具](http://snaps.io/chatbotplatform/)还是 fintech 应用，以下八点是所有使用移动技术的公司都应该确保实施的重要措施。

## [](#1-penetration-testing)1。渗透测试

渗透测试试图通过识别漏洞来预测安全漏洞。可利用的漏洞包括远程攻击、物理入侵和社会工程。

渗透的方法和手段涵盖了[广泛的方法层次](https://www.forbes.com/sites/ericbasu/2013/10/13/what-is-a-penetration-test-and-why-would-i-need-one-for-my-company/#48d46a1518a0)。企业应该审查他们的目标和风险不利的关注，以确定容忍的水平。

## [](#2-acceptance-testing)2。验收测试

一旦移动应用程序准备好进行部署，企业就需要通过测试来确保应用程序符合系统接受的强制性标准。最终用户接受度、技术要求、业务目标一致性和操作流程都有需要审查的安全考虑因素。

从操作角度来看，应用程序应在现有环境协议内运行，以避免引入新的或已知的安全风险。很少有广为人知的协议包括限制用户访问崩溃和调试日志、投资良好的加密和安装本地会话超时。

## [](#3-architecture-testing)3。架构测试

在移动应用发布到 App Store 和 Google Play 之前，公司应该针对架构测试源代码中的已知漏洞和攻击媒介。在推出之前，错误和缺陷更便宜、更容易、风险更低。可以更改代码或架构，在数据丢失之前消除漏洞。

开放 Web 应用安全项目(OWASP)提供了一个有用的问题清单，在解决架构和安全问题时要考虑这些问题。

## [](#4-thirdparty-vendor-testing)4。第三方供应商测试

由于第三方供应商根据定义不是企业 IT 环境的一部分，因此他们没有相同级别的安全意识或顾虑。移动应用程序可能是安全的，但如果后端系统和 web 服务存在安全漏洞，数据窃取和其他攻击的风险仍然存在。

当移动应用程序本身由第三方开发时，这个问题变得更加关键。第三方供应商可能会对安全要求做出错误或不正确的假设，不了解企业 IT 标准和策略的全部内容，或者缺乏在企业环境中工作的经验。

## [](#5-using-code-from-thirdparty-developers)5。使用第三方开发人员的代码

从头开发一个应用程序既耗时又具有挑战性。开发人员可以使用免费的开源库和框架来节省时间。然而，确保这些资源的安全至关重要。

众所周知，黑客发布库是希望应用程序开发人员将它们集成到他们的最终产品中。一旦这些应用程序被发布给最终用户，黑客们就可以获得他们想要的信息。

始终验证免费和开源资源的安全性。虽然这些外部贡献在应用程序开发中是可以接受的，但开发者应该始终确认它们的安全性。研究库和框架的来源实体的声誉，以及它们的历史和与其他贡献的成功。逐行检查代码，以充分理解它的作用，同时密切注意网络钓鱼欺诈。此外，仅使用来自已由内部或其他可信验证资源验证的可信资源的代码。

## [](#6-assess-developer-skillsets)6。评估开发人员技能

如果你不确保你的应用程序是安全的，所有的应用程序用户都面临风险。移动应用程序开发人员关注前端问题，如用户体验(UX)和用户界面(UI)或后端实现。大多数时候，安全性并不是他们最关心的问题，尤其是对于面向用户的应用程序。但是，他们的团队中应该有安全专家。

> 作为应用程序开发人员，你是公司产品或服务的最后一道防线。

企业解决方案应该始终将安全性作为开发人员或任何第三方开发人员的核心能力。如果我们谈论 Android 和 iOS 平台，那么这两个平台都提供了大量关于提示和指南的文档，以显著降低应用程序安全问题的频率和影响。

工程文化和发布过程往往会给快速的应用发布带来压力。通常，在紧张的期限内生产的压力会影响安全审查。企业应该确保安全性仍然是发布过程的一个重要部分。

## [](#7-assess-team-response)7。评估团队反应

移动应用测试应包括评估负责支持的企业 IT 团队的成功和失败。it 团队应该有适当的策略和过程，以及度量它们的度量标准，这些度量标准说明 IT 在系统中如何响应、关键性能指标(KPI)和问题解决方案。

解决问题的一个关键因素是了解安全补丁到达用户手中需要多长时间。例如，苹果的审批过程可能长达一周。此外，用户需要接受并下载补丁，这不是开发人员可以控制的。针对这些情况制定一个适当的策略也是至关重要的。

评估 IT 支持和响应，无论是内部的还是外包的，都可以暴露可能导致安全问题的流程问题。

## [](#8-know-industry-security-standards-and-regulations)8。了解行业安全标准和法规

安全性不仅仅是企业策略的问题；在许多情况下，某些安全要求是由网络安全和隐私法律强制执行的，比如 [HIPAA](https://www.hhs.gov/hipaa/index.html) 。否则，他们可能会受到 ISO 认证标准、、 [ENISA](https://www.enisa.europa.eu/topics/standards/standards) 或 [OWASP](https://www.owasp.org/) 等许多标准的管理。

> 根据行业和地区要求保持应用程序和数据的合规性是许多组织中 IT 部门最头疼的问题之一。

企业安全测试意味着确保移动应用程序开发从一开始就包含安全标准。通过建立在整个流程中优先考虑安全性的程序，所有企业都可以降低数据泄露的风险，这种风险会导致法律、财务、技术和公共关系方面的噩梦。
