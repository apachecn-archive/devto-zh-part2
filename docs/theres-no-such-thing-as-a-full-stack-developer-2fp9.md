# 没有全栈开发者这种东西

> 原文：<https://dev.to/remotesynth/theres-no-such-thing-as-a-full-stack-developer-2fp9>

语言很重要。给一些东西贴上标签和下定义有助于把我们头脑中抽象和模糊的东西变得“真实”一旦这个抽象的概念变得“真实”，它就通过迫使甚至那些反对它的人在被定义的结构内争论来框定辩论的术语。

> “语言很重要，因为谁控制了语言，谁就控制了对话，因为谁控制了对话，谁就控制了结果，因为谁策划了辩论，谁就已经赢了……”
> 
> 埃里卡·琼

让我们看看“全栈开发人员”的概念这是一个非常新的概念，但最近它的使用越来越多。在 Hacker Noon 或 Medium 等网站上，似乎有源源不断的帖子试图帮助初级开发人员或有抱负的开发人员成为“全栈开发人员”越来越多的公司正在发布招聘“全栈开发人员”的职位。

然而，我想说的是，仅仅通过创造和使用这个术语，我们现在已经将“全栈开发人员”固化为一个概念，这个概念正在重新定义我们(尤其是雇佣我们的雇主)如何看待完成这项工作所需的技能。曾经被我们许多人嘲笑的荒谬的工作需求列表被归入“全栈开发人员”一词

让我们来看一个(绝非全面的)列表，它通常被列为“成为全栈开发人员的基本技能”:

*   HTML/CSS 包括:
    *   前端框架，如 Bootstrap 或 Foundation
    *   像 Sass 这样的 CSS 预处理程序
    *   响应性和/或适应性设计
*   JavaScript 包括:
    *   JavaScript 框架，如 Angular、React 或 Vue
    *   JavaScript 工具链包括 TypeScript、Babel 和 ESLint、npm 等
    *   使用 Jasmine 和 Mocha 等工具进行 JavaScript 测试
*   取决于雇主需求的后端开发可能包括:
    *   服务器端编程语言（Professional Hypertext Preprocessor 的缩写）
    *   Node.JS/JavaScript
    *   红宝石
    *   计算机编程语言
    *   C#或 Java
*   数据库开发包括:
    *   像 MS SQL 这样的 RDBMS
    *   像 MongoDN 这样的 NoSQL 数据存储
    *   像 Redis 这样的内存存储
*   Web 应用程序架构包括:
    *   利用无服务器/云服务实现微服务架构
    *   部署到各种平台，包括 AWS、Azure、Heroku 等。

让我们承认这是一个不可能的列表——即使假设在其中的一些或大多数方面只有最低水平的专业知识。我还忽略了理解 web 应用程序安全性、管理版本控制、配置 web 服务器等基本上都是假定技能的事情。

> “全栈”开发人员是一个垂死的品种。仅仅是跟上 JS & React 生态系统就是一份全职工作！Redux、MobX、GraphQL/Apollo/Relay、Jest、Enzyme、Babel、Webpack、ESLint...再加上频繁的 React 发布，每年的 JS 版本，无数的 npm 包！[# JavaScript](https://twitter.com/hashtag/javascript?src=hash&ref_src=twsrc%5Etfw)[#大问题](https://twitter.com/hashtag/greatproblem?src=hash&ref_src=twsrc%5Etfw)
> 
> — Cory House 🏠 (@housecor) [March 16, 2018](https://twitter.com/housecor/status/974700015414382592?ref_src=twsrc%5Etfw)

以这样的方式列出来使得“全栈开发人员”的需求看起来很可笑。然而，作为一名雇主，我仍然可以*暗示*在“全栈开发人员”这个可接受的术语下的所有那些相同的需求招聘人员和业内培训开发人员的其他人正在阅读诸如“如何成为全栈开发人员的 6 个基本技巧”或“什么是全栈开发人员？”试图使定义正常化，使它看起来更合理，更容易接受。

最终的结果是，即使公司没有在这个列表中得到他们想要的一切，他们仍然成功地提高了成为一名成功的开发人员的要求——并使其在第一时间要求一切变得可以接受。他们让初级开发人员和未来的开发人员渴望获得所谓的“全栈开发人员”的资格

> 在阅读了 Hacker Noon 上的一篇文章后，我想起自己并不喜欢“全栈开发人员”这个术语。我认为这给了公司一个不切实际地要求所有东西的简单方法，并设定了一个无法实现的标准，尤其是对初级开发人员。
> 
> — Brian Rinaldi ([@remotesynth](https://dev.to/remotesynth) ) [March 6, 2018](https://twitter.com/remotesynth/status/971074236596084738?ref_src=twsrc%5Etfw)

实际上，我们已经让“全栈开发者”这个术语来框定这场争论。我们不应该。它是一个定义不存在的东西的术语——不能存在是因为它是一个不可能的标准。我们可以从拒绝使用这个术语开始。我们可以尝试减少向初级开发人员和有抱负的开发人员灌输的废话，这样他们就不会把这个术语视为未来成功的障碍。我们可以要求我们的雇主不要使用这个术语——列出你的要求，而不是把它们藏在一个可笑的标题下。如果你在一家好公司有一份好工作，头衔是“全栈开发人员”，试着说服他们，最好使用一个更清晰的头衔，更好地反映这个角色的具体情况。

希望我们越不愿意接受这个术语，我们就越不会看到它被使用，我们就越能重新控制关于如何成为一个成功的开发者的讨论。
