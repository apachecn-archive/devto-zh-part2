# 激励团队，而不是人

> 原文:[https://dev.to/rkoutnik/incentivize-teams-not-people-2k31](https://dev.to/rkoutnik/incentivize-teams-not-people-2k31)

任何称职的程序员都愿意在团队成员有困难时帮助他们。毕竟，软件开发是一项团队运动。不幸的是，管理层经常会完全忽略这一点，并产生有害的激励，破坏团队的合作和沟通。如果你是软件经理，这里有一个快速的自我测试:

列出你所有的激励措施，从加薪和升职到“谁得到新项目”。确保包括像常规的团队午餐或“月度最佳员工”这样的事情。现在，浏览列表，并针对每一项问自己以下问题:

“如果鲍勃走过去对爱丽丝说‘你有时间吗？我需要一些帮助。，这种激励会促使 Alice 帮助 Bob 吗？"

让我们看看一些常见的管理反模式，以及它们对 Alice 的回答有什么影响。

最糟糕的是将现金或其他奖金与个人绩效目标挂钩的计划。将个人绩效与薪酬挂钩意味着，对 Alice 来说，更好的财务决策是找个借口，专注于自己的工作指标。鲍勃不高兴，因为他仍然被封锁。Alice 也不开心，因为她必须在专注于工作和落后于绩效目标之间做出选择。你真的认为更多的现金会提高士气，足以抵消这种大规模的团队合作吗？

另一方面，一些奖金计划根据*公司*的表现奖励工程师奖金(最大的团队！).如果整个公司做得好，爱丽丝会赚得更多，而不是只关注爱丽丝自己的目标。现在，爱丽丝的外在和内在激励是一致的。她可以帮助鲍勃，因为她知道公司希望他们两个都能成功。

一些公司(比如我的雇主网飞)更进一步，完全取消了奖金，而是提供高于市场水平的薪酬，并明确信任员工在任何特定情况下都会做对公司最有利的事情。网飞的开支政策“符合网飞的最大利益”就是一个典型的例子。

晋升可能是另一个症结。如果鲍勃和爱丽丝都希望得到一个更高级别的职位，他们就会陷入同样的困境。谷歌因在做推广决策时只关注功能开发而声名狼藉，而将其他一切(和所有人)都抛在一边。不出所料，这意味着他们正在推出*另一款*聊天应用，同时让谷歌阅读器寿终正寝。

可能不如同步帮助一个同事那么直接。工程师还有许多其他的方法来改善团队/组织/公司:写文档，提供产品反馈，参加 UX 测试，进行内部会谈，等等。一种选择是坐下来为所有这些活动创建单独的绩效指标。更好的选择是去除不正当的激励，让团队做他们最擅长的事情:构建伟大的产品。
