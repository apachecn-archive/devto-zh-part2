# 软件文档，怎样才能做得更好？

> 原文：<https://dev.to/ardennl/software-documentation-how-can-we-do-better-29e3>

我长期从事前端开发的一个好处是，我不再害怕文档，不管它看起来有多无聊。开始使用新的库和/或框架更容易，而且我认识到如果我真的喜欢以更快的方式做一些事情。

这也让我意识到文档和版本管理真的很令人困惑，而且显然相当困难。

不仅仅是软件，甚至是我们使用的语言。几年前，我们很多人从未听说过 Ecmascript。现在我们都知道“ES”版本，如 ES6，它也被命名为 ES2015 和 ECMAScript 2015。你可能没有意识到，一件事有三个名字会让新手感到困惑。见鬼，这让我很困惑。我只是认为 ES6 和 ECMAScript 2015 表面上是一回事，但当我第一次查找信息时，我不确定一篇*独家*谈论 ES6 的文章是否与博客文章*独家*谈论 ECMAScript 2015 是一回事。(注意我是如何把 ECMA 写成大写和非大写的，因为这显然是它的工作方式)

如果我们谈论框架，我们会看到迭代进行得很快，而文档总是滞后。不仅仅是官方文件，还有所有解释如何用旧方法做特定事情的博客文章。我完全搞不清楚用`React.component`、`React.createClass`和简单的`extends Component`来找 React 的例子。我花了一段时间才弄明白那里到底发生了什么。

当谈到工具时，我在 BabelJS 中遇到了一个很好的例子。我目前正在写一篇关于编译的文章，所以我发现把事情做好很重要。当你浏览巴别塔网站时，它会告诉你安装`babel-preset-env`。当你进入`babel-preset-env` Github 页面时，它告诉你它已被弃用，你应该使用`@babel/preset-env`。困惑了吗？

当然，我们希望我们的软件能够改进，变得更好，并且每隔一段时间做出一些突破性的改变，但是我最大的问题是:我们如何确保不会有一大群人因为看到相互矛盾的文档而感到困惑？

我建议软件创建者/ OSS 维护者应该对官方文档进行突破性的修改，而不仅仅是在一个变更日志中。当然，这只有在官方文档是最新的情况下才有效。

作为开源用户，我们不能忘记我们可以积极地为文档做贡献。事实上，大多数 OSS 维护者非常欢迎文档方面的帮助！

我希望在这个问题上写得更多一点，所以我很想听听你的意见。

*抱歉，如果有什么不清楚的地方，我是在喝咖啡前写的*
