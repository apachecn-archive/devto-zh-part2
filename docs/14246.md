# 前端开发(在这种情况下反应)和坚实的原则

> 原文:[https://dev . to/hatsrumandcode/front-end-development-react-in-this-case-and-solid-principles-592m](https://dev.to/hatsrumandcode/front-end-development-react-in-this-case-and-solid-principles--592m)

作为一名多年的全栈开发人员，我主要在处理后端代码时看到有人提到 SOLID(更不用说强制使用了)。也许几年前实现起来比较困难，但是我们有 Knockout、Angular、Vue 和 React 这样的库和框架，这至少会让事情变得简单一些。

"[React 是固体吗？](https://dev.to/kayis/is-react-solid-630)”是一篇关于 React 如何让您使用可靠原则的短文。因为 React 是一个库，它不会强迫你应用任何原则...

## [](#what-does-solid-stand-for)固体代表什么？*

SOLID 是由 5 个面向对象编程设计原则的第一个字母组成的缩写。基本思想是，如果你遵循这些原则，你的软件会变得更好。

**S** 单一责任原则

**O** 笔/闭原则

**L** 伊斯科夫替代原则

**I** 界面分离原则

**D** 依赖倒置原则

如果你需要提醒，[这里有一个链接](https://medium.com/@cramirez92/s-o-l-i-d-the-first-5-priciples-of-object-oriented-design-with-javascript-790f6ac9b9fa)指向一个好的介绍。

**文章的结论是:**

虽然 React 不会把原则强加给你，但至少它通常会让你遵循这些原则。有时它会因为 JavaScript 而变得更容易，有时 JavaScript 会让它变得更难，但总的来说，用 React 编写可靠的应用程序是可能的。”

作为一名 C#开发人员，我确实发现将 SOLID 应用于后端代码要容易得多，这可能是因为我花了大部分时间在后端代码上。

我目前正在学习反应，至少在尝试...我知道我会尽量遵循至少单一责任和开/关原则。我不太确定依赖反转，因为前端测试使得测试和模仿你的代码变得更加容易，而 C#需要你在一些 DI 机制的帮助下使用接口进行抽象。利斯科夫替代也许是，但界面分离可能不是。

我做了一个 spike 来找出实现依赖注入的方法。我调查的一些事情是:

*   InversifyJS
*   重复可观察
*   使用模块系统
*   属性类型
*   Redux

因为我们使用 Redux 来管理状态，所以在需要的地方(如果需要的话)将它用于 DI 是有意义的。我们目前没有在我们当前的项目中强制使用 TypeScript，但我们希望采用它，因为我们以前在 Angular SPA 上工作时尝试过它。使用 TypeScript 会使应用 SOLID 更容易，并且更符合 C#，我认为这很好。C#开发人员应该会觉得它很熟悉，前端开发人员会接触到 C#中已经使用的概念。对感兴趣的人来说，这可能是进入全栈的好方法？
