# 建还是买？—永恒的 IT 问题。

> 原文永恒的提问者 https://dev.to/mongodb/build-or-buy

我一直在思考关于 IT 解决方案系统的“构建或购买”的想法。

我说的“建造或购买”到底是什么意思？

**构建** —基于您的团队整合所需的部分来构建 IT 解决方案。常见的例子是为您的应用程序构建定制的数据库或存储系统。这些解决方案通常位于(公共或私有)云中，或者可以在内部托管。

**购买** —使用来自公共云提供商的“即服务”解决方案，该解决方案能够抽象出您的 IT 基础架构管理需求。您可以让供应商确保正常运行时间和安全性，而您则专注于应用程序开发。常见的例子是使用像 [MongoDB 的 Atlas](https://www.mongodb.com/cloud/atlas) 或 AWS S3 服务。这些很容易开始工作，因为它们不需要资本支出，通常在几分钟内即可使用。我向最近经常谈论这个话题的人提出了这个问题，谷歌的[凯尔西·海塔尔](https://twitter.com/jaydestro/status/965999841997357058)。

> 我两者都主张。当待售的东西不工作时，你将别无选择，只能建造它。得出这个结论是困难的部分。
> 
> —凯尔西·海托华(@凯尔西·海托华)[2018 年 2 月 20 日](https://twitter.com/kelseyhightower/status/966000517892554752?ref_src=twsrc%5Etfw)

在什么情况下，您认为已经达到了平台在规模和资源方面的极限？此外，您何时确定自托管解决方案不再像平台即服务那样有价值？

### [](#why-build)为什么要建？

对于企业解决方案中的许多团队来说，为诸如数据存储之类的东西构建解决方案往往是一项常见的任务。在部署大型存储阵列时，主要组织往往会考虑一些问题，这些问题也会给团队带来压力:

*   我们将如何对此进行备份？
*   谁来提供长期维护？
*   成本会保持合理吗？
*   在我们存储数据的方式中，我们是否需要包含任何特定的业务或监管规则？

回答这些问题意味着要规划一个包括应用程序生命周期的长期解决方案，具体来说，您需要使用这些数据的应用程序保持可用多长时间？

### [](#why-buy)为什么要买？

基于服务的应用托管已经成为许多希望减少其 IT 架构总占用空间的企业的选择。从供应商处购买系统、谈判和交付所需的准备时间的日子已经一去不复返了。任何企业只需一张信用卡就可以轻松使用可扩展的解决方案，如 MongoDB 的 Atlas 或 T2 的 AWS。购买能力减少了交付应用程序所需的时间。这种向“购买”的转变也让开发商有了许多其他选择:

*   通过基于 API 或 GUI 的界面实现自助服务。

*   自我补救失败响应

*   发出警报。

*   处理常见管理任务的自动化流程。

*   使用可扩展的解决方案(向上和向下扩展)。

### [](#whats-going-to-work-for%C2%A0me)什么对我有用？

这些只是企业和开发者转向服务来托管他们的应用的几个原因。您的用例将最终要求您将使用任一解决方案将出现的所有潜在风险纳入计划。有一个有效的观点是，你真的应该买，直到它变得明显是时候建立。这是在构建应用程序时选择要利用的技术时避免服务供应商锁定的原因之一。

**提示**:

*   考虑像 JSON 这样的开放格式来存储翻译成多种不同语言的数据。

*   如果选择服务，确保供应商在您的情况发生变化时允许您的数据提供给其他人。(成本、竞争、信用)

*   制作清单并记录您的系统架构，而不考虑其托管的未来增长和规模。

*   只使用你的团队可以支持的。

最后一个技巧很关键，我指的是什么？如果您没有人手或资金来处理现场支持，请不要选择内部解决方案。如果您没有验证您拥有的数据是否允许在此环境中，请不要使用云解决方案。
