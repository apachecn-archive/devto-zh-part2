# 进化建筑简介

> 原文：<https://dev.to/hatsrumandcode/evolutionary-architecture--2n7o>

几周前，我参加了一个由 Thoughtworks 组织的关于“[构建进化架构](https://info.thoughtworks.com/evolutionary-architecture-london.html)”的讲座。这是一个非常有趣和重要的主题，它解决了大多数软件开发团队在某个时候会遇到的问题。其中最主要的是[科技债务](https://martinfowler.com/bliki/TechnicalDebt.html)，以及如何监控、减少和防止同类科技再次发生。

本主题讨论了微服务这一新生事物，以及这一新架构如何让我们重新思考如何设计软件。

多年来，软件架构被描述为“以后很难改变的部分”。但是微服务表明，如果架构师将可演进性融入到架构中，变化会变得更容易。”

“增量变化对进化机制至关重要，而不受控制的进化会导致不良的副作用。当基础架构师计划以意想不到的方式不断应对变化时，可预测性是不可能的。相反，更喜欢**可进化性而不是可预测性**。

演讲的重点之一是“**适应度函数**，描述如下:

从数学上来说，函数从某个允许的输入值集合中获取输入，并在某个允许的输出值集合中产生输出。在软件中，我们通常也使用功能这个术语来指代实际上可以实现的东西。然而，正如敏捷软件开发中的验收标准一样，演化架构的适应度函数可能无法在软件中实现(例如，出于法规原因需要的手动过程)，但是架构师仍然必须定义手动适应度函数来帮助指导系统的演化。虽然自动检查更可取，但有些项目不能自动完成所有的健身功能。因此，对于建筑师来说，将架构验证明确地解释为适应度函数仍然是有用的，原因将变得显而易见。”
