# 导言//建立一个分散的博客

> 原文:[https://dev.to/luke/building-a-decentralised-blog-3oec](https://dev.to/luke/building-a-decentralised-blog-3oec)

在开始我在 [Tessian](http://tessian.com/) 的新工作之前(对此我非常兴奋)，我想专注于我在业余时间一直在做的一些项目。我将使用 dev.to 来记录这些努力，部分是为了让自己保持理智，部分是为了让世界知道我正在做的所有酷东西。

其中一个项目叫做[期刊](https://github.com/lukebarnard1/journal)。这是一个自我托管的博客应用，由[矩阵](https://matrix.org)协议支持，这是一个开放的、去中心化的、联合的信息系统。这个想法是，显示、讨论和分享博客所需的所有数据都存储在一个矩阵室中。Matrix rooms 通过用户参与(写博客、评论)的 Matrix 服务器联合在一起，Matrix 拥有将事件流式传输到 web 客户端的 API。因此,《华尔街日报》受益于一个分散化的后端，支持流式更新。

我目前正在设计这将如何根据我选择的要求工作的细节。POC(在任何版本发布之前，请参阅 GitHub 资源库)展示了 journal 在原理上是如何工作的，并在 Matrix 社区中引起了相当大的兴趣。新的设计希望解决 POC 的主要问题，并产生一些只需少量设置就可以在生产中轻松运行的东西。

POC 缺少的一个关键需求是简单的博客文章共享。概念上很简单，分享一篇博客文章的 URL 是在网络上接触多个受众的标准方式，但是 POC 缺乏链接到特定文章的方式。

另一个未满足的需求是像静态网页一样快速地呈现文章。在最初，journal 被设计成一个通过 Matrix 进行所有联网的网络客户端。然而，这不能提供服务器端的渲染，而这是快速显示文章和在社交网站和其他链接共享媒体上显示内容预览所必需的。

我对此有一些有趣的想法，希望能在后续的博客中看到。昨天晚上我一头扎进了 [Next.js](https://nextjs.org) 作为上述问题的一个潜在解决方案，但我会推迟发布 journal 的新架构，直到我检查了它的合理性。
