# 古怪的开发者宣言

> 原文：<https://dev.to/codemouse92/the-cranky-developer-manifesto--24km>

我一周中 90%的时间都花在了别人身上。当你经营一家软件公司时，这是必然的:培训人员、维护工具、完善 devops 工作流程、与开源开发者协调等等。

最近的一篇 dev.to 文章， *[如何修复开发人员的倦怠](https://dev.to/lewismenelaws/how-to-fix-burnout-as-a-developer--4opl)* ，让我想起了过去的一年。**我筋疲力尽了。**我花了所有的时间做管理和开发工作，几乎没有时间编码。

现在，我已经开始腾出更多的工作时间来编码，不管我的盘子里还有什么。我必须这么做，只是为了保持理智。然而，尽管我在工作中热爱编码，但饼干面团中仍有一些坚果壳:

*   我必须遵循公司的标准。是的，也许对我来说更容易，因为我帮助编写了它们，但是它们仍然存在:编码风格、批准的库、许可政策...

*   我必须确保在我们精心打造的构建系统中一切正常。我建立了这个系统，所以我比任何人都了解它，但这并不意味着我总是喜欢使用它。我构建它是为了确保我们写出干净、高效、稳定的代码。

*   在我所有的编码决策中，我必须考虑项目和团队的需求。

*   在大多数情况下，我必须像其他人一样提交代码审查。我要为我写的代码对我的员工负责。

*   性能和便携性始终是我最关心的问题。它应该工作，工作良好，并在我们所有支持的平台上工作。

当所有这些目标和期望都合理时，编码仍然是一种相当愉快的体验，但我仍然在为别人编码。

问题是，我并不是天生如此。我最快乐的时候是在众所周知的 TARDIS 里，做我自己的事，不管其他人(或物理定律，就此而言)怎么说。

这就是个人项目如此重要的地方！一个人做一件事，我想做什么就做什么。我可以设定我自己的目标，并在那里规划我自己的道路。除了我自己，我不用对任何人负责。

而如果我想写可怕的代码，那是我的选择！是的，我知道这听起来很可怕，但这都是为了逃避期望。程序员通常从错误中学到的东西比其他任何东西都多，坦率地说，在不影响其他人的情况下，自由地彻底搞砸一些事情是非常美妙的。

因此，为了实现这一点，我为我的个人项目——一个名为 Elements 的音乐库应用程序——设定了一些规则。

需要注意的是**这些规则*绝不*用于严肃的生产级项目或在公司内部**。

## 规则 1:这是我的项目

永远不要忘记这是你自己的个人项目。如果有人知道它——尤其是在 Github 上——人们通常不会羞于与你分享他们的想法。

小范围内，那还好。有人会想出一个你绝对喜欢的主意。然而，要记住的重要事情是**你是这个项目中唯一重要的最终用户。**

我知道这听起来真的很自私，但这实际上很有疗效。我们花了太多时间为他人设计软件，让我们的想法满足他们的需求，以至于我们实际上没有*让*多想想我们自己。个人项目是弥补这一点的好地方:只有你的想法、你的偏好、你的设计感受和你的工作流程在这个特定的最终结果中起作用。

重要的是，你要为自己设定这个界限。如果你想用你喜欢的字体和配色方案构建一个文本编辑器，那就去做吧。如果你只想支持你选择的操作系统，而忽略所有其他的，没有什么可以阻止你！如果你想让整个界面只通过一套只有你知道的深奥的键盘快捷键工作，设计标准是该死的，给你更多的权力！

现在，很可能您想让您的个人项目对其他人可用，但这是您故意选择的一个特性，并且代价很高！你最好把你的用户群限制在特定的个人身上，而不是整个群体。比如《元素》就是为两个人打造的:我和我妈。现在甚至没有其他人加入进来，这就保持了它的乐趣！

然而，这有一个重要的推论，它是让你立足于现实所必需的:你需要成为你自己的最终用户。建造你将要使用的东西。不管是不是治疗，你都不想养成以设计的总成本来写方便代码的习惯。你自己在为*设计*，但你仍然在设计。

## 规则二:该做的时候就做

在像这样的个人项目中，你应该让大家知道**你不需要对截止日期或时间表**负责，除了那些你决定创造的。慢慢来，尽情享受吧！

这条规则也应该适用于你自己。我们已经花了这么多时间试图按照计划完成代码，目标是在截止日期前交付生产。没有什么比紧迫更能扼杀项目的乐趣了。所以，除非给任务设定目标日期实际上能给你带来乐趣，否则就不要设定任何目标日期。只是滑行。在这个领域已经够难得的奢侈了。

这条规则也很重要，因为它使下一条成为可能...

## 规则三:走极端

有一个时间和地点来避免爬行特征主义(或 feeping-creaturism)，但不是现在！鉴于我们花了大量时间在实用性和现实性的范围内自我审查我们的项目特性，我们需要一个机会让我们的想象力自由驰骋。

这意味着如果你想让你的定制代码编辑器有一个键盘快捷键来远程启动你的咖啡机，你应该*绝对*把它放在你要构建的特性列表中。是的，这个特性可能对这个星球上的其他任何人都毫无用处，因为你用了一个树莓派和一些自定义代码来实现它，但是*谁在乎*？您可以享受能够按 Ctrl+Alt+J 并冲泡一些新鲜咖啡的额外好处。

然而，要做到这一点，你需要为自己设定一个简单的规则:你应该使用一个问题追踪器，并确定你想要构建东西的顺序。如果你花了三个月的时间让文本编辑器与咖啡机对话，但是你不能从你的编辑器中运行代码，你可能会很不高兴使用你写的程序。更糟糕的是，如果您开始使用咖啡功能，然后决定也连接烤面包机，那么您可能会在进行兔子试验时推迟代码编辑器的最基本功能。

相反，你应该把你的想法写在你的问题追踪器上，然后决定你是否以及何时想去实现它们。也许，在四个月内，您会意识到您实际上并不那么喜欢 toast，而是花时间集成您最喜欢的静态代码分析工具。

简而言之，不要让任何事情阻止你*计划*在你的项目中添加任何你想象的功能，但是为了你自己的健康和快乐，一定要努力优先考虑这些功能。

## 法则四:惑人以稳

我们经常因为*bug*而阻碍自由编码。专业项目首先需要稳定；程序每十分钟崩溃一次，能不能做吐司都无所谓。

另一方面，在个人项目中，**稳定性不会自动成为一个因素**！你是这个项目中唯一重要的最终用户(规则 1)，你可能知道如何不破坏软件。

你应该一直保持对 bug 的跟踪，但是可以随意无限期的推迟修复它们。如果它们没有打扰你，并且你不想调试它们，那就不要调试！调试有时很有趣，但有时也很麻烦。你想让*期待*参与你的个人项目。

## 规则五:设定自己的做法

标准对于确保干净的代码和流畅的协作非常重要。在工作中，你几乎肯定会遵循某些惯例。我们还必须考虑好的实践，而且通常是流行的实践。

然而，在你自己的个人项目中，你应该从其他人强加给你的所有这些负担中休息一下。相反，在进行项目的过程中，制定出你自己的个人标准，并坚持下去。

一致的风格有助于确保代码易于继续使用。如果你发现你的风格经常不一致，考虑使用 AStyle，`clang-format`，或者任何数量的其他自动编码风格工具来帮你完成这个重任。

至于其他事情，颠倒通常的规则:不是实践>偏好，而是偏好>实践。我对此是完全认真的！在特定情况下，使用生成器代替 for 循环是否被认为是更好的做法并不重要；重要的是你更愿意写哪个*！如果你愿意，你随时可以回头重构它，但这并不意味着是生产代码；这是有趣的代码！*

## 规则六:随意发牌

我个人不喜欢 GPL，出于道德原因，我不会在这里深入讨论。在工作中，我们必须小心谨慎。然而，在 Elements 中，我遇到了一个独特的问题:使我的编码体验最有趣的库都是 GPL。在拖了几个月的后腿之后，我终于决定许可证在这件事上一点也不重要！我仍然可以编写我的代码 BSD-3，在 GPL 下许可整个项目，并满足所有的法律要求。

无论你是否同意许可证的目的，你都可能从 GPL 中获得最好的开源编码体验。给定你正在使用的库和工具，找到最合适的许可，然后继续。

还有另一条甚至更极端的路可以走。**如果你无意共享或发布代码或软件，你可以完全取消许可。**在这一点上，你实际上只是在*学习*。您可以使用完全不兼容的许可证中的部分内容，甚至不必为自己的作品设置许可证。只要记住*永远不要*让那个项目重见天日。不要放在 Github 上，不要发布在你的站点上，*没事*。建立一个本地存储库并私有地托管它。

重点是找出阻力最小的路径，无论是技术上还是法律上。把自己放在一个可以使用所有需要的库和代码的位置。

## 规则 7:你不是外交官

这不是一个大型的公共项目，而是你的项目。如果你把它发布在 Github 或其他类似的网站上，你可能会遇到问题，收到来自外部贡献者的请求等等。因此，虽然你永远不应该表现得*粗鲁*，但你应该为你的互动方式设定一些清晰的界限。

因此，在我的 Github 项目中，我警告人们我的期望。简而言之，虽然我不想阻止任何人做出贡献...

*   特征请求可能由于任何(或没有)原因而被拒绝，

*   如果我不想修复它们，错误报告可能会被吹掉，

*   拉取请求可能会被无限期地忽略或以任意理由被拒绝，我不欠任何解释。这不是针对个人的。

## 结论和警告

我相信仅仅为了娱乐而开发项目对一个开发者的健康是至关重要的。它让我们的技能保持敏锐，允许我们探索和实验，最重要的是，提醒我们为什么我们首先热爱编码！为了最有效，这些个人项目*应该*不同于生产代码，这就是上面的规则。

然而，正如我之前提到的，你应该**永远不要**将古怪的开发者宣言应用到产品级软件项目中。

如果您真的想将古怪的开发人员的宣言应用到您的项目中，请考虑在您的存储库中包含下面的文件`CRANKY.md`:

```
# THE CRANKY DEVELOPER'S MANIFESTO

I am developing this project for the sole purpose of my own enjoyment.
I make no promises about release date, features, usability, stability,
practicality, or compliance with any normal standards of software
development.

In pursuit of my unhindered enjoyment of this project, the only end-user
I choose to care about in this project is myself, and maybe a few select
friends. The timeline, the features, and the implementation are all
solely at my discretion. I reserve the right to make arbitrary decisions, 
and change them at a moment's notice, without owing anyone an explanation.

If you see something here you like, you're welcome to fork the code under
the terms of the LICENSE and do as you wish with it.

If you're still intent on treating this as a viable project, you're welcome
to submit issues and pull requests. I may respond, or I may leave it sitting
indefinitely. If I ignore your bug report or brilliant contribution until
doomsday, don't take it personally.

If you decide to try and *use* this software, you're taking your sanity
into your own hands. As long as it runs on my machine, that is all I care
about. It may be unstable, or not support your system. I offer neither
warranty nor technical support.

Long story short, I'm just coding for the love of coding! 
```
