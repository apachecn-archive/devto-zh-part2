# 作为文档的用例

> 原文:[https://dev . to/burdettelamar/use-cases-as-documentation-3481](https://dev.to/burdettelamar/use-cases-as-documentation-3481)

本在 dev.to 上回复了一个[帖子](https://dev.to/burdettelamar/markdown-helper--recursive-2d4f)说:“你能进一步描述这个用例吗？”

这让我开始思考....

首先，当我为一个项目写文档时，我应该多考虑用例，少考虑特性。

用户不想要“功能”，他们想要解决方案。一个用例概括了一个问题和一个解决方案，而一个特性可能是寻找一个问题的解决方案。

第二，我应该考虑围绕用例来组织文档，而不是围绕特性。

在我的一个 GitHub 项目(最大的一个)中，我构建了一个“旅游页面”家族(与上一个/下一个导航链接在一起)。该导览涵盖了项目中的功能。

相反，如果我构建一系列“用例”页面，强调问题/解决方案范例，会怎么样？目录难道不会向用户显示项目解决了哪些问题吗？

我将花一些时间来探索这个问题。

同时，欢迎你提出任何想法。
