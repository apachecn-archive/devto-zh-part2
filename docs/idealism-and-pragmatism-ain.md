# 理想主义和实用主义

> 原文：<https://dev.to/joshua/idealism-and-pragmatism-ain>

阿波罗 17 号火星车的挡泥板[脱落](https://airandspace.si.edu/stories/editorial/duct-tape-auto-repair-moon)。这是一个大问题——月球上的灰尘对精密仪器和宇航服非常有害。

幸运的是，有一个解决方案——胶带。美国国家航空航天局建议宇航员将一些地图拼成挡泥板的形状，并用胶带将它贴在火星车上。这解决了他们的问题！

当然，这种解决方案并不理想——最初的挡泥板会比这好得多。然而，在资源允许的情况下，这是一个很好的决定——临时防护板起作用了。

制作软件时也会出现类似的问题:

> "嘿，我可以跳过这个测试吗？"
> 
> “这有点黑，你觉得可以吗？”
> 
> “这不是其他所有事情的处理方式，但它更简单。我觉得还可以吧？”

这些都是常见的问题，我问过很多次，也问过很多次的问题。

你如何判断一个黑客是不是太过了？你怎么知道你是有原则的，还是迂腐的？

像这样的决定属于一个范围。我认为这个光谱是“理想主义和实用主义”光谱。

# 实用主义

[![Duct tape on a bike wheel](../Images/6c03e700a7080a1d43ab989703b9ceea.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--DD7T1XmS--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/287rmg29997zgfkfv68e.png)

实用主义是“以一种基于实践而非理论考虑的方式理智而现实地处理事情。”

在软件环境中，这表现为考虑效用的一种方式。一个更加关注结果(交付、运输、发布或特性)而不是如何实现这些结果的开发人员是一个务实的开发人员。

在实践中，这通常意味着愿意做“黑客”。如果有一条通往好结果的非理想路径，走那条路可能是务实的。

举个例子，在浏览器中，也许你需要覆盖一个`window`方法来修复一个紧急的 bug。或者您可能需要禁用一个测试来使构建通过。这两件事都不理想，但都有好的结果。

# 唯心主义

[![Gavel](../Images/f7e930ba9b79c3fa2c1028db2f809e20.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--SCsRxqA8--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/myophxe8znbv9myg6pof.jpg)

另一个极端是理想主义。理想主义是珍视或追求崇高的原则、目的和目标”

在软件中，当我们有一个不违背的原则时，理想主义就会出现。一个专注于什么是“正确的”，或者“做事情的最佳方式”的开发人员可能在他们的态度上是理想主义的。

软件开发实践中的理想主义可以归结为阻止事情，或者对做得不正确的事情说“不”。它可能是决定不做黑客攻击，或者花更多的努力去做一些“应该做”的事情。

除了对错误的事情说不，理想主义的实践可能意味着写下原则，比如“记录每一个新特性”，或者它可能意味着创建林挺规则，迫使人们以一致的方式编写代码。确切的原则将取决于项目，但理想主义总是瞄准一些高标准，即使很难达到。

这些理想化的方法用一些短期的速度来换取长期的利益，比如一般的代码质量和一致性，测试覆盖率，或者完整的特性文档。

# 如何知道什么时候是哪个

[![Checklist](../Images/d2565a7fc3f566aafa16d29e677db578.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--lcSUlAXr--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/e4kezcofs452ctnklmnz.jpg)

通常需要做出决策，而答案属于这两类中的一类。你应该务实，还是理想主义？

## 其实有原则

没有写下一些原则，你不可能是理想主义者，但是如果不知道你的原则，你也不可能真正务实。对你的项目有价值的是什么？如果需要的话，将这些原则清楚地写下来，并在团队中分享这些知识，将有助于你评估你违反这些原则的程度。

也许你有一个“用户体验第一”的原则，你需要做出选择——延迟一周交付，或者交付一个有些 bug 的用户体验。写下这个原则有助于你认识到每一方的得失，并在这种情况下做出最好的决定，认识到成本和收益。

## 知道压力

开发软件时有压力是正常的——有截止日期、路线图和发布。通常，当这些压力之一出现时，更务实的选择是有吸引力的。

试着分析压力到底是多少。如果对某个结果有很大的影响(其他团队/用户依赖于某个可交付成果)，那就是很大的影响，但是也许截止日期只是一个被选择为可能的交付日期的日期，延迟交付的影响很小。

## 意识到自己的偏见

知道你在理想主义和实用主义之间的位置是很好的第一步。你是否经常发现自己“懒惰”，或者走捷径？还是你发现自己太过“固执”，经常说不？如果你不确定，你可以问问和你一起工作的人，他们可能很清楚你的自然倾向在哪里。

就我个人而言，我倾向于做出务实的选择，我也需要有意识地努力确保原则得到建立和遵守。(工具和规则在这方面帮助很大，因为它们是我无法忘记的严格标准)。

如果你知道你在做决定时天生就有偏见，当你感觉自己做了一个过于务实或过于理想化的决定时，你可以后退一步。

# 例子

[![Bitbucket pull request screen](../Images/9676d140fa6636bea230b18e168b4d59.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--2eLOi7Hn--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/85oq0jeiafqe50jgx7tf.png)

## 拉取请求

您在拉取请求中看到一些不理想的东西。也许某个测试被遗忘了，或者有些应该被重构。你应该拒绝批准吗？

这里要记住的是，你在你的拉动式请求中设定了一个标准，理想主义的方法对未来的工作很有价值([你走过的标准就是你接受的标准](https://hecate.co/blog/code-review-review-is-the-managers-job)

然而，可能会有压力(参见:*了解上面的压力*)使得跳过这些事情成为必要。

一个务实的妥协可能是，“好吧，让我们合并并发布这个，但要继续进行适当的修复”。

这里需要监控的重要事情是这些修复是否真的发生了。

## 钻营

有一种方法可以做一些明显不理想的事情——你正在破坏一个 API，或者你正在做一些明显错误的事情，你几乎感到尴尬。然而，它完成了工作。

这样可以吗？

你可以检查你的原则，看看它违反了多少原则。这是一个更差的 UX，更差的代码质量，还是一个冒险的改变？如果它严重违反了你的原则，最好不要实现这个黑客。

另一方面，黑客也有有用的时候。也许它加快了其他开发人员的速度，在某种程度上，他们的代码不需要依赖这个黑客。它可能允许您按时交付一个高影响力的版本，这是用户或其他开发人员所依赖的。

在你的决定中，你可以包括一个在未来消除这种攻击的计划。在这个计划中，确保你不仅包括**何时**你会修复这个黑客，还包括**为什么**你会修复它。**为什么**可能是一张票，或者是确保完成这件事的过程。黑客的寿命通常很长，所以你应该确定你对它存在很长时间没有意见。

## 坏习惯

外部压力导致了务实的方法——结果是最重要的，但是实现结果的方式是工程师的领域。这意味着，如果不加以控制，坏习惯和做法可能会蔓延开来。

在这种情况下，重要的是要意识到这种情况正在发生(见:*实际上有原则*)，并与之斗争。这可以通过明确地提出原则，并通过拉式请求，鼓励你的团队遵守这些原则。

## 先例

有时，新类型的第一个代码被检入。如果你是第一次做某件事，要意识到你是在为以后所有类似的工作树立一个先例。你是在做第一个 GraphQL 解析器，第一个 API 模型，还是第一个共享库？你开创了一个先例，很多开发者都会效仿！在这种情况下，理想主义的方法总是更可取的。

# 一个伪二分法？

有一系列实用主义和理想主义，但这些方法并不是截然对立的。很多时候，我们既可以有理想的结果，也可以有理想的路径。通常，即使在短期内，实现高效用的最佳方式是以理想的方式做事。然而，对于一些困难的决策来说，情况并非总是如此，有时需要做出权衡。
