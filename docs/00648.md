# 如何管理技术债务

> 原文:[https://dev.to/apastuhov/how-to-manage-technical-debt-3p8g](https://dev.to/apastuhov/how-to-manage-technical-debt-3p8g)

最初发布在我的博客[apastuhov.com](http://apastuhov.com)上。

* * *

今天，与 5 年前相比，更多的人考虑技术债务。因为每个项目都有负债，但不是每个人都在推动。即使他们开车，在大多数情况下，他们也只会保持一个雪球。

我想分享我如何更喜欢管理技术债务，以及它给我带来的结果。

## [](#definition-and-the-problem)定义和问题

维基百科告诉我们:

> 技术债务(也称为设计债务或代码债务)是软件开发中的一个概念，它反映了由于现在选择一个简单的解决方案而不是使用一个需要更长时间的更好的方法所导致的额外返工的隐含成本。
> 
> 如果技术债务没有偿还，它会累积“利息”，使得以后更难实施变更。未解决的技术债务增加了软件熵。

简单来说，技术债务意味着一系列无效的解决方案，这些方案是你或你的团队错误地制定的。不要忘记，每个人都有可能犯错。因此，如果你不管理你的债务，你将会遇到下一个问题。

*   士气低落的团队
*   没有控制项目风险的能力
*   增加软件熵
*   每个人都想从头开始重写整个项目

## [](#causes)原因

原因可能很多，但我从我的角度定义了最常见的原因。

*   需求仍在开发过程中被定义
*   开发在任何设计发生之前就开始了
*   商业压力
*   缺乏过程或知识
*   最后一分钟规格变更
*   糟糕的技术领导力
*   缺乏合作
*   过时的技术
*   糟糕的开发实践

您注意到这些原因大多与企业或客户有关。这是真的——在大多数情况下，技术债务是与你的经理合作填补的。你应该听下一个乐句。

> 现在就做，因为我们真的需要。

然后你通过增加技术债务来达成某种妥协。

免责声明:在文章的最后，你会找到一个链接，链接到根本原因的完整列表。

## [](#technical-debt-and-poc)技术债务和概念验证

如果你在进行概念验证，不要担心技术债务。因为有人会用更稳定的解决方案取代 PoC。项目中的每个人都应该在同一页上的问题。

## [](#popular-solution)通俗解

大多数情况下，开发人员认为在他们的代码中写`TODO` item 就足够了。但是想象一下，如果一个大企业有 40 个微服务存储库。你将如何收集所有待办事项？扫描来源？好吧，但是你如何跟踪进展呢？如果你看不到你做了多少，你就不会感到快乐。

在这里，我想导致我的技术债务管理方式。

## [](#more-precise-solution)更精确的解

我建议使用下一个结构作为表格。你可以用额外的列来扩展这个结构，但是请不要收缩。

*   数字
*   问题
*   原因
*   解决办法
*   如何预防？
*   开始
*   解决
*   状态
*   代理人

看起来很大，但让我深入其中，详细描述每一点。除了数字-一个自动递增的东西。

### [](#issue-cause-solution-prevention)问题-原因-解决方案-预防

用于代码名称的问题，这将有助于您找到和识别一个项目。但是任何问题都有根本原因，确定原因非常关键。

填写原因栏后，回答下一个问题:问题发生的原因/方式/时间？问题将帮助您确定根本原因，并了解谁犯了错误以及原因。

当你知道了根本原因，你就可以解决问题。这是描述你的解决方案的最佳时机。你可以和团队一起集思广益，写出解决方案，这样每个人都会有相同的想法。

不要忘记，在未来你需要防止这一问题。如果你有相同的问题超过 3 次-你有一个问题在你的开发过程中，一些事情正在出错。

### [](#initiated-vs-solved)发起 vs 解决

已启动意味着您在某个时期发现或创建了一个问题。你可以写日期，但是我更喜欢写一个 Sprint 或者迭代的编号。通过这种方式，你可以分析一个表格，并看到 rush-sprint 和增加的技术债务之间的相关性。

当你解决了一个问题，也不要忘记写下来。如果你把事情做对了，你可以建立这样的图表。

[![Resolution chart](../Images/070676c6d3f8cbe20cf9d77c85d9ca12.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--cvDnGLuW--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/17discf68arrue5xt6zv.png)

你可以用决策图来定义你什么时候做了很多无效决策。还有，你可以发现你忘了固定债务很长一段时间。非堆积面积图最适合这种情况。对于纵轴-使用每个 sprint 或日期的总问题数/已解决问题数。

### [](#status-and-assignee)状态和受让人

你可以使用吉拉或任何其他任务跟踪系统，或者根本不使用它。但是，请不要从表格中删除已完成的项目。否则，你将无法跟踪进度和完成的数量。另外，不要忘记你写了如何防止将来出现这种问题。

### [](#example-of-an-item)一个项目的例子

这里我提供了一个技术债务的例子。

| 圆柱 | 价值 |
| --- | --- |
| 数字 | Thirteen |
| 问题 | 过期的自述文件 |
| 原因 | 创建登录页面后，文件夹结构发生了变化 |
| 解决办法 | 更新文件 |
| 如何预防？ | 在同一个拉取请求中用结构的变化来更新自述文件 |
| 开始 | Twenty-one |
| 解决 | Twenty-five |
| 状态 | 完成的 |
| 代理人 | Aleksey Pastuhov |

这里我想指出的是，你可以及时解决问题，这只是简单的优先排序。

## [](#summary)总结

请记住，你应该反复监控你的债务，并在冲刺阶段完成任务。

这是我更喜欢保持和管理技术债务的方式。如果你做得对，你会有积极的结果:

*   一个积极的团队
*   你将能够控制技术风险
*   如果没有需求或设计，你会有很多事情要做
*   你将能够向你的经理或客户争论你需要一个 sprint 来稳定代码
