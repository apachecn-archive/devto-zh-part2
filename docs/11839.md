我一直认为自己是一个典型的开发商，从 IT 到商业，从 fullstackear 到客户电话；我认为这提高了我作为一个开发者的逻辑。

现在我属于商业领域，我意识到学习科学和担任开发商的好处。今天，我对需要有效组织和解决问题的活动采用发展战略。

例如，有一种良好做法(在 it 领域)是模块化开发，模块化代码基本上是将您的程序分解成部分，以形成一个或多个元素，从而使其更易于查看或维护。

[![Fuente: StackBlitz](../Images/2926e18b07f0efb121bd1307b8b9ad8c.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--IbWV_jLF--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/espizncvz8ep6zfactn5.gif)

好吧，我发现你可以把所有东西都模块化起来，从邮件获取客户的需求列表，或者在特定情况下对用户提出疑问，到创建客户关系步骤或未完成步骤的大纲。

虽然使用开发技术有助于我发挥业务作用，但也发生了相反的情况，即在业务领域，使我对客户或最终用户的情况有了更广泛的了解，这对构建理想产品至关重要。偶尔，当你第一时间听到顾客的抱怨或痛苦时，你会间接提高你在 UX 的技能。

[![Fuente: Desktop](../Images/3e6f76970f8df7c93a47aa921eb23759.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--Zlt8vRpq--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/no9edls4qmimi3l4sm5s.gif)

我认为，作为开发者，我们忽略了概念，忘记了所有用户概要信息，从而导致了用户的冲突或麻烦。(开发人员的逻辑非常专注于解决一个问题，但并不总是考虑用户体验，这种观点使您能够与客户保持联系，特别是对于没有 UX 区域的规模不大的公司)

# [](#por-qu%C3%A9-desvirtualizaci%C3%B3n)什么虚拟化？

从我的角度来看，要与客户/用户直接合作，芯片必须变得越来越富有同情心、清晰甚至隐喻性；意识到并非总是正确的，并非每个人都知道技术术语。这就是我所说的“虚拟化”或“极客”世界，也就是虚拟化。
