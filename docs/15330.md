# 最小视窗优先

> 原文:[https://dev.to/nimmo/smallest-viewport-first-3 切克](https://dev.to/nimmo/smallest-viewport-first--3cck)

当涉及到设计/构建(视觉)用户界面时，术语“移动”和“桌面”实际上提出的独特考虑是:

*   Viewport size
*   能力

## Viewport size:

当我们设计或构建任何 UI 时，我们应该从最受限制的地方开始。

如果我们*从设计大视窗开始*,当我们尝试缩小视窗时，很可能会遇到问题。有时我们可能会侥幸逃脱，但通常不会。

如果我们以正确的方式开始，我们将*总是*侥幸成功。

## 能力:

假设我们想提供一些利用用户位置的功能。大多数人自然而然会想到的是`"oh! If they're *on mobile* then..."`

但是一旦你这么做了，你就在基于一个`capability`(在这里是 GPS)对`device type`做一个假设，任何设备理论上都可能有，或者没有。

无论多少次我们可能*认为*我们可以，我们不能基于设备类型做出假设。我们不能假设用户的意图，他们的限制，甚至他们的视窗大小。

像“移动”和“桌面”这样的术语把重点放在了错误的事情上。如果用户的视窗是 320 像素，这并不意味着他们在手机上。他们可能有四个并排的浏览器窗口，或者他们可能有视觉障碍，需要超低分辨率的屏幕。如果我们使用他们的视口大小来打电话询问他们的设备，然后我们基于此对他们的需求做出进一步的假设，那么我们最终会给我们自己*和*我们的用户带来问题。

人们很难不首先考虑直接在他们面前的屏幕。但是改变这种心态是非常有益的。

你很容易就完成了一个大视口模型，然后你去“缩小”它，发现它并不像你最初想象的那样工作，或者你发现你已经以完全不合适的格式布置了一些数据，现在你必须以不同的方式开发一些东西，因为你没有提前计划，或者其他什么。当你正确地做这件事的时候，你不会面对这些问题。

更糟糕的情况是，有人为一个大的视窗设计一些东西，然后把它交给开发人员，并说“你为较小的视窗处理它”——现在我们让不同的人为不同的视窗尺寸设计产品的相同部分。

这不会有好结果。尤其是如果我们有设计头脑的人交给别人(比如我！)没有任何设计技能的人。

### TL；速度三角形定位法(dead reckoning)

视口和功能很重要，设备类型不重要。所以也许我们可以停止欺骗自己认为它是？
