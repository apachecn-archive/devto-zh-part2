# 先读我:内部读物简介

> 原文：<https://dev.to/pullrequest/read-me-first-an-intro-to-internal-readmes-3k2>

最成功的开源项目都有知识性和娱乐性的读物。有徽章、截图、动画演示和入门指南。有些人甚至把它分成单独的文档站点，与自述文件中的章节相匹配。

快速浏览这些开源 repo 的自述文件之一，您可能会对这个项目有一个完整的了解。您将能够访问目的摘要、谁创建了它、他们为什么创建它、它为谁服务、如何安装它以及如何做出贡献。

它通常与我们组织内部的重要存储库不同。奇怪的是，尽管它们是业务的关键组成部分，但最终却常常是过时和空的 READMEs。

OSS 维护者是痴迷的文档狂热分子吗？

我不相信维护者和贡献者对文档有奇怪的亲近感。

他们热衷于开发帮助他人改变世界的软件。要把他们的项目送到合适的用户手中，需要一些
’[营销](https://www.pullrequest.com/blog/marketing-for-engineers/?utm_medium=referral&utm_source=devto)。”

当你投入到一个新的流行项目中时，很明显有人花了几个小时来记录它，这样更多的人可以使用它并为它做出贡献。

在自愿做了数百小时的工作后，他们希望它尽可能容易被人们理解、使用和贡献，这是有道理的。如果没有经过深思熟虑的自述文件，潜在用户可能不会发现回购协议，也不会错过这个伟大的功能。

这是否意味着贵公司的每个存储库都应该有一个同样健壮的自述文件？

一个有几千字，建立统计徽章，和 GIF 走查？

不，你们团队的代码库和开源的不一样。只有几个潜在的贡献者会参与你的回购，你们可能都在同一个办公室，如果人们遇到困难，也有机会提问。

很可能大部分代码只适用于您的业务用例，尽管有一个空的自述文件，您的团队仍然会使用它并为项目做出贡献。

这使得自述文件成为大多数内部回购的补充。短跑之间要做的事情清单上的事情。它们通常充满了很久以前回购协议创建时添加的模糊信息。

许多开发团队认为，因为他们没有试图向世界推销回购，所以根本没有内部自述的受众。

那么，为什么要花时间在阅读材料上呢？从根本上说，拥有一个有用的自述文件可以让您的团队更容易地推出更干净的代码。

开发团队中的每个人都受益于一个带有清晰定义的自述文件的项目。新员工将能够快速建立符合团队标准的贡献代码。总的来说，让更多的人知道如何运行和修复项目会让整个团队受益。

#### 为你阅读中的小变化干杯

大多数内部存储库已经包含了最基本的内容，比如描述其用法的一句话。虽然可能有很多可以添加的内容，但从一开始就很难知道哪些内容对你的公司和项目至关重要。不要期望一夜之间为每个自述文件写好几页。

要开始进行改进，有一个简单的方法来建立基础，让团队开始这个过程。选择你的三个最重要的存储库和你的团队需要知道的三件最重要的事情。例如，如何运行，如何测试，以及为什么做出关键决策。

在此基础上，您可以鼓励团队中的每个人尽可能地对自述文件进行增量修改。在相关 pull 请求的代码审查期间添加一个小注释将会提高您的 READMEs 的整体质量。
