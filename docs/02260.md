# 手动测试的作用

> 原文:[https://dev.to/pbeekums/the-role-of-manual-testing-4pjg](https://dev.to/pbeekums/the-role-of-manual-testing-4pjg)

我是自动化测试的忠实粉丝。数学就这么算出来了。当计算机可以在几秒钟内完成时，为什么要花几个小时来运行一系列测试用例。有了这样的好处，我对自动化测试的热情让我觉得所有的测试都应该自动化。然而，这与事实相去甚远。

在软件开发中，纯机械的手工测试几乎没有空间，但这不是手工测试软件的唯一原因。自动化测试能很好地告诉你软件是否按照你想要的方式工作。他们很难告诉你软件是否按照它应该的方式工作。

让我们以谷歌日历为例。默认视图是显示一周的事件。可以很容易地编写一个自动化测试来验证默认值是否总是一周，如果有人提交的代码导致默认值为其他值，那么测试就会失败。自动化测试不能告诉你一周是否是用户的最佳默认时间。

用户测试和研究通常会得出一周是最佳默认视图的结论。软件终究是为用户而建的。如果用户没有实际使用这些功能，很难为他们找到好的功能。

然而，与用户交谈并非没有挑战。有些用户会反应过度。他们会说“我讨厌这东西！”同时继续使用该功能。其他用户将反应不足。他们会说“这很好”,但从不使用该功能。如果不真正了解这个人，评估反馈的重要性可能会很困难，这将花费大量的时间。

这是手工测试的亮点。使用产品提供了背景、视角和与用户的共鸣，如果我们只是构建一些东西并试图观察用户的反应，这是不存在的。如果你对软件中的交互感到恼火，那么这是一个比较用户反馈的好标准。

我已经记不清有多少次我认为用户反应过度，但当我自己开始使用这个功能时，才意识到交互有多糟糕。对于 [Maleega](https://maleega.com/) ，我创建了编辑邮件线程摘要的权限。测试人员给了我压倒性的负面反馈。但是 Maleega 的 beta 测试者的数量远远不足以达到统计意义，我头脑中的用例告诉我，从长远来看，这是必要的。只有当许可妨碍了我的时候，我才意识到我是多么的错误。

还有谁是产品的好用户的问题。用户角色有助于理解这一点，尽管最容易填写的部分在确定用户身份时往往是最没用的。年龄、性别、职业和地点是相当清晰的领域，便于简单的划分。但是决定谁是你的用户的最重要的部分是他们的习惯。

我在 Maleega 的最初角色包括自由职业者和顾问，因为这些人在工作中大量使用电子邮件。因为他们的时间是有报酬的，所以他们也更愿意为实际有效并节省他们时间的订阅服务付费。

一些 beta 测试者确实被 Maleega 吓跑了。有些人真的很喜欢它。虽然我有很多关于这里的特定功能和那里的缺失功能的反馈，但我错过了决定某人是否喜欢使用 Maleega 的重要因素。只是在把所有这些反馈和我自己的使用结合起来之后，我才意识到那个因素是一个单一的习惯:整理电子邮件。

喜欢整理电子邮件的人讨厌 Maleega 缺少文件夹和标签，这是一个被故意遗漏的功能。不喜欢整理电子邮件的人喜欢 Maleega 为他们提供的一点点组织，几乎不需要任何努力。如果我不是自己产品的长期用户，我永远不会意识到这一点。我需要的视角只能来自每天使用该产品。

这种用法是手工测试最重要的作用。它可能不遵循特定的测试用例，但是它仍然在测试。它的实用性来自于不重复的机械动作。测试用例通常是围绕确保功能正常工作而构建的。在手动测试中，我们真正想要的是确保我们了解用户的感受。在这项工作中，没有任何自动化可以取代人。
