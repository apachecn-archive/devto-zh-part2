## [](#building-bridges)建筑桥梁

你知道金门大桥从构思到建成花了大约 65 年的时间吗？

1872 年，一个名叫查尔斯·克罗克的人提议在旧金山和马林县(这两个镇由金门大桥连接)之间修建一座桥。

直到 1917 年，另一个名叫约瑟夫·施特劳斯的人，后来成为这座桥的总工程师，提交了一份建造这座桥的非正式建议。几年后，在 1921 年，他提交了一份估价为 1700 万美元的建造方案。

1933 年，这座桥开始动工，4 年后的 1937 年完工(我知道，它的建造速度也让我吃惊)。

它最终花费了 2700 万美元，比预算高出 60%，以今天的美元计算，相当于 15 亿美元。

我不是桥梁历史学家，但建造桥梁和构建 web 应用程序或任何软件项目有很多共同点。

#### [](#why-do-we-wing-it-with-software)我们为什么要用软件即兴创作？

许多人，甚至包括我自己，都倾向于匆忙投入到软件相关的项目中，好像写代码是首先要做的最重要的事情。

我写代码和发布代码已经有 20 年了，我看过几十个团队的几十个项目。我可以非常肯定地说，如果你花一些时间提前计划好事情，你的情况会更好。

在过去的几年里，你可能听说过“敏捷”或“快速运输”这样的术语，这些东西在正确的上下文中是很棒的，但这并不意味着你应该尝试开发一个复杂的 web 应用程序，而不花一些时间预先计划事情。

#### [](#whats-on-the-agenda-for-this-guide)本指南的日程安排是什么？

本指南将向你展示如何将一个真正的网络应用创意转化为一个可行的计划。您将能够将这些相同的步骤应用到您自己的项目中，并且您使用什么 web 框架都没有关系，因为我们不需要编写任何代码。

不管这个想法是来自你的头脑，还是来自你刚刚咨询过的潜在客户。

当我和客户做咨询时，我经常记录对话，然后从那里我会把所有的事情分解。事实上，您可以将这种分解作为“项目发现”文档或演示来提供(甚至为此收费)。

它当然有很大的价值，因为完成了相当多的棘手的规划工作，如果您不能或不愿意自己做实际的编程工作，它甚至可以交给别人来做。

这个计划是花费你时间的一个很好的方式，因为如果它需要改进，你可以不需要修改任何代码就可以做到，因为你还没有编写任何代码。

**也不要被“计划”这个名字**吓到。我们不会坐下来花一年时间写一份 500 页的商业计划，然后雇佣一个委员会。**一旦你掌握了窍门，你就能在几个小时或更短的时间内规划出大多数项目。**

## 让我们制定一个计划

在我们开始创建一个很棒的计划之前，我们需要首先定义我们正在构建的 web 应用程序的类型。

对于我自己的想法，我倾向于自然地对着麦克风说话，并解释我希望这个网站做什么。我觉得这是一个很好的实践，因为它模拟了客户在咨询过程中会对你做什么。

能够将一个想法的冗长描述提炼为一个主要目标是拥有的一项重要技能。

下面是我开始漫谈[用 Flask](https://buildasaasappwithflask.com/?utm_source=devto&utm_medium=website&utm_campaign=start-and-finish-web-app) 构建 SAAS 应用程序课程的示例应用程序将会做什么时记录的文字。

##### [](#this-guide-is-an-inside-look-at-how-i-started-a-4000-line-flask-application)本指南深入介绍了我如何开始一个 4000+线烧瓶应用程序:

> 我想创建一个基于视频的培训课程，教开发者关于烧瓶。我读过的大多数教程都是零散的，没有涉及到创建生产就绪的应用程序。
> 
> 我觉得这是一种耻辱，因为通常当你创建你自己的项目时，你希望它们是设计良好和可维护的，换句话说，是生产就绪的。
> 
> 我最不想做的事情是创建一个类似待办事项应用的东西，我不赞成在课程中创建 10 个不同的应用，因为这最终会是 30 个小时的视频，不会真正教你任何“真实”的东西。
> 
> 我的直觉告诉我，一个连接不同点的更大的应用程序可能对人们更有价值，因为它可以显示系统的不同部分是如何连接的。
> 
> 我在教程中很少看到的东西是处理支付，所以如果我包括一种与 Stripe 这样的流行支付网关集成的方法，那将会很酷。

> 然后，该系统的用户可以注册一个特定的每月计划，他们将每月自动计费。这也将是很好的，如果我展示如何做一次付款。
> 
> 肯定需要想出一些有趣的东西来建立付费墙的保护，这样人们就不会太无聊，因为这个项目将是相当大的。
> 
> 也许我可以创造一个游戏。我不希望课程的重点放在游戏上，所以它不能是超级复杂的东西或涉及大型第三方库。
> 
> 这个游戏还需要足够普通，人们不需要太多思考就能理解。它还应该具有某种类型的软实时方面，这样我就可以处理 ajax 请求和 JSON 响应，这是人们要求我在其他学习材料中演示的内容。

> 
> 应该有一个像主页，服务条款和隐私政策静态页面夫妇。不需要详细介绍法律页面，但最好能包含一个模板，因为自己一个人去找总是很痛苦的。
> 
> 嗯，还有什么。需要确保包括 CLI 组件来帮助管理项目，如设置种子数据和数据库迁移。

#### [](#how-to-turn-the-above-into-a-plan)如何把以上变成计划

现在来说说有趣的事情，你如何将上面的话浓缩成一个主要目标？这个目标非常重要，因为它将是你的应用程序的基础。其他所有功能都将源于这个目标。

与此相关的是，你可能在野外听到的一个常见术语是“MVP”，代表最小可行产品。这是你的产品在向世界展示之前最起码要做的事情。

在 BSAWF(用 Flask 构建一个 SAAS 应用程序)案例中，接受客户的定期付款。

我之所以得出这个结论，是因为从技术上来说，如果我所做的一切都是为了支持这一点，并在付费墙后面放入一个虚拟的“将受保护的数据放在这里”页面，那么一切仍然是有意义的。

一旦你定义了这个目标，坚持你的观点是很重要的，因为如果你养成了添加一百万个“次要”特性的习惯，那么你的应用程序将会失去焦点并且会被延迟。

那些附带的功能甚至可能不是你的客户想要的。

实际上，当您发布 MVP 时，您会希望开始从客户那里获得反馈。现在，您可以根据反馈而不是猜测来开始构建您的应用程序。

这正是我在 BSAWF 课程中所做的，因为我发布了该课程的第一个版本，虽然它很成功，但我确实在应用程序中引入了一些人们并不真正关心的东西。

因此，对于版本 2(这是本指南的基础，也是视频课程的基础)，我听取了以前学生的客户反馈和博客帖子的评论。

#### [](#exploring-the-apps-features)探索应用程序的功能

主要目标是接受客户的定期付款。仅仅使用自然语言，我们已经成功定义了两大特征。

我们需要一种方法来处理:

*   支付
*   用户

这两件事都很大，如果你告诉我“嘿，尼克，给我建一个支付系统”，如果我试图解决这个问题而不分解它，我会在我的代码编辑器中徘徊很长时间。

不仅如此，当我看到所有的部分最终是如何组合在一起的时候，我肯定会把东西设计得很差，并最终多次重写。

让我们更详细地谈谈付款，好吗？

##### [](#payment-features)支付功能

当将特性分解成步骤或更小的特性时，没有真正神奇的公式或约定来确定它是步骤还是子特性。

用你最好的判断力，不要纠结于这样的事情。只要有可能，尽可能详细地描述。你钻得越深，问题就越小，也就越容易推理。

我也喜欢在列表中嵌套一些东西，比如:

*   通过价格表查看多个订阅计划
    *   挑选并订阅一个特定的计划
        *   输入信用卡详细信息以订阅计划
            *   用条纹处理信用卡
*   升级或降级订阅
*   取消订阅
*   创建一次性购买(微交易)
*   将优惠券应用于订阅和一次性购买
*   保存所有交易的发票
    *   查看所有交易的帐单历史记录
    *   确保本地发票与条带同步
*   检测和跟踪即将到期的信用卡
    *   当用户的卡即将到期时警告用户
*   更新信用卡信息

突然间，我们的支付系统看起来不再那么疯狂了。它正变得越来越清晰，我相信你的大脑正在构思如何在你最喜欢的 web 框架中实现这些东西。

即使你不知道具体要做什么，你也可以通过谷歌找到解决方案。你可以直接键入“我如何取消 Stripe 的订阅”并找到某种类型的答案。

##### [](#user-features)用户特征

现在来说说用户特性。我们将以处理付款的相同方式处理细分:

*   通过提供电子邮件地址和密码来创建帐户
*   登录和注销
    *   跟踪所有登录活动(IP 地址、时间戳等)。)
*   需要时收集额外信息的能力
    *   用户名(网站的公共身份)
*   重置密码
    *   提供身份以启动密码重置请求
    *   点击电子邮件中的链接后提交新密码
*   账户管理
    *   编辑登录凭据
    *   编辑个人资料信息(用户名等。)
    *   更改订阅状态
    *   查看账单历史

一旦我们开始谈论每一步，就不再那么可怕了。我们还可以看到，在帐户管理下，这是可以调整支付设置的地方。

通过这种方式计划事情，我们可以开始考虑“屏幕”，它可以是网站上的实际页面，但我们将在稍后讨论。

现在，让我们继续向前并且计划游戏。

##### [](#game-features)游戏特色

此时，你可能开始明白了。在我这个项目的早期脚本中，我推迟了定义我们将要玩什么类型的游戏，因为它在大计划中不是那么重要。

最终我决定我想创造一个叫做蛇眼的骰子游戏。很简单，每个用户帐户将免费获得一定数量的硬币，然后用户可以购买额外的硬币作为微交易和/或通过成为每月订户获得额外的硬币。

然后，用户可以下注这些硬币，同时试图猜测掷骰子的结果，如果他们赢了，他们将根据赔率获得更多的硬币。

[![Snake Eyes Build a SAAS App with Flask game](../Images/8f975433b7067a64452f2a6c719360a3.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--DPR9yO4D--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/84vh500qhaeik7etp77p.jpg)

我设想在按下 bet 按钮后，掷骰子将发生在服务器端，然后客户端将使用 ajax 请求部分更新页面。这很好地演示了部分页面重载。

这是游戏的细目分类:

*   用户可以用硬币下注
    *   除非他们有硬币可以赌
    *   除非他们事先没有下赌注(限速)
*   创建下注历史以显示终生下注历史

这里真的没有太多，但值得一提的是，我们的用户功能最终会发生变化，因为帐户管理功能需要一种方式让用户查看他们的下注历史并购买更多的硬币。

由于我们还没有编写任何代码，所以更改计划非常容易。我不会在这里这样做，但是您只是在列表中添加了更多的步骤。

##### [](#cli-features)CLI 特性

与管理功能类似，我不打算为您计划事情，但这次是出于不同的原因。

我从来没有费心去计划这些，对于某些事情，我认为这是可以的。

首先，让我解释一下 CLI(命令行界面)功能是什么意思。

在 web 应用程序的开发过程中，您倾向于执行一些任务，比如播种数据库、数据库迁移、运行测试等等。

其中一些命令运行起来可能会很乏味，尤其是使用 Flask 时。在 BSAWF 案例中，我创建了 10 个 CLI 命令来提高您的生活质量。

它们是:

*   `add`它允许你在数据库中植入每个模型的虚假数据
*   `babel`用于处理国际化(i18n)管理
*   `cov`针对代码库运行测试覆盖工具
*   `db`用于封装了 Alembic 的数据库迁移
*   `flake8`用于对代码库进行静态分析
*   `loc`用于计算代码行数(Python、HTML、CSS 和 JS)
*   获得所有路线、终点和方法的 ASCII 列表
*   `secret`用于生成唯一的秘密令牌
*   `stripe`用于管理订阅计划
*   `test`运行 py.test 驱动的测试套件

能够运行`snakeeyes add all`,让数据库重置，并填充成千上万的虚假但相关的用户、发票和赌注，这对开发非常有用。

我没有把这些计划出来的原因是，在我个人因为不得不一直把某些东西打出来而感到恼火之前，有一半的东西甚至还没有出现。

在这种情况下，你可以说 CLI 的特性是基于用户的反馈而不是猜测，很多事情都是这样做的。尤其是对于那些不一定是项目核心的“次要”特性。

## [](#beyond-the-plan)超越计划

该计划旨在采用一个模糊的概念，如“接受付款”或“与其他开发人员发布和共享代码”，并将其分成可管理的、定义良好的步骤，您可以开始编码。

在你的项目发布之后，你将会根据真实的反馈来添加或删除一些特性，但是我个人认为不值得让你的计划保持同步。

如果你还在计划阶段，并且在编写代码之前已经添加了新的东西，那么一定要回去编辑计划的其他部分。

但我认为最初的计划是达到目的的一种手段。你写它是为了达到最初的目的，然后你采取行动，继续生活。

回顾过去的计划，看看你的产品有多大的变化也是很有趣的。

我还发现这些计划对讨厌的咨询客户很有用，他们不明白你为什么要收费。他们带着这个想法来找你，然后困惑于你引用他们 300 小时的工作，并且用几句话解释为什么是徒劳的。

这是您与他们一起检查计划/项目发现文档的地方。

大多数时候，他们的目光会变得呆滞，给你一个老的“好吧好吧，现在我知道为什么了。继续努力，我很高兴雇佣了你！”线。

如果你有几分钟时间，我强烈推荐你阅读[门的问题](http://www.lizengland.com/blog/2014/04/the-door-problem/)。

这是我读过的最好的文章之一，它把一些看似简单的事情分解成一个史诗般的清单，比如“让我们给我们的电子游戏加一扇门”。

#### [](#from-steps-to-screens)从步骤到屏幕

这已经不仅仅是计划一个项目了，但是一旦你有了完整的计划，我喜欢浏览所有的功能和步骤，并开始将它们映射到网站屏幕上。

这些将是你网站上的真实页面，例如忘记密码屏幕或输入你的信用卡信息进行支付。

我没有在这一步花很多时间，也没有使用任何花哨的线框应用程序。我甚至没有勾画出这些屏幕，我只是在思维导图中列出来。

思维导图只是一个用来描述列出“事情”的策略的花哨词汇。

#### [](#modeling-your-data)为您的数据建模

此时，我将开始考虑项目中的数据模型。

我也不为此打开任何软件。只有一张纸和一支笔。这里的目标是开始询问你将如何使用书面形式的数据。

例如:

*   用户应该能够看到他们的投注历史列表

我们马上知道下注历史将需要存储在数据库中，也许作为一个`bets`表，但是确切的名称现在并不重要。

我们还知道用户也需要被存储。

如果用户应该能够看到他们自己的下注历史，那么我们立即开始意识到我们有一对多的关系发生。

我们可以更进一步，说`a user has many bets`和`a bet belongs to a user`。如果你是一名 Rails 开发者，`has_many`和`belongs_to`应该会让你想起一些事情。

这是描述这种关系的一种非常自然的方式。

#### [](#is-it-time-to-write-code-yet)是时候写代码了吗？

不完全是。现在你应该把你计划好的东西都拿出来，开始查看各种技术栈，看看哪些框架和数据存储最适合手头的问题。

您拥有做出明智决策所必需的所有信息，而不是仅仅猜测并在 6 个月后得出结论，如果您使用 xyz web framework，事情会容易 100 倍，即使您必须从头开始学习语言和 web framework。

但是挑选技术堆栈是另一天的故事...

#### [](#practice-practice-practice)练习练习练习

如果你还没有一个项目的想法，不要担心，你仍然可以实践。只要去你最喜欢的网站，开始思考他们的主要目标是什么，然后开始设计功能和可行的步骤。

如果你喜欢冒险，你可以录下你的朋友和家人向你解释如果他们知道如何编程，他们会想构建什么类型的应用程序。

然后把这段录音(可能会很搞笑，可能是有史以来最糟糕的 10 个想法)转化成一个计划。你知道规矩的！
