## [](#o-que-faz-uma-pessoa-desenvolvedora-de-software)**一个软件开发人员是做什么的？**

无论在哪个领域，所有这些职业都是软件开发人员，所以有一个共同点。

软件开发人员有责任使用[自动化](////www.eusoudev.com.br/tdd/)解决业务、人员、生产力或任何问题，即通过软件开发。

让我们想想一个实际例子，分析一个虚构公司的案例:

> 这家公司生来就是为了解决一个问题。创建该公司的人意识到，随着 web 的增长，信息越来越难以找到。为了找到一篇科学论文或一份蛋糕食谱我们需要装饰网页的地址并知道我们要找的信息的确切位置。公司创始人看到的是，每个人都需要一个地方来更容易地找到信息。那么，为什么不创建一个搜索互联网的网站，我们只需要放上几个关键词，比如‘炒掉家里的任何东西而不喷溅’就可以找到我们需要的东西呢？
> 
> 就这样！汇集了一系列软件开发人员来创建这个网站，然后创建了一个公司，通过它的地球数据组织 Toodle 来统治世界。互联网上的信息搜索网站。

我们意识到有些事情已经发生了。有一个问题，并通过软件开发得到解决。所以基本上，这就是我们的职业存在的原因:创造软件来解决人、物或过程等问题。

## [](#e-onde-entram-frontends-backends-ou-mobile-devs)**E onde entram 前端、后端或移动设备？**

## [](#plataforma)**平台**

当我们开发软件时，我们需要服务于各种平台。例如，如果是桌面程序，我们的平台就是操作系统。

作为 backends，我们拥有操作系统(Windows、Linux、BSD 等)作为运行软件的平台。作为 mobile devs，mobile 操作系统(Android、iOS 等)是我们的平台。当我们的专业是前端时，我们有浏览器(Chrome、IE、Firefox 等)作为平台。

这意味着:我们需要非常了解我们的软件将在哪些平台上运行。后端人员需要了解操作系统可用内存的潜在问题，避免处理器过载或滥用网络层。mobile 和前端也是如此，在这两个方面工作的粗心大意的人可能会锁定手机，因为他们放置了一个例程来进行密集处理，而不考虑平台上的可用容量。没有人希望使用应用程序或进入手机内存耗尽的网站，也没有人希望浏览器标签从您的计算机中提取所有资源。

## **Precisa se pre cupar com diversas Plata forms？**

如上所述，由于我们必须使用自己的平台，因此我们需要深入了解最常用的平台。

后端人员可能需要了解不止一个 Linux 发行版才能根据硬件投资的可用资金使用。

移动用户需要了解同一系统的多个版本，这些版本在不同的系统之间变化很大，例如 Android 是其 API 的第 27 版，这意味着新功能或旧功能将停止工作。

前端用户，除了根据自己的意愿实现其 API 的各种可用浏览器外，还需要在同一浏览器的不同版本中运行，而同一浏览器有时还没有您希望用于解决应用程序中的某些问题的功能。

## [](#lida-com-dados-vindo-de-terceiros)**是否处理来自第三方的数据？**

来自第三方的数据将是使用外部 API，如合作伙伴或 API，这些 API 为我们的应用程序的某些功能提供了所需的信息。

后端人员通常将所有第三方 API 调用集中到他们的工作中。它们使用可用的资源隐藏来自第三方的数据，这样，每当有人请求信息(frontends 和 mobile apps)时，就无需请求信息(这种信息并不经常更新)。

但是，有些东西不能留在后端，如用户体验跟踪 API 等。一个很好的例子是 Google Analytics 和 Hotjar 的使用。这些工具必须位于前端或移动应用程序中才能进行跟踪。

与这些 libs 一样，还有第三方信息变化如此之大，以至于不值得留在后端应用程序中，因此请求此类数据的责任在于客户端运行的应用程序。这只会增加服务器响应时间，导致页面或应用程序加载延迟，并导致软件用户退出。

## [](#precisa-persistir-dados-na-plataforma)**是否需要在平台上保留数据？**

软件始终需要存储在某个位置的数据，以供将来参考，甚至与其他软件共享。

对于后端用户，他们需要使用 PostgreSQL、mongodb 等 DBMS 将此数据存储在数据库中。通常，后端用户认为，他们的工作比较困难，因为他们除了基本堆栈(编程语言和开发工具)之外，还需要处理多个软件，但实际上 orms 只是另一个要使用的 API(与浏览器 API 或 Android API 相同)，而且使用 orms 更容易使用这些应用程序

ORM 是一种开发技术，在这种技术中，开发人员无需了解 SQL 即可处理数据，只需在编程语言中使用他们的知识，然后让 lib(第三方或非第三方)来处理 SQL。

过去，即使在具有大量数据和特定数据库问题的一些企业中，也有只负责数据建模的人员，但现在，后端人员自己负责数据建模。大多数情况下都使用 orms。

移动用户还需要保留数据，无论是以文本格式还是使用平台的本机数据库直接保留在用户设备(手机)上，例如使用‘t0’SQLite 的 Android。为了便于工作，也有适用于移动应用程序的 orms，如 Android，即“T2”ormulate“”。

前端人员最终也会处理用户设备上的持久性，在这种情况下，他们使用浏览器处理持久性，并根据需要或数据大小使用 cookies、localStorage、sessionStorage、indexdb 等。

## [](#precisa-se-preocupar-com-sess%C3%A3o-de-usu%C3%A1rio)**是否需要担心用户会话？**

考虑数据持久性的有趣之处在于，在许多情况下，我们需要通过用户会话持续存在，也就是说，当用户终止与后端的连接、关闭浏览器或移动应用程序时，数据将被销毁。

无论如何，我们需要处理用户会话。

在所有情况下，通常使用编程语言功能来控制会话，编程语言已经抽象了此会话创建、保留和终止工作。

## [](#precisa-se-preocupar-com-a-camada-de-rede)**需要担心网层吗？**

以前，所有数据都存储在本地软件中，直接存储在用户计算机上的桌面应用程序中。就像我们手机里的计算器一样，它存储着我们所做的操作的历史记录，我们以后可以访问它。

如今，数据通过网络以多种格式(如 JSON、XML 等)传输到后端的服务器上，并可从其他位置(如其他后端、前端或移动应用程序)访问。

在任何情况下，我们都需要通过网络随时消耗和发送数据。

无论如何，我们还需要了解 HTTP 动词、不同的网络协议，如 FTP、SMTP、SSH 等。

这是因为后端经常需要使用 SMTP 向用户发送电子邮件，需要通过 FTP 访问其他服务器上的文件和/或通过 SSH 访问其他服务。

对于前端或移动用户来说，如果我们只对服务器发出 POST、GET、PUT、UPDATE、DELETE 请求，那么对 HTTP 协议的了解就足够了，但今天我们不仅要处理这些请求，而且还需要使用 SSH 访问某些服务或使用 FTP 获取网络文件。

除了这种协议工作之外，我们在所有情况下都需要担心互联网连接速度慢或崩溃，后端的优先级比我们能够提供高速和高可用性的受控网络流量低，但在前端和移动中，我们依赖用户的互联网，其范围从千兆位到 256 kb/秒的流行带宽。

## [](#precisa-se-preocupar-com-o-funcionamento-offline-da-aplica%C3%A7%C3%A3o)**是否需要担心应用程序离线运行？**

如上所述，互联网可能会崩溃，在这种情况下，需要注意的是在前端运行的应用程序。

我们需要与用户保持某种形式的视觉交流(由设计人员开发)，向用户表明其处于脱机状态。

最近，前端的 pwa 已被广泛讨论，它除了具有类似的应用程序体验之外，还具有基本的离线操作。

在 mobile 应用程序中，这以前是一种模式，在这种模式中，他们使用缓存技术来存储某些数据，并在出现通信故障甚至连接中断时不允许用户离开应用程序。

在后端，人们通常不会创建脱机工作的应用程序，因此服务器必须运行才能将数据发送到前端或移动。如果第三方工具不响应，则此处不包括弹性案例，因为这不是 internet 故障，而是应用程序故障。

## [](#precisa-se-preocupar-com-infraestrutura)**是否需要担心基础设施？**

在任何情况下，我们都可以关心基础设施或不。这是因为，在当今的大多数企业中，团队负责维护基础架构，并提供我们在服务器上运行系统所需的一切。

但是，在某些情况下，如果服务器不存在，则需要后端、前端或移动人员(视具体情况而定)进行干预。

后端人员需要将应用程序放入空气中，并需要服务器站起来。

前端人员需要一台服务器将静态数据传送给最终用户，而且在大多数 SPA 应用程序中，只有一台服务器负责为前端提供服务，因此第一个字节的传送速度比等待后端处理某些数据或加载后端框架要快得多。当然，并不是所有的公司都这样做，所以它就像**也许**在工作表中。

mobile 用户不需要将服务器放在空中，因为应用程序是通过本机平台(Apple、Google、Microsoft 应用程序商店等)交付的，但可能仍需要将应用程序托管在特定服务器上，而不是使用商店，在这种情况下，可能会留给 mobile 用户。

## [](#precisa-se-preocupar-com-cache-de-dados)**是否需要担心数据缓存？**

如前所述，有些数据可以存储在更快的内存存储(如 isso 或 localStorage)中供将来参考，我们需要在所有领域都关注这些数据。

还有一些更具体的情况，例如静态文件，用户的浏览器可以存储这些文件，这样前端应用程序就无需再次下载。

在大多数情况下，客户端(移动和前端)需要缓存的数据是我们知道不需要一直请求服务器的数据，因此应用程序响应速度比提交请求和等待响应更快，在这种情况下，如果用户的 internet 较差，提交可能会花费较长时间，如果后端遇到问题，提交可能会花费较长时间

## [](#bibliotecas-e-frameworks)**Bibliotecas e 框架**

库和框架大大加快了我们的工作和软件交付，而且我们在各个工作领域都广泛使用了它们(无论是好的还是坏的)。

作为后端应用程序，我们需要掌握加快构建后端应用程序的库和框架，如 python libs、[【scrapy】](https://scrapy.org/)和[【SQL alchemy】](http://www.sqlalchemy.org/)或[【django】框架](https://www.djangoproject.com/)

作为移动工作时，我们应该有(我会引用 Android 的话，因为我不处理 iOS 的任何东西)libs 如[【gson】](https://github.com/google/gson)、[【改建】](http://square.github.io/retrofit/)、[【event bus】](http://greenrobot.github.io/EventBus/)以及[【Android framework】](https://developer.android.com/guide/platform/)本身以及最近

引用前端时，我们甚至不需要评论这一点，因为 internet 上的一个主要笑话是每天诞生的 libs 和 JavaScript 框架的数量。

所以，是的…在任何情况下，我们都需要学习和关心 libs 和 frameworks。
