# 软件开发中产品所有者的角色

> 原文：<https://dev.to/apium_hub/the-product-owner-role-in-software-development--263>

对于大多数转向[敏捷方法](https://apiumhub.com/tech-blog-barcelona/agile-transformation/)的公司来说，产品负责人是新事物。然而，每个月你都会看到[产品所有者需求](https://dev.to/apium_hub/agile-product-owner-job-offer-in-barcelona-4cn4)的显著增长。为什么？我们将在本文中讨论它，更多地关注于[软件开发项目](/software-projects-barcelona/)中的产品所有者角色。

## 谁是产品负责人？

一个伟大的产品所有者基本上是他产品的企业家。PO 是负责交付高质量数字产品的敏捷团队的成员。换句话说，PO 定义了用户故事，并对 backlog 进行了优先级排序，同时为团队维护了功能的概念和技术完整性。

敏捷产品所有者在质量控制中的作用是巨大的。阿宝是一个关键的成员，谁接受故事作为完成。根据我们与不同行业的公司合作的经验，如果一个公司没有采购订单，那么在大多数情况下，开发过程会变得一片混乱，当最后期限没有满足时，产品交付的功能并不是真正需要的。

但是，我不得不说，在我看来，要完成敏捷软件开发项目中需要完成的所有工作，产品负责人应该具备一些技术知识，这样才能更有效。为什么？因为产品负责人应该能够与技术团队交流，并理解对推进项目至关重要的概念。此外，PO 应该能够向其他利益相关者解释技术概念。PO 就像是开发团队和涉众之间的中间人。但是除了拥有技术知识，PO 还应该从最终用户的角度了解产品。这意味着 PO 也应该有商业背景。

不幸的是，现在世界上很少有人真正符合这个“理想候选人”的标准，因此许多大学和学院开始开设课程来培养产品所有者。如果你想听听我们的意见，那么我们认为将技术人员、开发人员或 CTO 转化为 PO 比将业务人员转化为 PO 要容易得多。其实以巴塞罗那为例，这里大部分的 po 都是前工程师。

让我们看看 PO 的主要职责范围。

## 产品负责人在软件开发项目中的角色

创建和维护产品积压

在软件世界中，没有什么是永恒不变的，产品负责人根据客户和市场需求调整产品积压是很重要的。还有，好 PO 知道什么时候说不，怎么说不，这可能是最明显但也是最难掌握的一条。对一个新想法或新功能说“是”很容易，这只是产品待办事项中的另一项。然而，良好的积压工作管理包括创建一个可管理的产品积压工作，其中包含可能会实现的项目。将项目添加到待办事项列表中，知道它们不会发生任何事情，只会产生“浪费”和错误的期望。为了避免整个开发过程花费太长时间，项目失去焦点，以及开发的解决方案可能实际上没有解决业务问题的情况，PO 应该对一些特性和变更说不。但是在这些情况下，产品负责人应该解释为什么一个反馈项目将会或不会被开发，并有效地将团队推向富有成效的对话和解决方案。

根据商业价值或投资回报率对待办事项进行优先排序

每个用户故事必须按照相对重要性排序。不应该有 5 个高优先级。重要的是要知道哪个用户故事是第一，哪个是第二，等等。而且这不仅仅是从业务的角度来看，PO 要考虑开发部分以及一些在做 X task 之前根本无法开发的功能。因此，采购订单应该从两方面分析需求，并提出最佳解决方案和最佳优先顺序，从一开始就为产品增加更多价值。

**用户故事**

PO 应该知道如何写用户故事。简单的例子是:作为用户，我想<一些目标或目的>，让<受益，值>。[这里](https://apiumhub.com/tech-blog-barcelona/user-stories-tips/)你有一篇文章解释了这一点。

**在每次冲刺的开始传达愿景和目标**

这有助于让团队保持在正确的轨道上。产品负责人代表客户的声音，并与利益相关者一起创建产品愿景。每一个决策都考虑到了产品愿景。这确保了可持续的产品开发，为开发团队提供了清晰度，并增加了产品成功的机会。

**让客户和利益相关者参与进来，以确保团队正在开发正确的产品**

开发团队不应该花时间向客户解释技术问题，这是采购专员的工作。换句话说，产品负责人是团队对外的代言人，他应该确保所有的沟通渠道都是开放的，并且项目拥有成功所需的适当数量的支持。PO 负责定义实现目标的界限和约束。它们可以包括截止日期、成本限制、内存限制和最低速度。

参与每日的 Scrum 会议、Sprint 计划会议以及 Sprint 回顾和回顾。

对于产品负责人来说，具备良好的沟通能力，能够适应不同的团队和性格类型，显得尤为关键。产品负责人应向利益相关者更新当前状态、进展、可能的困难和问题。

QA

通常情况下，阿宝是唯一一个可以接受故事已经完成的团队成员。这包括验证故事是否满足验收标准，是否有适当的、持续的验收测试，以及它是否符合其[对完成](https://dev.to/apium_hub/definition-of-done-examples-for-software-projects-9c0)的定义。

**ROI**
T3】产品负责人负责提供最好的投资回报。他们负责 sprint 发布和产品层面的所有经济决策。预算、时间和质量可以根据需要进行调整，每个产品 backlog 的成本和收益也可以用于区分用户故事的优先级。

**解决冲突**

任何不能处理冲突的人都不应该成为产品负责人。在数字产品开发中，拥有强大的冲突解决技能对于阻止争端升级和关注真正重要的事情非常重要。有时候，阿宝必须经历一些冲突才能找到解决办法。

**按时交付**

产品负责人负责确保团队满足期限和目标。PO 负责根据里程碑交付最优的[工作软件](https://dev.to/apium_hub/working-software-go-live-strategy)。
