# 展望未来:网络发展预测

> 原文：<https://dev.to/remojansen/a-look-ahead-web-development-predictions-1j86>

在这篇文章中，我想分享一些关于 web 开发世界的预测。做预测是危险的，因为预测很有可能是错误的。然而，我喜欢思考未来，因为它令人兴奋，它可以帮助我们识别潜在的未来职业机会。记住，所有的创新都是从幻想开始的。

在过去的几年里，我一直在阅读一些博客帖子和公告，并观察了 web 开发、DevOps 和云计算领域的一些突破性技术，我开始能够想象十年后 web 开发的世界会是什么样子。这些是我现在能想象到的 web 开发世界中最重要的一些变化。请注意，这些预测不是基于正式的研究，它只是纯粹的猜测。

# 基础设施将不再是问题

云计算已经存在了十多年了。亚马逊在 2006 年推出 AWS，微软在 2010 年推出 Azure。我不知道您是否还记得云计算刚出现时的大肆宣传，但它的前景是这样的:

> 立即扩大或缩小 IT 功能，以满足任何大型组织的近实时需求。
> 
> *   [*【云计算的伟大承诺】*](https://www.forbes.com/2010/06/30/pharmaceuticals-mobile-salesforcecom-technology-cloud-computing.html) 作者 Forbes.com

云计算的承诺从一开始就是正确的，但我觉得直到现在，由于无服务器技术的到来，以及十多年后，它才成为一个兑现的承诺。这一过程中的一个重要里程碑是 Docker 等容器技术和 Kubernetes 等容器编排技术的出现。

集装箱技术得名于航运业使用的[联运集装箱](https://en.wikipedia.org/wiki/Intermodal_container)。我希望你暂时停止考虑 web 开发和云计算，想想联运集装箱的到来意味着什么。

在联运集装箱到达之前，各种不同形状和重量的物品都必须用手来装载、容纳和卸载。将物品从一个地方移动到另一个地方需要大量的时间和人力。多式联运集装箱允许将物品装入集装箱，然后使用一种独特的标准形状通过不同的运输方式(从船舶到铁路到卡车)进行转移，而无需卸载和重装货物。在 20 世纪 40 年代末和 50 年代初，开始使用标准化的钢制集装箱。不久之后，轮船、卡车、火车和起重机都进行了改造，以优化集装箱的容纳能力。结果，今天世界上最繁忙的贸易港口(上海)每年处理 3200 万个集装箱，运载 7 . 36 亿吨货物到世界各地。联运集装箱实现了前所未有的生产力水平。

[![](../Images/93554a2e69f99f59e5470cce967a31d7.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--LTRrfWSF--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/t78y6v2bsaaac6qnes8i.jpg)

这个故事最令人印象深刻的部分是，多式联运集装箱革命尚未结束。物联网和人工智能的到来将使运输过程更加高效。整个运输链有朝一日可能会 100%自主。卡车、起重机、船只和火车都可以使用物联网传感器提供的数据和人工智能提供的算法在世界各地运输集装箱，而无需一个人的干预。虽然这听起来像科幻小说，但实现这一场景所需的所有技术如今都已具备。

现在让我们回到 web 开发和云计算的世界。我们已经有了容器，也有了容器编排工具。我们也开始有一些完全由云提供商管理的容器编排工具，这意味着我们根本不需要考虑容器编排。我们也有一些无服务器的技术，如[AWS Lambda](https://aws.amazon.com/lambda/)y[Azure functions](https://azure.microsoft.com/en-us/services/functions/)和一些托管分布式数据库，如 [Azure Cosmo DB](https://azure.microsoft.com/en-us/services/cosmos-db/) 。在我看来，管理服务器很快就要成为过去，就像联运集装箱彻底改变了航运业时用手装卸货物一样。

联运集装箱革命的下一步也是云计算革命的下一步。很容易想象物联网和人工智能将对云基础设施管理产生非常重大的影响。例如，可以基于物联网和人工智能提供的预测来扩展集群，并且还可以在服务器故障发生之前预测它们。无服务器技术、物联网和人工智能的结合将允许我们创建可扩展的应用程序，这些应用程序将具有难以置信的弹性和成本效益，开发者甚至不需要考虑这些问题。基础架构将不再是我们所有人关心的问题，我们将能够将全部时间用于业务需求和我们的用户，而不是浪费大量时间来处理与基础架构相关的问题。

今天，对知道如何实现大规模分布式系统的专业人员有很高的需求，但在未来，他们的需求可能会很低。随着组织越来越关注用户需求而不是基础设施，对技术技能的需求将会减少，对软技能的需求将会增加。

# 语义

语义网是这样一种网络，其中所有的信息都以一种计算机可以像人类一样理解的方式进行分类和存储。语义网的想法已经存在很长时间了，但我认为它还没有起飞。然而，我确实相信它会在某个时候发生，因为随着人工智能技术的进步，语义网变得越来越相关。

在未来，系统和服务将需要能够在没有人类干预的情况下相互理解和通信。例如，我们应该能够向 Cortana、Siri 或 Alexa 等虚拟助理请求各种各样的服务。然后，虚拟助理技术应该能够与第三方提供商(如优步或多米诺骨牌)实现的大量系统进行通信。在某种程度上，这些系统能够自动发现和理解彼此将变得至关重要。语义网将在这方面发挥重要作用。

# AI 辅助开发& UX

我们的开发工具将包含由人工智能驱动的辅助。例如，Guess.js 是一个库，它可以通过预测用户工作流和预取资源来帮助我们减少网站中的丢失时间。

> 用于在网络上实现机器学习驱动的用户体验的库和工具。
> 
> *   [Guess.js](https://github.com/guess-js/guess)

微软最近还展示了一款名为 IntelliCode 的人工智能辅助开发工具:

> **IntelliCode** 通过使用数千个公共代码库训练的机器学习模型来生成建议——今天，它使用超过 2000 个 GitHub 存储库，每个存储库都有超过 100 颗星，以确保您从最佳实践中受益。该模型在您的 IDE 中与您的本地代码上下文一起使用，以提供。NET 相关的 API，这些 API 可能与你正在编写的代码行最相关。随着时间的推移，我们将不断发展和改进该模型，因此随着我们的进展，推荐标准会越来越好。
> 
> *   [*【Visual Studio IntelliCode 简介】*](https://blogs.msdn.microsoft.com/visualstudio/2018/05/07/introducing-visual-studio-intellicode/) 微软

这只是两个例子，但还会有更多的例子。在 AI 的帮助下工作将成为常态。

# 分散

另一个重要趋势是具体服务的分散化。我不相信每一项服务都会变得分散，但我确信很多服务都会。随着越来越多的人意识到我们的敏感数据被脸书这样的公司拥有和管理的影响，使用分散服务的想法将变得越来越有吸引力。此外，似乎很多政治领导人对控制互联网上的内容很感兴趣。例如，英国首相特雷莎·梅(Teresa May)长期以来一直倡导监管互联网。分散系统将是规避这些监管的一种方式。

# 编译后的 JavaScript 将继续存在

开发 JavaScript 时使用编译器会越来越主流。我相信现在的领导者是 TypeScript，但即使 TypeScript 消失了，也只是因为它被另一个编译器取代了。编译器可以帮助我们更早地发现一些问题，并且不必担心编译目标的底层细节。然而，编译器也可以自动优化我们代码的某些方面，并生成可供其他开发工具使用的元数据。这种元数据是至关重要的，因为我们可以说它使其他工具能够理解我们代码的某些方面。例如，元数据可以用于允许依赖注入库能够理解我们的应用程序中的依赖。在某种程度上，我们可以说元数据帮助我们的代码变得“语义化”。我可以想象，在未来，有人会有一个疯狂的想法，将编译器与一些人工智能技术连接起来，以创建能够减少构建时间、进一步优化代码和自动检测更多问题的工具。在编译器生成的元数据的帮助下，AI 将为新一代编译器和新一代工具提供动力。

# 总结

总之，人工智能技术将会是一件大事。大公司意识到了这一点，它们在人工智能技术上的投资证明了这一点。AWS、微软和谷歌都在人工智能上下了很大的赌注。

我们在媒体上听到了很多关于人工智能应用于人机交互的突破，但我相信对 web 开发世界最重要的影响将发生在机器对机器的通信中。例如，我们的应用程序和托管它们的云服务之间的交互将由人工智能来管理和增强。
