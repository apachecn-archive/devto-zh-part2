#### 为什么是“军”？为什么不是“盟友”或“倡导者”或“赞助者”？

这是蚂蚁大军，而不是征服开发者的军队。但不是“军蚁”,因为它们很可怕。现在我很痒。

我在演讲中提到过这一点，但几年前，我的一位同事提到,“攀登公司阶梯”这个比喻是无效的，因为它暗示你是独自一人在做这件事。可能还包括你也在往上走的路上踩人。

事实上，你需要一个由倡导者、盟友、导师、同事和管理者组成的庞大支持系统——一支军队。我不仅需要这些人来帮助我的职业发展，还需要他们来帮助我完成所有跨职能的宣传工作。

#### 角色不是一刀切。

仅仅因为你过去曾与从事业务开发的人一起工作过，并不意味着你突然知道业务开发中的每个人都做什么。就像每个公司都不一样，每个角色也可能不一样。了解业务发展在你的公司意味着什么，并了解业务发展团队中不同的个人是做什么的。销售、营销、产品、工程以及你可能遇到的所有其他组织也是如此。

记住，这是一条双行道，我不在乎你在哪边行驶。仅仅因为某人以前与开发者权益倡导者一起工作或互动，并不意味着他们知道你如何给你的开发者权益活动调味。

#### 不要认为有人在“窃取你的销售线索”。

(我将在这里稍微挑一下销售😃)

我已经厌倦了开发者支持者告诉我所有的销售都是接受你的引导。

是的，这里应该有一种健康的紧张感。并不是每一次对话或接触都需要销售，就像不是每一个开发人员都需要开发倡导者的帮助一样。让我们来看看这些情况，从最近的会议展位工作中清理出来，因为它在我的脑海中记忆犹新:

1.  有人走近展台，寻求 API 方面的帮助。
2.  一个人走近展台，寻找销售机会来使用一项服务，或者提到将服务集成到他们的产品中。
3.  一个人走近展台，迫切需要开发资源。

这些非常简单，也不是非常全面，但是它们给了我们一些可以利用的东西。

在情况 1 中，这将是 devrel 的亮点。如果没有 devrel，可以找工程部门的人，甚至是产品部门的人，如果看起来需要一个特性请求的话。销售应该有所收获，你应该教育他们，因为这是 devrel 的一个很好的例子。

情况 2 很有趣，因为有时这个人会直截了当地问“这里有人是销售部的吗？”这有多简单？更简单的是，有人说“我们已经是客户了，但是我们也想评估服务 x。”但有时它会被掩饰为“我是大公司 x 的某某人，我们希望将这一点整合到我们的产品中，但我们需要谈论适合我们的定价层”。所有这些我都很乐意玩点连接器，甚至可能向团队提供一个技术概述。我对长期领导这个项目不感兴趣，但你知道谁感兴趣吗？销售团队的不同成员。

如果会议上有人代表销售，那太好了！如果没有，我会跟进。**这不仅仅是把他们抛在围栏外的情况，或者他们完全有理由“窃取你的领先优势”。**我通常会通过电子邮件或主持会议( **在下一节*中会有更多相关内容)与客户负责人或销售代表会面，解释情况和参与者，然后一旦我们内部做好准备，我会邀请外部各方通过电话或电子邮件进行沟通，让他们做自己的事情。然后我开玩笑说“如果这个成功了，记得我的名字”。最后，我会跟进内部人员，提醒他们如果有必要进行黑客马拉松或技术深度探讨，我很乐意提供帮助。告诉我的经理和我一起工作有多愉快。

最后，情况 3。这种情况经常发生，我确信这与 IBM 的庞大有关。虽然我很乐意为我遇到的每一个项目工作，但我不能，不管你给我多少名誉和财富(说真的，别说了)。我们在 IBM 内部为客户提供付费顾问和开发团队，但是您的公司可能有第三方开发人员或公司，您可以将这些人介绍给他们。这既不是销售，也不是开发者的宣传，但是，我还是会尽我所能地做点连接。

#### 介绍你自己、devrel、你的指标和你的专长。

如果不知道你是谁，不知道你做什么，没有人会和你一起工作。我并不是说去每一个工作岗位，总是和每个人建立联系，但是我是说在互动方面要考虑周全。

正如上一节所提到的，我不会把事情丢给其他团队。我天生好奇，这是福也是祸。

在许多场合，我和别人握手，说“很高兴见到你，我能请你为我做些什么？”这是一种微妙但又不那么微妙的询问电梯间推介的方式。接下来通常是笑声和对他们工作描述的非正式审视，实际的日常活动，以及有希望留下难忘的互动印记。他们问我同样的问题。

如果我幸运的话，他们已经知道什么是 dev rel 或 developer advocation。不管怎样，我的推销看起来有点像这样:

> 我是 Amara，一个专注于 AI、Watson、AR/VR 和一点点游戏开发的开发者倡导者。我最喜欢的对话是与开发团队开始他们的人工智能之旅。如果我们的产品今天对他们不起作用，我希望他们以后想和我们谈谈。

现在它改变了，但有时我会补充说，如果它不是一个很好的适合，如果这似乎没有被理解，我会阻止开发人员使用我们的产品。这是我个人品牌的一部分，也是信任的一部分，对开发者的支持至关重要。我曾经在开发团队中，我们的管理层被销售产品，这些产品并没有解决我们的任何问题，而是制造了更大、更棘手的问题。如果我能阻止这一切，我知道我为所有相关人员做了正确的事情。但我跨职能的哥们，在我告诉他们之前，不一定了解我的包袱和品牌。

#### 总结起来

我认为这是一个关键点，devrel 如何在有军队支持你的情况下工作。

Devrel 是一种自下而上的方法，首先针对开发人员。再次选择销售，我的希望是自上而下的对话最终在中间相遇，而不是拳打脚踢。

但是，如果我没有在这个过程中的正确点上保持参与，或者如果我没有让我的跨职能团队参与，有人会觉得在决策过程中权力较小，这是站不住脚的。至少会毁了某人的一天。
