系统集成是一个越来越多地被利用的过程，公司正在认识到其业务的价值。这个过程包括把不同的系统作为一个整体一起工作。

举个例子，如果你的公司有一个销售部门。所有销售联系数据都将存在于 CRM 中，比如说 sales force——因为它是 CRM 游戏中的主要参与者。如果你想抓住你所有的活跃客户，并自动将他们注册到你全新的忠诚度计划软件中，会怎么样？你会怎么做？数据转储？Excel 文件？手动复制粘贴？

## 系统互操作性正在成为主流

从商业软件的角度来看，我们看到了系统互操作性的大幅提升。如果您想要构建一个与 Salesforce 集成的产品，您可以通过他们的开放 API 来实现，创建编码块来链接两者。如果您已经有了自己的内部软件，您可以将您的“呼叫”直接构建(编码)到您的软件中，或者为它开发一个 API 来与 Salesforce 和您可能希望与之集成的其他软件产品进行对话。

像 [Zapier](https://zapier.com/) 这样的应用程序甚至可以帮你完成繁重的工作；他们构建了自己的代码片段，允许您在商业软件产品之间发送、同步和转换数据。例如，您可以获取输入到 Salesforce 中的每个新销售线索，并向 Slack 发送一条带有销售线索信息的消息。

## 成功系统集成的秘诀…

是一个坚实的计划！因此，让我们来看看所涉及的步骤，以确保在您构建系统乐高杰作时一切顺利。

### 步骤 1:确定当前软件界面的范围

为了能够成功地集成不同的软件产品，您需要从数据流的角度了解什么是可用的。对于某些产品，会有 API 或数据文件输出，这可能是连接其他系统的唯一方式。对于你自己开发的软件，或者开源软件，你可以随意修改代码来处理数据流。

### 步骤 2:需求收集

把这些系统连接起来，你想完成的所有任务是什么？你能想出以后“可能很好”的其他任务吗？保留一份您希望连接的系统的列表，以及您希望在哪些系统之间操作哪些任务、数据流的方向以及是否将使用触发器或同步。连同这个列表，以及你当前的软件界面，你的系统集成商将能够让你知道哪些部分是可以实现的，哪些可能是艰苦的工作或不可能的。

### 第三步。概述系统集成架构

这一部分将包括将两个或更多系统集成在一起所需的所有文件。它可能涉及像数据流图、软件版本要求、硬件要求等文档。这里应该注意数据类型输入，数据本身是否需要转换，以及数据类型输出，以连接系统。

### 第四步。系统集成开发

流程的这一步实际上涉及到构建执行系统集成所需任务所必需的东西。这可能是更新版本、购买和连接新硬件，以及对系统间的数据流进行编码。测试应该是这个过程中不可或缺的一部分，尽管不是在实时系统上！当端到端测试可用时，需要进行端到端测试。

### 第五步。推广计划

对于关键系统来说，一次在所有系统中上线是不明智的。你可以尝试跨组上线，或者在安静的时候上线，以确保你不会一次给自己带来太多的麻烦。当你在你的实际系统上发布时，可能会有问题——这就是为什么你这样做的原因。

### 第六步。上线

一旦所有的错误都解决了，测试就完成了，并且你已经在一个较小的规模上测试了系统，验证了它们按计划工作，然后你就可以实际上使用你的整个集成系统了。

### 第七步。维护和支持

请始终计划在您的上线日期后进行持续维护和支持。即使您已经用较小的样本量测试了系统，您也可能会在上线后的最初一段时间内需要系统集成商的支持和修复——对于更大或更复杂的项目，这可能需要很长时间。
