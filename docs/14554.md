根据理查德·费曼的说法，理解一个概念的真正标准是能够把它和一个蹒跚学步的孩子联系起来。这就是我喜欢 Dev.to 上的[# explain like im5](https://dev.to/t/explainlikeimfive)频道的原因:用几个简洁的段落解释技术概念，没有太多的技术术语，这是一个有趣的挑战，看看我自己是否真正理解这个概念。我[五月](https://dev.to/dmerand/comment/2fih) [有](https://dev.to/dmerand/comment/2fn5) [一个](https://dev.to/dmerand/comment/2fn1) [的问题](https://dev.to/dmerand/comment/2fnn)带着压抑不住的对这些讨论进行评论...

但是有一件事困扰着我:我不确定提问者是否总是想要一个适合五岁孩子的答案！五岁的孩子通常不关心特定的术语，但对[# explain like im5](https://dev.to/t/explainlikeimfive)的回答往往非常专业。请求者可能希望这样，但这不是你想给一个 5 岁孩子的答案！请求者*是否真的*想要 5 年前的响应还是 7 年前的响应，或者诸如此类的东西，这并不重要。真正重要的是，当你的回答与问题不符时，你就有被忽视或误解的风险。

我觉得理解这些细微差别很有趣！我将尝试向一个理论上的 5 岁、10 岁、15 岁和 20 岁的孩子解释同样的概念。在这样做的时候，我希望从我自己的教学过程中学到一些东西，并希望找到一些行之有效的不同方法。我将尝试[解释 Git + GitHub，就像我是五个](https://dev.to/webdevdeja/explain-git--github-like-im-five--3pak)(十个，十五个，二十个)。

## #EXPLAINLIKEIMFIVE

#explainlikeimfive 的一个好方法是讲一个类似的小故事。五岁的孩子通常不关心特定的术语，但他们非常擅长理解隐喻。你的目标是让他们理解这个想法如何解决与他们可能遇到的问题非常相似的问题。

> 假设你正在写一篇短篇小说，你想写得非常好。你可以把故事写在一张纸上，然后拿另一张纸写另一个版本，然后继续在更多的纸上写更多的版本。过了一段时间，你会得到很多张不是你现在正在写的版本的纸。你可以把这些纸放在一个文件夹里，这样你就可以回去看你的故事的旧草稿。
> 
> 你把那个文件夹交给你妈妈保存，因为你知道如果你需要的话，你可以随时向妈妈要一个版本的故事。另外，妈妈会保留复印件，这样她就可以把你的故事分发给你的兄弟姐妹。妈妈也会为你的兄弟姐妹写的故事保存类似的文件夹。
> 
> `Git`是文件夹，你的纸片是`code`。你妈妈是`GitHub`。

请注意，在上面的故事中，技术术语没有集成到故事中，实际术语的列表被大大简化了(可以完全省略，而不会对理解产生负面影响)。

## # explainikeimten

十岁的孩子已经准备好了术语，但是如果你离题太多或者太专业，他们可能会迷失方向。你仍然可以成功地运用类比，但他们也倾向于更全面地引用专业术语。兰道尔·门罗的《事物解释者》是另一种有效的方法——用通用的词汇和概念解释事物是如何运作的，因为十岁的人往往已经理解了所有真正重要的概念。

> 把电脑`apps`当成故事。假设你正在写一个故事，你想做得非常好。你可以把你的故事修改几遍。但是你不想丢失你的修改——如果你之前有了一个好主意，你想以后再来看呢？因此，您决定将所有修订保存在一个文件夹中。你可以把`git`想象成保存修订的文件夹，而你的`repository`(或`repo`)就是保存你正在写的这篇文章及其所有修订的文件夹。
> 
> 你可能想要分享你正在写的故事！也许你想和朋友分享你的作品，或者你只是想找个地方让人们去阅读。`GitHub`就是这样一个地方——一个 web `app`保存着你的`repos`，给你一个`web site`让他们分享。从`GitHub`，你允许的任何人都可以得到你的故事的拷贝，并对他们的拷贝进行修改。在`git`那叫做`fork`。如果`forked`你的`repo`的人想要与你分享他们的改变，以便你可以在你的`repo`中拥有它们，他们可以在`GitHub`上给你一个`pull request`——有点像给你一个红色改变的版本，并要求你用那个作为你的故事。

## # explain likeim fifth

15 岁时，人们能够在他们感兴趣的领域真正出类拔萃。他们可以处理术语和抽象概念:隐喻虽然有帮助，但并不是一个你需要非常依赖的工具。但是你可能仍然会有短暂的注意力持续时间，所以保持你的描述切题，不要过多关注题外话。

> 假设您正在编写一些代码，但是您担心当您更改代码时，您可能想要`undo`您的更改。所以你需要一个`version control`系统，你可以告诉“我做了一些改变，请记住它们”。`Git`是版本控制系统的一个例子。您的`repository`或`repo`是您的代码，您在工作时将它们`check in`或`commit`到您的版本控制系统。稍后，您可能会为您正在处理的其他代码项目创建其他`git` `repos`。
> 
> `GitHub`是一款为你提供回购网站的应用。其他人可以去这个网站，下载你的 repo 来运行代码，或者在他们自己的副本中做他们自己的修改(如果他们愿意的话！).每个人都有自己的`repo`副本并能对其进行自己的修改的想法被称为`distributed version control`。`GitHub`赋予你管理`pull requests`的能力。`Pull requests`是其他人对您的`repo`副本所做的更改，他们希望您将这些更改包含在您的副本中。

## # explainikeimtwenty

对于一个 20 岁的人来说，你可以一次列出你所有的术语，然后用那些解释完整的术语来解释这个概念。一个 20 岁的人不需要提示就可以在字典里查单词，或者在维基百科上查概念，所以你也不需要过多的解释。你的工作是解释这些术语是如何相互联系的。你也可以岔开一些相关的话题，而不用担心对方会失去主线。

> `Git`是一个`version control system` ( `VCS`，类似于`CVS`、`SVN`，或者`Mercurial`。`VCS`是一个以`append-only`方式将`committing`代码更改为`code repository` ( `repo`)的系统，这样在任何给定的时间点，您都可以看到到该时间点为止对代码所做的所有更改的历史。
> 
> `Git`是一个`distributed version control system` ( `DVCS`，这意味着如果多人在同一个代码项目上工作，他们每个人都有一个`repo`的独立副本(一个`fork`),它可以与其他副本不同。来自其他副本的更改(在`git`中称为`remotes`)通过基于文本的`differential comparison` `algorithm`被`merged`到给定的`repo`。在给定的`repo`中的改变也可以是`branched`然后是`merged later`，给开发者一个机会在没有`committing`压力的情况下处理具体的想法。`Mercurial`是另一个`DVCS`，而`SVN`和`CVS`依赖于`server`上的中央“主”码`repo`，通常称为`trunk`。
> 
> `GitHub`是一款提供围绕`git` `repos`的社交网络的网络应用。`GitHub`会给每个回购一个唯一的`URL`，可用于`forking`或下载一个`repo`。`GitHub`还提供了`pull requests`，其中开发者可以`fork`一个`repo`，创建变更，然后请求将这些变更`merged`到另一个分支中。`GitHub`提供代码历史的`issue tracking`、`wikis`、`visualizations`等其他服务。人们经常使用像`GitHub`或`Bitbucket`这样的服务，因为他们不想管理自己的 git 服务器，或者因为他们想要像`pull requests`或`issue tracking`这样的额外特性。

注意，这个版本是第一次提到 Git 的替代品，比如 SVN 或 Mercurial。我假设一个 20 岁的人如果感兴趣，会想做进一步的研究，所以提供更大的背景，比如命名替代方法，对实现这个目标非常有帮助。
