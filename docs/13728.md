## [](#highlevel-takeaways)高级外卖

*   **了解为什么你的企业需要你开发每一个软件**。如果你从事商业项目，你应该知道为什么你会得到报酬。以及你的工资是如何产生的。每个代码提交都应该为您公司提供额外的价值。
    例子:

    *   吸引新用户或使当前用户对产品更满意的新功能。或者甚至导致顾客购买该产品的高级版本。
    *   一个 bug 修复，让现有客户更开心，让他们继续使用你的产品，而不是让他们考虑竞争对手的解决方案。
    *   一种改进，使工程/产品/支持团队更有效率，从而腾出时间做其他有益的工作
    *   防止将来可能导致客户失去信任的问题或事件的维护。此外，突发事件会占用你本该用于其他工作的时间。

我认为在商业开发中，每个软件系统都与一些商业目标相关。如果你不知道你的团队正在实现的目标，问问你的管理层。如果没有客观问题，公司有什么必要支付你的工作报酬？

*   **将业务 okr 或 KPI 分解为工程可交付成果**。

    *   需要更多用户？了解为什么你没有获得 MAU，并找出你的代码如何增加 MAU。
    *   需要产品有更高的可靠性？通过建立专门的团队进行预防性维护来进行投资。
    *   需要有更多的收入？注意让付费功能更有吸引力。确保你和你的团队所做的事情确实有助于组织实现至少一个目标。否则就是浪费你的才华和公司的钱。
*   **确定你的团队正在做的工作类型**。根据这本书，存在四种类型的努力:

1.  商业项目——这是产品经理/项目发起人要求你做的。此类项目的目标与业务目标紧密相关。执行此类项目可以提高 MAU、客户满意度或收入。

2.  内部项目 -这是你持续实现商业目标所需要的。来自你的团队的工程激励或来自外部团队的要求属于这一类。完成这类项目的结果在参与项目的人群中并不明显。但是当这样的项目被跳过或者执行得很差时，它会影响业务功能。

3.  *变更*包括在上面列出的两类项目中进行的交付物的实际部署。此外，该类别涵盖所有小型家务工作。

4.  *计划外工作* -处理事故和紧急情况。你可能没有时间在你的工作时间表中分配给它。这样做会分散你做其他工作的注意力。太糟糕了。

*   **在您的组织内实施变革管理流程**。一个团队所做的代码或基础设施的变更会影响到另一个团队。当他们将公司范围的数据库版本“升级”到没有您最喜欢的不推荐使用的功能的版本时，您不希望感到惊讶。更真实的案例——几乎无法恢复的数据库模式更改。我知道这可能很难，但您(或您的管理层)应该:
    *   至少为每个团队制定一个季度的产品路线图。
    *   让它在不同的团队中保持一致。
    *   如果有任何向后不兼容的变更，请提前沟通。

要想成功领导团队，你应该跳出团队/部门或活动产品的范畴思考。想一想，你将要带来的重大变化会如何影响作为客户的整个公司。总是评估有风险的变化。

*   **明确自己的开发流程，找到瓶颈**。即使在同一个组织中，过程也可能因团队而异。这些步骤可能包括以下动作(列表*非常*简化):

1.  评估客户的反馈。如果你的客户在公共吉拉为一些功能或开放支持票投票——你很幸运。以此作为输入。

2.  确定定义特性的请求背后的真正需求。定义软件的功能需求。区分特性的优先级，并将其放入产品路线图中。

3.  在组织内分配工程资源以实施该功能。定义非功能性需求。

4.  设计该特性应该如何实现。制定工作分解结构并计划执行。如果需要任何帮助，与外部团队沟通。

5.  写代码。用测试覆盖代码。执行对等评审。修复注释。部署到暂存区。在试运行中执行测试。查找 bug。修复 bug。将代码部署到生产环境中。

6.  为客户启用该功能。接收客户的反馈。**转到第 1 页**:)

每个步骤都涉及不同的技能和角色——从支持工程师和产品经理到软件和质量工程师。
**你的目标是找到约束——最慢/最忙的链条环节，让它更快**。

根据艾利·M·高德拉特的“约束理论”：

> #### Except for the bottleneck, improvement anywhere is an illusion.

只有在这种情况下，您才会看到功能交付的改进，并最终实现业务目标。

*   **设定在制品的限制**。正在进行的任务数量高于您的吞吐量意味着:

    *   你的组织已经为要做的事情支付了费用，但是你的客户并没有从中获得价值。
    *   因为队列中的工作正在等待资源，这意味着资金没有被用来为组织实现价值最大化。

在我的团队中，我们将进行中的标签数量限制在工程师的数量之内。一个工程师不太可能同时做两项工作。在这种情况下，其中一个票证可能被阻止或正在等待其他人提供输入。你应该专注于完成正在进行的工作，而不是开始新的任务。

*   **标准化你的团队正在做的工作**。你的小队很少面临独特的任务。常见任务可以是:

    *   执行数据库模式更改。
    *   更改基础架构配置。
    *   添加更详细的日志记录；为他们跟踪更多的指标和仪表板。
    *   添加一个 API 端点。
    *   添加外部团队提供的新 API 端点的使用。
    *   剖析和优化一些软件。
    *   更改数据转换的业务规则。
    *   重构模块以获得更好的可维护性。
    *   调查客户的需求。

这样做的目的是收集有关任务解决方式的历史数据，以及实施变更所需的时间。

这些信息应该帮助你实现两个目标:第一，培训新的团队成员——他们可以看看类似的问题是如何解决的；第二，为你的企业主提供估算。

*   **不养暴玩家**。如果一个团队中只有一个人可以完成某些任务，那么这个人会非常忙。团队变得格外依赖工程师。

最终，工作在等待这个人获得自由。工程师成为项目的约束。为了进行可持续发展过程，你需要至少有两个人能够完成某项任务。

为了消除总线因素，启动内部知识共享，并投资于不能描述为代码的东西的自动化和文档化。它帮助你培养团队成员。借口*“对我来说做那件事比 expain 容易”*永远不应该被批准。

*   **跟踪您的团队收到的所有工作请求**。产品经理和支持工程师会分散你团队的注意力。实现他们为你设定的目标。是的，这听起来像是一个悖论。但是它发生了。我认为原因在于:很难评估所有客户的需求并对其进行优先排序。

*   理想情况下，你应该为紧急的/计划外的工作安排一些时间。每个请求应该有一张吉拉票。来自外部工程团队的要求也应该被跟踪和区分优先级。例如，如果您的服务公开了一个由其他十个工程团队使用的私有 API 请做好准备，他们中的一些人会要求您对一些重要的支持进行定制。反之亦然——你公司内部的供应商可以根据自己的需要改变游戏规则。

*   **让您的企业进行试验**。工程团队应该提供验证产品假设的能力，在实现上投入最少或者根本不需要编码。

我认为，在本季度，我们的团队在该领域取得了成功:一些业务能够在不分散团队实现其他承诺的情况下进行一些实验。在旅程开始时，我并不清楚如何做到这一点，但在阅读了这本书并有了实际结果后，我看到了全貌。

*   当他们不给你资源/时间/预算时，让你的企业接受风险。作为工程师，我们可以建议一些维护任务和预防措施的优先级。很好，如果我们能提供关于可能的客户影响的见解。T2 总是没有足够的时间来修复所有的错误和构建所有的功能。你的企业主应该理解权衡并决定你的优先事项。

*   **冻结低优先级工作**。最好让你的团队处理几个正在进行的项目，并及时完成它们，而不是让正在进行的工作消耗资源，还没有为业务、客户或你的团队提供价值。

*   考虑使用云提供商。它们提供了减少思考和出借资源的机会，而不是购买或掌握更复杂/高效的算法。如果使用您当前的代码库/基础设施处理一些后台作业花费了大量不可接受的时间——考虑并行化，只在作业时间启用额外的计算资源。

*   考虑外包。您的部分业务或遗留应用程序可以交给外部供应商。它可以降低成本，释放团队中最聪明的头脑。但是要确保合同不仅包括系统的维护，还包括支持您可能的业务计划所需的变更的实现。此外，确保外包团队能够及时完成所需的变更。

## [](#other-engineering-tips)其他工程提示

*   **自动安装/配置开发、质量保证、试运行和生产所需的环境。**让环境尽可能地相互靠近——相同版本的操作系统、数据库和库。您应该能够快速访问它们:保持供应并为此付费，或者快速供应。手动指令应该失效，手动更改永远不应该应用。

将人员从部署过程中移除。最大程度的参与应该是点击*立即部署*按钮。为新队友建立开发环境应该在一天左右的时间内完成。

*   **设置输送管道，测量吞吐量**。传统上，它包括编写代码和将代码部署到产品中，以向最终客户交付价值。在我看来，它还包括识别需求(业务或工程需求)并对需求/工作安排进行优先级排序。

*   文档(更好的是，自动化！)所有事件的“神奇修复”。如果问题再次出现，您需要能够复制它们。将它们保存在项目的知识库中。你不能指望上一次解决问题的工程师会一直提供帮助。就是这样，你所拥有的系统中的变化应该是透明的和可重复的。

*   **主动找到你的软件的所有脆弱部分**。如果你在加入团队之前开发的系统上工作——做好惊喜的准备。事情会在你意想不到的地方发生。除了代码库和项目文档(如果你的团队有足够好的文档)之外，你要学习的资源还可以是:负载测试的结果、度量和来自生产的日志、已关闭 bug 的注册表、客户支持标签。

*   稳定基础设施，专注于发展，而不是救火。当你不仅需要开发新功能，还需要保持现有软件正常运行时，很难做出合理的估计，也很难不加班。我会就这个话题发表一篇独立的博客。敬请关注。

*   **将空闲时间纳入你的业务承诺**。如果您团队中的工程师按照您的计划 100%地工作，这意味着任何计划外的工作都应该排队等待(这是不好的)，否则承诺将无法实现。给你的工程师一些空闲时间是好的，因为你不能预测完成工作的实际时间以及需求的变化。

*   避免工程师和跨团队之间的任务移交。语境切换扼杀生产力。不止一个负责人会加强公司的乒乓效应，让事情更难完成。

*   **衡量你的代码多久可以被部署到产品中**。您知道您的企业每天需要多少次部署吗？其中有几个你能做到不影响作息？您希望至少在发生事故时知道答案，并且您需要推出修补程序以防止公司声誉受损。

*   **对所有代码变更进行问责和授权**。除了基础设施变更之外，它们还应该经过版本控制系统、对等审查流程，以及*有时*由业务/预算负责人或外部团队批准。

*   **尽快将您的工作代码投入生产**。在代码投入生产并为客户启用之前——做产品研究、创建吉拉入场券、设计会议、编写代码和审查拉式请求不会产生任何价值。

*   **发布更快的版本**并且小批量发布。对我来说，我们给客户一个新版本的后台软件，让*每月在我们的基础设施*上运行一次的时代已经过去了。每个合并的拉请求都应该单独部署(并回滚)。在这种情况下，您可以观察这种变化如何影响您的系统，并快速找到故障。

*   为所有大型/高风险变更的部署准备回滚策略。示例:更改包含大量记录的数据库表、极端重构、数据迁移、更换供应商。如果你认为你的测试是不够的(或者覆盖所有需要的案例是昂贵的)——我会在这方面投资。

*   在你的客户或企业发现之前了解你的事件。首先，它让您有更多的时间来调查和解决问题。第二——及时更新状态页面是你团队的脸面。只是关心你顾客的感受。

*   **建立一个充满激情的团队，在真正需要的时候，他们可以加班加点拯救企业**。它最终应该得到补偿，包括额外的休息时间来恢复和陪伴家人或朋友。您还可以设置随叫随到的轮换，让有人 24/7 值班，随时准备解决任何问题。

我很高兴凤凰计划的书被 DevOps 书友会选中。阅读这本书，与其他工程师讨论和回顾帮助我定义下一步来改进我们后端工程团队的开发过程。
