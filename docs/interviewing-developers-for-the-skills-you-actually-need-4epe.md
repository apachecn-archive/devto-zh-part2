# 面试开发人员以获得你真正需要的技能

> 原文：<https://dev.to/pbeekums/interviewing-developers-for-the-skills-you-actually-need-4epe>

雇佣软件开发人员是一项艰难的工作。优秀的开发人员很难找到，也很难面试。我对前者没有任何建议，但对后者有丰富的经验。

当我第一次开始面试开发人员时，我不知道从哪里开始。我得到的唯一建议是“试着确定你是否愿意和这个人一起工作。”我如何在一两个小时后发现我是否愿意和某人一起工作？最简单的方法就是复制我面试时的做法:让应聘者在白板上实现一些复杂的算法。

我这样做了大约一两年，最终面试了一位候选人，他创造了一个比团队中任何人，包括我自己，都想出的更好的解决方案。很明显是个好工作，对吧？如果他能做出比我们更好的解决方案，那么他就能教我们很多东西，帮助我们成为更好的开发人员。

事情并不完全是那样。这位开发人员无疑非常聪明，但他的技能不适合我们的工作。他从事计算机科学的能力远远超过我们，但我们的大部分工作都不是计算机科学。我们的大部分工作是数据库设计、软件架构、管理分布式系统、确保我们有足够的日志来调试客户问题，等等。这家伙擅长复杂的算法，但在我们真正需要他做的事情上却不怎么擅长。

显然，最重要的技能是你实际使用的技能。去想想。

我发现，测试工作所需的实际技能的最佳方式是采用产品中已经存在的东西，并要求候选人实现它。这很方便，因为你不需要为问题创建内容，它已经存在了。你不需要为了在面试中提问而学习一个新问题的解决方案，你已经是这个问题的专家了。最重要的是，如果一个人能够充分地建造一些他们在过去从事这项工作时会被要求建造的东西，那么他们很可能在未来从事这项工作。

难的是缩小问题的范围。如果你花了两个星期，甚至两天的时间来构建一个东西，你不能合理地期望有人在两个小时的面试中就能完成它。

虽然两个小时对于一个候选人来说还不足以构建一个复杂的系统，但是对于和面试官一起创建一个系统的规范来说，时间是绰绰有余的。系统需要做的主要事情是什么？主要成分是什么？他们是如何互动的？我们如何防止失败？我们如何从失败中恢复？

这些都是开发团队在开始编写代码之前通常会讨论的事情。与候选人进行同样的讨论是在这种环境下与他们一起工作的真实写照。我问自己“如果在我们建造项目的时候有这个人和我们在一起，项目会不会更容易一些？”如果答案是肯定的，那么“我们应该雇佣这个人吗？”大概也是吧。

在这次面试中，我总是抛出一些编码问题，但那只是因为这似乎是应该做的事情。谁会雇佣一个不要求他们编码的开发人员呢？

我不再相信这是必要的。让候选人设计一个我已经建立的系统的好处是，我知道它的所有细节。如果我觉得候选人的回答含糊不清，我可以提问，让他们澄清这些细节。

我还没有遇到过没有构建类似系统经验的人能够很好地回答复杂的技术问题。如果他们能很好地回答这些问题，那么他们很可能已经建立了一个复杂的系统。如果他们也有预防和处理失败的好方法，那么他们可能已经很好地构建了这个系统(或者知道如何在另一个机会下很好地构建它)。这可能是一个更好的指标，表明他们可以在真实的情况下编写代码，而不是要求他们当场在白板上写出一些算法。
