# 现实世界中的算法

需要以写代码为生的计算机科学家，不断地在美丽整洁的理论小世界和有点混乱的实用软件工程的“macgyverish”世界之间游走。

在理论世界里，可爱的令人想拥抱的物体在彼此之间传递和平与爱的信息。他们生活在一个美丽的星球上，那里有丰富的自然资源。

在现实世界中，物体有时会试图让对方着火。他们酒后驾车，他们和他们妻子母亲住在一个 100 平方英尺的公寓里，他们的母亲是在他们和另一个项目的火辣方法结婚后搬进来的。

但是尽管如此，计算机科学家总是试图从理论星球带一些可爱的东西回来。

有一次，连我都很幸运。我工作的一家大公司需要一个目录系统来帮助客户浏览他们的城市街区大小的建筑。你知道商场里的“你在这里”地图，上面有一个红色的大点，告诉你在哪里。就像这样，但是在大屏幕上。

总之，长话短说，项目经理预测这个项目将需要几个星期才能完成，因为从大楼的任何一个点 A 到任何一个点 b 的客户路由非常复杂。

事实上我自己也这么认为。但是我记得不久前我们的网络人员说过一些关于“最短路径算法”的事情。当时，他试图向我解释，在给定几种可能的路径的情况下，网络数据包如何以高效的方式传输。

一个简单的谷歌搜索带回了一个叫做 Dijkstra 算法的算法的完整实现。整个事情，从上到下肯定有 100 行 Java 代码。非常简洁。

因为我不知道它是如何工作的，也不知道它试图解决什么问题，所以我找到了维基百科的文章:

> Dijkstra 算法是一种寻找图中节点间最短路径的算法。
> 
> 例如，如果图中的节点表示城市，边路径成本表示通过直接道路连接的成对城市之间的行驶距离，则 Dijkstra 算法可用于查找一个城市与所有其他城市之间的最短路径。

哦，好吧。因此，我所要做的就是将建筑中的所有点建模为“节点”，使用“边”将它们链接起来，并为每条边分配一个距离。然后，算法可以使用这些信息来计算出从任意点 A 到任意点 b 的最短路径。听起来很容易。

我将代码复制粘贴到我的项目中，输入一些样本点，并在一对端点上进行测试。成功了。

我在另外几个端点上进行了尝试，添加了可供算法选择的替代路径，每次都运行得完美无缺。

我将代码部署到我们的开发环境中，并展示给项目经理。他简直不敢相信。在接到任务的几天内，我向他展示了该产品的工作演示。

然后他开始自己测试。当然，他可以找到破坏算法的边缘情况。他甚至几次骄傲地来找我，宣布他找到了一个。但在每一个案例中，我们都发现是他的输入出了问题，而不是算法。

那么这个故事的寓意是什么呢？嗯，至少对我来说，这是意识到计算机科学是一门有用的学科，它是一种与软件构建截然不同的活动。此外，它教会了我编程有强大的基本原则。非常好地学习这些基础知识是绝对值得的，因为花在学习这些知识上的时间将会得到数倍的回报。
