# 处理软件开发中的未知问题

> 原文：<https://dev.to/pbeekums/dealing-with-unknowns-in-software-development-3l5o>

众所周知，软件开发很难评估。由于这个原因，我知道很少有人认真对待瀑布过程。有许多因素造成了一定程度的不可预测性。一个是通常有一些开发人员以前从未构建过的部分需要构建。领域经验的缺乏会导致某些东西被遗漏，并且世界上所有的计划都不会抓住所有的东西。限制我们自己只建造我们以前建造过的东西值得吗？大概不会。

另一个因素是，随着越来越多的关于用户如何与产品交互的信息被发现，软件产品的各个方面将会发生变化。这些变化通常也是不可预测的。你会喜欢和那些拒绝让你改变需求的开发人员一起工作吗？因为这会打乱他们的评估。大概不会。

软件的最终目标是让人们发现有用和有价值的东西。发现这是什么需要时间，并带领我们走上不熟悉的道路。这使得在构建软件时处理未知成为生活中的一个事实。

这并不意味着我们不能有效地管理我们的时间。

在过去，我使用了许多策略来做到这一点。最有效的是时间拳。这个想法是选择一个固定的时间量，例如 2 小时，1 天，或任何感觉合理的时间，来做一些发现。在时间盒结束时，您与您的团队同步并讨论结果。这个发现可以是纯粹的研究，但通常需要一些原型制作。除非花时间实际构建，否则很难知道构建一个软件的真正内容。

显然，一个未知的任务可能会花费不可预测的时间，那么创造一个固定的时间来做发现有什么帮助呢？为什么不让别人想完成就完成呢？

问题在于，发现新事物就像试图杀死一只九头蛇。学习一件事，你会发现你还需要学习 3 件事。很容易迷失在潜在知识的海洋中。一个团队可以像一个指南针，帮助提供一个方向的焦点。其他一切都只是噪音。

相反的情况也可能发生。一个人很容易陷入试图理解新事物或让原型工作的困境。有一个固定的时间回到团队并讨论问题可以帮助那个人找到前进的方法。

并且没有理由为什么你不能为一个单一的特性设置多个时间盒。时间拳击不是强迫事情在一定的时间内完成。它们是关于提供一个检查的结构，以确保正确的进展。

更重要的是，时间盒提供了一个重新评估一项任务是否值得的机会。也许有一个特性起初看起来很重要。在经历了几个没有取得什么进展的时间之后，这个特性可能看起来不那么重要了。在剩余工作量未知的情况下开发一个功能的机会成本比开发其他功能的机会成本要高。时间盒提供了一个重新安排事情优先级的机会，因为它开始揭示开发一个东西实际上可能有多昂贵。

还有其他方法来管理软件中的未知。我只是碰巧发现时间拳击对我从事的项目来说是最有效的。需要注意的重要一点是，好的项目管理不是假装软件开发中不存在未知。好的项目管理就是对他们负责。
