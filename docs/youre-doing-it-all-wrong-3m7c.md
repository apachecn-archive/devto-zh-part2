# 你完全做错了

> 原文：<https://dev.to/codemouse92/youre-doing-it-all-wrong-3m7c>

在我写这篇文章的时候，我正在对一个 septenduple-boot 系统进行最后的润色。是的，**septencecuple**-一个盒子上的 17 个 Linux 发行版。我发誓我有合乎逻辑的动机，我一会儿就会谈到，但事实是，这个星期已经有半打的人以这样或那样的形式问我同样的问题。

> 为什么不直接用虚拟机呢？

现在，我对虚拟机并不陌生——我已经广泛使用了近十年。当我开始这个小项目时，我考虑并拒绝了这个想法。我知道我会经常听到这句话，我试着为此做好准备。

那项准备工作确实没有抑制住疼痛。

# 不请自来的建议的“乐趣”

我绝对不会拒绝别人的建议、新信息或技术建议。我通过这种方式学到了很多很酷的东西。然而，更多的时候，我发现自己被同样的愚蠢的建议所困扰，一遍又一遍。例如...

*   “你为什么用 C++？太过时了！语言 X 是未来之路！”
*   “为什么要编写自己的数据结构呢？用`std::`或者`boost`就行了！”
*   “为什么要用纯 HTML5 和 CSS 来搭建一个网站？Javascript 框架是您所有问题的答案！”
*   “OMG，你用 Vim？你为什么不用 Emacs？太优越了！”
*   “Apache2？说真的，Nginx 才是正道！”
*   “谁还会使用 Python 进行应用开发呢？C#好太多了！”

...当然，还有前面提到的关于虚拟机的建议。继续，继续，继续，继续，继续....嗯，你明白了。

原谅我最初的尖叫。

yaaaaaaaaaaaaaaaaaaaaaaaaaaaarrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrrggggggghhh！

*咳咳*

对于那些觉得有必要对我说以上任何一句话的人来说，这是一条新闻:在我从事编码工作的 8 年多时间里，我并不只是坐在那里用 BASIC 写“Hello，world”。我是一个聪明、能干、通情达理的人。我基于数百个因素做出谨慎的决定，尤其是潜在解决方案相对于我的目标的适当性。

我并不孤单。

# 什么会成为决策

任何项目的第一步都是决定使用什么工具和方法。任何有理智的人都会去评价**三个 f**。

## 特性

相对于开箱即用，该解决方案能满足我的项目的所有需求吗？这还包括性能、兼容性、文档、许可证和社区因素。

记住**只有实际的项目成员知道他们需要的所有特性**是非常重要的。我经常看到善意的(或相反的)人认为他们比项目组织者更了解项目。在我的*普通书呆子实地指南*中，我把经常做出这种假设的人称为“程序员 Takeitovericus”。

## 熟悉度

与节省的时间或增加的质量相比，我需要花多少时间来学习新的东西？

例如，我经常使用 Python 而不是 C#，因为我*知道*前者。这并不意味着我永远不会学习 C#，但这确实意味着如果我想相当快地开发出质量不错的东西，我不想被一门我不懂的语言所困扰。

现在，我们总是需要学习新的东西，但是请记住，我们的大脑带宽有限。当某个项目的特点(可能还有有趣的因素，见下文)符合我的时间和学习愿望时，我可能会选择一些不熟悉的东西。

目前，我正在使用 AppImage 打包一个游戏，这意味着我需要在多个操作系统上测试它。这是促使我在一台笔记本电脑上安装 17 个操作系统的第一个原因，但是我进一步被玩和学习一堆不熟悉的 Linux 发行版的可能性所激励！除此之外，在一个硬盘上真正安装 17 个操作系统的挑战有点令人兴奋。我还需要再做一次吗？大概不会。我喜欢吗？**那还用说！**

同时，我选择了**而不是**将 Arch Linux 或 Gentoo 包含在我要安装的发行版列表中。我现在也没有足够的精神带宽来安装或学习。(我使用 Antergos 来测试基于 Arch 的系统。)

## 好玩

我会喜欢使用这项技术吗？不，好玩不是第一位的，但却是重要因素！如果你喜欢这些工具，它们会帮助你度过难关。相反，当你*鄙视*你正在使用的工具时，你就没有动力继续下去。

这是我使用 Vim 而不是 Emacs 的一个原因。我确实知道 Emacs 是一个很棒的编辑器，但我就是不喜欢它。我喜欢 Vim 的特性、按键绑定和界面。如果我强迫自己使用 Emacs，增加的烦恼实际上会*阻止*我想要在那个项目上工作。

现在，是的，有时我们不得不使用我们讨厌的东西，但这是保持乐趣的更多理由！如果我们可以通过使用我们最喜欢的操作系统、编辑器、语言、工具包或其他东西来缓冲一个不愉快项目的打击，或者，如果我们可以保持我们的副业项目的乐趣，它可以防止倦怠。

最后，乐趣是一个因素，因为我们可能有意选择艰难的道路来迎接挑战、刺激或享受。我们已经知道有更简单的方法！在这种情况下，不请自来的建议感觉就像嘲笑一个马拉松运动员没有用摩托车更快地到达终点。

# 检查你的动机

所以，如果你看到或听到有人用你认为不理想的方式解决问题，你会怎么处理？即使你认为你可以为他们节省很多时间，你还是闭上你的嘴吗？

不一定，但首先，你应该先问问自己**“为什么我觉得必须建议 X？”**尽管我们愿意相信，但我们的动机并不总是单纯或无私的。以下是这些年来我自己在*抓住的一些动机:*

我发现学究式的论点通常属于这一类。有一次，我发现自己陷入了一场毫无意义的讨论中，对方小心翼翼地错误定义了某些词，其明确目的是贬低我使用一种语言的理由，并贬低我作为一个人的智力。在观察编程社区的八年中，我很少观察到严肃的学究式的争论是公开炫耀的手段。(与刻意幽默的学究式辩论形成对比。)

这并不意味着你不分享你的知识，但是如果你的一个主要目标是表现出色，如果你把这些知识藏在心里，通常对你、其他人和所有观察者都是最好的。

**你只是想要把另一个用户“转化”到你喜欢的技术上的满足感吗？**有时，我们对自己使用的技术充满热情，希望其他人也使用它们，我们发现自己向遇到的每个人都过度推销了这些技术。

这通常有助于记住——实际上每种技术都有它的位置！即使是那些我们本能排斥的技术，不管是 C、FORTAN、COBOL、Visual BASIC 还是 PHP，都有重要且有效的用例。在某个地方，有人成功地用那种语言创建了一个可行的项目。

不要轻视你不喜欢的技术的用例也很重要！我已经厌倦了听到同样愚蠢的台词...

> C(++)是一种痛苦的语言，只对系统编程有用，说真的，谁还会这么做呢？

我可以给你至少十几个从事系统编程的人的电子邮件地址，以及至少二十几个在应用程序开发中愉快而成功地使用 C++的人的电子邮件地址，他们用 c++创建了可靠、稳定和可维护的项目！(出于匿名、隐私等原因，我*不会*给你那些电子邮件地址，但你会明白的。)

C++可能不是你喜欢或需要使用的语言，但这并不贬低或否定使用它的各种新旧项目，也不贬低像我这样的程序员从使用它中获得的纯粹乐趣。

现在，我确实对上面列出的大部分事情有具体、客观的技术关注。如果这些问题看起来相关，我可能会时不时地提出来，但我不会让我个人对某项技术的厌恶蒙蔽了我对其潜在用途的认识。

这里有一个重要的平衡点:在极少数情况下，某个特定的库、程序或框架非常糟糕，任何人能做的最好的事情就是警告其他人远离它。也许它包含恶意软件或跟踪代码，在性能关键的代码中有严重的内存泄漏，或者在它的架构中是如此错综复杂，以至于人们永远不能希望完成任何像样的事情。

例如，我在一次工作中使用了一个 ERP 系统，我发现它的继承结构是如此的糟糕，以每一个公认的计算机编程标准来看都是不可接受的。最后，我写了一篇对该系统的严厉的技术评论，这篇评论成功地警告了至少一家公司不要走上无止境的道路。

然而，要小心这个警告！最终，你所能做的和应该做的就是用客观的、可证实的技术信息发出警告。尊重并谦逊地将最终决定权留给对方，相信她或他会评估他们的选择，无需进一步提示。还是那句话，对他们来说，从自己挖的坑里把自己挖出来要比你*挖的坑好得多！*

# 何时分享

一旦你排除了任何腐败的动机，你需要评估你的建议是否真的有用！

最简单的情况是，如果这个人问了关于特定事情的问题:他们明确邀请你分享你的建议和知识。

然而，并不是每个问题都是自动邀请建议！我经常在 StackOverflow 上看到这种情况——有人问如何在 Y 中完成 X，六七个人突然冒出来痛击 OP，告诫不要使用 Y，而是使用{在此插入技术}！

更糟糕的是，如果你正在写一篇关于解决问题的一种方法的教程、StackOverflow 答案、博客帖子或 dev.to 文章，不请自来的批评有时会像是一记重拳。

最糟糕的是，如果有人在描述他们如何完成了他们引以为豪的事情，反驳“你都做错了”是非常不礼貌的！

同样，这并不意味着我们不应该谈论替代的，甚至看起来更好的做事方法。但我们需要注意，如果我们只是开始说话，别人会如何看待我们。

# 如何分享

如果你确实觉得你有一些值得分享的信息，有四个非常简单的规则你可以遵循，以防止你的建议在野餐时变成雨。

## 1:获得权限

当有疑问时，请求允许。不要假设自己知道对方不知道的事情。如果你觉得你有什么要补充的，问问你是否能提出一个替代方案。例如...

> 我可以推荐一个 GUI 工具包来简化你的项目吗？

注意这句话是怎么说的:假设对方很聪明，而你的建议不一定是唯一的正确答案。

然而，有时提问对于媒体或对话流来说是不切实际的，在这种情况下，只需双倍下注于#2...

## 2:提问

**提问。我确实有一个朋友以一点也不伤人的方式问我关于我的九月计划。他只是说...**

> 出于好奇心...你为什么要对一个系统进行虚拟引导？

注意他的提问方式。他*不是*认为我不了解虚拟机，他也不是刚刚开始有了自己的想法。**他在询问我现在的方法**背后的逻辑，他在为*他*留有学习的空间。

在另一个例子中，我在寻求帮助使算法更有效。帮助我的一个人从获取更多信息开始。他的问题中有一个是我个人最喜欢的...

> 你想完成什么？

我经常问这个问题，甚至在我冒险分享建议或意见之前。我意识到我并不了解所有的细节，我需要知道更多才能有所帮助。

## 3:保持客观

如果你最终分享了你的建议，一定要把你对这个人、他们的项目以及有问题的技术的感觉排除在外。坚持事实。

追求这个，小心黄鼠狼的话。像“脚本语言”、“过时”、“最佳实践”、“行业标准”、“难以维护”和许多类似的短语是很好的迹象，表明您可能诉诸于站不住脚的、空洞的论点。

例如，我正在咨询一个项目，其中 Java 是被考虑的语言之一。我建议不要这样做，主要是因为性能和许可问题，因为它们适用于这个特定的项目。我对 Java 的看法无关紧要。最终，项目领导选择了一种不同的语言。

同时，在同一个项目中，我支持选择使用制表符而不是空格作为标准，*，尽管我总是更喜欢空格。我认识到我的观点是基于偏好，而不是客观事实；由于绝大多数核心开发人员更喜欢选项卡，我建议他们将他们喜欢的东西标准化，尽管事实上我也必须使用它！*

## 4:尊重界限

简单地给出你的建议，然后等待对方继续提问。不要逼迫、唠叨、纠缠他们听你和你的“智慧”。准备好回答问题，但是让其他人决定他们是否想采用你的方法。

最终，决定权仍在他们手中。如果他们决定不接受你的建议，不管你认为自己有多正确，都让它过去吧。

# 结论

在一个拥有数十种操作系统、数百种编程语言、数千个库和框架的世界里，选择合适的工具可能会令人望而生畏。许多“圣战”都是围绕编辑器、语言和标准展开的。Tabs v. Space，Vim v. Emacs，Windows v. Linux...这些战斗在继续，有什么好处呢？我们不能强迫别人采用我们的思维和工作方式，也不应该自动屈从于别人的建议。

最终，每个人只对自己的工作流程和项目负责，而不是其他人。我们可以从彼此身上学到很多东西，但我们必须明白，我们的想法并不是全部。**事实上，无论我们喜欢与否，每项技术都有其存在的价值。**

是的，这可能意味着十年后，仍然会有人为 Windows 服务器启动一个充满活力的 PHP 新项目...而那个事实只是让几个读者打了个寒噤！我们天生就不喜欢这样的想法，即人们正在用我们讨厌的技术*快乐地*建设*健康而充满活力的*项目。这一事实一直困扰着我们。

但是为什么要这样呢？也许我们只是对自己可能错了的想法感到不舒服。如果我们非常诚实的话，我们只是渴望得到一些幸灾乐祸的东西，而牺牲那些被我们抛弃的劣质技术的不幸用户的利益。

但是说真的，如果下一个十年带来了 Visual Basic 的复兴，这对我又有什么坏处呢？网？让他们拥有自己热爱的语言。

我仍然会愉快地使用 Linux 和 C++。
