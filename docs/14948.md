# 开发商和雇主如何维持和平

> 原文:[https://dev . to/spirodonfl/how-developers-and-employees-can-maintain-peace-30cc](https://dev.to/spirodonfl/how-developers-and-employers-can-maintain-peace--30cc)

在我的前两篇文章[这里](https://dev.to/spirodonfl/how-to-wage-cold-your-on-your-employers--e9d)和[这里](https://dev.to/spirodonfl/how-to-wage-cold-war-on-your-developers--17n0)中，我提供了我过去的轶事经验，强调了作为雇主或开发者你不应该做的事情。我认为这些方面是互相“冷战”的一种形式。

在这篇文章中，我想强调上两篇文章的积极成果以及我从中学到的东西。同样，我希望社会各界人士都能学到一些东西。

# 尊重对方的时间

我觉得这是我学到的最有价值的推荐。尊重时间是双向的，它最终会变得无比珍贵。

对于雇主来说，这意味着尽最大努力避免范围蔓延，并保持需求理性。这可能意味着尽可能不设定不切实际的最后期限。这意味着不要对你的开发人员在任何一天、一周或一个月的时间要求太多。

对于开发者来说，这意味着不要过度工程化解决方案。这可能意味着在工作之外给自己留出适当的社交时间。这意味着对时间估计要诚实。

# 互相帮助学习

在我个人看来，这也是一个非常重要的因素，因为它又是双向的。

对于雇主来说，这意味着要非常清楚一项任务的业务需求以及对业务的影响。这可能意味着解释业务的某些方面在技术栈之外是如何工作的。它可能意味着给你的开发者一个机会去使用或执行一些与你最初计划不同的东西。

对于开发人员来说，这意味着学习如何尊重地应对项目中的业务需求。它可以意味着帮助雇主从技术角度理解一些东西。这可能意味着诚实面对时间限制或资源债务，以及为什么这些会导致问题。

# 积极地互相加强

这一条稍微有点“包罗万象”，因为你可以将这篇文章的很多内容应用到这个主题中。但是我想到了一些特定的领域。

对于雇主来说，这可能是一些简单的事情，比如对出色完成工作给予适当的表扬，或者在截止日期前完成工作，或者低于预算。老实说，这只是简单地向开发人员传达(这可以扩展到任何员工关系)他们正在按计划进行。开发人员可能会缺乏安全感，即使他们非常自负，所以当他们值得的时候说几句好话实际上会有很大帮助。你帮助开发人员感到成功，这鼓励他们继续努力。

对于开发人员来说，你可能要考虑给你的雇主什么样的反馈。你上一次感谢雇主给你钱让你做的酷工作是什么时候？我有一些工作关系，这些年来，变成了友谊，但这需要这种程度的来回沟通。如果我不说“我喜欢在这里工作”或“这是一个有趣的项目”，我就不能指望听到“工作做得好”。再给我一个！”。

# 居然在个人层面上给对方一句废话

这是上一个话题的延伸。

在我个人看来，社交技巧是一门艺术。我天生内向，我发现社交环境很难处理。我非常努力地去战胜那些感觉。

无论是雇主还是开发商，都应该对彼此感兴趣。我的新宝宝是一个话题。如果你很难记住，记下你和别人谈论的事情。

很难看出社交活动和你的编程之间的 1:1 关系，但我向你保证这是有关系的。当你感觉和你的同事有联系时，差别是巨大的。

# 尊重对方的专长

这非常简单明了。

作为一名雇主，如果你能创建你要求你的开发者创建的网站或系统，你可能会去做。你可能不会要求开发者去做。这就像让一个建筑承包商给你建一栋房子，而你是一个药剂师，你进来开始告诉承包商如何建造你的房子。你的开发者并不是完全无知。他们有你可能没有的技能和经验。尊重它，对它有点信心。

作为一名开发人员，您也必须挺身而出。你需要诚实地面对你自身经历中的局限性。不要自负。不要做一个无所不知的人。你可以说“我不知道如何解决这个问题”或者类似的话。此外，深入你正在从事的业务并理解它是非常有益的。执行你自己的代码和工程化它还有很长的路要走。

# 互相挑战

是啊！这样就可以了！讨论方法是可以的。甚至可以争论。当一个好的案例出现时，坚持自己的立场或让步是可以的。

当你们都用建设性的反馈来挑战对方时，你将*而不是*觉得你们在争吵。这可能是一个有益的过程，你们都能从中学到很多东西。

这发生在我和一个经常一起工作的朋友身上。他是产品所有者类型的人，而我是开发者。有时我们会争论用户体验特性或 SEO 项目。我心中有一种方法，他心中有另一种方法。要靠我们两个人来赢得对方的好感。好好想想这个词。

把对方争取过来。

它不是“显示我更聪明”或类似的东西。

你希望有人站在你这边，看到好处。当然，这意味着你需要对跳到另一边持开放态度！

# 沟通就像你的生命取决于它一样

这对于远程团队来说也是一个大问题。

我曾经领导过一个大约 12 人的团队。其中一个在澳大利亚。他每两三个小时交流一次。比如时钟工作。即使它只是说“自上次以来没有更新，只是抨击代码”。起初，我对我必须阅读的邮件数量感到恼火。

当我的一名团队成员一天都不和我说话，然后在一天结束时，向我展示他们所做的事情，而他们却偏离了目标时，我很快，非常非常快地改变了我的想法。我宁愿每三个小时被检查一次，看看发生了什么，然后花 8 个小时或更多时间做任何事情，最后非常担心。

对雇主来说，这也可能是另一种情况。沟通的形式可能是解释需求，或者为什么你有一个时间表，比如你做的事情或者类似的事情。当你谈论来自雇主的沟通时，这是一个有点不同的心态，但它同样重要。您需要能够帮助您的开发人员理解您的需求，以便他们能够为您提供正确的答案。
