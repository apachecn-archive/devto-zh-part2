# 任何人都可以使用的 UX 审查框架

> 原文:[https://dev . to/attacous studio OS/a-UX-review-framework-any one-can-use-bhn](https://dev.to/assaultoustudios/a-ux-review-framework-anyone-can-use-bhn)

> 随着互联产品融入我们日常生活的更多方面，UX 将负责将战略转化为直观体验。
> 
> 福布斯（美国出版及媒体集团）

随着各种组织争相创造将客户需求置于业务需求之前的产品，对 UX 的需求变得越来越明显。这些组织通常拥有庞大的 UX 和设计团队，专注于为客户提供最佳体验。

但是，如果你或你的企业负担不起聘请 UX 设计师来审查你的产品或服务呢？

我分享这个框架的目的是让任何人都可以做 UX 回顾；不管你是不是 UX 设计师。你可以用这个框架来分析产品或服务，或者为潜在客户分析一件作品。你甚至可以用它来回顾你自己的工作，因为我所使用的工具和技术没有给偏见留下多少空间。

* *虽然这个框架可以用于任何类型的产品或服务，但它将主要关注 web。*

** *这篇文章中有很多关于开发的好东西——它不仅仅是为设计者准备的。*

### [](#objectives-of-the%C2%A0review)评审的目标

通过定位你的目标来开始你的评估是很重要的。这就定下了基调，并确保你和你的利益相关者都有相同的期望——无论你是展示你的评估还是通过电子邮件发送。

一些目标或成果可以包括:

1.  发现机会
2.  证实设计变更
3.  识别技术问题
4.  讨论 UX 的改进

这些目标将主要与你最初的简报有关——为什么你的客户首先要你做一个 UX 回顾？

* * *

### [](#business-objectives)经营目标

商业目标是你的客户想要达到的目标。有时目标可以很容易地由你自己确定，但它们也可能需要额外的研究。在研究的情况下，这取决于你的客户——有时他们的目标很明显，但通常有更深层次的需求，需要通过进一步的检查才能浮出水面。

分解这些目标很重要——让我们以“增加销售额”为例。他们是想卖出更多的特定产品，还是想增加店内流量？这些细节很重要，因为它让你的成果更加有形和可衡量。将特定产品的销售增长归功于 UX 的改进要比将一千种产品的销售增长与一个没有恰当定义的目标相匹配容易得多。

这些目标可以通过与你的客户和他们的利益相关者进行面谈来展开，另外的目标也可以通过市场调查来确定。如果你能接触到利益相关者的客户，通过[调查猴子](http://surveymonkey.com/)进行的简单调查可以提供有价值的见解。

最后，在你的回顾中陈述这些目标，提醒你的客户你正在试图解决的问题。这将在整个评估过程中重申你的想法，并使你的想法可信。

### [](#personas-and-userflows)人物角色和用户流

既然您已经了解了业务需求，那么是时候看看用户需求了。分解用户需求的一种流行方式是创建人物角色或人物角色表。简而言之，人物角色是一个虚构的用户，他们的需求、抱负和欲望通常与他们生活中的目标相关联。

以桑迪为例。她是中产阶级职业女性，有两个家庭。她喜欢学习技术，看纪录片，和孩子在一起。桑迪是极客玩具店的理想顾客，她可以在那里购买她和她的孩子都喜欢的电子玩具。

创建人物角色可能令人生畏，但却成为一个无价的工具。人物角色可以通过研究以及对你或你的利益相关者的商业目标的理解来建立。 [Shane Williams](https://medium.com/@shanepwilliams) 在[中全面介绍了创建人物角色——开始创建人物角色——要问的问题](https://medium.com/@shanepwilliams/getting-started-with-creating-personas-questions-to-ask-b15689e3eb91)。

现在我们知道了谁是你的用户，是时候看看他们会采取什么步骤来实现他们的目标了。这可以通过创建用户流程图来实现。这些图表展示了基于用户目标的示例用户旅程，并且通常可以与业务目标相结合，以获得完整的客户体验视图。

###### [](#example-of-user-flow%C2%A0diagram)用户流程图示例

有很多方法可以创建这些图表，无论是使用便笺还是像 [Draw.io](https://www.draw.io/) 这样的在线工具。在设计这些图的时候没有规则，但是重要的是你的图要易于理解和阅读。

### [](#working-with%C2%A0data)处理数据

数据是一篇好的 UX 综述的基石，没有数据，你可能会发现很难支持你的发现。谷歌分析为用户交互数据提供了一个很好的起点，前提是你知道你在寻找什么。

像某一天流量下降这样简单的事情可能表明了一个问题——也许用户对一个新功能有负面反应，或者你在社交媒体上发布了一些负面新闻。

设备信息(比如移动设备和桌面设备的流量)在决定使用哪种设备进行可用性评估时也很有意义。

假设设置正确，Google Analytics 中的事件跟踪提供了另一个有用的信息来源。如果你没有在 GA 中设置事件跟踪的专业知识，CrazyEgg 是一个很好的选择，因为它为不同设备上的点击和滚动提供了热图。

行为流报告也是非常宝贵的，它以流程图的形式提供了对用户旅程的洞察。它提供了离店、会话等数据，并能够突出显示或分析特定的用户路径。

[![](../Images/48519d3b9aa4bf74230e84bf366719f6.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--5tlNd30n--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/900/1%2AvAlYLVcZDPkSnHTf9tWngw.png)

###### [](#example-of-google-analytics-behavioural-flow%C2%A0diagrams)谷歌分析的行为流程图示例

进行[系统可用性等级(SUS)](https://www.usability.gov/how-to-and-tools/methods/system-usability-scale.html) 调查也是一种通过用户反馈快速评估系统可用性的简便方法。SUS 测试可以在产品、网站、应用程序或软件上进行，并提供一个缩放的结果，而不管反馈量如何。SUS 调查的数据也有助于可用性评估。

最后，您在上一节中设置的角色表也是一个很好的数据源。在检查这些数据源时，请牢记您的用户目标，您可能会发现一些行为模式，表明用户的旅程中存在差距。

您或您的客户可能并不总是设置数据工具。在这种情况下，我的建议是推迟审查，直到你有有意义的分析工作。安装 GA 既快速又简单，仅用一个月就能让您获得相关的见解。

### [](#usability-review)可用性评审

> 可用性统治着网络。简单来说，如果客户找不到产品，那么他或她不会购买。
> 
> 雅各尼尔森

可用性是评论的核心——产品可用吗？

是时候根据业务和用户需求对项目的可用性做出一些假设了。使用你的角色表、用户流和分析数据，站在用户的立场上。开始在不同的设备、浏览器或操作系统上使用应用程序或服务，同时跟随用户流。您将很快发现可能令人沮丧的旅程，或者无法帮助您实现作为用户的目标的功能。

您还可能遇到设计缺陷、损坏或混乱的组件或响应问题。截屏时记下所有这些潜在的问题。一旦你以用户的身份完成了项目，就要站在产品负责人的立场上，经历同样的过程。

您的下一个挑战是在整合时使反馈易于理解。请记住，你的客户可能对 UX 或设计没有任何了解，作为一名 UX 评论家，你的工作是使反馈简洁。如有必要，口头上详细解释你的想法，但书面反馈要简明扼要。确保反馈不被视为一系列问题，而是改进的机会，这一点也非常重要。

最后，不要进入解决方案模式。在这一点上，你只能识别可用性问题。我喜欢将我的反馈整合到几个桶中:

#### [](#1-user%C2%A0journeys)1。用户旅程

这应该涵盖与用户旅程相关的任何问题。如果你很难说服你的利益相关者这些问题，试着把他们放在用户的立场上。用户旅程缺陷的一些示例可能包括:

*   用户需要的信息在页面中的位置太低
*   重要页面在信息架构中隐藏得太深了
*   信息是不必要的，不能为用户提供价值
*   购买需要太多的点击
*   不一致的用户旅程
*   重要信息位于折叠下方

#### [](#2-general)2。一般

产品或服务中有哪些一致的问题？列出任何不特定于旅程或设备的东西。

#### [](#3-design)3。设计

拥有设计背景在评审的这一部分非常有用，但不一定是必需的。设计的某些方面通常是显而易见的:

*   设计不一致(即不同页面上的按钮大小不同)
*   对齐问题
*   糟糕的页面层次结构

#### [](#4-mobile)4。移动的

您的移动评论的准确性在很大程度上取决于您用于测试的不同类型的设备(平板电脑和智能手机)。这些问题可能包括:

*   响应性问题(即不支持移动)
*   字体太小等比例问题
*   某些页面需要缩放

#### [](#5-desktop)5。桌面

在大多数情况下，您已经在其他章节中介绍了桌面上的任何问题，但有时可能会出现特定于桌面的问题。还要记住你的分析——如果你的流量主要是移动的，你可能想跳过这一节。

### [](#accessibility-review)无障碍审查

易访问性经常被开发人员和设计人员所忽视——我自己对此也感到内疚。然而，随着我们越来越被数字所驱动，考虑残疾用户变得越来越重要。像红色按钮上的绿色文字这样简单的东西可能会完全破坏色盲用户的体验，并导致销售或销售线索的丢失。

[![](../Images/48eed68a06eac60341faa9beaa8a6c0f.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--I6jK4cOc--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/900/1%2Alk_TrO9KpgKOKRjcniClpQ.png)

可访问性涵盖几个方面，包括颜色、字体大小、字体类型、描述性文本或 alt 标签。考虑所有这些设计和技术方面很重要，因为它们可能会影响用户的体验。

我经常使用的一个测试网络可访问性的工具是[谷歌灯塔](https://developers.google.com/web/tools/lighthouse/)。Lighthouse 可通过 Chrome 开发者工具获得，并提供关于速度、渐进式网络应用功能、可访问性、最佳实践和网站 SEO 的详细反馈。

[![](../Images/cc5fa92f19052a88af643878a4b1f9bf.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--wNHSLFt4--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/900/1%2AmOcGhbsXSw4DqSLR0jo9Sg.png)

###### [](#google-lighthouse-results)谷歌灯塔结果

它不仅提供了解决方案，而且还证实了建议的变化，使其成为 UX 评论者武库中的一个强有力的工具。

> 当一个按钮没有可访问的名称时，屏幕阅读器会将它声明为“button ”,使依赖屏幕阅读器的用户无法使用它。
>

### [](#technical-review)技术审查

您是否在演示文稿中包含技术评估取决于您的定位方式以及您的观众的技术水平。

使技术审查有帮助的是，通常小的代码更改可以在转换率和整体用户体验方面产生很大的差异。

#### [](#1-performance-%C2%A0speed)1。性能/速度

[![](../Images/fd52de61abf4c51dfd50950c27fc6392.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--U5mZwPCk--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/900/1%2A9jCDQgaYN_xSxEAk6mdURg.png)

来自谷歌页面速度洞察的分数

分析网站速度的一个很好的工具是谷歌更有名的[页面速度洞察](https://developers.google.com/speed/pagespeed/insights/)。PSI 提供了一些提高页面速度的快速技巧，包括桌面和移动设备的资产缓存、图像优化等。这些建议有时会变得有点技术性，但谷歌提供了有用文章的链接，详细解释了每个建议。

还可以使用 Google Lighthouse 分析性能(前面提到过)。Lighthouse 提供了比 PSI 更多的细节，并基于更新的技术(如 [HTTP/2 标准](https://developers.google.com/web/fundamentals/performance/http2/))提出了建议。

#### [](#2-best-practices)2。最佳实践

最佳实践处理项目是否遵循给定设备或介质的标准。一个很好的例子就是当打印标准渗透到 web 中时，比如在标题中强制换行，这对于响应式 web 项目来说是一个糟糕的做法。

Lighthouse 也是一个提供最佳实践建议的好工具；但是，了解什么是最佳实践通常取决于您对所审查的渠道或设备的体验。

#### [](#3-seo)3。搜索引擎优化

当考虑一个网络产品的用户体验时，SEO 并不经常被考虑——但是它应该被考虑。SEO 不仅处理你的排名，还处理你的搜索外观。如果你的页面没有一个吸引人的标题和描述，你就失去了将用户带入你的网站的所有机会。通常你解决的一些关于可访问性的问题也会影响你的 SEO(比如让 alt 标签更具描述性)。

Lighthouse 提供了一些关于 SEO 的高级反馈，但要获得更详细的信息，我推荐 Moz。

### [](#opportunities)机遇

下一步是看看您或您的客户如何利用从审查中收集的大量数据来弥补他们的用户体验中的一些差距。这些建议可能基于您的评估反馈或您在使用产品时发现的机会，例如:

*   将相关文章添加到现有页面将允许用户在阅读完一篇文章后继续浏览网站
*   缩短表单的长度可以提高其可用性，这可能会带来更多的线索

小心做出类似*“增加一个横幅区* *会增加流量* *”的忽悠性陈述。*机会应该意味着改进的可能性，而不是改进的承诺。

### [](#metrics)度量标准

对一个确定的 UX 问题实施一个解决方案可能是不够的——您还需要能够衡量它的成功。这可以通过为特定的行动点设置指标来实现。

一个好的度量标准需要是有形的和可实现的。例如，“增加销售额”不是一个有形的指标。你如何准确地测量它，更重要的是，如何将它与 UX 变化联系起来？

分解你的指标，给他们一个期限，让他们回到你的目标和确定的机会:

*   通过添加相关文章将 X 页面的跳出率降低 5%
**   通过使表格更易于使用，每天在 X 页上多捕获 5 条线索*

 **您的客户可能也想设置他们自己的指标。在这种情况下，确保他们的期望是合理的和可实现的。

谁知道呢，你甚至可能成为他们团队中的实施伙伴或 UX 设计师。
