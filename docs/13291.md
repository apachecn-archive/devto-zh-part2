# 给有抱负的技术团队经理的五个建议

> 原文:[https://dev . to/robd waller/five-tips-for-aspirated-tech-team-managers-3n G5](https://dev.to/robdwaller/five-tips-for-aspiring-tech-team-managers-3ng5)

成为一名技术经理并学会如何擅长它是很困难的。没有大量可用的“文档”,开发人员也不经常讨论这个话题。随便看看你的平均[代码大会](https://www.phpconference.co.uk/schedule/)，很少有关于管理的会谈。可悲的是，这意味着大多数新的技术经理和首席开发人员在工作中学习，这并不总是理想的。

我已经做了两年的技术经理，这是一个真正的学习曲线。我可以分享很多想法和见解，但我认为其中有五点很重要。它们对任何有抱负的经理或首席开发人员都很有用，甚至对那些还没有考虑进入管理层的人也很有用。

### [](#great-developers-make-great-managers)伟大的开发者造就伟大的管理者？

人们通常认为最有技能的开发人员会成为最好的经理。除非他们是团队中最好的开发人员，否则开发人员不应该去担任管理角色。这是一个谬论和谎言，你不一定要成为一个伟大的开发者才能成为一个伟大的管理者。

重要的是你有经验，对代码和技术问题有很好的理解，但你不必成为团队中最好的。管理不仅仅是你的专业领域，所以不要让你的代码能力影响了你的管理角色，你可能会做得很好。

对于这一点，一个很好的类比是足球或“英式足球”，它像发展一样是一个高技能的职业。先说一个问题，以下这些足球经理都有什么共同点？

*   亚历克斯·弗格森
*   阿瑟·温格
*   佩普·瓜迪奥拉
*   Jurgen 在敲门
*   何塞·穆里尼奥

答:他们都不是特别擅长足球，至少不是特别出色。他们是比赛中最受尊敬的经理之一。

该理论认为，一般的足球运动员会成为更好的经理，因为他们有更高的同理心。他们从能力较弱的球员身上获得更多，因为他们更好地了解自己的局限性，以及如何最好地使用和部署他们。相比之下，伟大的球员经常不能理解为什么普通球员不能做他们能做的，导致沮丧和争论。

普通玩家也可能专注于发展其他技能，如对经理有用的沟通和策略。英国《金融时报》上有一篇关于足球经理背后的一些数据的好文章，涵盖了其中一些问题。对任何有抱负的经理来说，这都值得一读。

总体而言，足球和体育突显出，许多最优秀的经理人并不总是技术最娴熟的。所以请记住，技术上的弱点并不意味着你不能胜任管理工作。

### [](#mistakes-will-happen)错误会发生

这很简单，当你开始你的经理生涯时，你会犯错误。如果这是你的第一个管理角色，你不会知道如何正确应对每一种情况。在我担任经理的两年里，我犯了无数的错误；我雇错了人；我反应过度，变得不必要的愤怒；我太雄心勃勃了，这导致了失败。

现实是你会犯错误，没关系，承认错误，道歉，记下错误，改变，继续前进。

### [](#expect-the-unexpected)期待意想不到的事情

没有人能写出精确的管理指南，所以要有所期待，为意外做好准备。你可能有很多处理技术问题的经验，但当你成为经理时，你将不得不处理你技术专长之外的问题。

你会接触到人们的个人问题；他们的心理健康问题；他们的爱情生活；他们的论点；广泛的问题。期待这一点，并准备以尽可能慎重和敏感的方式处理这些问题。

### [](#learn-to-compromise)学会妥协

这一点很重要，你不会总是随心所欲，你必须在糟糕的情况下尽力而为。技术团队位于企业内部，因此技术问题和后果并不总是会推动企业决策。作为一名经理，你将不得不与其他有不同动机和目标的部门和经理谈判和妥协。

有时这些目标会与技术最佳实践相冲突，并可能对您的团队产生负面影响。你不能总是以好斗或好战的方式回应。你必须妥协，想出解决这些问题的办法。

做到这一点的一个方法是定义你的“底线”是什么，然后在其他事情上妥协。这是一个分两步走的过程，重点关注非技术人员可以理解的内容，以及可能对您的组织产生负面影响的内容。通常是在财务或声誉方面。

例如，代码原则不符合这些标准，因为它们对于非技术人员来说是不可理解的。例如，你不能在会议上说，“我拒绝在本周发布产品，因为 Dave 的代码不符合单一责任原则……”

```
// Dave's Code
// Convert every integer in the array so it is divisible by five.

public function everyValueDivisibleByFive($array)
{
    $newArray = [];

    if (!count($array)) {
        $result = 'error';
    }
    else {
        foreach ($array as $row) {
            if ($row % 5 != 0) {
                $remainder = $row % 5;

                echo $remainder;

                if (5 - $remainder < 3) {
                    $addition = 5 - $remainder;

                    $newArray[] = $row + $addition;
                }
                else {
                    $newArray[] = $row - $remainder;
                }
            }
            else {
                $newArray[] = $row;
            }
        }

        $result = $newArray;
    }

    return $result;
} 
```

戴夫的代码很糟糕，但却很管用。这是企业所关心的，任何其他部门都没有人会关心它的质量。这并不是说你不应该鼓励你的团队遵循代码原则。但是由于各种原因，你并不总是能够跟随他们，所以没有必要对他们充满敌意。

一个更好的“沙线”将关注安全等问题。对于非技术人员来说，这是一个更容易理解的话题，可以对一个组织产生直接影响。举个例子，你可以禁止某些工具，比如 [WebMin](http://www.webmin.com/) 或者 [PHPMyAdmin](https://www.phpmyadmin.net/) 。这些工具可以帮助开发人员并提高效率，但它们也暴露了 web 上的服务器和数据库，因此可能是安全隐患和业务威胁。

其他好的“沙上线条”包括:

*   **测试:**要求在发布前有时间进行 QA 和测试，以减少生产中发现的 bug 数量。
*   预算/时间表:要求预算和时间表与能够交付的相匹配，这样开发人员面临的压力就更小，产生的错误也更少。
*   **部署:**要求代码不要在星期五部署，这样在周末难以修复的问题就会减少。

所有这些问题都是可以理解的，很容易向非技术人员解释。它们也将作为坚实的原则来指导你的团队，并为他们提供时间和空间来开发无错误和安全的软件。总体而言，这对你的业务、声誉和利润率都有好处。

### [](#keep-coding)保留编码

当你成为一名经理时，没有理由停止编码。我现在很少在工作中编写代码，但我会继续从事个人项目，并定期参加技术会议。保持足尖在水中是很重要的，你没有理由不这样做。

参与开源项目，或者编写和维护自己的库。一周只要几个小时。但是如果你继续编码，它将帮助你了解你的团队成员在做什么；更广泛的技术社区正在发生什么；保持你对这个主题的热情。这将增加你的团队对你的决定的信任，并帮助你更好地为你的组织出谋划策。总的来说，这只能帮助作为管理者的你，所以继续编码吧。

从开发人员到经理的转变可能会很困难。但是我相信上面的提示和建议强调了你可以学习成为一个更好的管理者，就像你学习成为一个更好的程序员一样。只是需要时间，耐心和坚持。我也希望这篇文章鼓励更多的人考虑管理角色。更多有技术经验的管理人员只会对我们的行业、我们的客户、我们的工作以及我们的开发伙伴有好处。
