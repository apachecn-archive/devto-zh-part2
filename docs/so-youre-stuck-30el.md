# 所以你被困住了

程序员工作的一部分是寻找丢失的分号和复杂的缓存问题，仔细检查并绞尽脑汁研究应用程序中的每一行代码。这是关于代码的最令人沮丧的事情之一:我们用它来减轻大脑的复杂性，但这种好处的反面是任何有用的程序对我们来说都太复杂了，我们无法完全理解。复杂性创造未知，而每个未知都是一个等待发生的 bug。

代码的另一个令人沮丧的地方是它必须死记硬背。你不太可能通过盲目的实验发现你需要的关键字或 API。每种语言和库都有问题，可能在文档中有描述，也可能没有。在这种情况下，经验是不可替代的。

大多数程序员(我猜)都熟悉在一个问题上停滞不前几个小时后，不知道如何前进时产生的深深的、炽热的、令人崩溃的挫败感。关于这个话题，这里有一条来自创造堆栈溢出的人的推文:

我头部的这种特殊用法可能缩短了我几年的寿命。如果你是一个新的程序员，你可以通过遵循一个类似于我在这里推荐的过程来避免这些。每一步都建立在前面一步的基础上，并且不断升级，大多数情况下，在到达最后一步之前，你很久就会摆脱困境。

## 1。有自知之明

如果你照顾好自己，关注自己的感受，一切都会变得更好。睡眠不足、饥饿、恶劣的工作环境或个人生活中的压力都会直接影响你解决问题的能力。理想情况下，当你开始工作时，你应该处于一种平和的心态。当你感到沮丧时，你可以给这个问题贴上标签(“我卡住了”)，深呼吸，然后开始解决它。

## 2。时间盒你的挫折

一旦你确定自己被困住了，就设定一个 20 分钟的计时器。关掉空闲时间和电子邮件，戴上耳机，礼貌地推迟提问和与同事的对话，开始搜索问题。尽可能多地阅读相关内容。尝试几种不同的搜索查询。你可能会惊讶于有多少人有你的特殊问题，即使它看起来是独特的或专有的。如果你足够幸运有一个错误信息，用引号搜索它。考虑所有的系统(库、包、框架、API、服务等等。)与应用程序的错误部分进行交互，并问自己问题是否出现在与您想象的不同的地方。

## 3。橡皮鸭

向想象中的同事解释这个问题。手边有橡皮鸭或填充动物玩具吗？甚至更好。我的桌子上一直放着一只鸭嘴兽标本佩里。尽可能详细地描述这个问题，并假设你的同事不太熟悉你正在谈论的代码。

如果这对你来说很难，试着给同事写一封电子邮件或短信，也许能帮上忙。先不要发送信息——只是把它写出来并修改，直到你满意你已经清楚地表达了问题。一定要提到你所做的研究和你已经排除的任何可能的原因。

## 4。画出来

在一张纸上或白板上，勾画出在问题区域中交互的方法、类和文件的基本图表。这里没有花哨的壁画——一堆圆圈和线条应该没问题。画一个通过这个系统的数据。如果您需要仔细阅读代码以确保您的绘图是正确的，请花一分钟来这样做。

## 5。休息一下

到现在为止，你应该对这个问题有相当的了解了(或者知道你不了解它的什么)。此时，让电脑进入睡眠状态，让大脑休息一下。任何对精神要求不高的事情都可以。与朋友或同事轻松交谈，小睡一会儿，散散步，吃点午饭，看一段有趣的视频，去健身房，或者洗个澡。当你完全在做别的事情时，你可能会想到解决办法。

## 6。请求帮助

到目前为止，你已经达到了你一个人所能做的极限；你可能在这个问题上花了 30-90 分钟。是时候引入增援了。找一个有相关经验的同事，寻求帮助。如果你没有同事或者一个都没有，那就去找你的导师。如果你没有良师益友，现在是时候把它列入你的任务清单了。Stephanie Hurlburt 的 Twitter feed 列出了一长串愿意在广泛的话题上提供指导的人(也在该网站上汇编了)。

不要担心浪费他们的时间。你已经研究和思考过这个问题，所以你可以完整简洁地描述它。对你来说，继续独自尝试可能会变成更严重的时间浪费。

## 7。隔离问题

如果你仍然停滞不前，是时候从头开始了。创建一个小的 Git repo 或 Codepen，看看能否添加足够的代码来重现问题。一次一部分，围绕问题复制环境和代码，直到您看到它发生。然后试着依次删除各种片段，确保你没有包含任何不必要的东西。你最终得到的是问题的最小可能的例子(一个 [MCVE](https://stackoverflow.com/help/mcve) ，这将使别人更容易帮助你解决它。与您在上一步中寻求帮助的人一起回顾这一点。

## 8。写下来

写下问题的完整描述。包括代码样本、Git repo 或 Codepen 的链接、截屏(如果适用)以及您所做的尝试和研究的总结。如果公开发布，不要包含任何会给公司带来安全风险的内容，因为这是下一步。

## 9。找个哥们问问互联网

找一个朋友(可能是第 6 步中的那个人)，让他们通读你在第 8 步中写的文章，以确保它有意义。然后发布到网上。Stack Overflow 和其他 Stack Exchange 站点(如 Server Fault、Software Engineering 或 Database Administrators)是您及时获得问题答案的最佳选择，但他们的社区可能会对提问者怀有敌意，所以不要独自前往-请您的好友投票支持您的问题，并帮助您抵御粗鲁的评论者或投票来结束问题。运气好的话，你会在一两天内得到一些好的反馈和想法。

## 10。找到解决方法

如果您仍然没有解决您的问题，是时候执行 b 计划了。您可以考虑更改受影响的功能，添加一些回退代码，或者切换到不同的软件库。这听起来很像放弃，但妥协是所有软件的核心，所以不要为此自责。

## 最后:记录解决方案

一旦你解决了这个问题，考虑写一些关于它的文档来减轻其他人的沮丧。这可能是代码中的一条注释、公司内部文档中的一个页面、一封电子邮件、博客或 dev.to 上的一篇帖子，或者是一个自行回答的堆栈溢出问题。

再说一次，注意你的心理健康。编程本身就令人沮丧。我们都需要休息。如果你经常感到精疲力竭，是时候休息一天，改变你的日常生活，或者寻找更好的工作了。

祝你好运！
