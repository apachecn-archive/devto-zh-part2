# 新开发人员的一般想法/指南

> 原文:[https://dev . to/lube kpl/general-ideas guidelines-for-fresh-developers-1ej 3](https://dev.to/lubekpl/general-ideasguidelines-for-fresh-developers-1ej3)

对于缺乏经验的软件开发人员来说，这是一个非常通用的指南。这些年来，我花了一些时间才意识到这些，我认为当我开始编码时，这对我会有很大的帮助。

1.  你的代码应该是可读的。为什么？这个伟大的报价将解释这一切。

    > 事实上，花在阅读和写作上的时间比远远超过 10 比 1。作为编写新代码工作的一部分，我们不断地阅读旧代码。...[因此，]使其易于阅读会使其易于书写

    *   罗伯特·马丁
2.  试验...一些客户/项目经理认为花钱进行测试是一种浪费。你的生活中不需要那种消极。测试非常有用，原因有很多！首先，它们检查你的代码是否真的做了它应该做的事情。第二，重构——你有没有尝试过重构一个类并且*希望*每个方法返回和之前一样的结果？测试也是一种帮助新队友理解什么、如何以及为什么的文档。

3.  尽管代码从不说谎，而注释有时会说谎，但对代码进行注释通常是个好主意。尽量避免使用解释复杂代码的注释(参见第 1 点),使用注释来解释业务逻辑。这段代码为什么会这样。

4.  作为一名开发人员，照顾好你的~~子~~代码是你的责任。你认为重构已经到位了吗？找到你的客户或项目经理，向他们解释为什么重构是一个好主意，并且是必需的。技术债总是要还的。

5.  有没有想过开发商周围的橡胶鸭子是怎么回事？当你被一段代码卡住时，你可以试着向一只橡皮鸭解释它。这种方法被称为[橡皮鸭调试](https://en.wikipedia.org/wiki/Rubber_duck_debugging)，它是开发人员武器库中非常强大的武器。谈论你的代码并试图向某人解释它将帮助你摆脱困境，并以不同的方式思考你的代码。
    有时甚至一只鸭子也无济于事。和其他人聊聊，最好是其他开发者；)

6.  你听说过敏捷吗？有这样一个想法，每天开会，团队成员互相交流，分享问题。试试看，当每个人都知道你在做什么时，分享想法和寻求帮助会更容易。

7.  文档。比考试更可怕。告诉我，如果没有他们的文档，你会如何使用你最喜欢的 API 或第三方服务？与你的队友和其他团队一起工作时，让他们开心，并记录面向公众的事情。

8.  记住软件开发的原则( [DRY](https://en.wikipedia.org/wiki/Don't_repeat_yourself) 、 [KISS](https://en.wikipedia.org/wiki/KISS_principle) 、 [SOLID](https://en.wikipedia.org/wiki/SOLID_(object-oriented_design)) 和 [YAGNI](https://en.wikipedia.org/wiki/You_aren't_gonna_need_it) )，并试着在日常生活中使用它们，例如提取一个类或一个模块(DRY)将有助于保持代码的可维护性。

9.  有时当你通读代码时，你会发现一些奇怪的地方，或者仅仅是一段糟糕的代码。当然这是可行的，但是如果你能很容易地改进它，为什么不呢？这就是所谓的童子军规则——“永远要让露营地比你发现它的时候干净”。

10.  “衡量两次，削减一次”——在进行实际开发之前思考(或写下)事情总是更好的。

11.  你可能会感到惊讶，但你也是人。每个人都会犯错。你会犯错误。把它做完。我得说，犯错是工作的一部分。试着充分利用它，当你犯了一个错误，试着尽可能多的从中吸取教训。

12.  不要犹豫提出问题和反对观点。问题可以引发关于为什么需要这个特性、我们应该如何实现它以及这个实现的风险是什么的讨论。

13.  渴望学习和传播你的知识。书籍、博客帖子、RSS 提要、会议、播客、本地用户组会议。这些都是学习新事物和谈论你的经历的好地方。

14.  始终关注您使用的新库和新版本的框架/库。更新的版本带来了新的和令人兴奋的功能！

15.  充分利用你的工作流程。你在用 vim？太好了。Atom？很好。崇高？耶。在纸上写下你的代码，然后扫描它们？不管什么能让你开心。你想知道哪个编辑器/ide 最好？最好的是让你觉得舒服的那种。

16.  尽量专业一点。不要使用“我们会看到”和“希望”这样的短语。如果你把你的车带到一个机械师那里，他会告诉你希望他能修好它，你会有什么感觉？

17.  永远不要害怕问。如果你不知道或者不确定，看在上帝的份上，去问吧。
