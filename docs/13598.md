# 我如何每天使用 CRC(班级责任协作)卡(第二部分)

> 原文:[https://dev . to/defmy func/how-I-use-CRC-class-respons ibility-collaboration-cards-every-day-part-2-579g](https://dev.to/defmyfunc/how-i-use-crc-class-responsibility-collaboration-cards-every-day-part-2-579g)

在我的上一篇文章([我如何每天使用 CRC(班级责任协作)卡(第一部分)](https://dev.to/defmyfunc/how-i-use-crc-class-responsibility-collaboration-cards-every-day-part-1-35c1-temp-slug-2334002))中，我解释了如何通过后门将 CRC 卡的使用引入到您的流程中。在这篇文章中，我将解释如何开始使用“CRC 系列”技术来做“刚好够用”的前期建模和设计，以给你的软件开发过程可预测性和可持续性。

我希望还能解释为什么我认为“CRC 系列”技术不是灵丹妙药，但它可能是对现有流程的最大改进之一。如果处理得当，它会极大地提高跨职能团队的一致性和对软件的理解，进而提高整个组织中所有角色对软件的理解，从而带来更可预测、更可持续的软件交付。

我在这篇文章中对人们如何工作做了很多假设。我的经验主要集中在不同规模的组织中的自主、授权、工程领导的团队，这些团队已经或正在向“技术核心”组织转变。YMMV。

### [](#the-crc-family)华润世家

在这一部分中，我将解释两种直接相关的技术，我知道是“裸 CRC”和“物理 CRC”。一般来说，当我对人们说:‘请帮我做 CRC’时，我的意思是，‘请在这三种技术中的一种上做些变化’，即。我交替使用它们，并使用任何合适或必要的技巧。

#### [](#what-is-naked-crc)什么是‘裸 CRC’？

“裸 CRC”是您使用 CRC 卡的地方，但不要填写它们，您让它们为空。你可能在日常生活中做过类似的事情，即使你从未在软件中做过:“如果盐瓶是街角的商店，胡椒是你，那么你左转”。这项技术是轻量级建模的极致。你用面对面的交流来描述合作者和责任，而不是用书面语言。我非常相信“适合你大脑的代码”,而 Naked CRC 确实测试了它的极限。

#### [](#what-is-physical-crc)什么是‘物理 CRC’？

“物理循环”是指您使用自己制作的卡片，并在团队中分发。然后，你们作为一个团队站起来，通过你们所拥有的场景进行游戏，其中每个人都假装是其中一个对象，并在整个场景中扮演自己的角色。这种技术对于团队合作非常有用，可以确保每个人都有相同的理解。

#### [](#why-the-crc-family)为什么是‘华润世家’？

纯属趣闻，我发现这 3 种技术比其他任何东西都更能帮助我交付可预见的软件，简单且被所有人理解。在开发生命周期中恰当地使用这三种技术，可以迫使您在团队和业务的约束下工作。它们作为一种指导力量(几乎是工作的适应功能… [你好，进化架构！](https://www.thoughtworks.com/books/building-evolutionary-architectures))你的团队可以解决的问题的大小，他们如何分解这些问题，他们如何相互沟通和与外界沟通，以及他们如何开展工作。如果团队使用这些技术无法解释和理解工作，那么根据他们目前的理解/状态，这可能不是团队应该尝试做的工作，因此，做更多的尖峰，将工作分解得更细一些，找人帮助，无论什么，无论什么，直到你可以使用这些技术解释你的问题。

### [](#leveraging-crc-techniques-to-do-just-enough-upfront-designmodelling)利用 CRC 技术进行“刚好够用”的前期设计/建模

因此，现在您在回顾型活动的流程结束时使用 CRC 卡，以更好地了解您的当前状态并塑造您未来的工作。我现在将解释我通常什么时候使用它们来指导软件的设计。

所有这些场景都是我认为的“理想”场景。我在团队中工作，我不能决定所有的规则(该死的团队！你知道你想要像国王一样被统治)但是当一个问题发生在团队的过程中并且我相信一个 CRC 可以解决它的时候，我建议这些类型的行动在回顾中作为可能的解决方案。

#### [](#scenario-1-ticket-kick-offs)场景一:门票开球

我发现，由票上的一个开发者在开球时运行**裸 CRC** 有很多好处。这应该以这样一种形式完成，即每个人在开始时都能理解它和输出。(门票开球看起来有点像“3 个朋友”)。

**例**，这里有 3 张空白卡片:

我可能会按照特定的顺序放置它们，当我放置第一个时，会说“这是控制器。它负责编排视图和模型。它还负责射击统计。”当我放第二张卡片时，我可能会说“这是风景。它负责管理用户交互并将这些事件传达给控制器。当用户单击保存按钮时，将引发一个事件，控制器将理解该事件，并在我们当前正在发送的统计数据中挑选出我们感兴趣的新数据。”当我放第三张卡片时，我可能会说“这是模型，模型中不需要新的数据，但是，由于您请求发送的属性是两个字段的合并，我们将在这里这样做。”全部 3 个下来，我可能会说“这是我们打算做出的更改，以支持此票证，让我们播放场景和 AC，看看更改的范围是什么。”

当您创建 ACs 时，您根据它们来判断解决方案，并相应地更改解决方案。您可能已经注意到，我没有提到任何测试，所以在这个例子中，我将讨论需要更改以支持它的测试代码。但是很明显，IRL，你随波逐流，相应地改变，去支持你需要支持的东西。

**结果**，在这个过程中，希望你们都能对完成这张票据所需的工作达成共识。第一件应该很快变得显而易见的事情是‘这张票太大了吗？’。如果在门票启动中的每个人都不能跟踪正在发生的事情，那么你可能有一张太大的门票。除了这种情况，一旦每个人都同意票证涉及的内容以及通过 CRC 流程对建议的实施所做的更改，我们就各走各的路，每个人对将要做的事情以及该票证将对他们的角色产生的影响都有更具体的想法。

积极的副作用，我经常要求三个朋友中的开发者将结果具体化，并在他们制作彩票时想象出来。切换到 CRC，这样他们就可以向团队的其他成员传达他们将要做出的改变。由于他们做了工作，而且事情并不完全符合每个人的期望，我希望这一点能够反映到那些参与启动的人身上，帮助他们根据需要改变他们的工作。如有必要，显著差异应作为一种气味来“重新认识”。

**为什么裸 CRC 超过 CRC？**它更轻便，证明了理解，你只需做足够的“准备工作”就能让它工作。对于“不太懂技术”(无论这意味着什么)的人来说，它没有 CRC 那么令人印象深刻，它迫使跨学科的合作和一致的理解，因为人们更有可能挑战，它鼓励参与和参与，因为如果你在“三个朋友”期间关闭。最后，在大多数情况下(虽然不是全部！)它作为一个自然的限制因素，限制了一张票的“大”和“复杂”程度，因为它强制要求票的大小适合团队负责人的大小。最后，我相信“为了证明你理解一个主题，教它”,而裸 CRC 比所有其他技术都需要对问题有一定程度的理解，同时也是一种优秀的教学辅助手段。

#### [](#scenario-2-epic-level-kick-offs)场景二:史诗级开球

首先，我通常使用 CRC 来帮助设计解决方案。我为系统所有最初受影响的部分生成新的卡片，然后我和团队一起根据需要将新的合作者和职责添加到这些卡片中。一旦我们有了候选人，我就使用物理 CRC 来遍历所有当前已知的场景。

给每个团队成员一张卡片，让他们在场景中表演他们的角色。这可能包括这样的表述:“我是视图，当用户与按钮交互时，我触发带有该数据的事件”，“我是控制器，我监听‘按钮被点击’事件，当它被触发时，我解析来自事件的数据，寻找新的数据项并调用带有所需数据的 stat 库”。

**结果**，团队中的每一个人都应该了解期望的解决方案，因为他们参与了方案的制定，并作为一个团队经历了所有已知的场景。

**为什么物理 CRC 优于 CRC？**物理 CRC 真正突出了该解决方案存在的任何[【时序一致性】](http://connascence.io/timing.html)问题。通过一个人在场景演练中比在纸上谈兵时说了多少话，往往更容易发现其他事情(如耦合、过多的责任等)。当人们被迫表演他们在阅读时没有表演的东西时，他们会问问题。

#### [](#scenario-3-debugging)场景三:调试

就像橡胶鸭子一样，在调试过程中完成的任何形式的 CRC 都是有价值的。即使“解决方案”很简单，它通常也值得通过 CRC 来获得正确的抽象级别，帮助您发现您没有想到的事情。

### [](#summary)总结

希望我已经通过这两篇文章说服了你，CRC 值得一试:)我做了一些大胆的陈述，我希望你能看到它的价值。不幸的是，我所有的“证据”都是轶事，基于经验，而不是事实。

不管你是否相信我，我认为 CRC 是“技术领导”工具箱中最棒的团队协调工具之一。通过结盟带来了许多其他的好东西。CRC 极大地提高了代码“可理解性”。它不仅帮助人们适应当前的代码，还帮助团队在他们看来什么是好的方面达成一致。这有助于团队找出“适合他们头脑的东西”。一个队好终究是另一个队坏。不仅如此，CRC 比其他任何工具都更加清楚地揭示了你所不知道的事情。这有助于团队弄清楚他们做了什么和不理解什么，他们需要调查和跟踪什么，龙在哪里。

对于复杂性也是如此。允许团队理解他们系统的复杂部分，同时允许团队表达对他们来说什么是复杂的，什么是不复杂的，为他们在正确的层次上抽象事物。
