## 控制台日志！！

您可能已经被教授或讲师多次告知，控制台日志记录对于找出错误和问题是多么重要。但是说实话，在我短暂的编码“生涯”的早期，我拒绝了这种方法，只是因为我更专注于学习函数、查询和大量的编码概念(你知道肉和土豆)。这当然对他们自己的权利很重要。或许，当一个人开始他们的编码之旅时，我甚至可以说我选择的路线是正确的。

(大而机警！)

但是，我在这里要说的是，一旦你熟悉了任何语言的语法和概念，控制台日志应该是你需要掌握的下一件事。说真的，我自己直到最近才意识到这一点。实际上，让我向你描述一下在理解控制台日志的魔力之前，我在编码中的思维过程。

# 面临问题时**前**控制台日志

1.  谷歌一下这个问题。
2.  看过去的活动。
3.  问我朋友 Matt(坐我旁边的同学)。
4.  开始怀疑我的人生选择。
5.  仔细阅读我的代码中的每一行，查找拼写错误:'(
6.  向我的导师寻求帮助。

# 面临问题时**后**控制台日志

1.  用控制台日志运行我的代码，收到错误消息。
2.  阅读错误信息。
3.  修复错误消息指出的代码位置。

当然，这并不简单，但这基本上是主要的想法。

在我看来，任何新的开发者都更喜欢后者。但是就像我之前说的(至少对我来说)，如果我不习惯先了解一门语言的基础和概念，那么在我看来，尝试控制台日志记录一切是不太有效的。因为我不知道控制台日志的内容/位置。

因此，也就是说，一旦你觉得你掌握了基本原理和概念，接下来试着掌握控制台日志的艺术，这是跨越任何编码障碍的最好方法之一。
