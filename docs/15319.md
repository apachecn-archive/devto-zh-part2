#### 问题 1:同一个角色有这么多头衔(开发者成功工程、技术支持、技术账户管理、解决方案工程)。在你的公司你叫它什么？

迈克:所以我给自己起了个头衔——不知道其他公司有没有用。给我们实际做的一些背景:Nylas 目前有 22 个人，大部分是工程师，为电子邮件、日历和联系人构建 API。在这类工作(开发人员支持工程)中，命名是很难的——更重要的是要看你拥有的客户和产品的类型，以确定你的角色是什么。

玛吉:在 Segment.io，我的头衔是成功工程师。我在以前的公司也是一名成功的工程师，那是一个很小的公司，只有 30 个人。现在，我们在 Segment.io 是一个更大的团队(仅成功工程就有 13 人)，所以它更加孤立，更加专注于支持。现在，我每天都和代码打交道，而在我的上一家公司，我可能每个月查一次代码。完全不同——同样的标题，但是不同的公司和不同的产品。

凯利:我会重复迈克的话，我也虚构了一个客户工程师的头衔。我刚开始的时候，是创始人在做支持。因为我们是一个错误监控工具，而且我们正在用代码清理杂草，这是有意义的。我们在成功方面提供支持和用户教育(当我们有时间的时候)。标题没有你在做什么重要

**David:** 我的团队实际上隶属于客户支持，但我们也与跨职能平台团队合作:DevRel、BD、开发支持、产品、工程，我们都在 Slack 的平台上一起工作。我的团队之所以成立，是因为我们是处理我们构建的前 80 个集成的代理人，这是一个超级技术性的问题，需要深入研究日志。当我们推出平台时，我们结束了。作为帮助调试和 QA 平台的人，我们最终只是给自己起了个头衔，也就是“开发者支持”。

#### 问题 2:你如何与工程或其他技术部门一起工作，以确保你照顾好你的客户？

**David:** 我们的支持票现在分为使用集成的客户和构建集成的开发人员。我们花在跨职能部门工作的大部分时间都是和平台工程团队在一起。很长一段时间，我们处理所有的手动 QA，现在已经交给不同的团队，所以我们可以专注于对开发人员的实际支持。我们与产品团队(平台内的少数产品团队)合作，确保他们了解开发人员在特定功能中寻找什么。在一个特性发布之前，我们会构建一些东西来确保它能正常工作。

Kelly: 我们与客户成功团队坐在一起，非常符合他们的目标，但我们实际上是在工程的保护伞下。一部分原因是我们得到的关于职业发展的反馈更多地转向了工程。我们要做的一件很酷的事情是建立我们自己的管理工具，所以我们花了更多的时间与工程师团队一起编写代码，并使用我们自己的产品来帮助我们的客户。

Maggie :我们开发了自己的 Zendesk 工具，我们一直都依赖这些工具。我们显示客户信息，他们的账户有多大，谁是他们的客户经理。我们依靠所有这些东西来确保客户在写作时获得适当的支持。我们使用许多其他的集成，如 [JIRA](https://www.atlassian.com/software/jira) (与我们的工程师和产品团队沟通)。

迈克:尼拉斯现在总共有 22 个人。开发成功团队只有我和娄。我们与 Nylas 的所有部门(销售、产品、工程)密切合作。我们是支持的前线。因为我们的业务是 API，我们的客户大多是开发人员，所以这是非常技术性的。我们帮助销售的方式是，我们接到销售电话，承担所谓的销售工程角色，就如何与我们的 API 和最佳实践相集成向人们提供技术建议，让他们快速加入进来。我们帮助通知产品决策，因为我们与客户的沟通如此频繁，我们对他们在功能要求方面的需求有着深刻的见解。通过工程，我们将把我们看到的任何错误升级给他们。我们与工程团队一起进行日常沟通，让我们每天都能发现任何客户问题。

#### 问题 3:你如何保护你的工程团队不被客户支持票轰炸？你如何平衡他们的日常工程工作？

Mike: 开发成功团队保护工程时间很重要。工程师们有他们自己的长期项目，他们很难取得进展，当他们每天都被客户的问题打断时，他们需要切换上下文来修复错误。为了保护我们的工程团队，我们需要建立信任。每次我们把不该上报的事情上报给他们，下次你想让他们看罚单时就很难了。我们的工作是对我们正在查看的所有票证进行优先排序和鸟瞰，并显示最重要的票证。我们考虑保护工程师时间的因素有:

*   有多少帐户受到此问题的影响？
*   是否存在我们需要遵守的与客户的 SLA？
*   该客户是我们试图达成的交易的一部分吗？

对我们公司来说，我们非常依赖内部文件。因此，当我们与我们的工程师交谈时，他们告诉我们“这就是你看到此错误信息的原因”，我们会记录下来。我们非常依赖这一流程，不仅仅是我们的团队在关注它，来自销售、客户成功经理和市场营销的团队也在关注它。我们在各个部门都会遇到同样的问题。我们用收纳盒纸来保存这些文件。

考虑到我们几乎支持每一种语言和框架，而我只有几年的工程经验，当我第一次开始一个问题时，我经常不知道我是否能在一个小时内解决它，或者在一天结束时我是否能知道更多。我和我的同事(我们也是一个两人团队)在早上和时间盒中运行那些配置重或深的杂草。net(一些我不知道的东西),通常看两个小时，然后重新评估是否升级。我们两个工程团队中的一个人负责解决我们下午无法解决的所有问题。

大卫:我们也有类似的流程。我们的每个工程组在 Slack 中都有一个所谓的分类通道，支持人员(无论是我们提供开发人员支持还是客户团队提供客户支持)可以进来说“我遇到了这个问题，我们中没有人能解决它。”工程师们轮流在不同部门工作。在站台上，有两个人，他们所做的就是分类。但是我们试图通过尽可能多地了解 API 来保护它们。T3】

#### 问题 4:你如何获得关于需要添加到产品中的特性的度量标准？

**David:** 对于 slack 来说，支架是产品的一部分。这是自产品问世以来我们所做的宣传。如果你为懈怠买单，你就是在为支持买单。在过去的几年里，我们推出了一款企业级产品，并扩大了其中的一部分。现在，作为一个团队，我们正在考虑如何从一对一的电子邮件支持转向更广泛的企业支持。我们如何看待一对多而不是一对一的解决方案？

**Mike:** 在 Nylas，我们有一个基本的支持 SLA，所有客户都可以免费自动使用。作为一家小型创业公司，每个团队都受到资源的限制。很明显，我们希望每张票都有 30 分钟的解决时间，但这是不可能的。因此，我们有其他层次的服务级别协议，人们将为之付费。一般来说，对于新客户，我们也会优先考虑他们的机票，以帮助他们快速登机。

**Maggie:** 在细分市场，我们有不同的定价等级，从企业到免费开发者计划。因此，如果您在团队计划或企业计划中，响应时间会更快，您的问题会更快地升级。企业计划有一个专门的支持或客户成功经理。

Kelly: 我们非常依赖 salesforce 与 Zendesk 的集成。因此，如果某个项目在企业计划中，它将在队列中上移。但很多时候，我坐在销售团队和客户成功团队附近，这更为随意。起初，我觉得和客户成功团队坐在一起有点傻，但实际上，能够联系到有背景并能帮助快速解决问题的人是一个巨大的好处。

#### 问题 5:如何将功能请求传递给正确的团队/如何代表客户发言？**

凯莉:我们真的很想得到顾客的反馈。这是我们经常收集的东西，因为我们重视反馈。错误监控是一个新的想法，所以对你解决程序中的错误的方法进行分组对每个客户来说都是独一无二的，我们觉得当我们有足够多的反馈并能与我们的工程师紧密联系时，这是最好的。一旦我们看到临界质量，我们将开始建造一些东西。

玛吉:对于我们的任何主要功能要求，我们都会将它们放入 JIRA 报告中。我们有一个与 Zendesk 的集成，我们可以将 Zendesk 与 JIRA 报告集成。我们还将客户资料放入票据中，这样无论他们是大客户还是小客户，我们都会记录所有信息，以便产品团队可以查看、研究并决定将反馈放入哪个 sprint。

#### 问题 6:你如何帮助你的客户支持他们自己，这样你才不会淹没在支持票中？**

迈克:创造大量的内容。最有效的是知识库文章。我使用一个叫做 Loom 的工具，它可以帮助你制作视频并与他人分享链接。我们还主持每周办公时间。每周有 2-3 个小时的时段是开放的，我们的客户可以每周打电话进来和一个 n 工程师(基本就是我)聊天。他们喜欢谈论一切，从他们的集成和 ti 如何良好地工作，到深入到一个具体的问题。如果他们没有 SLA，许多客户会利用这一点来获得 roe 面见时间。

麦琪:是的，我们有面向公众的文件，我们一直在努力让它们变得更强大。每当客户来信询问问题时，他们都会查看文档。我们团队中有一个人只关注文档。

#### 问题 7:开发者成功最难的部分是什么？

Maggie: 我们的许多客户来信希望我们了解所有的事情——不仅仅是你自己的产品，还有所有的集成，你的所有合作伙伴，你可以在 StackOverflow 上找到的问题。他们期望你在所有技术方面都是专家，显然没有人是，所以我们必须自己做大量的研究。最难的部分是知道什么时候该反击，说“嘿，这是你可能需要自己研究的东西，这不在我们的范围内。”

迈克:对我来说，最难的部分是长时间以来成为唯一的支持者。当你试图让客户满意时，这是一项相当情绪化的工作，因此没有像娄这样的团队成员真的很难。我非常兴奋，因为我们刚刚雇用了另一名 DSE，我将继续为我们的团队雇用更多的技术角色，这样那些日子就结束了。

大卫:我认为对我们来说，最棘手的事情之一是我们作为一个平台的增长速度突然变得很难跟上。即使我们是负责质量保证的人，事情还是被我们忽略了，在没有质量保证的情况下发布，我们不得不灭火。

Kelly: 我想说的是我们获得支持请求的方式。我们确实让人们写客户支持，与他们的成功经理交谈，但我们也有大量的开源库，所以你总是可以在那里解决 GitHub 问题。这很好，因为一旦你解决了某个问题，你可以在那里引用它，每个人都可以查看过去的 GitHub 问题，但这也是客户工程和开发这些库的平台团队之间有趣的重叠。如果他们没有被 GitHub 问题压垮，确保流程能够持续，并知道在我们的其他客户支持问题中在哪里优先处理这些问题。

#### 问题 8:你最喜欢工作的哪一部分？

凯莉:我觉得很酷的是，在我开始提问之前，你很少知道别人的问题的答案是什么。我从客户的问题中了解到的信息，以及我从未想过要自己去查阅的技术。

Maggie: 我们的客户群非常国际化，所以让来自世界各地的客户用不同的背景和语言写信是一件很有趣的事情。他们可能会问非常不同的问题。

大卫:对我来说，这个角色感觉更像是过去的互联网，你可以帮助某人建立他们的网页，然后“嘭”的一声[他们太兴奋了]！

迈克:我的同事是首席技术官和技术主管，所以我的角色是基于关系的。建立这种关系，并在整个整合过程中与某人合作，这很酷。我喜欢当我能够帮助那些人并让他们高兴，并且能够以一种找到优雅的方法而不是一遍又一遍地做那些小的可重复的过程的方式来自动化过程，这真的令人满意。

#### 问题 9:在你担任这个角色之前，你最希望有人告诉你的一件事是什么？

Kelly: 我认为最重要的是关于错误处理的所有知识。但考虑到这不太可能，也许只是理解这个问题，弄清楚用户在他们的问题中得到什么，以及他们认为他们技术的哪个部分出了问题。这是第一步。一次。你可以筛选一下，这样一切都更容易找到。

Maggie: 知道顾客实际上在问什么。可能和你想的大不一样。我会调试 20 分钟我认为他们在说什么，然后他们会回来说“这不是我在问的”，我不得不重新考虑这个问题。总是再次确认他们真正想要的是什么。

**Mike:** 回到我上夜班的时候(那时我一个人在做支持工作)，感觉到我必须尽快解决每一个问题。我希望我能早点知道，你可以对客户的要求说不，并诚实地告诉人们你的期望。

大卫:我认为诚实是很重要的一部分，当“是的，API 有问题，今天不会修复，但我们正在努力解决它”时，不要害怕告诉开发者然后他们至少知道并可以围绕它进行构建。

凯莉:这挺好的，因为大多数开发人员以前都经历过。他们就像“那有很多工作要做，我不打算建那个”。

#### 问题 10:一旦你深入研究了代码并知道你可以为它打开一个拉取请求，你实际上提交了 PR 还是把问题带到了工程中？

**Mike:** 在 Nylas，是的，作为一名成功的开发工程师，你可以通过 Zendesk 获得客户支持请求，通过 [Kibana](https://www.elastic.co/products/kibana) ，使用 [Honeycomb.io](https://honeycomb.io/) 深入日志，找出问题的原因，编写代码，提交给 [Phabricator](https://www.phacility.com/phabricator/) ，然后实际部署它。我们在 Nylas 编写了一个命令(CLI 命令),因此您可以从头到尾使用它，然后返回给客户并说它已修复。类型。我们用这种方式解决的问题范围更小。对于大的基础设施变化，我们会升级到工程。

大卫:我有提交权限，但我尽量不使用它们。现在有足够多的工程师，我会把它交给公关，然后交给工程师审查，要么自己提交，要么让他们来做。

凯莉:很多人都在摸索这条线在哪里。我们经常发现，如果有些事情不清楚，我们可以做一个更好的例子。我的同事更新了许多我们的开源示例，说明如何配置 Bugsnag 的不同部分——这更常见(随机的小 QR 很有帮助，但不是针对我们的工程师日常工作的)。

#### 问题 11:你有摆脱日常工作的建议吗？

麦琪:对我来说，这取决于有多少人在为票的事烦我。如果我有来自许多不同团队的人说我现在需要完成一些事情，我会很紧张。如果不需要尽快回复，有时可以等到明天。你从顾客那里获得了紧迫感。

大卫:让人们知道你没有忘记他们会有很大的帮助。

#### 问题 12:面试这个角色时，你会寻找什么样的关键特质？更专业或有更强支持背景的人？

凯莉:我们寻找那些对陷入代码困境无动于衷的人。你必须习惯于从很少的信息开始，然后不断学习。拥有技术背景并能够与客户良好沟通是一种独特的品质，很难找到能同时做到这两点的人。

David: 我们在招聘开发人员或客户支持人员时，筛选的一个因素就是耐力。开发人员支持团队的成员现在都是工程师，他们决定转行做别的事情。你必须从整天埋头写代码到在其他代码间切换的心态。人们一天中遇到的问题是很重要的，所以我们寻找真正有兴趣提供帮助的人。

迈克:我们现在正在招聘一名成功工程开发人员。他们 60-70%的交流是通过电子邮件、电话等。另外 30-40%的人在钻研问题。我们测试他们的沟通技巧和技术能力-我们有一个小的编码挑战贯穿。以前的编码经验肯定是我们寻找的东西。另一方面，合伙人工程师的角色是:20-30%的交流，更多的时间花在研究 bug 上。

玛吉:对我们来说，这很相似。有技术背景是一个很好的基础，但我们真的强调口头和书面的沟通技巧。

#### 问题 13:投资什么样的工装？

大卫:我们对 [Zendesk](https://www.zendesk.com/) 有了一个非常基本的设置。它被捆绑到[基巴纳](https://www.elastic.co/products/kibana)和其他实例中。我认为 Slack 的代码基础是如此不同寻常，以至于我们围绕它构建了大部分自己的工具，随着我们引入更多的商业内容，这一点有所稳定。

凯莉:我们有支持票，我们使用 BugSnag 来找到在我们网站上遇到错误的人。它非常元，但当某人的集成不起作用时，它真的很有帮助。通常这是我最不喜欢尝试调试的事情，但是每当有人的集成失败时，我们都会发送一个自定义事件，我会使用 Bugsnag 来查看是否是错误的密码，或者是更大的问题，如网站关闭。使用他们抱怨的工具来解决问题很有趣。

迈克:在内部，我们使用网络应用程序。我们的一些账户列表页面——因为我们是一个快速成长的初创公司，所以这个页面的加载速度一天比一天慢。随着我们雇佣越来越多的 DSE，我们将有更多的时间从事影响力更大的项目，在这些项目中，我们构建工具来支持我们自己。我喜欢的一个非常棒的工具叫做 Honeycomb.io。这是一个获得结构化日志数据并根据几个字段进行分解的好方法。你可以查明为什么一件特定的事情发生得非常快。

#### 问题 14:如何防止一张罚单上的调查变成兔子洞？

大卫:我抓住别人，吸引他们的注意力。如果我意识到我要掉进兔子洞，那我就太过分了。

玛吉:我给自己大概 30-40 分钟，如果我还没想出来，我会问别人。有时候，我会要求客户重复或复述问题，或者在掌握更多数据后更新他们的问题。

凯莉:这是一个获取更多数据的好点。在那段时间里，我希望如果我不知道答案，我至少有下一个问题，可以问用户或问正确的团队问题，这样当我升级它时，他们会更进一步，我为他们节省了一些时间。

迈克:我每天都住在兔子洞里。通过电子邮件工作——这就是我们企业提供的价值。我们抽象出了开发人员在构建电子邮件集成时必须处理的所有兔子洞。我尽我所能用我的技术知识做到最好。我花了很长时间调试 Exchange ActiveSync 问题，但同时我还必须与客户沟通。我不能在调试时保持沉默，所以我会尽可能多地记录我发现的问题，这样当我将问题升级到工程时，他们就成功了。
