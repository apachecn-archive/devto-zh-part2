# 构建自主用户界面

> 原文:[https://dev.to/lsoares/building-an-autonomous-ui-3d9p](https://dev.to/lsoares/building-an-autonomous-ui-3d9p)

#### 一个自主的 UI 不依赖于动画、声音、颜色、快捷方式和帮助来被理解和使用。这并不意味着它不应该有他们，但它不应该依赖他们。

#### [](#animation)动画

**动画有助于强化所选的视觉隐喻**(例如，向右滑动以表示卡片移除；显示有一些后台工作的加载程序)并创建视觉和谐(例如，软化过渡)。它们应该简单、微妙、使用一致(例如，淡入和淡出来表达相反的概念)。适度使用它们，如果它们能带来价值。动画不应该总是被用户注意到，因为它们会分散用户的注意力。它们不应该只是用来“让应用程序更酷”。

过度和沉重的动画可能真的很糟糕，会消耗大量的 CPU 时间或对某些残疾人(例如癫痫)有害，因此在某些情况下，应该可以[关闭它们](https://api.jquery.com/jquery.fx.off)。

一个很好的尝试是关闭所有的动画，并确认 UI 是否仍然可用(最好是对用户)。这有助于理解某些动画是否是为了掩饰或强化好的或坏的设计决策。

#### [](#color)颜色

颜色、形状、位置、对比、纹理是图形设计的基本元素。颜色与可能随文化而变化的心情和情绪有关；它们也可以用来突出元素，因此具有很高的表现力。然而，你不应该使用颜色作为单一的区分元素。如果你有几个需要区分的项目，颜色可以加强它，但它不应该单独使用。主要原因是可访问性(例如，色盲的人使用这种用户界面会有问题)。另一方面，你通常不知道用户显示器的颜色限制。

一般规则是用形状来区分物品，用颜色作为附加。光度本身也是一个区分元素。为了知道你是否通过了这个“颜色测试”，**试着在灰色阴影中使用你的界面**，看看它是否需要一些修正。

#### [](#help)帮助

一个好的界面不需要用户依靠帮助来执行任务。这并不是说应用程序不应该有用户手册或上下文帮助。相反，帮助必须存在，并且尽可能地上下文化(例如，恰当放置的工具提示；帮助的上下文链接)；你的目标应该是用户不需要查阅它，尤其是在重复使用的时候。**一个好的 UI 应该“不言自明”。为此，[极简的设计，自然的信息流](https://medium.com/@lsoares/what-developers-need-to-know-about-ux-974b99d61d05)，良好的信息架构，等等贡献良多。**

避免指令；你可以自然、透明、含蓄地去做。如果你的 UI 需要并依赖指令，“你做错了”。嗯，没有必要，但是试着像那样构建 UI。

如果用户过于依赖帮助，这可能是设计甚至信息架构出了问题的征兆。

#### [](#sound)声

适时的声音效果会有所帮助。它可以用来引起注意(例如，新消息到达)，加强一个概念(例如，成功的操作)，发出错误信号，等等。

像动画一样，**声音必须始终作为用户界面的补充，而不是交流的主要形式**。你不应该依赖声音，因为用户可能没有声音设备或静音，在嘈杂的地方或有某种听力障碍。

如果加入某种声音真的能带来价值，那么就应该慎重使用。不恰当的使用会令人讨厌和疲劳。

#### [](#mouse)鼠标

鼠标的依赖性有时也是负面的。典型的例子是鼠标悬停和鼠标右键操作，它们在只触摸设备上变得不可用。你可以使用右键，但仅仅是为了灵活性和速度——同样，这也是一种交互的替代方法。对于导航，也要考虑键盘快捷键。

同样适用于鼠标滚轮；应该有一种替代它的方法。最后，完全依赖鼠标也会影响可访问性。

#### [](#keyboard)键盘

键盘快捷键(例如 Ctrl+C 复制，Tab 导航)不应该是唯一的交互方式——大多数是为高级用户准备的。用快捷方式做的事情应该在 UI 中明确提供(隐藏在菜单下或不隐藏)。

#### [](#navigation-controls)导航控件

你不应该期望用户点击浏览器的“刷新”按钮来从错误中恢复或强制数据更新。此外，尽量避免依靠“后退”和“前进”按钮来导航。**软件在导航、错误恢复和数据更新**(特别是 [SPA](https://en.wikipedia.org/wiki/Single-page_application) s)方面应该是完备的。

在原生安卓应用中，你可以在应用 UI 本身提供“菜单”和“返回”的替代选项。

#### [](#permissions)权限

在 web 和原生应用程序中，您可以请求硬件和软件许可(例如，摄像头、麦克风、位置、显示通知)，但您的应用程序应该在没有它们的情况下工作(例如，如果用户拒绝许可)。如果依赖于它们来使用，你应该向用户解释。

#### [](#messages)消息

不要假设用户在对话框中阅读错误/警告信息。很多时候，他们没有阅读他们的文本就拒绝了他们。[不要因为错过就惩罚](https://userexperiences.co/design-ui-don-t-punish-the-humans-53807f955517)用户。该应用程序应该有办法重复信息或显示内联(根据上下文)代替。

#### id

不要创建依赖于 id(数据库标识符)来完成的 UI 用例。例如，为什么要在数据表上显示标识符？为什么您会让用户将一个对象分配给另一个请求其 id 的对象？友好的称谓是给人类的；标识符用于机器。

#### [](#conclusion)结论

作为一般的经验法则，试着设计每个“应用程序中的位置”,让用户不需要知道他们做了什么才能到达那里。他们可能真的很健忘，或者总是去喝咖啡。也就是说，每个地方都应该有自己理解的必要信息，允许用户继续手头的任务。

在设计 ui 的时候，我们需要决定交互的手段是什么，并把它们分为主要的和辅助的。一般来说，动画、声音、颜色和帮助应该被看作是一种辅助——来加强或补充主要的概念和想法。当然，有时它们会有很大的价值(例如，提供上下文帮助的内嵌图标，作为快捷方式的面包屑)，但至少你应该尝试在不依赖它们的情况下设计 UI。一般的指导方针是关闭所有功能，并测试 UI 的“行为”。

在 web 应用程序/网站中，很明显，现在你应该避免依赖目标受众的显示能力，比如屏幕尺寸和分辨率、颜色模式等。拥抱[响应性和适应性设计](https://css-tricks.com/the-difference-between-responsive-and-adaptive-design/)。此外，要小心，远离非标准的依赖，如 Flash、Java 和 Silverlight。

不要假设用户最初会看到隐藏的信息，如工具提示、未选中的标签、下拉选项等。请记住，某些[交互设计模式](https://en.wikipedia.org/wiki/Interaction_design_pattern)本质上是某种东西(辅助)的替代物，而不是交互的原始形式。例如，面包屑不应该是唯一的导航形式；这些只是主导航的快捷方式/替代方式。

有些项目是您希望用户看到的，有些项目是他们以后可以发现的(取决于任务的重要性)。然而，你应该以不需要事先指导的方式展示物品。以这样一种方式设计用户界面，它可以自我发现。以自我解释、自主和独立的用户界面为目标。

请注意，上述规则应仅作为参考，因为它们可能有例外(但是，要打破这些规则，您需要了解它们)。只有和用户一起测试，你才能确定他们满足了。仅仅重复 [5 个用户就足以检测大多数 UI 问题](https://www.nngroup.com/articles/why-you-only-need-to-test-with-5-users/)是远远不够的。
