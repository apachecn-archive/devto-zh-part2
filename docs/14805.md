# PostgreSQL 或 MySQL 数据导入到谷歌大查询

> 原文:[https://dev . to/pedrojmfidalgopt/PostgreSQL-or-MySQL-data-import-into-Google-big query-2 kii](https://dev.to/pedrojmfidalgopt/postgresql-or-mysql-data-import-into-google-bigquery--2kii)

如今，云平台提供了巨大的处理能力，亚马逊决定更好地利用现有的机器，一旦它们的处理能力主要用于圣诞节销售(今年剩余时间将停止销售...)，微软租借 Office 或 Azure，谷歌租借处理 CPU 和数据处理服务。所有东西都是按月支付的。我们有新的趋势，永久支付租金，但取决于业务，并考虑一个人可以在基础设施和手工劳动中节省什么，这可能是值得的。为实际业务案例研究选择最佳解决方案时，应制定适当的中长期评估计划。

无论哪种方式，所有的技术和平台都征服了它们的利基市场，并且它们会一直存在下去。传统数据库将继续有自己的空间，因为访问云平台需要非常好的互联网连接，而在一些地方这还不可能。而且你总是会遇到敏感内容不能放在云端的问题。例如，考虑政府安全信息、保险单、工资或税收信息等。

Google BigQuery 允许任何小型企业使用 SQL 查询方法存储和处理大量数据，它是作为基于云的大数据分析 web 服务提供的。它允许使用传统的 csv 文件数据类型导入数据，甚至可以使用最新的 json 结构化文件类型。谷歌提供了一个很好的[教程](https://cloud.google.com/bigquery/docs/loading-data)来解释这一点，并且[不久前我甚至为 MySql 也写了一个](https://www.codementor.io/pedrojmfidalgopt/mysql-data-to-google-bigquery-analysis-g0606jofn)。可以使用简单的鼠标操作来完成事情，或者对于更喜欢命令行的人来说，可以使用 bq load 命令将数据加载到 BigQuery 创建的表中。要将数据从 [Postgres 移动到 BigQuery](https://www.alooma.com/integrations/postgresql/bigquery) ，可以使用类似的方法，因为这只是使用适当的可接受格式导出数据的问题。可以使用 SQL 或任何能够使用 ODBC 驱动程序或其他工具连接到 Postgres 的工具来实现这一点。

这些方法对于独特的或特定的批量数据导出和加载很有意义，因为它们不需要太多的集成，可以在需要时手动执行。只需使用任何导出脚本从现有数据库中导出数据，并在创建了适当的数据结构后将其导入 BigQuery，等待正确格式化的数据。
但是，免费数据库正在获得更大的市场，与 Oracle 或 SQL Server 等昂贵的每核心解决方案相比，许可成本让任何 CFO 都考虑使用免费数据库。如果你检查一下数据库[排名](https://db-engines.com/en/ranking)的对比，PostgreSQL 出现在第四位，MySQL 第二。它们是免费的数据库，将自己呈现为非常高性能和可伸缩的免费解决方案(令人印象深刻的是 MySQL 数据库在其桌面工作台中提供的工具)。PostgreSQL 也正在成为一个非常常用的数据库，能够存储要求苛刻的事务或数据仓库解决方案。但最终，即使它可以存储大量数据，处理这些数据也需要 CPU 的能力，这就是谷歌或 AWS 的亮点，因为你的手掌上有数千个 CPU。

对于小企业来说，分析大量数据并不困难，任何处理网络数据的公司都知道，facebook 提供了数百万条日志，linkedin 拥有数百万用户。两年前，我为一家美国公司开发了一个数据迁移过程，该公司使用蜘蛛在网上搜索价格。他们最终拥有数百万行数据，这些数据必须合并到一个数据库中，这样他们就可以比较供应和销售价格，并做出良好的管理决策。考虑到他们使用的小型 SQL Server 机器，对如此多的数据进行分析变得非常困难。在处理大型数据集时，人们应该真正考虑像 Google BigQuery 或 Amazon Redshift 这样的平台作为可能的数据分析引擎选项。

作为结论，如果需要定期的数据迁移，ETL 过程应该构建得使我们的生活更容易，并且不应该依赖于永久的手工迭代(或者至少应该最小化...).如果可能的话，我们应该能够在预定的时间框架内运行接近实时的数据迁移，只通知我们他们已经完成，没有错误。人们应该在这些免费数据库和 Google BigQuery 之间寻找其他数据集成解决方案，这似乎是可以接受的。如果读者在 Google 中搜索这些主题，就会出现几个商业解决方案，它们都指向同一个方向，Google 桌面集成。基于网络的平台将所有这些都放在适当的、经过深思熟虑的工作台中，它们允许数据映射、数据转换、进程调度。一些曾经在桌面端开发的步骤现在可以迁移到云端。可以定义复制频率，以便在每个定义的时间段内迁移数据，而无需任何手动干预。这些平台的最大优势在于，您可以从一个门户网站访问所有的数据源和数据目的地。它们不局限于 PostgreSQL 或 MySQL，有很多连接器可用。
如果你不相信我也在 youtube 上搜索它们，你会发现很多可用的教程，你将能够评估它们的真正威力，几乎与一些可用的桌面 ETL 解决方案相比较。它们是要花钱的，但另一方面，你将不再需要担心服务器维护或数据库备份以及数据库管理员的工资...
