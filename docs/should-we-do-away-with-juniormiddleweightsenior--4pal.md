# 我们应该取消初级/中级/高级吗？

> 原文：<https://dev.to/charliedevelops/should-we-do-away-with-juniormiddleweightsenior--4pal>

有时候你必须从一种情况中退一步来看清它的真实面目，我认为这对于软件开发人员被贴上初级、中量级和高级的标签来说尤其如此。

出于某种原因，这已经成为软件开发行业公认的一部分，如果你是一名初级开发人员，想找一份初级工作并学习相关知识，你可能最终会被忽略，因为你没有——在某些情况下——必要的三年经验！

听起来很傻，对吧？需要三年经验的初级工作？

那是因为事实如此。

然而，在招聘广告中看到这个确实有它的用处。这是对你申请的公司类型的一个非常明确的警告。在软件行业，似乎有一种越来越多的企业在玩职位头衔游戏:在你发布的任何职位描述前打上“初级”字样，你就可以拿到低得多的薪水。

可悲的事实是，这些企业将会发现留住员工变得更加困难，因为新员工会很快意识到自己的价值，并发现自己所做工作的价值与薪酬之间的差距。从长远来看，随着消息传开，他们会发现招募优秀的候选人越来越难。

找第一份工作的开发人员和已经在这个行业工作了三年的人有很大的不同。你可能会认为我在提倡引入另一个开发人员阶层的新术语，但事实恰恰相反。

从我自己的经验来看，我发现在一个开发人员被简称为开发人员的扁平结构中工作最容易获得职业发展。没有高级开发人员，没有中量级或初级开发人员…只有开发人员。

也许这是心理上的，但我发现这种结构对鼓励会议中来自新程序员的意见和输入有很大帮助，同时也减轻了那些已经做了很长时间的人的压力，他们可能觉得(这通常是不必要的)他们需要带领团队并几乎承担起管理角色。这意味着对开发人员的评判仅仅基于他们所知道的和他们的行为。

这并不是说每个人都以同样的速度学习。我鼓励人们考虑在开发者中，年份不是一个可靠的比较指标。由于构成行业的复杂技术知识的数量，拥有不可思议的聪明的长期程序员不知道关于基本技术的非常基本的东西并不罕见。类似地，有很多非常新的程序员，他们的能力远远超出你根据他们在行业中的时间所能预料的。

我确信，在行业中工作三年后，一群开发人员的编码能力可能会看起来非常不同，这是基于他们每个人获取新材料的相对速度。在这种情况下，初级、中量级和高级的标签在为工作选择最佳候选人时毫无用处。

答案？雇主应该从能力倾向和开发测试以及面试中收集技能水平的信息，而不是依赖于工作经验的年限。作为一个行业，我们应该挣脱困扰开发者求职的标签。当花时间去发现人们有什么能力，而不是他们在工作场所呆了多久，每个人都赢了！
