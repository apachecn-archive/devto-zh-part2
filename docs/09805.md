# 用户故事能让软件项目迟到吗？

> 原文:[https://dev . to/jaymeedwards/are-user-stories-making-your-software-project-late-51e 2](https://dev.to/jaymeedwards/are-user-stories-making-your-software-project-late-51e2)

在我的职业生涯中，我见过的由于误用用户故事而失败的项目比敏捷团队中通常使用的任何其他实践都多。

你可能已经知道用户故事只是 scrum 或敏捷团队中编写需求的一种简单格式。

今天我想帮助你避免一些我在那里看到的关于什么是用户故事以及它们如何帮助(或伤害)你的坏建议。

# [](#why-user-stories-were-created)为什么要创建用户故事

在敏捷开发方法出现之前，团队在构建软件产品之前编写描述软件产品设计的大型文档。

由于项目是预先设计的，*它也预先得到了资助*。

文件越详细，对成本的估计就越准确。

但是公司发现，当软件完成时，客户想要不同的东西。

敏捷社区的早期领导者想出了一个好主意，一次只构建和发布一个软件产品的一小部分。

这将让团队*改变主意，一旦项目开始，他们就要建造什么。*

因为记录你可能不会构建的东西需要很长时间，所以业界开始使用用户故事来描述需求。

这个想法是为了描述一个特性为客户提供了什么价值，它可以被优先考虑。

# [](#the-pressure-for-traditional-budgeting)传统预算的压力

但是一些领导和企业主仍然想在批准预算之前知道他们到底能得到什么。

他们强迫人们评估和研究每一个想法，因为他们太害怕在不知道他们会得到什么的情况下资助一个团队。

许多敏捷教练告诉团队，用户故事只是对话的占位符。

一旦团队开始工作，他们将与客户或产品经理交谈，了解全部细节。

问题是，一旦进行了这次谈话，许多额外的细节出现了。

**这导致估价大幅上升**。

# 对于非敏捷领导者，你需要的不仅仅是用户故事

如果你的利益相关者(产品经理，客户等。)关注成本，*在给出任何估计之前，你最好创建详细的验收标准，详细描述用户故事*。

验收标准读起来就像一个测试脚本，并准确地描述了某人如何使用软件，或编写一个自动化测试，以验证它已经完成并正常工作。

如果你没有接受标准，你对用户故事的粗略估计将会改变。

每次这样，你都会失去利益相关者的信任，因为*你看起来像是估计错了*。

因此，将设计的详细程度和项目的要求与利益相关者对不确定性的容忍度相匹配。

如果他们关注的是成本、确定性和可预测的截止日期——只写用户故事和承诺评估而没有验收标准是愚蠢的。

这就像用 *10%的必要文档来对瀑布项目进行评估*。
