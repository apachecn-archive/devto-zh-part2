# 开始开发运维工具

> 原文:[https://dev.to/thaiwood/beginning-devops-tools-4e4](https://dev.to/thaiwood/beginning-devops-tools-4e4)

尽管 DevOps 是一种文化转变，而不是一个职位，但仍然有工具可以帮助打破孤岛和与运动相关的工具。这意味着如果你申请一份“DevOps 工作”，你可能会使用某些工具。这就是为什么我在我的[书的标题中使用这个词](http://rubyfordevopsandautomation.com/)这也是初学者容易被 DevOps 吸引的地方，在工具层面上。真的，这没那么糟。“我如何使用快速行动的组织所使用的工具？“这是一个很好的问题，只要你不忽视这个想法，心智模型和文化也可以发挥巨大的作用。因此，考虑到这一点，如果你想成为“DevOps”，你应该学习哪些工具？没有确切的工具，这取决于你的组织，它的需求，需要和目标。这是 DevOps 给你的。但是这些工具确实有一些类别。就像编程语言一样，一旦你知道了一种语言，学习另一种就容易多了，因为你已经知道的许多概念将会被映射。

类别包括:

*   结构管理
*   源代码控制
*   沟通
*   CI/CD
*   监视

对于配置管理，有几个选项:

*   厨师
*   木偶
*   盐
*   Ansible

这些都需要学吗？可能不会，当然也不会一蹴而就。最好先了解一下这些工具一般是如何工作的，它们遵循什么样的范例，等等…然后选择一个并了解更多。

例如，Salt 遵循主/从(客户机/服务器)。Chef 遵循类似的模式。但是 Ansible 没有中央服务器。

对于源代码控制，尽管周围有其他人，但我只想说学习 git，除非您的组织对您有其他要求。如果你或你的团队想要一个学习和尝试的地方，你可以使用 [Try Git](http://try.github.io)

对于 CI/CD 来说，重要的是考虑您试图完成什么，但是许多工具可以用来完成类似的目标。不要让你的团队陷入困境，更重要的是建立一个基本的管道，并从那里开始改进。选项包括:

*   詹金斯
*   雄蜂
*   特拉维斯
*   Gitlab

你会注意到我也列出了交流。起初，这可能看起来不像是一个 DevOps 或 SRE 的事情，但真正的沟通是帮助您打破孤岛的东西。它还可以帮助您更快地解决事件。这可能包括聊天机器人之类的工具，甚至是你联系相关人员的系统，比如 PagerDuty。

对于监视来说，就像上面的一些工具一样，考虑您想要监视什么以及监视到什么程度比现在担心特定的工具更重要。如果你要监控某个指标，你应该能够回答一些关于它的问题。第一，这个数据告诉我什么？如果它没有讲述某种故事，很可能它不应该被监控，或者您需要调查数据对您的组织意味着什么。其次，为了确定你是否应该对这个指标保持警惕，“我希望有人(可能是我)对这个指标做什么反应？”还有“什么时候？”以及“有多快？”

如果您对某个指标没有采取任何措施，那么它就不适合发出警报。
