# 创业公司在购买和管理 AWS 计算资源时犯的 5 个错误

> 原文:[https://dev . to/veermanhas/5-错误-创业-边买边做-管理-AWS-计算-资源-4ec](https://dev.to/veermanhas/5-mistakes-startups-make-while-buying--managing-aws-computing-resources-4ec)

如今，大多数初创公司或诞生于云的企业都依赖公共云，如 AWS，而不是内部 IT 基础架构。然而，由于缺乏内部 IT 团队，他们被迫采用 DIY 方法来调配和管理 AWS 云资源，以满足其部署和运营需求。

由于没有合适的技能，他们需要学习和理解云带来的几个抽象级别，还需要自己熟悉孤立的监控工具。最重要的是，他们被需要吸收、分析和行动的大量信息淹没了。当我们学习一项新技术或工具时，依赖我们在过去的项目中已经知道或经历的东西是人类的一种自然倾向。

初创公司和新手将管理内部基础设施管理的相同类比应用到云管理，并最终咬紧牙关，支付比预期更多的钱。这是因为，他们最终假设了太多的事情(大多是错误的)。

这篇文章讨论了初创公司和新手在购买 AWS 计算资源时会犯的两个主要错误和三个常见错误:

*   不使用付款人的帐户概念:当团队创建一个帐户时，他们必须避免在一个帐户中创建他们所有的资源。最佳实践之一是创建三个 AWS 帐户——开发、QA 和生产各一个帐户。通过这样做，团队可以相应地调配资源。另一个最佳实践是创建第四个帐户，并将其作为付款人帐户。虽然团队可以向其他三个帐户提供资源，但不建议在付款人帐户中创建任何资源，而是将所有三个帐户链接到此帐户。有了这个，团队将得到一个很好的折扣，也将有资源的分离。

*   选择错误的实例模型:企业应该知道是使用随需应变还是 RI 或 spot 实例模型来启动实例。检查需求并进行相应的计算。

    *   如果您不知道服务器需要运行多长时间，那么按需实例是最好的，因为按需实例每小时收费，并且可以根据需要随时丢弃。在这里，你可以随时使用，也可以想用多少就用多少，但是费用很高。基本上，一台服务器一年的费用是 1*24*365 美元= 8760 美元。
    *   相反，如果您知道您需要服务器多长时间，您可以选择保留实例，因为与按需相比，您可以以折扣价获得它，但是，如果您想在使用一段时间后丢弃该服务器，您无法摆脱它，因为它已经付费，云提供商不会收回它。
    *   如果您在很短的时间内需要一台服务器，或者确实想测试一些东西，那么 Spot 实例是理想的选择，因为其价格通常是“随需应变服务器”成本的十分之一和“预订服务器”成本的五分之一。然而，问题是你需要出价购买，比如每小时 0.1 美元。因此，在有人以更高的价格出价之前，该服务器是可用的。在这种情况下，在它消失之前你有两分钟的时间采取行动。
*   不使用 CloudWatch 警报监控 AWS 计算成本:一些 AWS 新手用户还不知道 CloudWatch 警报功能。他们仍然认为需要手动查看 AWS cost explorer 控制台。使用 CloudWatch 服务，用户可以为特定实例启用警报，以快速了解成本何时达到阈值点。这里需要注意的一点是，用户必须知道与 CloudWatch 服务相关的所有限制，比如它处理大量定制日志数据的能力，近乎实时地对事件做出反应等。

*   没有灾难恢复(DR)计划:灾难恢复在业务连续性中起着至关重要的作用。即使来自 AWS 的中断有时也会影响企业的服务，而不仅仅是灾难或其他灾难。让我们回忆一下 2017 年 2 月的 S3 服务中断，当时大约有 148，213 个网站和 121，761 个唯一的域宕机。解决这种中断的最佳方案之一是为 DR 购买实例。没有多少初创公司和新手意识到跨区域复制和设计良好的 AWS CloudFormation 模板可以在为开发和运营需求持续提供计算资源方面发挥重要作用。此外，在所有情况下，使用地区级灾难恢复比 AZ 级灾难恢复更有用。

*   没有使用良好的云可见性工具，仅仅依靠可信顾问:首先，许多初创公司使用可信顾问功能来发现他们的 AWS 帐户中的漏洞，降低成本，提高性能，并提高安全性。但是，该功能并不提供对资源使用情况和云优化所需的其他指标的完全可见性。即使 Trusted Advisor 提供实时指导来帮助按照 AWS 最佳实践调配资源，它也不允许用户使用上下文洞察和操作功能来监控、分析、修复和调整 AWS 云。像 TotalCloud 这样的云可见性工具，可以帮助用户从单个窗格视图中了解完整的云使用情况，只会有所帮助。
