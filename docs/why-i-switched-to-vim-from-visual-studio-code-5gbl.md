# 为什么我从 Visual Studio 代码切换到 Vim

> 原文：<https://dev.to/ayoisaiah/why-i-switched-to-vim-from-visual-studio-code-5gbl>

大约六个月前，我开始使用 Vim 作为我的主要编辑器，我可以说这是一次值得的经历，因为它促使我在编辑文本时以更有效的方式思考。

这篇文章解释了我转向 Vim 的理由，希望它能帮助你决定投资学习 Vim 对你来说是否合理。

## 我的历史与文字编辑

自从我三年前开始学习编码以来，我已经尝试了许多文本编辑器，并且在很大程度上我对每一个都有很好的体验。

我想写一点关于这些经历的东西，这样你就能明白我的想法，并最终理解我是如何使用 Vim 的。

### 崇高的文字

[![Sublime Text Editor on a Macbook](../Images/c52b58c650fe6c8cfbf8419593723a08.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--AyqGMwBk--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://freshman.tech/assets/images/vim/sublime.jpg)

我用的第一个编辑器是[崇高文字 3](https://www.sublimetext.com/) 。我之所以选择它，是因为当时它在 web 开发人员中非常流行，大多数教程都推荐它作为初学者友好的选择。

在这一点上，我只是刚刚开始学习编码，所以我只利用了它的最基本的功能，尽管我确实学习了如何安装插件来增加功能。Emmet 是我最喜欢的软件之一，因为它帮助我更快地编写 HTML，这太棒了。

我用 Sublime 用了一年多，甚至完成了 freeCodeCamp 的前端证书，在里面做了我所有的项目。老实说，我对 Sublime 没有任何真正的问题，除了不时出现的弹出窗口鼓励我购买该产品的完整版本。

可以肯定的是，免费版本在其功能集上是不受限制的，但你必须处理弹出窗口，它有时会很烦人。因为我没有 80 美元来购买文本编辑器，所以我开始寻找替代方案。

### Vim 第一次进站

大约在这个时期，我决定第一次尝试 Vim，因为当你看程序员的文本编辑器时，它总是谈话的一部分，并且受到许多人的青睐。

Vim 肯定更难开始，虽然我能够学习基础知识，但我不明白为什么每件事都要这么复杂。

我的工作效率很低，所以在摆弄了一段时间后，我放弃了 Vim，并决定看看在界面和工作流程方面更接近 Sublime 的 Atom。

### Atom

[![Atom Text Editor on a Macbook](../Images/b0f0a38ad4f1724040c19d4d33a468cd.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--OA7PZRjH--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://freshman.tech/assets/images/vim/atom.jpg)

从 Sublime 切换到 Atom 并不困难，因为 Atom 很大程度上受到 Sublime 的启发，所以它们有很多共同点，这让我很容易开始使用它。

我使用的所有插件或等效插件都可以在 Atom 上获得，所以我从一开始就做得很好。因此，我能够毫不费力地完成我的工作。

然而，基于电子构建的 Atom 对我的 RAM 和 CPU 产生了巨大的影响。由于当时我只有 4GB 的内存，这对我来说很重要，需要在那个部门进行升级，我在我的计算机上安装了额外的 4GB 内存。

对于 Atom，性能从一开始就没有 Sublime 好。我对此并不感到惊讶，因为基本上我尝试过的所有其他电子应用程序都有或多或少的相同问题。我的电脑启动速度很慢，但一旦加载完毕，情况还不算太糟。

随着时间的推移，尽管定期收到更新，但我对这方面缺乏改进感到沮丧。所以我再次寻求改变。

### Visual Studio 代码

大约在这个时候， [Visual Studio 代码](https://code.visualstudio.com/)在编程论坛上引起了很多讨论，并且在各种博客上有很多好的报道。由于它也是基于电子构建的，我并不指望它在性能上与 Atom 有太大的区别，但我还是试了一下。

一开始我并没有被打动。它似乎没有提供任何我没有从 Atom 获得的东西，性能也没有明显提高。所以我卸载了它，继续使用 Atom。

然而，随着每个月的发布，会有一些炒作，我会再次下载并试用它。相比之下，当 Atom 陷入困境，尤其是在性能方面时，情况似乎总是会变得更好。

最终，VS 代码在特性集和性能上变得明显优越，所以我开始全职转向它。此时，我从电脑上删除了 Atom，VS 代码占据了主导地位。

从 Atom 切换到 VS 代码并不像从 Sublime 切换到 Atom 那样无缝。这部分是因为界面有点不同，我花了一些时间来适应。

[![A package.json file open in Visual Studio Code](../Images/efb8a8fb934b497d04e81fafed7b8b8c.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--U1Dfe1xC--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://freshman.tech/assets/images/vim/vscode.jpg)

默认的键绑定也与我习惯的 Atom 有很大不同。这个问题可以通过安装一个扩展来轻松解决，这个扩展将键绑定重新映射到 Atom 的键绑定，从而稍微减轻了我的过渡。

对于编写大量 JavaScript 和 TypeScript 的开发人员来说，VS Code 可能是最好的文本编辑器，因为它对这两种语言都有全面的现成支持。但是，也很好地支持其他语言。举例来说，在一些插件的帮助下，使用 Rust 真的很容易，我去年曾经尝试过。

在我使用 VS 代码的几个月里，我没有太多抱怨。虽然性能没有 Sublime 好，但是比 Atom 好很多，而且还在继续提高。

尽管如此，我仍然着眼于学习和使用 Vim。每当我在 Reddit 或 Hacker News 这样的讨论论坛上读到关于文本编辑器的内容时，一定会有人对 Vim 大加赞赏，并提到它可以提高个人的工作效率。

当我的一个朋友[转到 Vim](https://lanre.wtf/blog/2017/08/08/on-vim/) 后不久又唱同一首歌时，我就明白了。我只想知道维姆的优点。

## 第二次尝试 Vim

由于当时我没有太多进展，还没有发布[恒星照片](https://stellarapp.photos/)，我决定切换到 Vim 冷火鸡，这意味着在这个过渡过程中我没有在我的计算机上保留任何其他文本编辑器，以迫使自己只使用 Vim。

这当然没有我第一次尝试时那么难。我用`vimtutor`来提醒自己基本的命令，并从那里开始。

Vim 非常简单，没有我在其他文本编辑器中认为理所当然的所有特性，但是在大多数情况下，我可以使用插件添加我需要的功能。

也就是说，这在某种程度上成了问题，因为它导致我安装了比我实际需要的更多的东西，并且我的`.vimrc`文件被填充了许多我不理解的东西。

因此，我后退一步，不再试图让 Vim 像我用过的其他文本编辑器一样工作，而是学习它的独特之处。我很快意识到，没有插件，Vim 实际上可以自己做很多事情。

在必要的地方，我仍然可以添加插件，但是，由于没有使用它们来满足我的每一个需求，我被迫学习 Vim 的做事方式，在大多数情况下，这比我习惯的方式要好。

只用键盘工作真的让人大开眼界，不接触鼠标能有多快。只需几个笔画就可以完成所有事情，由于 Vim 的模态特性，每个组合键在每种模式下都有不同的含义。乍一想，这听起来可能很复杂，但是一旦你习惯了，它实际上是非常直观的。

我已经习惯了在其他地方只使用键盘，也喜欢使用 [jumpapp](https://github.com/mkropat/jumpapp) 在应用程序之间切换，使用[萨卡人键](https://github.com/lusakasa/saka-key)在浏览器中快速导航。

我喜欢 Vim 的一个原因是，使用 dot 命令或宏可以多次重复每个更改，这确实节省了我编辑代码的大量时间。这可能只是在这里或那里节省的几秒钟，但它累积起来。

与其他编辑器相比，我发现 Vim 最糟糕的一点是颜色和字体质量，但我怀疑这与我的终端模拟器(我使用 Terminator)有更大的关系。

我知道让 Vim 在终端上看起来真的很好是可能的，因为我已经在[r/UNIX born](https://www.reddit.com/r/unixporn)上看到了其他设置的截图，但是我还不能创造出我完全满意的东西，所以我目前正在探索 Gvim (Vim GUI)所能提供的东西。

## 总结起来

在花了六个月的时间使用 Vim 作为我的主要编辑器后，我可以说这绝对是一次有价值的经历，我不认为自己会很快切换到其他编辑器。

现在，我希望提高我对编辑器的掌握，并学习一些高级流程以变得更加高效。为此，我目前正在阅读德鲁·尼尔的《实用 Vim 》,这似乎是这方面最好的书籍之一。

如果您在当前的编辑器中已经有了一个高效的设置，我认为没有必要像我一样突然切换到 Vim。根据你的工作情况，这可能不太现实。

尽管如此，我认为学习 Vim 编辑文本的方式仍然是有用的，并且您不需要使用 Vim 本身来这样做。大多数编辑器都有一个模拟 Vim 键绑定的插件，这样你就可以很容易地体验 Vim 的方式，而不用扔掉你现有的工作流程。

这里有[崇高](https://github.com/lunixbochs/ActualVim)、[原子](https://github.com/t9md/atom-vim-mode-plus)、 [VS 代号](https://github.com/VSCodeVim/Vim)的。

我会继续在这个博客上分享我所学到的东西，所以如果你对 Vim 感兴趣，不要害怕尝试它。你可能会喜欢它。
