# JavaScript 标准化 Jory Burson 访谈

> 原文：<https://dev.to/remotesynth/javascript-standardization---an-interview-with-jory-burson-i0o>

跑步活动的一大好处是，你会不时有机会邀请你欣赏的人来演讲。对我来说，Jory Burson 就属于这种情况，他将是今年 10 月 25 日至 26 日在波士顿举行的[jsmobilecoff](https://jsmobileconf.com/)第二天的主题演讲人之一。

Jory 是波士顿著名的网络平台咨询公司 Bocoup 的标准联络员和前首席运营官。我第一次有机会见到 Jory 是在大约七年前，当时我参加了 Bocoup 的一个培训课程。这些年来，随着我对她的了解，我看到她在 Bocoup 和 Bocoup 的宣传工作中担任领导职务，成为 Ecma International、JS Foundation、W3C 和 MDN 产品咨询委员会的代表。

Jory 将在[jsmobilecof](https://jsmobileconf.com/)上[讨论 JavaScript 在 Ecma TC39](https://jsmobileconf.com/schedule/keynote-day2-1.html) 中的标准化工作，所以我认为这是一个讨论她的背景、她如何参与 web 标准以及她在 TC39 中参与的一些工作的好机会。

问:你有一点独特的背景。你的简历上说，你曾在俄克拉荷马州从事过教授媒体知识、管理餐馆甚至养牛等工作。这种背景是如何让你加入 Bocoup 并关注开发者权益和开源的？

考虑到我的角色，我确实有一个奇怪的背景。很多从事 web 开发的人都是这样，我认为这是一件非常好的事情。我们正在为许多不同的人开发工具、产品和服务，从不同的经历、视角和思维来应对这些挑战，只会让体验变得更好。对于新的开发者，我想说你永远不知道你的生活经验何时何地会派上用场——请不要在你前进的道路上抹去或最小化你是谁的那一部分！

我在 4-H & FFA 的成长过程中非常活跃——我展示了猪、牛、马，并且非常喜欢它。我交了很多来自各地的朋友；我们互相帮助照顾动物，一起学习，一起比赛。这一经历无疑是我渴望成为一个社区的一部分并为其做出贡献的根源。

大约从 1997 年开始，有几个夏天，我会在正午时分来到这里，上网玩这个关于展示马匹的游戏。你必须制作一个关于这些虚构的马的网页，人们在上面投票。这真的很傻，但我喜欢它。你的网页看起来越好，你能更好地讲述这些假马的故事，你就做得越好。

我认为我们中的许多人来到这个领域是因为一些其他的激情或兴趣将我们带到这里，随着时间的推移，你会学会将平台视为一种工具，而不是一种娱乐设备。我不知道我是什么时候意识到这一点的，但在新闻学院时，我开始思考我们的通信媒介在多大程度上直接或间接地控制着我们的经历。我希望能够颠覆这一点，我希望能够表达我自己的故事，但我对这一切是如何运作的理解不够。于是我上了研究生，学的是传播学。

我的论文工作是关于媒介素养，也就是我们如何生产、消费、获取和分析信息和媒介。在那次研究过程中我学到的东西在此后的几年里给了我很大的帮助，尤其是在把实践和理论联系起来的时候。当我成为一名全职教师时，我开始为本科生收集这些想法，帮助他们理解和处理网络和其他媒体的限制。我的主要课程是电子通讯——学生们必须制作一个网页和博客，报道不同类型的故事，并将所有媒体嵌入他们的网站。

2011 年，我的合伙人在波士顿的一家初创公司找到了一份工作，所以我们从俄克拉荷马州开始了大搬家。我加入 Bocoup 是为了帮助运营和发展开发人员培训业务——我的课堂经验和非传统背景最终在文化和角色上都非常适合。Bocoup 一点也没有被它吓跑——他们真的欣赏和赞美不同的背景，并鼓励你不断探索你感兴趣的东西。所以我今天的成就也要感谢他们。

问:你在 jsMobileConf 上的演讲集中在围绕 JavaScript 的标准化问题上，这个问题似乎有一段漫长而复杂的历史，早年被微软和网景的政治所主导。值得注意的是，从 1999 年到 2009 年，发布的标准之间有 10 年的差距。你认为这段复杂的历史会如何影响语言标准的现状？政治对这一过程的影响仍然如此之大吗？

历史绝对重要。这种语言最初的几个版本是关于直接的市场竞争——Netscape/Sun 希望发布一个 JavaScript 标准作为对微软的防御策略。微软希望通过推动 VBScript 和 JScript 来保持其主导地位——他们的加入是为了将一些东西排除在语言之外。ES1-ES3 编纂了 JavaScript 和 JScript 之间预先存在的共享位，但许多功能和语法问题仍然存在争议。

如今，市场已经发生了变化，参与 TC39 的公司对该语言的商业兴趣与当时不同。他们可能仍然是竞争对手，但这不是他们真正直接竞争的地方，他们在这里帮助保护和投资于支持他们实际竞争产品的网络平台。

第一个 JavaScript 标准在不到一年的时间里就发布了，对于一个共识标准来说，这是非常快的。该标准的下两个版本也很快发布了。然后，我们进入了两个漫长的缓慢或毫无进展的时期。当时委员会发生了很多变动，很多开发人员开始转而使用 ActionScript 和 Silverlight。

今天，委员会已经承诺每年发布一次，因为我们需要并且想要响应开发者的需求和不断变化的网络环境。就开发方式而言，它更像是一个开源项目。另一方面，对于开发人员来说，这是一个相当快的变化速度，我们知道在每一个版本中，你都需要保持 100%的掌控，这种感觉是无法抗拒的。所以需要找到一个平衡点。

任何人类的合作努力都有政治因素在起作用。今天的 TC39 政治不同了。我们正在应对文化变化、政策更新、开放流程——这些具有挑战性的人力资源也对规范产生了巨大影响。关于特性仍有争论，但保护语法，使它更安全，更容易理解，这符合每个人的利益，在这方面有很多一致之处。

**问:从 ES6(现在称为 ES2015)的大规模更新开始，这个过程变成了一个更加一致的每年规格更新的系统。你能分享一下新流程是如何运作的吗？过程的变化会影响对新语言特性的思考吗？**

这个过程实际上是大多数开源项目和开放标准工作之间的最大区别之一，我认为这非常好。

TC39 目前每两个月召开一次面对面的会议。在整个一年中，代表和社区成员致力于不同的专题提案或问题，并在会议期间介绍他们的进展、引导讨论、征求反馈等。如果合适，他们可能会要求将他们的提案升级到下一阶段(有 4 个阶段)。如果没有人反对，这项提议就可以通过了。一旦提案进入第四阶段，这意味着它将被合并并作为规范下一个正式版本的一部分发布。在此之前，整个事情仍然是一个草案规格。

在 11 月的会议上，我们的特色是冻结。没有新的东西进来。在正式成为标准的一部分之前，属于第 4 阶段提案的任何内容。然后编辑们开始润色规范文本，在一月份，我们有一个 90 天的规范文本冻结期，以获得知识产权许可。6 月，Ecma 大会(所有代表的团体，而不仅仅是 TC39 的参与者)投票接受该标准作为“官方”Ecma 标准。因此，今年 11 月，代表们将投票接受新规范的第 4 阶段提案，这将成为 ES2019。

然后这个过程重新开始！

该过程确实会影响新功能。首先，因为这是一个年度循环，如果你没有得到什么东西，后果会更小——总会有下一年。另一方面，这可能会在年底导致很多活动和压力，以完成一些事情。另一件事，这可能不是一个很受欢迎的观点:委员会真的必须能够说“不”——我们在语言中保留的东西可能比加入的东西更重要。我认为这个过程让人们在说不的时候心理上更加安全，因为我们更容易把投反对票当成“下一个选项”或者“不是不，但不是现在。”

问:我认为任何积极参与标准团体的人都有非常固执己见的倾向。团队如何确保每个人都被倾听并有机会参与？

标准工作的性质需要意见。不需要的是那些观点被牢牢地抓住，以至于不能被改变。能够宽松地持有强烈意见并进行建设性辩论的人是标准工作的良好候选人。

我们面临的挑战之一是，小组的规模已经大幅增长，我们有许多人符合这种类型，但可能没有足够的时间听取他们对某个特定主题的意见。在会议期间，我们使用 Brian Terlson 自制的工具来帮助对给定主题的问题进行排队。因此，它有助于防止人们互相谈论和打断。

我们还大量使用 GitHub 进行会议间的交流。有一个内部反射器，用于元笔记、会议规划等等，全年都非常活跃。
