# 我认为对于软件架构师和技术领导者来说，这是一个成功的三角形

> 原文：<https://dev.to/ggonchar/what-i-believe-is-a-triangle-of-success-for-software-architects-and-technical-leads-2gek>

近年来，我一直在多个组织中担任软件架构师，包括科技公司和高增长初创公司。既是个人贡献者又是架构团队的管理者。我意识到有三个最重要的原则帮助我做好日常工作。我总是试图让所有的决策制定者都参与进来，试图理解业务和成本影响，并花时间编码。还有一些我认为重要的原则，但这三条让我感到惊讶。它们是我个人的成功三角(感谢 HBO 给了我这个名字灵感)。

如果你不相信为什么需要架构师或技术领导，你可能想先看看我的另一个故事——[软件架构师和自主团队](https://ebaytech.berlin/software-architects-and-autonomous-teams-328138202df1)。

# 让你的船员分担任务

[![http://time.com/5304209/blanchett-hathaway-kaling-oceans-8/](../Images/436d8975827ce92c70bad7a971640ad0.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--oRzyIp3E--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://imagesvc.timeincapp.com/v3/mm/image%3Furl%3Dhttps%253A%252F%252Ftimedotcom.files.wordpress.com%252F2018%252F06%252Foceans-8-crime-caper-movie.jpg%26w%3D1600%26q%3D70)

很容易陷入这样一个陷阱，即在你的决策中只需要说服管理层就足够了，或者只需要作为架构师建立一个规则。如今，我们有独立的团队使用隔离的微服务，团队必须平衡相互冲突的优先级。抄近路或走最短的路很容易。如果你不能说服人们为什么某个决定是重要的，他们可能根本没有足够的时间去遵循它。确保每个人都理解任务，并有高度的积极性。之后，通过亲身参与，成为使命的一部分。

### 让贡献者参与决策

在将任何想法称之为决策之前，先将它展示给直接参与其中的人。首先，你得到了一个有价值的反馈，其次，你确保这个决定被分享和理解。在@ebaytechberlin，我们发现以拉请求的形式呈现的架构决策记录效果很好。但是在提出拉取请求之前，您还应该采取其他步骤。

### 首先在白板上讨论技术观点

我发现写得很好的提议很难改变。一旦任何人对某件事投入了巨大的努力，情感依恋就开始发挥作用。在这种情况下，你可以偏向最初的提议。最好先在白板上展示你的想法，并讨论其他人的想法。这允许进行更多的讨论，并通过朝着正确的方向共同行动来即时采纳提案。该工具可以不必是白板，而是例如拉请求。想法是最初的提议应该支持变更的灵活性。

### 确保你记录了缺点

我们都知道没有灵丹妙药。几乎任何技术决策都是一种权衡。通常，有逻辑的人不同意决策，因为他们不以同样的方式看待权衡。确保你解释了为什么你当前项目的某些优势超过了劣势。在架构决策记录中包括权衡、结果和缺点，并记录为什么它们没有解决方案的优点重要，这是一个好主意。

### 向所有受影响的利益相关者展示决策

一旦做出决定，工作就没有结束。这只是旅程的开始。通过向其他利益相关者解释团队采取某个技术方向的原因，您增加了以正确的方式平衡优先级以做出决策的机会。您还应该确保不仅工程师，而且产品经理和工程经理都理解技术愿景。确保你的技术目标有对非技术人员有意义的名字和价值。“重构事件收集器”是一个书呆子的名字，没有一个产品经理会理解它的价值。“移除不合理的事件跟踪代理以更快地添加新事件的处理”是一个冗长但已经很有意义的名字。

# 了解现金流量

[![https://www.rogerebert.com/reviews/the-wolf-of-wall-street-2013](../Images/01d88dddf8f455e38379e46729e2243d.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--9G8igR4i--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://static.rogerebert.com/uploads/review/primary_image/reviews/the-wolf-of-wall-street-2013/hero_WolfOfWallStreet-2013-1.jpg)

每一个成功的工程组织都需要财务上的成功，才能继续其使命。这应该是技术领导的主要关注点之一。

### 不同的解决方案花费多少美元

了解某些云或硬件服务的成本，以及您的组织的平均工程小时成本。与雇佣一个有经验的顾问相比，你自己学习技术的成本有多少。好的架构也是一个以长期最优成本实现业务目标的架构。如果不对你的决定给出粗略的数字，就很难做出正确的决定。

### 参与产品需求梳理

作为一名架构师，产品细化是我唯一经常参加的团队会议。作为一个技术幻想家，你需要非常清楚你的未来系统需要如何工作。当产品经理介绍新功能时在场，捕捉某些产品决策的所有细微差别，我觉得非常有用。但是请记住，如果你是一个架构师或者其他不属于团队的领导，你应该在团队会议中做一个倾听者。这种出勤的主要目的是学习业务需求。只有当一个重要的决定必须做出，并且它没有朝着你认为是最佳的方向发展时，才积极参与。

### 花些时间坐在顾客椅上

与运营部门或客户支持团队一起度过一周可能是个不错的主意。我不是说你应该停止工作一周，然后坐等客户来电。但是，如果你不直接与 UX 或前端打交道，就很容易对你的产品如何使用和操作做出错误的假设。

### 知道度量标准

了解你的产品的另一个补充方法是从数据指标中学习。谁是你的客户，你有多少客户。人们在你的平台上最常用的功能是什么？某个功能应该处理多少流量应该由数据决定。您可以从产品管理中获得许多想法，以学习该领域的最佳实践。

### 知道战略目标

了解你企业的战略重点。没有人能预测未来，你也不应该建造不需要的东西。但它有助于了解船航行的方向。你在市场上的竞争优势是什么，你需要保持的关键优势是什么。这会让你知道在哪里投入更多的时间，什么对生意更重要。

# 从内部了解你的车

[![https://www.filmofilia.com/fast-and-furious-6-photos-58-149253/](../Images/f129afb93fb06360653929699d056a77.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--WvLMMRWn--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/ts3taid3mo6iivoj76wz.png)

5 年前是 JavaScrip 专家的人今天不会自动成为专家。“我有几十年的经验”在今天的技术领域不是一个论点。事情确实在变，你需要跟上时代。此外，在实践中展示产品和代码库知识的人在提出决策时会赢得更多信任。

### 用编码保持平衡

没有一个理想的时间你应该花多少时间去实践。这很大程度上取决于你在这个项目上呆了多长时间，你对这个产品了解有多深，以及你对这个技术栈了解有多深。主要思想是编码不再是技术领导的主要职责。如果你编码太多，你将没有时间做你最重要的工作。

作为一名技术领导者，你可以通过做出正确的技术决策，而不是通过你自己交付的代码，对组织产生更大的影响
。同时，你自己也需要深入了解你的代码库和产品。您需要尽可能多地编码，以保持这些冲突优先级之间的平衡。

### 找时间编码

作为一名技术领导者，你会发现你在会议上花了太多时间，工作时间变得支离破碎，你不再能足够集中精力，代码贡献成为一个挑战。

提示#1 组织你的工作。根据经验，当时间窗口少于一小时时，我甚至不会尝试编码。尽管我试图将没有会议的一天完全用于实际工作，而在会议较少的日子里，我会审查拉动式请求、创建文档或与人交谈。

技巧#2 在“低”建筑季节做出更大的贡献。每个项目在开始时都有更多的设计决策。你会高度参与其中，动手机会会少一些。只要接受这一点，只要项目在发展，你就会有更多的时间做出更大更重要的贡献。

### 慎重选择你的编码任务

如果你不属于一个单独的工程团队，那么如何对产品代码做出贡献可能并不明显。

提示#1 作为一个想法，在最具挑战性或最不愉快的工作中迈出第一步。你不应该只拿糖果。你需要把手弄脏，以身作则。如果你在努力推进痛苦的重构——你也应该是做这件事的人。

提示#2 你应该为产品代码做贡献，而不仅仅是做原型。只是到团队中来说“嗨，我能和你一起写一个 sprint 代码吗”会有用，但不是一个最佳的主意。对于团队来说，这看起来像是老大哥在看着他们。与其这样，不如找一个真正需要帮助的团队。这可能是一个多成员度假的团队，或者只是一个还不了解情况的新团队。

### 待得深

恭喜你，现在你是技术主管了。你将不再整天编码，因为你有不同的职责。你应该至少在几个学科上保持深入。自从我不是一名软件工程师以来，已经有将近 5 年的时间了，我也没有花一整天的时间写代码。但是我仍然可以向我最有经验的同事暗示一些棘手的 Java 错误的解决方案。重要的是不要成为只有广博知识的“万事通”。我很难想象自己会成为一名优秀的架构师，在不深谙技术的情况下提出最佳解决方案。

作为技术领导或架构师，有很多方法可以产生影响。不要害怕花时间学习你的产品、代码库和解释你的技术愿景。你需要做的最重要的工作是做出正确的决定，并把它们变成现实。实际上，你 99%的工作只是通过学习和贡献来准备技术愿景。
