# 在技术面试中开始做的 10 件事

> 原文：<https://dev.to/amandasopkin/10-things-to-start-doing-in-technical-interviews-1d6a>

有很多关于技术面试的建议，所以我将保持这一个简短。几年前我开始面试软件工程的候选人，我发现我希望更多的候选人在编码风格的技术面试中做几件事。以下是在技术面试中你应该开始做的十件事(如果你还没有开始做的话):

1.  用一句话(或更少)重新表述问题描述，以确保你理解。例如，“所以你要我翻转位，切换它，并扭转它？”

2.  如果还没有提供，请确保您有一个示例输入和一个示例输出。为什么？我让候选人假设他们应该在我的问题中返回 true 或 false 布尔值，而不是索引，这会影响他们如何编写解决方案。澄清这一细节有助于你在回答中取得成功。

3.  一开始就澄清边缘案例。我可能会说还不值得编写这部分代码，但是提到它们是有好处的。

4.  接受评论/问题，不要置之不理。候选人通常对建议或问题反应不佳。可能是因为他们认为他们不应该需要它们，或者把一个建议看作是他们做得不对的信号。如果你接受我的建议并给予考虑，我可能甚至不会记得我必须提供它。如果你反应古怪或忽视它，我会记住这一点，这将损害你的机会。

5.  开始编码前先问一下。如果我不在乎的话，你不会想用次优的方法来浪费时间。说“要我把这个编码起来吗？”因为这经常给我一个机会提出“你能想到一个更好的解决方案吗？”

6.  永远要提到强力解决方案和其他替代方案，即使它们对你来说似乎是超级基本的。这表明你已经考虑过这个问题，这是一个很好的方法来组织对替代方案的讨论。即使有一个显而易见的选择，提及你评估了其他选项也显示了超出仅仅解决问题的技术知识水平。

7.  问你真正想要答案的问题。有时我会收到一些问题，我认为这些问题只是为了让候选人看起来更好。真诚的兴趣在面试中肯定看起来不错，但是跳过“你在 X 公司面临的最大挑战是什么？”如果你不在乎我的回答。此外，关于小吃的问题最好在现场解决。

8.  提供多个测试案例。如果你有时间的话。

9.  试着根据问题的上下文来命名，而不是根据它的类型。例如，使用像“nums”这样的名称来引用数字数组，而不是像“arr”这样的名称

10.  尊重面试官的时间。如果你面试迟到了，我要求重新安排时间，那就选择这个选项。期待你的面试官因为你迟到或想要多 5 分钟而让时间变长会让你表现不佳——通常我只预定了一个小时的房间，并在我们结束后有一个完整的时间表。

面试愉快！
