# 缓慢释放导致的快速死亡

*   嘿，我们能调整一下那个按钮上的复制品吗？
*   是的，完成了。
*   太好了！当它上线的时候让我知道。
*   嗯，大概 2-3 个月后。我们将在下一个版本中批量发布。
*   ...
*   🙃

这种类型的对话在很多场合都会发生。不管你是在一个大团队，一个小团队，一家初创公司还是一家老牌公司。一次又一次，你会在会议上听到这种说法，或者在一个松散的渠道读到这种说法。我们不遗余力地采用敏捷实践，拥抱 Scrum，做日常站立练习，并在更短的开发周期内工作。但是部署到生产中仍然是艺术多于科学，小的变化可能需要很长时间才能到达客户手中。

通常情况下，一个问题会很快从开发转移到暂存，然后它会在那里等待一会儿，再等一会儿，就会被其他票证挤在一起，直到砰的一声！让我们部署到生产中吧！当团队焦急地监控日志，以确保每个人都没事时，雪崩般的变化流向客户。

然后，一天过去了。没有人抱怨。至少，不比平常多。太好了。我们再次松了口气，可以开始下一个版本的工作了。

这是缓慢释放的死亡。我们经常关注中断生产的风险增加，但要考虑的远不止这些。

## 整容会议的兴起

当你不能快速出货时，一个微妙但昂贵的事情发生了，那就是小的修改(文本、颜色、按钮的位置)现在需要更多的讨论。这并不是说改变本身很关键，而是一旦你部署了它，可能需要几个星期或几个月才能更新。因此，你需要花更多的时间预先制作它，以确保它尽可能接近完美。如果发布一个改变只花了你几天的时间，你就不会犹豫了，只要尝试一种方法，然后在下一次冲刺中改进它。但是发布周期的缓慢增加了每次提交的成本。

## 在干草堆里找虫子

大型发行是一个悖论。一方面，一次提供这么多新功能感觉很好。但另一方面，当错误发生时，确定错误的原因会困难得多。您的团队必须找出众多变更集中的哪一个引起了问题；如果他们不得不切换上下文来回想他们几周或几个月前写的代码，这项任务会变得更加困难。即使一切顺利，您的客户可能仍然会被应用程序中一次交付的大量新概念所淹没。

## 与市场脱节

最终，当你的发布周期缓慢时，你会失去人们想要的东西。敏捷的全部目的是与你的客户建立一个快速的反馈回路，这样你就可以尽早纠正这个过程。如今，由于创新的速度加快，人们的需求和行为正在快速演变，这一点变得更加重要。但是，当你花了几个星期来运输，你开始支付昂贵的知识费用，在你的市场竞争。交付速度更快的同行正在做出明智的决策，而您必须猜测如何让您的产品变得更好。

## 你发布的速度是你团队的脉搏

不可否认，每个软件团队都是不同的，即使是在同一个组织内。当然，还有技术方面，但文化也是你如何向客户交付改进的决定性因素。

即使他们的运作方式不同，仍然有办法比较这些团队。你可以看看开发周期的速度，了解他们做得如何。不管是什么原因，缓慢的发布节奏对企业来说都是痛苦的。如果控制不好，太快也是有害的。

这就是为什么我们要建造战斗小队。帮助团队掌握他们开发周期的脉搏，了解他们项目的健康状况。让你的顾客开心不应该有那么大的压力。
