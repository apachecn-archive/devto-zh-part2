# 抵制“快速解决方案”

> 原文:[https://dev.to/nholden/resist-the-quick-fix-4 fpl](https://dev.to/nholden/resist-the-quick-fix--4fpl)

我经常遇到表面上看起来像是“快速解决方案”的问题也许这是一个可以通过调整一个变量来解决的 bug，或者是一个可以通过改变一个条件分支来添加的新特性。

当我注意到快速修复的机会时，我很自然地开始考虑实现。打开我的文本编辑器进行修改是很诱人的。我喜欢编码；这是我成为一名开发人员的重要原因。发货代码让我觉得很有生产力。因为我几乎总是独自工作，或者在一个有很多信任和自主权的小团队中工作，所以没有什么能阻止我创建一个新的分支机构并继续工作。

不过，大多数时候，抵制住编写快速解决方案的诱惑，转而编写一个公共待办事项，会产生更好的结果。

待办事项会花费更多的时间，但最终会减少工作量。有很多次，我分享了一个待办事项，并了解到我团队中的其他人也在思考类似的问题，同时解决这两个问题会更快，并带来更好的解决方案。其他时候，我为一些我认为是问题的事情创建了一个待办事项，但是通过讨论这个待办事项，我明白了根本没有必要改变。

创建待办事项给问题带来的可见性不仅仅是摆脱额外的工作。陷入在竖井中快速解决问题的循环会让我感到混乱，别人会怀疑我是如何度过时间的。公开待办事项向我的团队展示我集中精力的地方，并允许他们在有意义的时候做出贡献。

待办事项将一个问题的背景从我的大脑中转移到一个集中的地方，这样其他人就可以在他们自己的工作中考虑这个问题。用散文对问题进行深思熟虑的解释比提交消息、注释或“自我记录”代码更容易被发现。它还可以通过让那些本来不会审查代码解决方案的人了解问题，为对话带来更多的视角。

通常一个“快速”的解决方案并不是这样。有几次，我已经开始编写一个补丁，随着时间的流逝，我发现自己深深陷入了代码库中的一个线程，以至于忘记了我最初要解决的问题。用英语解开一个问题的练习可以抖出问题的细微差别，从而防止陷入兔子洞。

待办事项也能减轻认知负荷。在处理一个问题的时候，很容易发现一些其他的问题，默认快速修复会迫使我把这些都记在脑子里。一路上注意待办事项中的问题，然后回到最初的问题，这让我保持专注。

也就是说，有时候，突然出现的问题看起来很容易快速解决。尽管我默认使用待办事项，但有时我会利用我的判断力，从钻研一些代码开始。在这些情况下，时间盒是有帮助的。如果 20 到 30 分钟后，我没有感觉离某种解决方案很近，我会停下来，创建一个待办事项，即使我打算马上回到同一解决方案上。

但是大多数时候，从一个公共的待办事项开始，而不是快速修复，对我的代码和我的同事来说结果更好。
