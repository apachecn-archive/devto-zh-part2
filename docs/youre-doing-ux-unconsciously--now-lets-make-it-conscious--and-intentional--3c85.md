# 与 UX 的初次相遇

我现在知道用户体验(UX)是关于产品或服务为用户创造的体验。但是在今年(2018 年)革命大会之前，我根本不知道 UX 是什么！

我在 RevConf 的第一次会议，我去的是克里斯·德马斯的“聚焦焦点”。他说的一件事给我留下了印象。

> **“20%的人有残疾，所以即使对于内部应用程序，可访问性也是必须的。”**

我的下一节课是约翰·阿泰德的“开发者的 UX”。有两个短语引起了我的注意。

> **“UX 债务是技术债务”**
> **“UX 是所有人的问题”**

最后，我很欣赏艾莎·布莱克的《网络无障碍:如何成为 A11y》。有一句话让我明白了一些事情。

> **“无障碍让每个人的生活更美好”**

听完这些演讲后，我不禁想到:

*   我现在正在开发一个内部 web 应用程序，可以访问吗？
*   我如何改进当前正在开发的应用程序的 UX？
*   UX 设计公司如何在我的团队中工作？

我现在准备更深入地研究 UX，并告诉我的队友我在会议上了解到的一切。

# 团队

我的团队为我们的组织开发内部 web 应用程序。我们是一个五人团队，由一个业务分析师(BA)，一个质量保证分析师(QA)，一个后端开发人员，两个前端开发人员(包括我:})组成。

我们的*业务分析师* (BA)将业务目标转化为需求。他协调关于产品的沟通，并收集最终用户的反馈。他还创建了我们的线框(他用微软的油漆做的，它们看起来棒极了！).

我们的*质量保证分析师* (QA)使用 BA 的需求文档来创建用户测试用例。在特性开发完成后，她手动运行她的测试用例，看看是否一切正常。

我们的*开发人员*利用技术实现 BA 的要求。有时由于技术限制，他们可能不得不对需求提出一些修改。

# 向团队讲述 UX

当我从会议回来时，我告诉我的团队我所知道的关于 UX 和无障碍的一切。我也让大家知道我的雄心勃勃的目标，为内部 web 应用程序创建一个设计系统。

该团队不仅同意将 UX 作为首要目标，而且他们真的很兴奋！每个人都在分享如何改善我们产品的用户体验的想法。我听到了他们以前的经历，在不同的项目和不同的公司，解决 UX 的问题。

我简直不敢相信！但这是真的！

当我第一次开始做这个项目时，我注意到视图开始在较小的笔记本电脑和设备上崩溃。后来，我和另一个前端开发人员采取主动，说服项目负责人给我们时间将 UI 重构为组件，并更新样式以支持响应式设计。

通过允许用户能够在任何尺寸的屏幕上使用应用程序，我们改善了用户体验。我们在演 UX！反过来，这也为我们的应用程序为用户提供了更多的使用机会。

因此，每当你根据用户的需求和反馈做出设计决策和改进时，你都在做 UX！

既然我们对 UX 有了更多的意识，那就让它成为有意识的吧

# 让 UX 有意识——团队努力

我们团队做的第一件事是获得专家的建议和帮助。我们的团队向组织业务方面的 UX 设计师寻求咨询，他们给了我们一些提示和可行的建议。

这个建议对我们创建项目模型的业务分析师特别有帮助。他现在更加注意确保交互的一致性和导航的直观性。他还通过改进错误信息和工具提示的措辞来确保应用程序对用户友好。

团队中的另一个前端开发人员专注于使页面上的内容更具可读性。他能够通过不显示不重要的列来简化应用程序中的表格。他还致力于通过区分重要内容的优先顺序，让所有屏幕尺寸的视觉效果看起来井井有条。

从用户的角度来看，我们需要尽快展示相关信息。我注意到启动页面会在呈现之前等待来自 API 请求的所有数据加载。然后引导我更改启动页面，使其尽可能快地呈现，然后我让 API 请求在后台加载。这改善了用户体验，允许用户快速查看大部分必要信息，而不必等待所有数据进入。

我们的质量保证分析师也有助于使 UX 有意加入我们的团队。她现在有一个“UX 改进/UI 增强”的标签因此，我们需要的任何改进都记录在质量保证记录中。通过观察我们的用户，她注意到能够浏览应用程序是至关重要的。所以跳转现在是回归测试的用户测试用例。所以，我们现在知道，如果我们打破了禁忌，我们就不能通过 QA！

# 下一步——制定 UX 战略

在我的团队中，我们通过对用户更加感同身受，让 UX 变得更有目的性。这有助于我们进行深思熟虑的 UI 改进。

实际上，UX 不仅仅是改进用户界面。这是关于在设计过程的每一个环节考虑用户的需求和反馈。这是一个客观的过程，对用户进行调查和采访，然后基于这些学习和反馈对产品进行测试和调整。

我的目标是制定一个 UX 战略，利用完整的 UX 进程。随着团队的动力和项目的变化，这种 UX 战略将是有益的。
