# 质量保证工程师的众多头衔:测试员

质量保证工程师是一个宽泛的术语，可以涵盖各种各样的角色和职责。它可以指更专业的角色，如自动化工程师或技术支持。它可能被用来描述负责 DevOps 实践的人，或者负责 Scrum Master 职责和特性测试的人。

在 Metal Toad，我受益于不断创造自己的角色，为我认为重要的事情增加责任和义务。另一方面，我必须创造我的角色。在我被聘用之前，Metal Toad 的质量保证仅仅是一个特定高风险项目的测试员角色。没有稳定的质量保证过程，没有 CI/CD，并且很少有所有权来承担责任。结果，我在日常生活中戴了很多帽子。我所做的工作包含了各种各样的角色——Scrum Master、Tester、Architect、DevOps，甚至是一个小产品所有者和 UX /设计。

那看起来像什么？当我转换角色时，我要实现什么？

我很高兴你问了！让我们从我的角色开始的地方开始，看看当我戴上我的测试者帽子时是什么样子。

### [](#tester)测试仪

作为一名测试人员，我的主要职责之一是为几个敏捷团队验收测试网站和移动应用。在项目期间，我根据验收标准审查每一张吉拉票，以确保它符合规定的客户和业务需求。验收测试的一部分是探索性测试——我通常会将功能分支合并到我们的主分支中，这样我就可以确保它与现有的代码和功能相集成，而不会引入回归或错误。我通过不同的浏览器和设备进行测试，既作为匿名用户，也使用我们创建的任何角色登录。

我这里还有一点我的 UX /设计帽。通过验收测试，我专注于我们正在创建的用户和管理员体验。它很少被作为明确的接受标准，但它是我们工作中最重要的部分之一。我的目标是给我们的客户一个简单、直观的管理体验。当我审查功能的时候，我会试着从我们客户的角度来看，他们可能没有更新和维护网站或应用的经验。一些对工程师来说显而易见的东西，对于扮演站点管理员角色的人来说，可能是——而且经常是——令人困惑或不直观的。

当我认为一张票需要返工时，我会尝试探索 bug 或回归的根本原因，或者提出具体的改进建议——我不想只是把它扔回墙上。清晰的、感同身受的交流总是很重要的，尤其是在这里。很容易把 QA 的反馈当成个人意见，所以我尽量让我的反馈更具协作性。我给出反馈或要求改变的目标和团队中的每个人一样——创造我们产品的最佳版本。

我还在每个项目的生命周期中进行可访问性测试。科技行业越来越意识到可访问性需求——以及在我们创建新产品和应用程序时满足这些需求的重要性，而不是事后的想法(或者更糟，完全被忽视)。目前，我使用一个名为 [Pa11y Dashboard](https://github.com/pa11y/pa11y-dashboard) 的产品进行我的可访问性测试，如果你感兴趣，你可以在这里查看我的设置[！仪表板让我们可以跟踪一段时间内的可访问性问题，这让我们可以向客户展示一些有意义的东西。我在 Google Sheet 中捕获可访问性错误，并使用它在吉拉创建门票，我们可以将这些门票带入我们的 sprints 中。](https://github.com/angelariggs/pa11y-setup)

偶尔，我们会遇到一个需要更深入研究的 bug 或回归。当这种情况发生时，我会花时间写一份正式的测试计划，创建并实现测试用例，并报告我的发现以及改进或未来验证的建议。在最近的一个例子中，我们的技术副总裁要求对一个一直困扰我们客户的特性给予额外的关注。我首先与客户会面，向他们保证我会优先考虑这个问题，我们最终会得到一个如预期那样运行的改进功能。我创建并执行了大约 25-30 个测试场景，写下了我的发现的摘要，并与我们的一位工程师会面，讨论改进的建议。我们回顾了我的测试结果，他重新编写了特性以符合假设和需要的功能。这个测试和特性变化以一次客户会议结束，在会上我做了一个修改后的特性的演示。我还花时间总结了探索、故障排除和更新功能背后的过程——所有这一切的预期结果之一是重新建立对 Metal Toad 和我们团队的信任和信心，我们希望确保他们感到受到了照顾。

建立信任和信心的需要是我们首先提供质量保证实践的主要原因。通过在我们的软件开发生命周期中包含 QA 实践，我们向客户表明我们认真对待我们的工作质量，并确保在每一步我们都在生产我们引以为豪的工作类型。我们向自己保证——从工程师和产品所有者一直到我们的首席执行官——我们有适当的检查来防止和捕捉错误、回归或其他不合标准的代码。我们给予*自己*信任我们的代码和过程的能力，对我们正在创造的产品有信心。
