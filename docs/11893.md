你好，
我是一名前端开发人员，我在一家 Saas 公司工作。
各种应用/项目、敏捷方法、吉拉入场券、git、团队。这是它的要点。

我有一个问题。技术上的，也许是行为上的，也许是道德上的:
当我被指派一个新的故事时，我努力保持在范围内。

我举个例子:

一个不超过 30 个 React 组件的应用程序，一个相当复杂的 Redux 存储和一些 API 调用。这不是一个庞大的应用程序。

我的故事是有条件地在列表中添加一个新的嵌套层次:现在一些条目有了子项，需要改变 UI，可能会对状态/存储造成一点混乱。

在我看来，代码应用程序很难阅读，例如，每个 react 组件都是用 redux 包装器封装的。或者状态中包含太多派生数据。或者文件结构是错误的，组件有很长的名字，它们是按类型组织的，而不是按特性，现在有很多这样的组件。
或者没有关于代码格式化的策略，我不想要 semi，我的同事想要。等等。

我没有责怪别人，我是几个月前，在我们的 React 之旅开始时，制作这个应用程序的团队的一员。

在这段时间里，我明显地改进了代码，我觉得我有责任，我只是不能接受它。事实上，它变得如此复杂，以至于我很难去解释它！

假设此刻只有我一个人在做这个应用的前端。

我要回顾一下时间，因为我开始发送带有重构的 PR:pre-commit 上的更漂亮以保持它在 git 上的一致性，按功能组织我的故事周围区域中的文件，删除嵌套的 redux HOCs，只传递 props 1，max 2 level down 和其他小重构。我也在编写最初分配给我的真实故事，但这显然需要更多的时间。

我本可以提出改变，完成我的故事，也许以后，在这个过程中实现这些改变。

做这样的“大”改变，不严格地与指定的故事相关是不对的。这可能会破坏别人的计划。只是不太好。但是我强迫我的利益相关者接受这些改变。

尽管现在代码已经处于更好的状态，我终于可以有效地使用它了。

我应该做什么？
