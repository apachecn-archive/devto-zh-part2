尽管云彻底改变了我们管理应用程序的方式，但许多公司仍然从服务器的角度来看待他们的系统，即使他们不再使用物理服务器。如果您将服务器的概念从等式中去掉，并开始将基于云的应用程序视为工作流、分布式逻辑和外部管理的数据存储，会怎么样？该报告研究了新的**无服务器**运营方法。

像许多软件趋势一样，对于什么是无服务器没有一个清晰的观点。首先，它包含两个不同但重叠的领域:

*   无服务器最初用于描述显著或完全整合了第三方云托管应用程序和服务的应用程序，以管理服务器端逻辑和状态。这些是典型的“富客户端”应用程序，例如单页面 web 应用程序或移动应用程序，它们使用云可访问数据库(例如 Parse、Firebase)、认证服务(例如 Auth0、AWS Cognito)等庞大的生态系统。这些类型的服务之前被描述为“[后端即服务](https://en.wikipedia.org/wiki/Mobile_backend_as_a_service)或**“BaaS”。**
*   无服务器也可以指服务器端逻辑仍然由应用程序开发人员编写的应用程序，但与传统架构不同，它运行在无状态的计算容器中，这些容器是事件触发的、短暂的(可能只持续一次调用)，并且完全由第三方管理。一种思考方式是“功能即服务”或**“FaaS”**。 [AWS Lambda](https://aws.amazon.com/lambda/) 是目前功能即服务平台最流行的实现之一，但还有许多其他的实现。

尽管名字如此，它实际上并不涉及没有服务器的代码运行。之所以使用“无服务器计算”这个名称，是因为拥有该系统的企业或个人不必购买、租赁或提供服务器或虚拟机来运行[后端](https://en.wikipedia.org/wiki/Front_and_back_ends)代码。

随着**无服务器**而来的是:

*   没有服务器管理(不需要管理任何形式的机器)
*   按执行付费(从不为闲置付费)
*   自动扩展(根据需求扩展)
*   作为应用程序逻辑的一个单元

**无服务器**模式鼓励将开发重点放在定义良好的业务逻辑单元上，而不考虑与如何部署或扩展该逻辑相关的过早优化决策。因此，开发的重点是单一的功能或模块，而不是具有较大表面积的服务。**无服务器**将软件开发人员从部署问题中解放出来，让他们专注于按照逻辑封装线分解应用程序。

一个典型的例子是上传到文件存储的图像，该事件调用一个无服务器函数，该函数创建一个缩略图并将该缩略图存储回文件存储，同时在 NoSQL 数据库中记录缩略图的位置。对 NoSQL 数据库的插入可以触发进一步的功能。这种缩略图创建功能仅按需运行，并且产生的唯一成本是调用该功能的次数。如果很多人上传图片，该功能将实时扩展，然后在他们完成后扩展到零。

然而，缺点是系统监控和调试变得困难，并且基于服务可用的日志记录，尤其是在利用来自其他服务的事件时会发生供应商锁定，而**无服务器**计算可能比其他服务便宜，并且部署机制和 ide 通常不准备管理无服务器功能。
