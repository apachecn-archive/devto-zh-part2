# 使用 JavaScript 为开发人员设计模式-第一部分

> 原文:[https://dev . to/olivermensahdev/design-patterns-for-developers-using-JavaScript-part-one-b3e](https://dev.to/olivermensahdev/design-patterns-for-developers-using-javascript----part-one--b3e)

大多数使用 JavaScript 的开发人员努力编写可读、可维护和可重用的代码，因为用 JavaScript 编写“坏”代码很容易。这个问题源于这样一个事实，即我们今天使用的 JavaScript 不是 Brendan Eich 在 1995 年写的 JavaScript。最初，JavaScript 是为交互式网页编写的，但今天，JavaScript 已经发展起来，因此它被用于许多平台，如服务器、移动设备、桌面等等。因为现在除了网页之外，它还被用在其他平台上，如果不小心的话，开发人员将会开始编写混乱的 JavaScript，变得很难阅读和维护。

当谈到编写优雅的、可维护的和可重用的代码时，设计模式是开发人员的工具带下所需要的，以帮助构造和组织 JavaScript 代码。每个模式由许多属性组成。到本系列第一部分结束时，您应该能够理解各种设计模式，以及如何使用它们来编写优雅的代码。

设计模式的概念来自一位名叫 Christopher Alexander 的建筑师。早在 1977 年，他写了一本名为《模式语言》的书，讲述了他在建造建筑、城镇和城市时遇到的常见模式。Alexanders 的设计模式概念让软件世界进入了 Gand of Four，Erich，Richard，Ralph 和 John 的世界，他们在一本名为 design patterns 的书中将他的概念转化为软件开发的通用模式，这本书通常被称为四人帮。根据这本书，开发人员在编写软件时会遇到一些常见的问题，那么什么模式可以帮助他们解决这些问题。比如说；创建一个服务层来操作数据，其中数据操作与控制器分离，模块模式可以帮助解决这个问题。此外，假设我们有一个复杂的对象，Facade 模式可以用来抽象复杂的对象并使其更简单。那么什么是设计模式呢？

设计模式是经过验证的概念，它提供了如何解决问题的描述或模板，可以在许多不同的情况下使用。那么，我们为什么还要关心所有这些设计模式呢？作为程序员，我们会遇到问题。一旦问题没有解决，这样的人就会想出解决问题的办法。万一有人面临类似的问题，为什么还要再解决一个问题呢？然后这个人利用之前已经解决的来解决他的问题。此外，我们有来自不同领域的不同开发人员，如来自函数式或面向对象编程语言的开发人员，设计模式为他们提供了一个公共词汇，以便他们作为一个集体进行良好的交流。

**我们开始吧？**
现在我们已经看到了开发人员编写糟糕的 JavaScript 代码是多么容易，以及编写软件时可以使用的常见解决方案。设计模式以多种形式存在。我们有处理对象新实例创建的模式。这些模式可以分为创新设计模式。此外，正如 creating 处理创建对象一样，结构设计模式处理实际对象本身的构成。最后，我们有一个行为模式，它更多地处理对象如何相互关联以及它们如何操作，而不是结构或获得某个事物的新实例

注意:第二部分将只关注创造性的设计模式。
