# [](#introduction)简介

这将是一个关于我如何看待网络发展的个人帖子。然而，我不会简单的说这个应该喜欢那个，因为我更喜欢它，而是会试着去思考背后的原因。

# [](#the-web-development-is-the-future)网络发展是未来

随着渐进式网络应用程序的引入，很明显网络应用程序将在未来取代本地应用程序。但是，一点也不。NativeScript、ReactNative、ReactVR 和大型呈现器等技术告诉我们，web 开发不仅仅是关于浏览器，相反，它可以通过任何类型的技术来呈现。

除此之外，JavaScript 语言现在可以用于服务器端编码，还有其他几个应用程序，这在语言创建时是难以想象的，比如机器人和物联网。

这让我相信 web 技术将允许我们在任何平台上开发任何类型的应用程序，无论是现在还是将来。

# [](#the-web-development-is-the-past)网络发展已成过去

然而，我们需要改变我们开发 web 应用程序的方式。我们日常使用的语言已经发展了几十年，变成了今天的样子。但是我们永远不应该忘记语言是如何被创造出来，以及它们被创造出来的原因。例如，Javascript 只用了 10 天就完成了，而且仅仅因为他们想，任何开发人员都可以粘上一些 HTML。

Web 语言并不是为任何当前的应用程序而创建的，因此，为了给 web 提供更好的开发环境，人们创建了其他一些技术和语言，比如 Typescript、JSX 和 CSS-in-JS，这只是其中的几个例子，但它们都是新技术的顶级应用。

这些年来，浏览器本身没有太大变化。当然，web 语言已经标准化，浏览器现在实现了标准化功能，新的标准允许我们开发新的功能，但最终，我们开发的是同样的语言，同样的语法，以同样的方式解释。我们正在开发一个有 25 年历史的系统，我们必须维护它。

# web 开发需要进化

我认为新的网络技术需要被创造、标准化并取代现有的技术。我们需要保留我们目前拥有的所有好的东西，从不好的东西中学习，并应用社区这些年来接受的东西。

## [](#markup-language)标记语言

##### [](#the-bad-about-html)关于 HTML 的不好

HTML 本质上没有什么不好，但是我们需要开始注意一个好的编写程序的重要性。HTML 是为了用可预测语法编写简单的元素而创建的，我认为我们已经不再有这样的语法了。有了 Angular、React 和 Vue 这样的库，以及 Web 组件这样的未来特性，我们不再是编写简单的元素，无论如何，我们有了可预测的语法。我们需要知道元素需要的属性，以及如何应用它们。

##### [](#the-good-about-html)关于 HTML 的好

HTML 非常容易学习，因此我们可以在几秒钟内开发一个框架应用程序。语法很简单，可以很好地与其他模板引擎(如 ejs)和其他语言(如 PHP)配合使用。

##### [](#what-we-could-learn)我们能学到什么

我们需要一种简单的方法来定义定制元素和独立模板。Vue 定义模板方式非常简单。我也喜欢 Angular 使用模板的方式，即使一个模板可能被许多组件使用，但大多数情况下 Angular 模板只被一个补集使用。此外，这种新语言必须用于模板引擎，其他语言没有太多的努力，必须简单有效。

##### [](#what-i-think-should-be)我认为应该是什么

HTML 是一门伟大的语言。然而，我认为这种语言必须不断发展，不再是 XML 的扩展，而是独立成为一种完整的语言，一种允许发展的语言，同时保持与同一种语言的旧实现的向后兼容性。此外，需要有一些自己的模板逻辑操作，一个能够与当前和未来的语言集成的语法，并保持隔离。

我认为一种受 QML 启发的语言对网络非常有益。QML 管理的树结构类似于 HTML，但它意味着为复杂的应用程序编写和处理自定义元素。不一定非得是现在的 QML。对我来说，这只是一个想法，但我认为我们真的需要一个为当前应用程序制作的标记语法。我希望在这种语言中看到的另一件事是类型化属性，以及在新组件中定义它们的简单方法。

## [](#style-language)风格语言

##### [](#the-bad-about-css)关于 CSS 的不好

和 HTML 一样，CSS 并没有什么错，它是一门简单的语言，但是很难掌握。也有一些需要改进的地方。一个是需要一种在定制组件中应用独立样式的方法，并且能够轻松地重新设计组件的样式。如果你在想*“影子 DOM 有那个”*，是的，它有。但是 Shadow DOM 仍然是一个实验，规范还在开发中，一些东西已经被删除和替换。让我们面对现实吧，给 CSS 添加一个新的特性太慢了，即使我们有了它，我们也需要等到大多数浏览器都支持它，并且为非标准化的属性加上供应商名称的前缀。

这给我们留下了第二个需要改进的地方。当前给老浏览器带来新功能的方法是填充新功能。CSS 真的很难填充，实际上是不可行的。

##### [](#the-good-about-css)关于 CSS 的好

像 HTML 一样，CSS 是一种非常简单的语言，可以完成复杂的事情。如我所说，很容易学习，而且大多数编辑器都有自动完成功能。

##### [](#what-we-could-learn)我们能学到什么

SCSS 让我们学多少，我们就能学多少。我以前认为 SCSS 代表超级 CSS，我知道，我知道。我刚开始做网站开发，但你能怪我吗？语法与 CSS 相同，它允许嵌套选择器、应用混合、使用变量和许多其他事情。SCSS 在很多情况下已经成为开发网页风格的标准方式。

##### [](#what-i-think-it-should-be)我认为应该是什么

在复杂应用程序中应用样式的新语言应该类似于 CSS，具有 SCSS 允许我们拥有的特性。我要说的是，SCSS 允许我们拥有的功能应该是标准的。需要支持更好的交叉文件引用。

我认为它需要一些当前规范的标准化，包括规则的初始值和标准化的单位值，这将在每个设备上显示 nice，我不只是说 web 消费设备，但其他渲染器引擎应该显示相同的结构。实际上，它需要能够在该语言的旧实现中很好地发挥作用，并在需要时以简单的方式进行聚合填充。

## [](#scripting-language)脚本语言

##### [](#the-bad-about-javascript)关于 Javascript 的不好

Javascript 坏了。它生来就坏了，从那以后一直坏着。如今，越来越多的新特性被添加到规范中，但我们只是在一条断腿上贴了一些创可贴。那是行不通的。即使是现在，当 Javascript 引擎以几乎相同的方式解释脚本时，这种语言也有几个问题，而且它们是根深蒂固的，你不能再让 Javascript 作为一种合适的语言工作了。

我相信，这是唯一需要替换的东西。Javascript 是一种可以在多种设备、多种平台上运行语言，但它刚刚被打破。我们不需要不断给 Javascript 添加新特性，我们需要编写并标准化一种新的脚本语言，用于开发复杂的应用程序。

##### Javascript 的好处

Javascript 是一种非常容易学习的编程语言，而且功能强大。网络中的每一样东西都有一定量的 Javascript，现在，甚至移动和桌面应用程序都运行 Javascript。我觉得这太棒了。启动一个 Javascript 程序不需要复杂的结构应用，只需要写一个函数然后调用它。简单就是美。在大多数情况下，使用 polyfills 在旧浏览器上添加新功能既简单又准确。

##### [](#what-we-could-learn)我们能学到什么

项目和语言，如 Typescript、Flow、Props 和其他一些，告诉我们，即使我们不愿意，我们也需要这是一种类型化的语言。大型企业级应用程序需要一种可预测的工作方式，而 Javascript 根本不可预测。

扩展操作符、装饰符、模板字符串等新特性向我们展示了一种语言需要从它本来的样子进化到社区想要的样子。

甚至库也应该是我们需要学习的东西，比如不可变的。这给 Javascript 带来了类似不变性的东西，并告诉我们为什么这很重要。甚至有人提议给当前 Javascript 带来不可变的特性。

但是，我们最需要的是了解 Javascript 的怪异之处，并创造一种合适的语言。我们不需要三重相等运算符，我们需要一个不做奇怪事情的相等运算符。

##### [](#what-i-think-it-should-be-like)我认为它应该是什么样子

毫不奇怪，我认为 Typescript 需要成为它所说的 Javascript 超集，或者至少是类似的东西。当然，目前 Typescript 编译成普通的旧 Javascript，有时甚至使 ES7 特性对 ES5 浏览器可用，但是如果浏览器能正确理解 Typescript 代码的真正含义，那会更好。这将允许利用 Typescript 拥有的类型系统，更好的是，作为 Javascript 的超级集合，用 Javascript 编写的每个程序都是有效的 Typescript 程序。

好吧，既然我们在做梦，我们可以弃用关键字`var`和运算符`===`而支持新语法`let`和`const`，以及简单的`==`而不进行任何转换。我甚至会说`const`应该能够存储一个常量对象和函数，在其中不可能递归地为它的任何属性分配新值。es 将是真正的类，并且可以用其他语言写成任何其他类。这将是一个你可以在运行时检查的真实的东西。和其他几个能带来更好语言的特性。

这种语言仍然是相同的函数 first language，有很好的语法，可选的类型，当没有赋值或类型声明时，退回到我们亲爱的`any`类型。好吧，如果我们还有任何类型，在处理等式的时候，应该有一个类型转换，对吧？不完全是。即使没有指定类型，变量在求值时仍然有一个类型，这在某种程度上是三重检查相等的当前行为。

## Web 应用程序

##### [](#the-good-about-the-web-apps)网络应用的好处

web 应用程序不是为操作系统编译的，也不是在虚拟机上运行的，而是在浏览器内部运行的。但是它们可以被编译成一个操作系统，可以在虚拟环境上运行，它们可以被用来编程服务器，机器人，以及其他任何东西。

web 应用程序真正好的地方是，它们只是使用标准开发的。你可能会注意到，我在这篇文章中经常使用这个词，原因是因为我们所知道的 Web 只是一系列复杂的标准，可以由任何人在任何地方以任何方式实现。要做到这一点，你只需要尽你所能正确地实现关于你想要使用的技术的当前规范，你甚至可以决定在没有正确描述的事情上做什么。我认为这就是美。我认为这就是未来。

##### [](#what-i-think-it-will-be-like)我认为它会是什么样子

也许不是一年，也许不是两年。但不久之后，原生 web 应用程序将出现在我们的生活中，它们将像我们目前拥有的任何其他原生应用程序一样运行，并将同样执行。我说的不是 Cordova 或 NativeScript，而是 web 应用程序与操作系统的深度集成。

新的操作系统只需要实现网络技术就可以使用，新的网络呈现器出现在我们现在只能梦想的设备中。

应用程序将会像过去那样开发，那时我实际上没有编码，用一种特性的语言，没有捆绑器，传输器，或者我们现在有的任何东西。因为我们会从现有的东西中学习，并不断进步。
