# 我最后是怎么写技术书的？

在这一节中，我将试着向您提供一些背景信息，解释我是如何结束与 TypeScript 的密切关系，并接受撰写有关它的技术书籍的邀请的。

### 我的第一个 JavaScript 步骤

我认为自己非常幸运，因为在我职业生涯的早期，我在一家公司找到了一份工作，这家公司有一个由 JavaScript 支持的大型假期预订系统。在我职业生涯的头几年，我一直从事 PHP、Java 和 C# web 应用程序的工作，这个应用程序与我当时见过的任何应用程序都完全不同。应用程序使用 AJAX 调用按需加载“小部件”(HTML、CSS 和 JSON 的组合)。它基本上是一个带有“web 组件”的单页面 web 应用程序(SPA ),但是 SPA 成为主流还是几年前的事情。

该应用程序运行得非常好。然而，我们经常遇到一些常见的 JavaScript 问题，比如:

*   由“this”操作符指向错误的对象而导致的问题

*   类型强制问题，比如添加实际上是字符串的数字

*   由于缺少块范围而导致的范围问题

这个应用程序让我很早就接触到了带有客户端渲染的大型 JavaScript 应用程序。我发现这种构建 web 应用程序的“新”方式很吸引人，我非常喜欢使用 JavaScript，所以我决定集中精力学习更多关于 SPAs 和 JavaScript 的知识。

过了一段时间，我找到了一份新工作，这一次我打算从事一个新项目。我有了更多的选择空间，我想尝试解决我在之前的项目中遇到的一些问题。我决定像以前一样用“小部件”创建一个 SPA，但是有两个显著的变化:

*   我第一次决定使用 JavaScript 模块加载器(Require.js)。

*   我还决定使用 CoffeeScript，因为我喜欢这样的想法，由于使用了箭头函数，不必担心手动绑定“this”的值。

### CoffeeScript 对我不起作用

首先，CoffeeScript 确实在一定程度上有所帮助，但它从未完全解决我的所有问题。然而，几个月后 CoffeeScript 成了一个问题。我发现，随着我们的代码库变得越来越大，调试一些问题变得越来越复杂。也很难找到以前有过 CoffeeScript 经验的开发人员。所以我最终删除了 CoffeeScript，用 Require.js 和 AMD 模块回到了普通的 JavaScript。

### 打字稿来了

仅仅几个月后，微软第一次展示了 TypeScript。TypeScript 最初并没有引起 web 社区的兴趣。我认为这是因为微软在 web 开发社区中名声不好。许多人认为 TypeScript 与 JavaScript 完全不同，因为当 TypeScript 发布时，大多数人都不知道 ES6 规范的细节。当他们看到一个带有 TypeScript 中的类的代码示例时，他们认为它是 TypeScript，而实际上，它是 ES6。因此，他们认为微软并没有试图改善生态系统，而是用一个新的生态系统来取代它，而事实是微软非常有远见！

我意识到了微软过去的错误，但我也熟悉微软好的一面。我有一些使用 C#和 Visual Studio 的经验，这些工具的开发者体验简直太棒了。除此之外，我已经是 Anders hels Berg(C#和 TypeScript 之父)的粉丝，虽然 JavaScript 和前端开发是我的激情所在，但我在后端使用 C#的冒险经历让我怀念并羡慕 c#的类型安全。最后，TypeScript 被宣布为能够在 Windows、Mac 和 Linux 上运行的开源项目，这在今天看来似乎是显而易见的事情，但在过去，这对于微软来说是非常不寻常的。

第一个预览版一发布，我就开始使用 TypeScript，这对我来说是一个非常令人兴奋的消息。它能够解决我过去经历的一些问题，它是开源的，它改善了现有的生态系统，而不是试图取代它。它勾选了我所有的选项。

我开始学习更多关于 TypeScript 的知识，我写了一些关于它的博客文章，我强调它是我在 Linkedin.com 的技能之一。然后，有一天我收到了 Packt 出版公司的邮件。他们告诉我，他们正在为一本即将出版的打字本寻找技术审稿人。

Packt Publishing 找到了我，因为我强调打字稿是我在 Linkedin.com 的一项技能。今天，成千上万的人已经把打字作为他们在 Linkedin.com 的一项技能，但几年前，只有一小部分早期采用者。我确信这大大增加了我被出版商联系的机会。在这件事之前，我并不太关心 Linkedin.com，但从那以后我学到了宝贵的一课:

如果你想增加伟大的事情发生在你身上的机会，你需要努力工作，但是你也需要让世界知道你在做什么。

我认为成为一名技术评审员是一个极好的机会。我一直想写一本技术书。这是我一生的目标之一，但我认为这完全不在我的能力范围之内，还需要很多年，所以我认为成为一名技术评审员可以帮助我更快地达到目标。

我没有预料到这一点，但最终我得到了成为一名作者的机会，就在我的第一份工作——技术评审员之后。

### 打字稿成为主流

在 TypeScript 成为主流之前的几个月,《学习 TypeScript》的第一版出版了。

随着时间的推移，web 开发社区开始意识到 ES6 的新特性，普通 JavaScript 应用程序的规模呈指数级增长，微软开始拥抱开源。然而，只有当谷歌的 Angular 团队宣布他们将采用 TypeScript 时，它才成为主流。

# 写一本技术书涉及哪些内容？

在这一部分，我将尝试描述写一本书的过程以及其中涉及的工作量。

### 你面前有一条崎岖的道路

每个人都可以写一本书；你不需要成为天才。然而，你需要时间、耐心和毅力。例如，Manning Publications 网站称您将需要:

> 时间和写作技巧，或者至少是获得它们的时间和意愿。

而 Wrox 网站上说:

> 如果你是一名程序员，并且你对为 Wrox 写作感兴趣——即使你以前从未写过任何东西——请随时联系我们。如果你对技术了如指掌，我们会非常乐意让你成为作者。Wrox 作者的职业不是作家——他们是程序员，我们独特的编辑流程将让你得到最好的。

我相信时间是最重要的，而毅力或意志是最基本的技能。我还想指出，我认为你无法独自完成这项工作。

图书销售会给你带来一些直接的额外收入。我在这里说不出确切的数字，因为它们在 NDA 名下。然而，我可以说，你可以赚足够的钱去国外度假，但却不够钱买车或还房贷。

你很可能需要保住你的全职工作来支付你的账单，这意味着你只有在晚上下班后和周末才有时间写作。你可能在工作之外还有其他的责任，这会占用你更多的时间。我不知道其他人是如何处理这种时间不足的，但在我的情况下，我很晚睡觉，几个月都没有睡好。

你会感到疲倦，而且你不会像你希望的那样花太多时间与家人和朋友在一起。

[![A cartoon that tries to represent expectations (easy path) and  reality (difficult path) ](../Images/183a7b6eff4ba965a4a34c0a3f4b6415.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--5x9WTT0l--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/o4gtttdpeje0tx19t7ma.jpg)

你可以想象这是很难的，这就是为什么我认为你不能独自做到这一点。就我而言，这一切都是因为我得到了家人和朋友的支持。

对此我的建议是做好准备。期待在你面前有一条漫长而疲惫的路。对你和关心你的人来说，这将是一个巨大的牺牲。特洛伊·亨特对这种牺牲有一些好的建议:

> 很多次我听到那些想更多地参与我正在做的事情的人说“我不能，我的妻子/丈夫/孩子会发疯的”。这对重要的另一半来说是一个完美的位置，当这是你们共同的愿景时，这绝不是一件坏事。当一方不断地想要朝着另一方不支持的方向前进时，事情就变得棘手了。
> 
> 为了达到这一点，我花了很多年的时间拼命工作，这给了我妻子很大的压力。我们有两个小孩，直到最近，她还在一个相当高的职位上工作；我的动力和雄心也成了她的责任。她支持我——几乎毫无疑问——因为最终，我们对如何生活有着共同的愿景。这种努力花了很多年才结出超出我个人满足感的果实，虽然这很重要，但对整个家庭来说并没有太大帮助。只是在过去的 18 个月里，她(和孩子们，就此而言)看到了一种我们都受益的回报。显然，其中很重要的一部分是经济上的，但它也转化为灵活性，这意味着有更多的时间和孩子在一起，更多的假期，更普遍的是，当我们去的时候，做我们想做的事情。这是她的牺牲，也是我的牺牲。
> 
> 现在我正在帮助她做同样的事情，至少在关注她的公众形象和以我们都知道的方式建立独立于公司生活的独立性方面。如果你欣赏我这些年来所做的，在推特上快速感谢她，因为她值得拥有！
> 
> 小贴士:和你的另一半谈谈你想在哪里投入时间，以及这对你们大家有什么好处。不要这样做，也不要让它成为你们之间的障碍；就你们一起前进的方向达成一致。

### 写作过程

写一本书的过程紧张而漫长，但并不复杂。

1.  创建一个大纲:对我来说，这个过程是从订购市场上所有现存的打字本开始的(当时只有几本)。我列出了每本书的内容，我喜欢的部分，我不喜欢的部分和遗漏的部分。然后我用所有这些信息拼凑出最初的轮廓。创建一个逐步介绍内容的大纲是至关重要的。在你进入下一阶段之前，大纲必须得到编辑的批准。

2.  **计划阶段**:在这个阶段，你需要对每一章进行评估，并为每一章创建一个包含交付日期的日程表。就我而言，写一章的初稿需要两到三周的时间。

3.  初稿阶段:一旦你有了提纲，你就可以开始写章节了。每一章都必须经过图书编辑的初步质量检查。本章随后进入技术审查阶段。

4.  **技术评审阶段**:在你写下一章的时候，一群具有不同专业水平和不同背景的技术评审人员会阅读这一章，并让你知道他们对这一章的感受。

5.  **终稿前阶段**:一旦你完成了所有章节，你必须将技术评审的反馈整合到该章节中，该章节将进入终稿前阶段。此时，您还有最后一次机会来升级过时的内容并修复任何遗留问题。

6.  最后阶段:你会得到最终版本，如果你满意的话，它会被送去印刷。

不幸的是，审查过程需要非常快，因为一些技术发展速度非常快。例如，TypeScript 每两个月发布一次。如果你花两个月的时间来复习所有章节，你的书还没发行就已经过时了。

整个过程(从大纲到发布)第一版花了我十一个月，第二版又花了十一个月。

我发现评论和内容升级比写初稿更难。我对此的建议是尽量不要惊慌。你的书永远不会完全更新和完美。努力工作，尽力而为，享受旅程。

# 写一本技术书的努力值不值？

在这一部分，我将试着解释成为一名作家给我带来的所有好处。

### 直接经济效益不大

正如我前面所说，从经济角度来看，这不值得。如果你把直接从图书销售中获得的收入除以你打算投入到书中的小时数，你会发现每小时的报酬并不多。然而，这本书将为你提供一些显著的间接收益(包括间接的财务收益)。

### 你会学到很多东西

有些东西是你每天都了解和使用的。然而，如果你写一些关于它们的东西，并试图给出一个简单的解释，你会意识到你并不像你想象的那样理解这些东西，结果，你会做更多的阅读并有更好的理解。

你还必须深入细节，因为技术书籍是比我们的电子邮件、闲聊和博客帖子更正式的媒介。这将引导你了解一些你以前可以忽略的低级细节。因此，你会学到很多东西，这些知识也会帮助你找到一份更好的工作。

### 它会帮助你涉足 OSS

当我在写我的书的两个版本时，意想不到的事情发生了。除了写书，我一直想做的另一件事是为开源做贡献。我知道如何做出贡献(使用 Git，GitHub...)但是我从来不觉得自己有一个值得开源发布的想法。然而，我在编写第一版(InversifyJS)时有了一个想法，在编写第二版(TsUML)时又有了一个想法。

我相信这是因为当你写一本书时，你必须考虑未来。你需要关注产品路线图，确保你的书在发布时是最新的。

例如，在这个特性被添加到 TypeScript 之前，我开始写一章关于 decorators 的内容。我用当时网上的一些公开设计笔记写了这一章。就在那时，我有了创建一个由 decorators 驱动的 IoC 容器的想法，这是我第一次没有太晚。我在网上搜索了一下，Angular 也在计划类似的东西，但当时没有可用的东西，所以我开始研究后来成为 InversifyJS 的东西。今天 InversifyJS 在 npm 上有超过两百万的下载量。

### 它会帮助你变得更加自信

发生在我身上的另一件事是，这本书在某种程度上帮助我克服了我的冒名顶替综合症。我不再害怕做一些事情，比如在博客帖子、当地用户组或会议上与社区分享我的想法，我决定开始都柏林 TypeScript meetup。

对我来说，克服我的冒名顶替综合症是这本书给我带来的最大好处，并最终让我成为了微软的 MVP。

### 间接财务收益很大

图书销售直接产生的收入很可能不会很大。然而，间接的好处，如学习或参与 OSS 和社区将帮助你成为一个更好的专业人士。

在你的简历中放一本书、一些 OSS 项目和会议演讲将有助于你脱颖而出，获得一份更好的工作。这意味着这本书会给你带来一些与图书销售没有直接关系的巨大经济利益。

# 怎样才能写出一本技术书？

我没想到这么快就有人邀请我出书。我原以为这是只为约翰·瑞西格、卡米尔·福尼尔、乔恩·斯基特或马丁·福勒等非常著名的工程师保留的特权，但我错了。

现实情况是，很多时候这些非常著名的工程师太忙而不能写书，或者他们不想写，因为写书是一项非常累人的活动。

事实上，对技术作者的需求很高，可用的作者不多。

我认为缺乏作者不是因为缺乏技能，而是因为缺乏信心。就像我一样，我相信许多其他工程师也想写一本书，如果他们有机会，他们会做得很好，但他们觉得自己还没有准备好。这只是臭名昭著的冒名顶替综合症的又一个例子。

### 成为技术的早期采用者

成为早期采用者并不总是一个选项。一些关键的解决方案不能依赖不成熟的技术。然而，没有什么能阻止你在下一个爱好项目中尝试新事物。

成为开源技术的早期采用者将有助于您确定改进技术和解决问题的潜在方法，这将为您提供为项目做出贡献的明确机会。

### 分享你的经历

成为早期采用者只是第一步。你需要让世界知道你过得怎么样吗？如何才能入门这项技术？这项技术有什么好处？你注意到哪些模式是有效的？当你是尝鲜的时候，这种东西网上是没有的，但是你会去创造。

如果出版商想在几年后出版一本书，你的名字很可能会出现在他的谷歌搜索中。你需要让世界知道你在做什么，你对什么感兴趣。如果他们有机会给你，这是他们能够找到你并联系你的唯一方法。

### 成为技术评审员

你现在就可以成为技术评审员。相信我；你现在就可以给一本书投稿。

> 如果你觉得自己还没有准备好成为一名 Wrox 作者，或者你想参与其中，但没有时间写书，你有一些选择。你可以成为一名技术编辑，利用你的编程知识来审查和验证一本书的技术资料。您的反馈将对最终图书的质量起到非常重要的作用。

即使是初级开发人员，也可以做技术审核人员。这本书很可能会被其他初级开发人员阅读，所以有你的投入将有助于作者确保这本书适合经验不足的公众。

通过联系出版公司，你可以成为一名技术评审员。以下是一些例子:

*   [Packtpub](https://www.packtpub.com/books/info/packt/contact-us)

*   [进程](https://www.apress.com/gp/services/tech-review)

*   [人员配备](https://www.manning.com/manuscript-reviews)

*   [乌鸦](http://www.wrox.com/WileyCDA/Section/id-105073.html)

### 成为作者

我强烈建议先做技术评审，因为这有助于你熟悉流程。正如我之前说过的，你不需要成为天才或者有惊人的写作技巧。你只需要:

> 获得它们的时间和意愿(写作技巧)。
> 
> 如果你是一名程序员，并且你对为 Wrox 写作感兴趣——即使你以前从未写过任何东西——请随时联系我们。如果你对技术了如指掌，我们会非常乐意让你成为作者。Wrox **作者的职业不是作家**——他们是程序员，我们独特的编辑流程将让你收获最大。

# 总结

写一本书需要很多时间和精力。这是一个巨大的承诺，但总的来说，这也是一次非常积极的经历。如果你曾经梦想出版自己的技术书籍，不要害怕，联系出版公司，试一试。你能做到的！
