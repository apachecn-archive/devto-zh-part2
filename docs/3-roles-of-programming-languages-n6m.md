### 编程语言的 3 个角色

[![](img/25d929007f5e4ddaba012c2bee68675d.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--w4Vq19LR--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/1000/1%2AlEcnNOr5onYegA8rk9BFaw.jpeg)

一种编程语言扮演三种不同的角色:它连接机器、开发者和团队。对于这些角色，现有的编程语言存在严重的缺陷。如果我们能解决上面列出的 3 个问题，它不仅能让开发者体验到极致，也一定会有丰厚的经济回报。

### 问责的问题

软件开发的最大危机是缺乏责任感。理想情况下，将有一个团队对整个事情负责，然后他们可以像结构化编程所承诺的那样，将责任委托给较低级别的团队。相反，我们在微服务世界中面临的是责任以分散的方式从一个人传递到许多其他人，导致企业所有者的巨大通信开销。更重要的是，这使得没有人能够对最终结果负责。

为什么这和编程语言有关？因为我们写的每个函数，每个类都是为了分解问题。语言不仅仅是描述逻辑，它还塑造了我们的世界观，即如何将大事分解为小事，以及如何将它们重新组合在一起。

### 可读性问题

我们经常需要阅读大量的代码来找到我们想要的。信息密度低。遵循代码中的因果关系往往是扭曲的。代码重用不可避免地会损害可读性，因为您需要跳过与您的用例无关的代码。非功能需求与已经复杂的功能逻辑纠缠在一起。错误处理是使代码不可读的常见问题。

让代码可读不仅仅是正确地命名变量。编程语言需要提供像协同例程、AOP、多调度这样的结构来支持某种编码风格。

### 异构计算问题

主流编程语言中的并行或分布式执行是作为编译器向量化提示或库来实现的。主流编程语言很大程度上类似于只有一个顺序 CPU 的 PDP-11 的世界观。如今，在同一台机器中拥有不同种类的执行引擎(SIMD、GPU)是非常常见的，并且在一个软件中使用更多的执行引擎也越来越常见。内存模型不再只是一个大堆，对于多个执行器来说，可以有多个阶段的堆。编程语言应该提供实际执行设备的更忠实的抽象，而不是在过时的计算模型中限制思维和表达。

### 远景

理想的编程语言应该提供这些特性

*   责任是从上到下分解的，总有一个团队对整体负责。
*   良好的逻辑局部性，不需要跳过多个地方来了解发生了什么
*   同样的语言也可以用来描述 SIMD/GPU/微服务以及它们之间的互操作

最终，它是关于人类的可读性。您可以从不同的概念层次检查代码，并在不伤害眼睛和大脑的情况下推理其正确性(符合您的心理模型)。
