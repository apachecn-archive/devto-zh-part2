# 为什么我是一名开发者

> 原文:[https://dev.to/vorahsa/why-i-am-a-developer-ack](https://dev.to/vorahsa/why-i-am-a-developer-ack)

我直到大学才学会编程。我小时候写过一些 BASIC 语言，但更多的是命令序列，在“子程序到底有什么用处？”。我对计算机或编程不是很感兴趣，我在大学里只学了计算机科学，因为我认为这是一个我应该了解的话题，而不是出于任何兴趣。

即使在大学里，编程课也不是我感兴趣的。我更喜欢算法和理论计算机科学(我是一名数学系学生，最终专攻数理逻辑，所以这很自然)。让我感兴趣的是一个与我的课程无关的课外项目。我爸爸有一个非常适合自动化的东西。我用我的初学者技能编写了一个小程序，它可以自动运行一点点，随着我了解的越来越多，我不断地扩展这个程序，直到它完成了几乎所有需要的事情。

回想起来，我认为那个项目引起我兴趣的原因是它很有用。我可以看到我所做的事情的实际价值，将一个手工过程一点一点地转换成一个程序是很好的，因为我可以很快看到新功能的运行。作为我的第一个 50 行 C 程序开始的程序，增长到几千行 Perl，我永远不会厌倦工作，当发现有用时添加新的小功能。有一次，我把它转换成了 Excel 电子表格(就是那种程序)，但是这样做的好处并没有大到让我懒得去完成这项工作。尽管如此，当我试图做一些有意义的事情时，我喜欢学习 Excel 和 VBA。

在我早期工作的几年里，我偶尔会尝试一些其他的编程副业项目，但是没有什么结果。我“知道”我应该制定某种计划，为整个项目做一个设计。这意味着我需要花一些时间来建立数据结构，然后才能让程序做任何事情。这导致我没有完成任何有意义的事情，因为在看不到计算机做我想做的事情的情况下，从事数据结构和基础设施方面的工作对我来说并不是很有价值。

当我转行做开发人员时，我的第一个主要项目是和一个优秀的 scrum 大师和产品负责人一起使用 Scrum。做小增量以提供用户价值的敏捷思维对我很有效，使它成为一个令人愉快的项目。我们做了一些小的设计工作，但仅仅是为了让接下来的事情正常工作，当进一步的需求需要时，重构设计。

随着时间的推移，我学会了从事情中获得乐趣，而不仅仅是让电脑在屏幕上显示一些东西。重构、清理代码、简化架构，所有这些都让我在代码看起来更好并且所有测试用例仍然工作正常的时候感到高兴。但是，即使在这里，我对不取得进展的不耐烦也开始发挥作用:如果一个架构清理需要几天时间才能完成，我很可能会沮丧地放弃整个事情。着眼于最终目标，每一小步都能产生有用的东西，这是我最擅长的工作方式。

了解自己的这一点也让我更擅长做副业。我可能对自己想要建立的东西有宏伟的计划，但我已经学会满足自己不断取得可见进步的需求。我从让计算机在短时间内做一些有用的事情开始，不管它离我想要的有多远。这是我积极培养的一种技能，一种把我想要的大东西分割成小块有用的功能，然后一个接一个实现的能力。即使我的宏伟计划从未实现，我仍能从工作中获得一些有用的东西。

我需要创造一些有用的东西，这让我觉得这伤害了我的职业发展。如果有一项新技术是我“应该”了解的，不做些实际的事情，我根本无法掌握它。我可以浏览一个教程，也许可以学习如何使用它，但是没有什么有用的目的，我看不出这种技术有什么好处。只是一段没有价值的代码。我需要一个使用新技术的有意义的项目，这样的事情通常比仅仅阅读一些网页或浏览一个简单的教程要耗费更多的时间。所以我经常觉得我对当前所有的热门话题都不够了解，这在咨询业务中并不太好。

总之，这就是为什么我仍然是一名开发人员。在我的职业生涯中，当计算机做了我告诉它的有用的事情时，我获得的喜悦感没有减少。我已经学会了扩展“告诉计算机做什么”的概念，以从事情中获得这种感觉，而不仅仅是在屏幕上显示东西，我总是试图以一种我总是获得那些小成功的方式组织我的工作。事情并不总是这样，我仍然会对事情感到沮丧，但我管理得足够好，仍然喜欢把编程作为一种职业和偶尔的爱好。
