# 开发人员移交清单

> 原文:[https://dev.to/stoft/developer-handover-checklist-255h](https://dev.to/stoft/developer-handover-checklist--255h)

时不时地，你需要把一些东西(代码、东西、诸如此类的东西)交给某人。在这些时候，有一个清单来检查是很好的。这是我倾向于使用的列表。所有的点都是*可选的*，用户应该争取*足够好*。我基于 Kruchten 和其他人定义的架构观点(C4 或其他方案也可以)。

| 项目 | 描述 | 例子 |
| --- | --- | --- |
| 商业案例 | 促使解决方案存在的主要业务案例的概述。 | 当我们的客户从我们这里收到有缺陷的小发明时，他们必须能够提出“索赔”。 |
| 概念模型和过程 | 解决方案中使用的概念模型的演练。这包括各种业务实体和所涉及的流程。 | 实体:索赔、接收方、产品
流程:提出索赔、结束索赔、反驳索赔 |
| 功能/统一通信视图 | 功能视图和用例视图的演练，包含一个或两个典型的用例，以及需要更详细解释的更复杂的用例。 | 作为客户，我可以创建索赔，列出我的索赔，显示索赔等。 |
| 发展观 | 开发这个东西需要知道的一切:
-组件、模块
-集成系统图
-信息流图
-集成依赖(DBs)
-开发环境(工具、存储库等)。)
-环境(开发、测试、生产)
-部署流程(构建链、ci/cd 等)。)
-使用的软件/支持系统 | 经典的三层架构，框架 x 中的 MVC 模式。我们也调用 CRM 系统...通常 IntelliJ + git
Wiki 用于文档，JIRA 用于问题跟踪。 |
| 流程视图 | 解决方案执行模型的详细说明，如线程等。专注于硬的东西。 | “GizmoService”是单线程的，因为它必须按顺序处理“GadgetJobs ”,并且必须保持顺序。 |
| 物理视图 | 可用物理环境的描述(应该与开发视图中的环境相关联)以及它们包含的内容。 | 通过 ELB 和 CloudFront 部署到 AWS EC2 实例。“测试”和“生产”的单独配置。 |
| 质量要求、约束 | 对质量要求和可能的约束以及如何实现它们的描述:
-可用性(SLA:s 等)
-安全性
-性能
-可靠性(容错、备份等)。)
-配置管理
-可测试性
-许可约束
-法律约束 |  |
| 操作视图 | 操作视图的描述，日常工作如何进行，涉及的组织和工作流程，以及如何执行维护和支持。 | 我们负责开发和部署，但“运营”团队负责正常运行时间、重启、操作系统更新等。如果是低级别的事件，我们会先处理，然后移交给运营部... |
| 可交付成果 | 导出源代码、文档等，或者访问 SCM 等。 | 代码
联系人
文档
脚本
系统账户 |
