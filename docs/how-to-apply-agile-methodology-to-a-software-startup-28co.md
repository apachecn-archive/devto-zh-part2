# 如何将敏捷方法应用于软件创业？

> 原文：<https://dev.to/apium_hub/how-to-apply-agile-methodology-to-a-software-startup-28co>

软件开发是一个非常有利可图和令人兴奋的职业选择，无论是对于应届毕业生还是已经建立的专业人士。加上软件启动工作环境，软件开发可以让位于创新和有吸引力的软件解决方案。然而，满足于每天的工作流程可能是困难的，尤其是议程上有许多项目，不同的个性相互冲突。

敏捷方法在这种情况下非常有效，因为它可以帮助团队获得更大的关注和生产力。这要归功于强调任务区分和协作流程的敏捷(没有双关语)工作流。但是你如何将敏捷应用到一个崭露头角的软件创业公司呢？让我们找出答案。

### 1。短期里程碑，长期目标

在敏捷软件开发环境中工作，允许有许多差异。目前，研究表明，2%的项目失败是由于不清楚、不明确的可交付成果。这通常是在没有制定出[必要的发展里程碑](https://dev.to/apium_hub/software-project-milestone-what-why--how-588j)的情况下设定最终目标的结果。

敏捷允许软件开发人员设置里程碑的不同轨迹，并根据他们当时的感觉在它们之间跳转。Jordan Rowland 是 ResumesCentre 的一名 UI 文案，他讲述了他写 UI 文案的经历:*“客户要求我在不同的时间间隔为他们的用户界面写文本。有些人在编码前做，有些人在最终发布前做——这两种方式对我来说都很好。我很高兴开发人员能够按照自己的喜好改进开发过程，而不用担心线性进展。”*

你今天可能想写后端代码，但是还不想为 UI 集成而烦恼。在敏捷中，你可以按照你选择的任何顺序轻松地完成这两项任务，并根据每个人的技能和一天的心情来分配任务。这导致了更高质量的最终产品，以及由于诱人的工作环境而留住了软件开发人员。

### 2。每日团队签到

软件开发是一个易变的过程——一个 bug 或未受监控的过程会导致修改和调查的连锁反应。然而，敏捷方法允许对项目更新采取更加协作、宽松的方式。Small Biz 最近公布了一项统计数据，称 [90%的创业公司失败](https://www.failory.com/blog/startup-failure-rate)是由于创业目标错位和业务增长管理不善。这意味着软件创业团队保持凝聚力并朝着共同的目标努力是至关重要的。

[EssaySupply](https://www.essaysupply.com/) 的首席执行官詹姆斯·斯科特讲述了他自己管理一家初创公司的经历:*“我的网站在发展成今天的样子之前，是一家小型外包公司。在早期，管理团队很困难，因为我们都是远程工作。从那以后，我们建立了日常生活，很少偏离它。”*

可以利用每日团队会议来确保每个人都参与当天的工作重点和里程碑。它还可以用来给每个团队成员分配工作，让每个人与办公室里的其他人分享他们的目标，以及向团队反馈有趣的想法。每日检入是敏捷开发的宝藏，应该在软件启动时尽快实现。

### 3。强调合作，反对个人化

虽然有点拗口，但是在软件启动环境中，协作应该总是优先于个性化。这一点再怎么强调也不过分，因为实施敏捷方法的确有助于保持健康的团队合作。你可以很容易地根据技能组合和性格来分配工作，这两者可能会很好地配合。这将阻碍个人主义者在不与他人联系的情况下追求自己的工作。

这也将有助于保持一个稳定的反馈系统，而不必担心有人在没有监督的情况下独自工作。Clyde Bate， [FlashEssay](https://flashessay.com/) 的联合创始人谈到了他作为项目经理的工作委派习惯:*“将工作分配给特定的团队成员总是很困难——而且从来都不容易。相应地委派工作的秘密是把人们分成小团体，让他们在没有你的持续监视下合作。这样，你将创造一个无微观管理的环境，并将今天的团队成员培养成明天的领导者。”*

协作应该优先考虑，即使软件创业团队成员在业余时间远程工作。项目管理平台在网络上非常丰富，这意味着任何人都没有理由单独工作，而不把他们的想法和进展反馈给其他人。

### 4。满足于“某事”

软件开发是一个长期的游戏，这意味着短的期限和快速的周转时间很少能结出具体的果实。据统计，46%的创业公司将“不称职”列为失败的首要原因。如果怀疑蔓延开来，还有许多其他项目等待探索，那么坚持一个项目是没有意义的。毕竟，如果客户询问他们项目的当前进展，简单地耸耸肩会损害开发团队的士气。

有了它，软件开发人员就可以将敏捷方法应用到他们的项目中，在任务之间转换而不会精疲力竭。一旦某个活动项目达到了某个里程碑或标准，团队就可以转移到另一个里程碑或标准，以保持新鲜感、“敏捷”和高效。

因此，团队的积极性将会保持很高，软件初创公司将会在每次客户签到时向他们展示一些东西。或者，一家软件初创公司可能会因为一个项目而陷入困境，忽略所有其他项目，在这两方面都没有任何成功。通过敏捷的工作流程，不惜一切代价避免[团队的疲惫和摩擦](https://dev.to/apium_hub/distributed-agile-teams-9-hacks-that-make-them-work-1c82)，以便在更短的时间内完成更多的事情。

**结论**

敏捷软件开发是线性设计哲学的一个更好的选择。可以在更短的时间内做更多的事情，并有更多的具体证据来证明这一点。敏捷方法不需要专门的培训或过于复杂的计划。它鼓励创造力，生产力和积极的发展，几乎没有停机时间。

确保在你自己的软件开发中测试敏捷方法，比如 Scrum，看看这种类型的工作流是否适合你的团队。毕竟，尝试一两个新技巧来找出最适合你的团队心态的方法是没有坏处的。
