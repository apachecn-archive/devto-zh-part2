# 自由网站开发结束后，我的责任是什么？

> 原文:[https://dev . to/raddevon/what ' s-what ' s-my ' s respons ibility-after-a freelance-gig-ends-jb4](https://dev.to/raddevon/what-are-my-responsibilities-after-a-freelance-gig-ends-jb4)

你刚刚为一个客户完成了一个网站。客户签字同意并支付了最后一笔款项。一周后，他们回来要求修改或修复一个错误。你对客户的责任是什么(也就是说，我什么时候才能最终安排那个早该安排的假期😜)?

## [](#what-is-the-norm)什么是“常态？”

简而言之，没有规范。在你开始任何项目之前，你应该有一份关于你将为客户建造什么的书面协议。一些自由职业者将这一点包含在他们的合同文件中，而其他人则使用单独的“工作声明”无论你使用什么样的文档，你都应该**清楚地列出作为项目的一部分需要完成的工作**。

唯一的规范是，作为自由开发人员，你应该做你已经同意的所有事情，你不必做任何不存在的事情(尽管有时你可能想做；我们稍后将对此进行讨论)。

## [](#how-to-set-expectations)如何设定期望值

设定期望的第一步是讨论它们，作为你初步项目讨论的一部分。当你觉得你接近完成一个项目时，向你的潜在客户概述你会做什么，也许更重要的是，你不会做什么。

这种讨论可能会导致你和你的潜在客户之间的一些谈判。也许你不想在项目交付后提供*任何*支持和维护。**那是你的电话。**听到这些，客户可能会问你是否会考虑提供一年的支持和维护。然后由你来决定你是否想这么做，以及你想用它来交换什么。也许你想对持续的支持和维护按月收费。也许你想收取年费。也许你自己做，也许你把它外包给别人。也许你就是嫌麻烦，决定拒绝。所有这些都是有效的回应。

现在，轮到你的潜在客户了。如果你拒绝他们的要求，他们可能会决定不想和你一起工作。这也可以。如果你讨厌做维护和支持工作，而你遇到一个有这种需求的客户，**那就不适合你**(你不应该试图抓住每一个客户)。通过拒绝他们的请求，早点把他们过滤掉，在为时已晚之前认识到不合适，你们两个人的生活都会变得更好。恭喜你！

## [](#the-contract)合同

如果潜在客户在讨论后仍然同意，你和你的新客户都需要在一份文件上签字，说明你将做什么。这应该反映出你们之前讨论的内容。

大多数人认为合同只是告诉你*将*做什么，但是，就像你在之前的讨论中所做的那样，列出你*不会*做什么也很好。如果你的合同中没有提到什么，客户有时会对此做出假设。如果你能**预见到他们可能期望你做什么样的事情，并在你的合同中明确说明你是否会做那些事情**，你就能让每个人都不那么心痛。

以下是客户*可能会*期待的一些事情:

*   永久支持
*   永久维护
*   具有新“小”功能的更新
*   免费迁移到新主机或云提供商

如果没有明确声明你*将*提供它们，其中一些可能听起来不合理，但一些客户无论如何都会假设它们。有些人可能有稍微缓和的期望，但仍然在你同意提供的范围之外。例如，客户可能意识到期望永久的支持是疯狂的，但是他们可能期望你支持他们“至少一年”,仅仅因为你构建了这个软件。仍然不合理，但他们可能会期待。

你不可能预料到你的客户可能期望的所有事情，所以尽你所能，继续前进。如果客户来找你，希望得到你合同中没有说明的东西，**礼貌地向他们解释这超出了项目**的范围，但是如果他们需要帮助，你很乐意与他们合作。

如果你发现自己向几个客户解释说，你不提供 X 是因为它不在范围之内，这可能是一个很好的期望，添加到你未来客户的合同中，只是为了在问题发生之前阻止它。

## [](#unless)除非…

当客户要求额外的工作时，是否提供由你决定。如果一个客户来找你，要求你做一些没有承诺的事情，而你做到了，根据这个客户的性格，会发生两种情况之一:你要么成为这个客户眼中的一个容易的目标，他们会试图看看他们能从你那里得到多少免费的工作，要么他们会意识到你为他们做了一些你不必做的事情，并将这个故事传播到世界各地。通常，前者会发生，所以**要非常小心**不要因为免费提供服务而成为受气包。

试着根据客户过去对你的表现来判断他们会如何看待你为他们做的事情。猜猜做他们要求的事情需要多长时间。他们是否尊重、感激、容易共事？他们要求的是 10 分钟还是 10 小时的事情？如果他们是一个伟大的客户，只需要你几分钟的努力，那就去照顾他们吧。你损失了一些计费时间，但是**你为一个你将来可能会成为回头客的客户创造了一个很好的客户服务体验**。他们更有可能在未来向你推荐其他潜在客户。

## [](#times-you-should-always-do-free-followup-work)次你应该总是做免费的跟进工作

我能想到一个场景，你应该**一直做免费的工作**，甚至在合同到期后:**你搞砸了**。如果你交付了 web 应用程序或网站，而你承诺的某个功能没有工作(并且在最终支付账单之前没有被发现)，你应该去修复它，不管花多少时间。不要让客户扩大“完成”的含义，但要确保所有明确的期望都到位。

这里有一个简单的场景。你刚刚为一个客户发布了一个营销网站。它应该在首页有一个按钮，弹出一个带有电子邮件选择表单的模态。你展示了网站，客户签字同意了，但是后来发现这个模式在 Safari 中不能正常工作，尽管浏览器承诺支持最新版本的 Safari。

在这种情况下，您应该返回并修复模态，以便它可以在 Safari 中工作。但是如果客户回来说他们发现了另一个问题呢？该表单收集的电子邮件不会进入他们的 MailChimp 帐户。这是您第一次听说客户需要将电子邮件发送到 MailChimp，而您只是在提交表格时将电子邮件发送给客户。编写一个 MailChimp 集成可能需要几个小时的工作，所以这可能是您想要划清界限的地方。

向客户解释你在最初的项目中没有考虑集成的范围。让他们知道你很乐意提供帮助，并询问他们是否希望你评估一个添加 MailChimp 集成的新项目。你现在已经实现了最初的协议，但是**你已经为客户画了一个清晰的界限**。

## [](#guidelines-for-postproject-work)项目后期工作指南

### [](#when-to-always-do-it)什么时候总是这样做

*   当你没有交付协议中规定的所有东西
*   当你交付了一个不像承诺的那样工作的特性

### [](#when-you-probably-shouldnt-do-it)在你可能不应该做的时候

*   当询价不是合同的一部分，并且需要很长时间来执行时
*   当要价不是合同的一部分，客户是个混蛋，不管要花多少时间
*   当事情发生后，客户拒绝签订维护合同(或者您拒绝进行维护)

### [](#when-you-get-to-make-the-call)到了就打电话

*   当问题是由您无法控制的事情引起时(一个 API 突然被否决，在您使用的库版本中发现了一个严重的错误，客户决定使用不同的提供者，等等)。)
*   事先没有讨论这个问题，但你知道这不会花很长时间

## [](#value-should-flow-in-both-directions)值应该双向流动

让这成为你在合同结束后被要求工作时的指导原则。你实际上是被要求免费做一些工作。你打算从中得到什么？

这听起来可能很自私…而且*就是*。没关系。除非每个人都有所收获，否则生意不会做成。期待工作有所回报是完全合理的。有时候，是钱。有时候，是善意。**价值永远不应该单向流动。**这意味着你的生意不景气。

不要勉强做这项工作。这对你们未来的关系定下了不好的基调。如果你确保价值是平衡的，每个人都会为此感到兴奋，每个人都会高高兴兴地回家。在项目结束后对工作说“不”没有坏处，但是如果“是”对每个人都有意义，你就不必总是说“不”。
