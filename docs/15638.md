# 杀死你的敏捷团队的三个短语

> 原文:[https://dev . to/_ Patrick god/3-phrases-that-will-kill-your-agile-team-1la 6](https://dev.to/_patrickgod/3-phrases-that-will-kill-your-agile-team-1la6)

我要说现在大多数开发团队至少*称*自己是敏捷的。但是经常会出现这样的问题，*他们真的敏捷吗？*

每天站起来，每两周开一次冲刺规划会议是不够的。作为一个敏捷团队工作也意味着**这个团队的每个成员都有正确的心态**。除此之外，你可能会体验到一个大公司中的敏捷开发团队也依赖于其他部门的思维方式。

如果销售和开发不能协同工作，任何敏捷范例都无法帮你摆脱困境。可悲的是，有三个短语你在公司里或多或少总会听到，能否战胜它们取决于你自己。

### 我们一直都是这样做的

当我以所谓的“数据仓库专家”的身份加入一家银行时(尽管我对数据仓库几乎没有任何经验——就头衔而言是如此……)，我的主管对这个年轻人能够改变整个部门寄予厚望。我获得计算机科学学位半年后就加入了这家公司。

所以，是的，可能有人能做到这一点，实际上我很有动力，但当你很年轻，没有经验，其他人都很乐意保持现状时，改变整个部门的想法真的很难。

当然，数据仓库部门与软件开发没有太大的关系。这只是一个例子，肯定有软件开发公司有同样的旧思维。

想想单元测试(甚至不是测试驱动开发，只是单元测试)或者版本控制。我知道有些团队仍然在使用 Visual SourceSafe。最新版本是 2005 年的。这太疯狂了。

所以拜托，为了你整个团队的爱，不要拘泥于“我们一直都是这么做的”这句话。这让你和你为之工作的公司一事无成。最后，每个人都会觉得自己像是来自[那个著名的社会实验](http://www.wisdompills.com/2014/05/28/the-famous-social-experiment-5-monkeys-a-ladder/)的五只猴子。

### 不是我的工作

听起来不太合作，是吧？

我知道，事实上在某些情况下，我们的同事真的会让我们心烦。开发人员似乎也是计算机、打印机和网络维修人员。几乎每次技术上的东西在其他部门不管用，他们就立马站在开发商办公室门口。

这是一种天赋，我们往往知道这么多，对不对？嗯，如果你有五到十分钟的空闲时间，为什么不友好地互相帮助呢？

当然，如果太多或者你此刻太忙，大家都会理解。我们都有负责特定任务的人，提及他们完全没问题。

还有，有些请求你真的应该[学会拒绝](http://programmergoals.com/learn-to-refuse-tasks/)。如果你是一名程序员，有截止日期和重要的后端任务，但有人要求你设计一个演示图表，那么你真的应该先和你的主管谈谈。

但是除此之外，试着互相帮助。尤其是在您的开发团队中。如今，获取你领域之外的知识变得越来越重要。如果您的主要任务是编写代码，但是您也知道如何将 web 应用程序部署到您的 web 服务器，那么从长远来看，整个团队都会受益。

谁知道呢，也许**你的知识对你的公司真的很有价值**，下一次加薪就在眼前。

### 我们不能

是的，你可以！好吧，这太老套了。不管怎样…

不要随便这么说。你是开发商。解决问题是你工作的一大部分。所以不要只说事情不可能。

说真的，有太多的例子证明，有了正确的心态，一切似乎都是可能的。你不必制造下一辆电动汽车或下个月开始火星之旅，但如果你的应用程序性能可以更好，尝试寻找一些优化。也许代码中有一些懒惰实现的算法。如果没有，也许有必要购买一台更好的服务器。

你明白我的意思。

努力推动你自己和你的团队，让每个人都乐于在这个特殊的地方工作。
