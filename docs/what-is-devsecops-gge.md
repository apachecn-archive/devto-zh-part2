# 什么是 DevSecOps？

> 原文：<https://dev.to/aditichaudhry92/what-is-devsecops-gge>

这篇文章最初发表在 Medium 上。你可以在这里看一看它[。](https://medium.com/@aditi.chaudhry92/what-is-devsecops-cb14cfd457b2)

随着每个人都向云迁移，有太多关于 DevOps 以及它如何使流程更快、更简单、更高效的宣传。作为一名开发人员，我喜欢事情变得更容易。不过，作为一名安全领域的人士，没有强调安全地迁移到云让我感到害怕。输入 DevSecOps。

DevSecOps 是关于在软件开发生命周期(SDLC)的早期引入安全性。目标是通过让 SDLC 中的每个人都对安全性负责来开发更安全的应用程序。我喜欢这个。让商业、技术和安全部门合作生产安全产品似乎是梦想成真。也许好得不像真的？让我们进行更多的调查，看看 DevSecOps 是否能成为我们在构建安全产品时都需要的银弹。

首先，我们来谈谈为什么需要 DevSecOps。几年前，软件产品遵循瀑布方法，这是一种开发产品的线性顺序方法，以“大爆炸”发布结束。

[![](../Images/1b765bd16043d21d9712e8de29720777.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--0wg1hznG--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/sm43hf8lxxk6ahm6mptx.png)

随着向云计算和资源动态供应的转变，开发人员在应用程序开发的速度、规模和成本方面获得了许多好处。这些好处非常适合 DevOps 运动的采用。DevOps 强烈主张在 SDLC 的所有步骤中实现自动化和监控。目标是缩短开发周期、增加部署频率和更可靠的版本，所有这些都与业务目标保持一致。就像我之前说的，作为一个开发者，我喜欢这样。但是稳定的基础设施和应用程序并不等于安全的基础设施和应用程序。这让我很担心。在瀑布生命周期中，安全检查被放在产品发布之前的最后。安全性被视为一个路障，是通往产品发布的最后一道关卡。应用程序的开发方式发生了很多变化，但人们对安全性的看法却没有改变。

如今，大多数团队使用敏捷方法进行软件开发。在敏捷环境中，重点是快速交付。通过使用迭代计划和反馈结果，团队可以不断地将产品交付与业务需求保持一致。对于交付一个有意义的产品来说，对不断变化的需求的适应性是非常好的，但是如果你每周发布一个新版本的产品，你什么时候测试安全漏洞呢？不幸的是，传统的安全流程跟不上敏捷/DevOps 环境的步伐，使得安全性成为软件开发中的主要障碍，通常被绕过。如果不绕过它，开发团队很少有足够的时间在产品上线前解决所有问题，这意味着一个不安全的应用程序存在于互联网的某个地方。具有讽刺意味的是，为了避免错过最后期限的风险而忽视安全性实际上给应用程序带来了更多的风险。SDLC 中的安全缺陷会导致严重的漏洞，如由不良代码导致的违规。这就是为什么我们需要 DevSecOps。

在 DevSecOps 中，两个看似对立的目标，“交付速度”和“安全代码”被合并成一个流线型的/自动化的过程。DevSecOps 的目的是建立在“每个人都对安全负责”的思想上。它是关于向左推动安全性和自动化核心安全任务。还记得碧昂斯的《不可替代》这首歌吗？我知道这是一种倒退，但碧昂斯唱“向左，向左”时，她的想法是正确的

[![](../Images/b15846f92f4f42a68d0eeb614f9586c3.png)T2】](https://i.giphy.com/media/G0bVqvhbRFd1C/giphy.gif)

我们希望将安全性推到 SDLC 的左侧，以确保应用程序安全性从一个代码开始。通过左移，团队可以快速发现和分析漏洞，然后调整他们的代码来减少这些漏洞。

[![](../Images/823639d4ded9d42f9c5ff710c4eb4893.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--8_QWCrkH--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/plyu8wytsofl6f263bjv.png)

DevSecOps 允许开发人员专注于编写高质量和安全的代码，使团队能够发布 titanium 应用程序。好处很简单:

*   从一开始就确保安全性可以最大限度地减少出现漏洞的可能性
*   让自动化安全工具在管道中运行的好处是，它让安全团队成员专注于高挂的果实
*   开发和安全团队之间更好的协作和沟通
*   提高整个安全和企业的运营效率

然而，DevSecOps 也面临着一些挑战。即使将安全性嵌入到管道中，仍然有办法绕过安全检查点。让我们举一个例子，如果发现了某个漏洞，就使用漏洞扫描器来阻止构建/部署。作为一名开发人员，你知道这个漏洞是什么，你知道你的代码有，但你真的需要做这个版本。因此，您找到一种方法来隐藏您知道将无法通过安全扫描的代码片段，从而成功构建。另一种常见的情况是，如果存在一个或多个特定严重性的发现，团队决定中断他们的构建。例如，一个团队可能会说他们不想破坏他们的构建，除非发现是高的或关键的。虽然这肯定有助于立即识别和解决高优先级问题，但结果是中低级别的发现进入了生产构建。

既然我们已经讨论了 DevSecOps 的优点和缺点，那么如何实际实现它呢？Jenkins 和 Git 等传统 DevOps 工具是构建 DevOps 管道基础的必备工具。市场上有许多安全工具，从开源到专有解决方案。其中许多可以集成到您现有的管道中。以下是我根据经验和互联网收集的列表:

*   Checkmarx:一种 SAST(静态应用程序安全测试)工具，用于分析应用程序代码中指示安全漏洞的缺陷。
*   WhiteSource:一个开源漏洞扫描器，它在后台自动持续运行，跟踪开源组件的安全性、许可和质量，并将其与 WhiteSource 的全面开源库数据库进行匹配。
*   zaproxy:OWASP Zed 攻击代理(ZAP)是一个易于使用的集成渗透测试工具，用于查找 web 应用程序中的漏洞。

DevSecOps 是解决我们所有问题的灵丹妙药吗？也许不是我们所有的问题，但对我和许多其他人来说，它是不可替代的，这是一辆炒作列车，我准备登上并采取所有的方式。
