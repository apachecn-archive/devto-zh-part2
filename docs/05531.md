在当今的软件世界中，许多团队正在采用 DevOps 和 DevSecOps 的文化。开发(Sec)运营的工作方式是在软件开发生命周期到达生产阶段之前尽可能早地进行持续测试——这是开发运维中“左移”的概念。当团队使用一种被称为测试驱动开发的方法编写代码时，可以实现这一点，在这种方法中，测试用例是在实际功能之前(用代码)编写的。这有助于定义最终适合的范围和功能。它为开发者提供了一个目标，而不是匆忙地思考特性应该是什么。在这种情况下，开发人员也是测试人员，甚至可能是测试用例的作者。可以编写的测试用例的范围没有限制——可以从测试应用程序的安全性、特性、错误处理、性能等等。

执行测试的目的是收集代码的质量度量，因此当它通过某个质量时，它就进入下一个阶段。这些度量还应该以一种对开发人员和管理人员都有意义的方式来收集和呈现，以便可以快速确定代码质量。此外，要进行大规模测试，自动化是关键。一个可重复的测试过程，加上一个可以轻松集成到当前生态系统中的工具，通常是首选。

# 开源在测试中的安全优势

利用开源知识和工具来实现开发和测试的目标对于开发人员来说显然是有利的一步。开源汇集了许多有着不同经历的人的知识来实现一个共同的目标。开源知识通常是由开发人员为开发人员共同创造的，因为在某个主题上没有预先的定义或知识库。开源所提供的是一个报告或贡献的渠道，安全性的话题对开源来说是有意义的。当有人报告零日漏洞时，这种知识对整个社区来说都是有价值的，并会反过来推动尽快找到修复方法，从而提高项目的质量。

# 开源安全的榜样

RSA 算法作为开源安全实现的一个示例，是在许多数学和计算机专家的研究和审查之后公开推出的，现在已经成为行业标准，并作为 go-to 加密算法在许多系统中广泛实现。

常见的漏洞、暴露和利用数据库是开源的知识数据库，安全专家随时掌握最新的安全数据。这些[开源安全漏洞](https://resources.whitesourcesoftware.com/blog-whitesource/open-source-vulnerability-management)和情报是安全专业人员的饭碗。但是，今天更有价值的是在这些数据库之上构建的东西——一种安全服务，配有工具，以现有的安全数据库为基础，以便尽可能快速、无缝地检测和修复安全漏洞。

# 将开源工具映射到 DevOps 实践和价值观

开源工具可以免费使用，而且进入门槛低。其次，它们通常是自身的产品，但也倾向于很好地融入其他现有产品，并专注于做好一件事——而不是试图做几乎所有事情的企业产品。开发人员和测试人员可以“即插即用”来尝试不同的工具，这似乎符合 DevOps 的第三种持续实验和学习的方式，即测试人员接触工具并了解它是否符合他们的目标——寻找可测量的小胜利。如果这不适合他们的用例，源代码的透明性使得它可以根据每个测试人员自己的需要进行修改。同时，开源项目也依赖于开发者的反馈，并以有机的方式为项目做出贡献，这对于项目在双向关系中的持续改进和成长至关重要。随着开源领域知识和工具的大量涌现，开发人员和测试人员有了太多的选择。随着开发人员和测试人员在现有项目的基础上实现更大的目标，开源项目的数量只会继续增长。
